## Q30. Do switch expressions have any return type limitations?

Yes. A switch expression is an **expression**, not a statement, so the compiler must be able to infer (or be told) a single result type — and that single fact drives all the limitations.

```csharp
string label = shape switch
{
    Circle    => "circle",
    Square    => "square",
    Triangle  => "triangle",
    _         => "unknown",
};
```

The rules:

1. **Single result type via best-common-type.** The compiler runs the same algorithm it uses for ternaries and array initializers across every arm's expression. If there is no best common type, the switch expression doesn't compile — even if every branch is "obviously" a reference type. You fix it by either casting on one arm or by giving the result a target type:
   ```csharp
   // CS8506 — no best common type between Cat and Dog
   Animal a = pet switch { true => new Cat(), false => new Dog() };
   //                                                            ^ target type Animal makes it compile
   ```
2. **Cannot return `void`.** Switch *expressions* must produce a value. If you want `Console.WriteLine` per branch, use a switch *statement*, or invoke the methods inside a delegate-returning arm. The compiler error is CS8917 / "switch expression must produce a value".
3. **Cannot return `ref` directly.** Switch expressions don't support `ref` or `ref readonly` arms — there is no `ref` switch expression. If you need a ref-returning dispatch, fall back to a switch statement with `return ref`.
4. **`ref struct` arms work, but the result type must itself be allowed in the surrounding context.** A `Span<T>` arm is fine; trying to assign it into a heap-allocated lambda capture or async method local is not.
5. **Exhaustiveness is checked, but not always enforced.** If the compiler can prove the patterns aren't exhaustive (open enums, nullable, non-sealed types), you get **CS8509** as a *warning*, and the runtime throws `SwitchExpressionException` if no arm matches. Don't rely on a `_ =>` to stay silent — write it explicitly.
6. **`null` is not handled by `_`.** A discard does not match `null` for reference types under nullable-aware analysis. Add `null => …` or `null or _ => …` if you mean it.

### Senior-level gotchas

- The "best common type" algorithm is **not** a join over the inheritance graph — it has to find a type that every arm is implicitly convertible to. `Dog` and `Cat` both deriving from `Animal` is not enough; you need at least one arm typed as `Animal`, or a target-type context.
- Pattern arms can use **list patterns** (C# 11+), **property patterns**, **relational patterns**, and **logical patterns** (`and`/`or`/`not`). They compose with `when` clauses for predicates that aren't expressible as patterns.
- Switch expressions do **not** introduce a new scope per arm — variables declared in patterns (`is X x`) are scoped to that arm's expression only, similar to `is`-pattern scoping in `if`.
- The compiler lowers a switch expression to a decision DAG and may emit a jump table, an if-else chain, or a hash table, depending on the patterns. The IL is often tighter than the equivalent statement form.

**Modern note (.NET 10 / C# 14):** No language change for switch expressions in C# 14. List patterns, slice patterns, and the C# 13 `params ReadOnlySpan<T>` interactions remain stable; the things you'd reach for here have all been GA since C# 11.

## Q31. What is `yield return` used for?

It tells the compiler "this method is an **iterator**" — generate the state machine for me. The body becomes a stream you can pull from with `foreach` (or `await foreach` for async iterators), and execution is **lazy**: each `MoveNext` call runs until the next `yield return`, then suspends, holding all locals in the synthesized state class.

```csharp
public static IEnumerable<int> EvenSquares()
{
    for (int i = 0; ; i += 2)   // infinite — caller's responsibility to stop
        yield return i * i;
}
```

What the compiler actually emits (conceptually):

- A nested compiler-generated class implementing `IEnumerable<T>`, `IEnumerator<T>`, `IEnumerator`, and `IDisposable`.
- A `_state` integer that encodes "before first call", "between yields N", and "after dispose".
- A `MoveNext` that's a goto-driven dispatch on `_state`, advancing through your code until the next `yield`.
- Locals lifted to fields on the state machine, so they survive across suspensions.
- `Current` returning the last yielded value.

What you can do:

- `yield return value` — produce the next element.
- `yield break` — terminate the sequence.
- `try { yield return … } finally { … }` — the `finally` runs when the consumer disposes the enumerator (e.g., `foreach` exits early or throws). This is the *only* way to bracket cleanup around suspension correctly.
- **Async iterators** with `async IAsyncEnumerable<T>` + `yield return` + `await` — same machinery, plus `IAsyncEnumerator`/`ValueTask`-based MoveNext.

What you can't do:

- `yield return` inside a `try` block that has a `catch` clause (only `try`/`finally` is allowed around a yield).
- `yield return` inside `lock` (the lock would have to span suspensions — semantically broken).
- `yield return` from `unsafe` blocks, anonymous methods, or lambdas.
- Pass `ref`/`out` parameters into an iterator method (their lifetime can't be lifted onto the heap).

### Real-world uses

- **Streaming I/O.** Reading lines from a file, rows from `IDataReader`, messages from a queue — the consumer pulls one at a time and the producer never materializes the whole sequence.
- **Composable LINQ pipelines.** Almost every LINQ-to-Objects operator (`Where`, `Select`, `Take`) is implemented with `yield return`. That's why `.Where(...).Select(...)` is one pass, not two — the iterators chain.
- **Infinite or unbounded sequences.** `EvenSquares()` above; primes; Fibonacci.
- **Async streams** for IAsyncEnumerable consumers — `await foreach` over a database cursor or an HTTP/2 stream.

### Senior-level gotchas

- **Deferred execution is a footgun.** Throwing in an iterator body (e.g., argument validation) does **not** throw at the call site — it throws on the first `MoveNext`. Idiomatic fix: split into a public method that validates eagerly and a private iterator method that does the yielding.
  ```csharp
  public static IEnumerable<T> ChunkBy<T>(IEnumerable<T> src, int size)
  {
      ArgumentNullException.ThrowIfNull(src);
      ArgumentOutOfRangeException.ThrowIfNegativeOrZero(size);
      return Iter(src, size);     // yields are inside Iter
  }
  ```
- **Single enumeration without buffering.** Iterating an iterator twice runs it twice. If the producer is non-idempotent (network call, randomness) you get different results. Materialize with `.ToList()` when you need to enumerate twice.
- **`IDisposable` is real.** `foreach` always wraps the enumerator in a try/finally and calls `Dispose`. If the consumer is *not* using `foreach` (manual `MoveNext` loop), they must dispose it themselves or your `yield`'d `finally` blocks don't run.
- **Allocation cost.** Each iterator call allocates a state-machine instance. For hot, tight loops over short collections, a plain `for`/`for-each` over an array is cheaper. `IAsyncEnumerable<T>` adds a `ValueTask` per item — fine for I/O-bound work, surprising on a CPU-bound stream.
- The compiler-emitted state machine implements **both** `IEnumerable<T>` and `IEnumerator<T>` and returns itself the first time `GetEnumerator` is called from the same thread that built it; subsequent calls allocate a fresh enumerator. This optimization avoids one allocation per enumeration when the typical pattern (`foreach` once) is used.

**Modern note (.NET 10 / C# 14):** No new yield-related language features in C# 14. C# 13 already lifted some restrictions around iterators inside `ref struct`–related contexts; behavior is unchanged in C# 14.

## Q32. How many generations does the Garbage Collector have?

**Three: Gen0, Gen1, Gen2.** That count has been stable since .NET 1.0 and is unchanged in .NET 10. There are also two special heap spaces that the GC reports separately:

- **Large Object Heap (LOH)** — objects ≥ 85,000 bytes (or arrays of doubles ≥ 1000 elements on 32-bit). Logically considered Gen2 for collection purposes, but allocated and (mostly) compacted differently.
- **Pinned Object Heap (POH)** — added in .NET 5. Where you allocate objects that need stable addresses (interop, async I/O buffers) without forcing the rest of the heap to fragment around pinned blocks. `GC.AllocateArray<T>(length, pinned: true)`.

So if an interviewer asks "how many heap regions does the GC manage" the honest answer is "three generations plus LOH and POH"; if they ask "how many generations" it's three.

### Why three

The **generational hypothesis**: most objects die young. Collecting Gen0 frequently is cheap because survivors are rare and promotion to Gen1 is fast. Gen1 is a buffer that absorbs the slightly-older-than-temporary objects so they don't pollute Gen2. Gen2 holds long-lived state — caches, singletons, statics — and is collected only when necessary because a full Gen2 collection is expensive (it walks the entire reachable graph and may compact the heap).

```text
Allocation → Gen0 → (survives a GC) → Gen1 → (survives a GC) → Gen2 → (stays)
```

### Trigger model

- **Gen0** is triggered when an allocation budget is exceeded. Cheap and frequent.
- **Gen1** is collected as part of a Gen1 GC, which also collects Gen0.
- **Gen2** is a *full* GC: collects all generations + LOH. Triggered by `Gen2 budget exceeded`, explicit `GC.Collect()`, OS memory pressure, or a manual call to `GC.TryStartNoGCRegion` failing to satisfy a request.

### Server vs Workstation, Background GC

- **Workstation GC** — one heap, one GC thread. Default for desktop apps and small console apps.
- **Server GC** — one heap *per logical core*, one GC thread per heap, in parallel. Default for ASP.NET Core when running in a server context. Higher throughput, more memory per heap.
- **Background GC** — concurrent Gen2 collection. The Gen0/Gen1 collections are still stop-the-world (very brief), but the Gen2 mark/sweep happens on a background thread while user threads keep allocating. Default since .NET 4.5.

### Senior-level gotchas

- **`GC.Collect()` is almost always a mistake.** It pauses your process, invalidates the GC's accumulated heuristics, and rarely improves anything. Legitimate uses: just before a measurement in benchmarking; after dropping a giant cache so subsequent allocations don't trigger an awkwardly-timed Gen2 (use `GCCollectionMode.Optimized`).
- **LOH fragmentation is the single biggest non-obvious GC pain.** Allocations ≥ 85K go to LOH and are *not* compacted by default. Long-running services that allocate variable-sized buffers (XML, large strings, byte arrays) bleed memory to fragmentation. Mitigations: `ArrayPool<T>.Shared`, pinned/preallocated buffers, or one-shot `GCSettings.LargeObjectHeapCompactionMode = LargeObjectHeapCompactionMode.CompactOnce; GC.Collect();` after a known cleanup point.
- **`finalizers` move objects to Gen2 immediately.** An object with a `~Finalizer` survives at least one extra generation because it's queued to the finalizer thread before being collected. Always pair finalizers with `GC.SuppressFinalize(this)` in `Dispose`. Better: don't write finalizers — use `SafeHandle`.
- **`WeakReference` is the right tool for caches that should yield to memory pressure**, not `GC.GetTotalMemory` polling.
- **Region-based heap.** Default since .NET 8. The GC manages the heap as fixed-size regions (4 MB / 64 MB depending) rather than one growing segment per generation. Reduces address-space pressure and makes large heaps practical without 64-bit address-space exhaustion. Mostly transparent, but it changes what diagnostic tools report.

**Modern note (.NET 10 / C# 14):** Generation count is unchanged. The runtime story worth knowing:

- **DATAS (Dynamic Adaptation To Application Sizes)** — opt-in in .NET 8, **on by default in .NET 9 and .NET 10** for Server GC. The heap size adapts to long-lived data size rather than scaling linearly with core count, so a process on a 64-core box no longer reserves 64 heap budgets when its working set is small. Hugely beneficial for containerized / multi-tenant / "bursty" workloads. You can still disable with `<ServerGarbageCollection>true</ServerGarbageCollection>` plus `DOTNET_GCDynamicAdaptationMode=0` if benchmarks demand classic Server GC.
- The .NET 10 GC includes additional refinements (better survivorship heuristics, improved physical-memory-pressure response). Tooling-side, dotnet-gcmon and EventPipe events surface DATAS state.

## Q33. What is `Interlocked` class used for?

Atomic, **lock-free** read-modify-write operations on a single memory location — the building blocks for safe concurrent counters, flags, and CAS-based algorithms without the cost (and contention behavior) of `Monitor`/`lock`.

```csharp
private long _processed;

void OnItemDone()
{
    Interlocked.Increment(ref _processed);   // atomic ++
}

long GetCount() => Interlocked.Read(ref _processed);  // atomic read of long on 32-bit
```

The API surface, grouped by what they do:

| Method | What |
|---|---|
| `Increment` / `Decrement` | Atomic `++` / `--` on `int`/`long`/`uint`/`ulong` |
| `Add` | Atomic `+=` |
| `Exchange<T>` | Atomically write a new value, return the old. Generic for reference types; specialized overloads for `int`/`long`/`float`/`double`/`IntPtr`/`object` |
| `CompareExchange<T>` | Compare-and-swap. The fundamental primitive for lock-free algorithms |
| `Read` | Atomic 64-bit read on 32-bit platforms (a plain read of a `long` is *not* atomic on 32-bit) |
| `And` / `Or` | Atomic bitwise update (added in .NET 7, broadly available since) |
| `MemoryBarrier` / `MemoryBarrierProcessWide` | Full fences |

### When to reach for `Interlocked` instead of `lock`

- The protected region is a single read-modify-write on a single value — the lock is pure overhead.
- High-frequency counters, hit/miss meters, sequence numbers, "have I started yet?" flags.
- Lock-free CAS loops: build a stack/queue/dictionary slot by attempting `CompareExchange` and retrying on failure.

```csharp
// Idiomatic lock-free "compute new value from old"
int old, computed;
do
{
    old = _state;
    computed = Transform(old);
}
while (Interlocked.CompareExchange(ref _state, computed, old) != old);
```

### Memory model guarantees

- Every `Interlocked` operation is a **full memory barrier** (sequentially consistent on the location). You can rely on writes before the call being visible to other threads after they observe the result, and vice versa.
- That's stronger than `volatile`, which only orders the read or write at hand.
- Stronger than required is sometimes wasteful; on ARM64 the JIT emits expensive `dmb` fences for `Interlocked`. `Volatile.Write`/`Volatile.Read` is cheaper when half-fence semantics are sufficient.

### Senior-level gotchas

- **Alignment matters.** `Interlocked` ops on a `long` field require natural alignment — true automatically for fields, but if you're playing with raw pointers or `[StructLayout(LayoutKind.Explicit)]` overlapping fields, you can get torn writes on 32-bit platforms.
- **The `ref` parameter is not optional.** It tells the compiler the location is shared; `Interlocked.Increment(ref obj.Field)` works but `Interlocked.Increment(ref Method().Field)` does not — the location must be addressable across the call.
- **`Interlocked` does not protect invariants across multiple fields.** If your "atomic" update is `_count` and `_sum`, you need a lock or a single struct field updated via `CompareExchange<T>` over the whole struct.
- **`CompareExchange<T>` for reference types** is the canonical *lock-free publish*: `Interlocked.CompareExchange(ref _singleton, newValue, null)`. Don't reinvent double-checked locking; this is correct under the .NET memory model.
- **Don't mix `volatile` and `Interlocked` on the same field as a "belt and braces"** — it makes reasoning about ordering harder, not safer. Pick one model per field and document it. (Q48 in this set goes deeper on the volatile/Interlocked distinction.)
- **High contention = wasted CAS retries.** If `CompareExchange` fails 80% of the time, your "lock-free" code is doing more work than a `lock`. Profile, don't assume.

**Modern note (.NET 10 / C# 14):** No major Interlocked surface changes in .NET 10. The bitwise `And`/`Or` overloads (added .NET 7) and the unsigned overloads continue to be available; `Interlocked` is fully vector-friendly via the JIT's intrinsics.

**Sources:**
- [Interlocked Class — Microsoft Learn](https://learn.microsoft.com/en-us/dotnet/api/system.threading.interlocked?view=net-10.0)

## Q34. What is the code generated by compiler for auto properties?

For a classic auto-property, the compiler synthesizes a **private backing field** plus the two accessors. For:

```csharp
public string Name { get; set; }
```

…the emitted IL is approximately:

```csharp
[CompilerGenerated]
private string <Name>k__BackingField;

public string get_Name()
    => <Name>k__BackingField;

public void set_Name(string value)
    => <Name>k__BackingField = value;
```

A few facts about that backing field:

- The field name uses an **invalid C# identifier** (`<Name>k__BackingField`) so user code can't refer to it directly — IL allows angle-bracketed names, C# does not.
- The field is decorated with `[CompilerGeneratedAttribute]` and `[DebuggerBrowsable(DebuggerBrowsableState.Never)]` so debuggers and reflection tools render the property, not the field.
- The accessors are `[CompilerGenerated]` too — useful when reflection-based code wants to tell synthesized accessors from hand-written ones.

### Variations and what they emit

**Get-only auto-property** (readonly auto-property, since C# 6):
```csharp
public string Name { get; }
```
- Backing field is emitted as `initonly` (the IL `initonly` flag = C# `readonly`). It can only be assigned in the constructor or in a field initializer.

**`init` accessor** (C# 9):
```csharp
public string Name { get; init; }
```
- Setter is normal IL, but it's marked with a **modreq** on `IsExternalInit` (`System.Runtime.CompilerServices.IsExternalInit`). Older compilers see the modreq and refuse to call the setter, which is how downlevel safety is enforced for object-initializer-only assignment.
- Inside the constructor, an object initializer, or a `with` expression, the JIT/compiler permits the call; everywhere else, it errors as if the property were read-only.

**`required` modifier** (C# 11):
```csharp
public required string Name { get; init; }
```
- Adds `[RequiredMember]` on the property and `[RequiredMember]` on the containing type. The compiler enforces that *callers* assign every required member at construction (object initializer or a constructor decorated with `[SetsRequiredMembers]`).

**Static auto-property:** identical, but the backing field is `static`.

**Static abstract / virtual interface members** (C# 11):
- An interface can declare `static abstract Name { get; set; }`. The implementing type emits a normal auto-property; the interface emits no field.

### Senior-level gotchas

- **Reflection over auto-properties** finds the backing field via `BindingFlags.NonPublic | BindingFlags.Instance`, but its name is angle-bracketed — don't string-match by name; check `[CompilerGenerated]`.
- **Source generators** that emit partial classes can't `partial` an auto-property's accessors prior to C# 13. C# 13 added **partial properties and indexers** — useful when a generator wants to provide one accessor and let the user provide the other.
- **Records** lower their positional parameters to init-only auto-properties (with `IsExternalInit` modreq) plus a synthesized primary constructor that assigns them.
- **Auto-property vs hand-written backing field** is essentially zero-cost — the JIT inlines the trivial accessors. The gain of "I'd write a backing field for performance" is illusory; only stop using auto-properties when you need to inject behavior.

**Modern note (.NET 10 / C# 14):** The headline addition is the **`field` contextual keyword** for "semi-auto" properties. You can now add validation or a side effect to one accessor while still letting the compiler generate the backing field for you — the old workaround required giving up the auto-property entirely.

```csharp
public string Name
{
    get;
    set => field = value?.Trim()
                 ?? throw new ArgumentNullException(nameof(value));
}
```

What the compiler emits is essentially the same as before — a `<Name>k__BackingField`, a getter that returns it, and your setter body with `field` replaced by a reference to the synthesized backing field. `field` is a **contextual** keyword: it's only treated specially inside accessor bodies of a property, so existing code that has a local or field literally named `field` still works (with a warning recommending you `@field` or rename it).

Combines with `init`, `required`, partial properties, and static auto-properties as you'd expect:

```csharp
public required string Email
{
    get;
    init => field = NormalizeEmail(value);
}
```

This is a meaningful refactor lever: previously, adding a single trim/normalize step to a DTO meant rewriting every property as a manual backing field + accessor pair. Now it's one line.

## Q35. How is Polymorphism implemented in C#?

C# implements polymorphism in three different mechanisms, each with its own runtime story. Naming them and explaining where the dispatch happens is the senior-level answer.

### 1. Subtype (inclusion) polymorphism — `virtual` / `override` / `abstract`

Most people mean this when they say "polymorphism": a base reference holds a derived instance, and a virtual method call dispatches to the most-derived `override`.

```csharp
public abstract class Shape { public abstract double Area(); }
public class Circle(double r) : Shape { public override double Area() => Math.PI * r * r; }
public class Square(double s) : Shape { public override double Area() => s * s; }

Shape s = new Circle(2);
double a = s.Area();   // dispatches to Circle.Area
```

**How it works:**

- Each class has a **method table** (the runtime's vtable equivalent — accessible via `RuntimeType` / `MethodHandle`). Slots in the method table for virtual methods are populated with the override's address as you walk the inheritance chain.
- The IL instruction is **`callvirt`**. The JIT emits a load of the object's method table, an indirect call through a fixed slot, and the call.
- `sealed override` lets the JIT *devirtualize* the call (turn it into a direct call) at most call sites because no further override is possible.
- The **tiered JIT** also devirtualizes when it can see the concrete type at compile time (e.g., `new Circle(2).Area()`) or at tier-1 recompilation when guarded devirtualization is profitable.

**`new` keyword is *not* polymorphism.** A `new`-shadowed method is dispatched by the static type of the reference, not the runtime type. Use `new` only when you have to (versioning, name collisions); never as a substitute for `override`.

### 2. Interface polymorphism

```csharp
public interface IRenderable { void Render(); }
public class Button : IRenderable { public void Render() { … } }

IRenderable r = new Button();
r.Render();
```

- Different vtable layout: the runtime uses **interface dispatch maps** to find the implementer's method. The JIT emits `callvirt` against an interface method, and the runtime resolves through a per-type interface map.
- **Default interface methods** (DIM, C# 8+) — a method body can be defined on the interface itself. The class can opt to inherit it; calling code can also call the interface method directly. Useful for evolving an interface without breaking implementers.
- **Static virtual interface members** (C# 11+) — interfaces can declare `static abstract` members, enabling polymorphism *over types*, not instances. This is what powers the `INumber<T>` / generic math story:
  ```csharp
  public static T Sum<T>(IEnumerable<T> xs) where T : INumber<T>
      => xs.Aggregate(T.Zero, (a, b) => a + b);  // operators are static abstract
  ```
- **Struct + interface = boxing trap.** Calling `IRenderable.Render` on a `struct Foo : IRenderable` boxes the struct to the heap unless you constrain through generics: `where T : IRenderable` keeps it on the stack and devirtualizes.

### 3. Parametric polymorphism — generics

```csharp
public static T Max<T>(T a, T b) where T : IComparable<T>
    => a.CompareTo(b) >= 0 ? a : b;
```

- The CLR specializes generic types **per value-type argument** (`List<int>` and `List<long>` get separate JITted code), and **shares one code path** across all reference-type arguments (`List<string>` and `List<object>` share). This is unlike Java erasure.
- Constraint-based dispatch (`where T : IFoo`) gets devirtualized for value types and goes through a constrained call for reference types; the JIT often picks the best path automatically.
- Generic specialization is the reason `Span<T>`, `INumber<T>`, and `EqualityComparer<T>.Default` are essentially free at runtime.

### 4. (Optional) Ad-hoc polymorphism — overloading

`int Add(int, int)` vs `double Add(double, double)` — resolved at *compile time* by overload resolution. Often classified as polymorphism in language theory; usually omitted in C# interview answers because it's not runtime dispatch.

### Senior-level gotchas

- **`base.Method()` calls are non-virtual.** They use IL `call`, not `callvirt`. Don't expect a base call to re-enter the most-derived override.
- **Constructors and virtual calls are dangerous.** A virtual call from a base constructor dispatches to the derived override *before* the derived constructor has run — its fields are still default. C# allows it; Roslyn doesn't even warn. Avoid.
- **Hiding by signature, not just by name.** A derived `void Foo(int)` does not override a base `void Foo(object)`; it hides it. The CS0114 warning catches the common mistake — make hiding intent explicit with `new`.
- **Devirtualization is heuristic.** `sealed` on the class or override is the strongest hint. Adding `sealed` to a class your team owns is a free perf and JIT-friendliness win.
- **Pattern matching is not polymorphism**, but it's the modern way to express "switch on subtype" and is often a cleaner alternative to a `virtual` dispatch tree when behaviour lives outside the data:
  ```csharp
  double Area(Shape s) => s switch
  {
      Circle { Radius: var r } => Math.PI * r * r,
      Square { Side:   var x } => x * x,
      _ => throw new NotSupportedException()
  };
  ```
  Use virtual methods when behavior belongs to the type; use pattern matching when it belongs to the consumer. Keep one consistent.

**Modern note (.NET 10 / C# 14):** No language-level changes to polymorphism in C# 14. Static virtuals in interfaces (the load-bearing C# 11 feature for generic math) and the runtime devirtualization improvements continue to be the modern stories. The .NET 10 JIT improves devirtualization heuristics further (better profile-guided guarded devirtualization), so virtual calls on hot paths increasingly pay zero overhead.
