## Q1. Difference between `readonly` vs `const`

Both create values that can't be reassigned, but they live in completely different worlds.

**`const` is a compile-time literal.** The compiler substitutes the value directly into every call site at compile time. There is no field at runtime — the IL just contains the literal where the constant was used. Because of this:

- Must be initialized **at the declaration site** with a constant expression.
- Limited to **types the compiler can encode as a literal**: numeric primitives, `bool`, `char`, `string`, and `null` for reference types. (`decimal` is special — it's allowed via `const` even though it isn't a CLR primitive, because the compiler emits a `DecimalConstantAttribute` rather than an IL literal.)
- Implicitly `static` — you cannot have a per-instance `const`.

**`readonly` is a runtime-assigned field.** The field exists in the assembly's metadata and is assigned exactly once: either at declaration, in a constructor (instance ctor for instance fields, static ctor for `static readonly`), or — since C# 12 — through a primary constructor parameter capture. After construction, the runtime enforces immutability.

```csharp
public class Limits
{
    public const int MaxRetries = 3;                      // baked into IL of every caller
    public static readonly DateTime Epoch = DateTime.UtcNow; // assigned at type init
    public readonly Guid InstanceId;                       // assigned per-instance in ctor

    public Limits() => InstanceId = Guid.NewGuid();
}
```

### The versioning hazard of `const` (this is the senior answer)

Because `const` is inlined at compile time, **changing the value of a `public const` is a binary-breaking change** even though the source change looks trivial. Consumers compiled against the old version still carry the old literal in their IL until they're recompiled.

```csharp
// Library v1
public const int DefaultTimeoutMs = 5000;

// Library v2 — change this and ship a new DLL...
public const int DefaultTimeoutMs = 30000;

// Existing consumer DLL still uses 5000 until it is itself recompiled.
```

**Rule of thumb**: use `const` only for values that are genuinely immutable across the lifetime of the API contract (mathematical constants, protocol magic numbers, well-known names). For anything that might tune over time, prefer `public static readonly` — it's resolved at runtime, so consumers pick up the new value the moment they load the new DLL.

### `readonly` shallow vs deep

`readonly` only locks the **reference**, not the object behind it.

```csharp
public readonly List<int> Items = new();
// later: Items.Add(1);  // legal — the field still points to the same list
// Items = new List<int>(); // illegal — the reference is frozen
```

For deep immutability you need an immutable type (`ImmutableArray<T>`, a `record`, etc.) or `init`-only properties.

### Related modern constructs

- **`init` accessors (C# 9)**: object-initializer-friendly equivalent of `readonly` for properties.
- **`required` members (C# 11)**: forces callers to initialize a member during construction; works alongside `init`.
- **`readonly struct` and `readonly` instance members (C# 7.2 / 8)**: tells the compiler the struct/method does not mutate `this`, eliminating defensive copies when invoked through `in` parameters or `ref readonly` returns.
- **`ref readonly` parameters (C# 12)**: pass-by-reference with read-only semantics — a stricter alternative to `in`.

**Senior-level gotchas:**
- A `static readonly` reference type field assigned in the static ctor is **not** thread-safety free — concurrent reads during type initialization are safe (CLR guarantees it), but if its target object is mutable you still need synchronization on writes.
- Replacing a `public const` with `public static readonly` is itself a binary-breaking change (different metadata kind), even if the value matches. Plan the API surface up front.
- Values used in attribute arguments, `switch` case labels, default parameter values, and `[CallerMemberName]`-style scenarios **require** `const` — `static readonly` won't satisfy the compiler.

## Q2. What is a `sealed` keyword used for?

`sealed` slams the door on inheritance. It's used in two places:

**1. On a class** — prevents any further derivation.

```csharp
public sealed class CurrencyAmount { /* ... */ }
// public class Foo : CurrencyAmount {} // CS0509: cannot derive from sealed type
```

**2. On a `virtual`/`abstract` method or property override** — stops the polymorphic chain at this level. The member is still virtual to *callers*, but a subclass can no longer override it again.

```csharp
public class Base       { public virtual void Tick() { } }
public class Middle : Base { public sealed override void Tick() { } }
public class Leaf : Middle { /* cannot override Tick */ }
```

### Why it matters — design

This is Eric Lippert's argument: **inheritance is a contract**, and the base class has to be designed for it (Liskov, the fragile-base-class problem, ensuring subclasses can't break invariants). If you didn't *design* a class for inheritance, sealing it is the safer default. Many senior codebases use Roslyn analyzer **CA1852** (`Seal internal types`) to enforce this for `internal` types automatically.

### Why it matters — performance

The JIT can do more aggressive optimizations on sealed types:

- **Devirtualization**: a virtual call through a sealed class reference can be turned into a direct call, which then becomes inlinable.
- `is`/`as`/pattern checks against a sealed class can be reduced to a single type-equality compare instead of a hierarchy walk.
- Hot-path code that boxes or pattern-matches on small sealed hierarchies often benefits measurably.

The CoreCLR's `class devirtualization` pass exploits this; you can see the wins in BenchmarkDotNet on tight loops.

### Where `sealed` doesn't apply

- **Structs are implicitly sealed** — value types can't be inherited from, so `sealed struct` is redundant (and the compiler will reject it on a struct declaration).
- **Static classes are implicitly `sealed` and `abstract`**.
- **`record` types accept `sealed`** the same way classes do; useful for closing record hierarchies used in pattern-matching exhaustiveness checks.

```csharp
public abstract record Shape;
public sealed record Circle(double Radius) : Shape;
public sealed record Square(double Side) : Shape;
// switch on Shape becomes effectively exhaustive
```

**Senior-level gotchas:**
- `sealed override` is the right call when you're overriding a hot virtual method *just* to fix behavior at one level — it lets the JIT devirtualize from there down.
- Sealing public types is an API decision, not a perf decision. Once a type is unsealed and shipped, you can't seal it later without a binary break. Default-sealing internal types is cheap; default-sealing public types deserves discussion.
- `sealed` on a class does not seal its members for reflection or `DynamicMethod`-style code. It's a language/compile-time guarantee.

## Q3. Name all the access modifiers for types

C# has **seven** access modifiers as of C# 14. Six are accessibility levels you stack on members and types, and one is a special **file-scoped** modifier introduced in C# 11.

| Modifier              | Visible from                                                                      |
|-----------------------|-----------------------------------------------------------------------------------|
| `public`              | Anywhere — any assembly                                                           |
| `internal`            | The same assembly only                                                            |
| `protected`           | The declaring class and its derived classes (any assembly)                        |
| `private`             | The declaring type only (default for class/struct members)                        |
| `protected internal`  | **Union**: same assembly OR derived class in any assembly                         |
| `private protected`   | **Intersection**: derived class **and** in the same assembly (C# 7.2)             |
| `file`                | The same source file only — top-level, non-nested types only (C# 11)              |

`file` is the new one most candidates miss. It exists primarily for **source generators**: a generator can emit a helper type with `file` visibility and be confident the name won't clash with another generator (or with user code) in the rest of the project.

```csharp
// Generated.g.cs
file sealed class HelperBuffer { /* ... */ }   // invisible outside this file

// MyService.cs — completely unaware that Generated.g.cs has a HelperBuffer
file sealed class HelperBuffer { /* a different one */ } // also legal, no conflict
```

### Where each modifier is legal

This is where most candidates trip up — *not every modifier works on every declaration*.

- **Top-level (non-nested) types**: only `public`, `internal` (default), and `file`. You **cannot** declare a top-level `private` or `protected` class — there's no enclosing scope for those to mean anything.
- **Nested types**: all seven are legal.
- **Class members**: all seven are legal.
- **Struct members**: only `public`, `internal`, `private` — structs don't have inheritance, so `protected*` makes no sense.
- **Interface members**: default to `public`; since C# 8 you can specify `private`, `protected`, `internal` on members with a default implementation. Static abstract members default to public.
- **Enum members**: implicitly `public` and unmodifiable.

### `InternalsVisibleTo` — the escape hatch for `internal`

```csharp
[assembly: InternalsVisibleTo("MyLib.Tests")]
```

Lets a friend assembly see your `internal` and `private protected` members. Standard pattern for unit-testing internals without making them public.

**Senior-level gotchas:**
- The default accessibility for a type declared at namespace scope is `internal`, but for a class **member** it is `private`. Mixing those up is a classic source of "why is this not visible?" bugs.
- `protected internal` reads "protected OR internal" (more permissive); `private protected` reads "protected AND internal" (more restrictive). The English ordering is misleading — read them as set operations.
- `file` types can be referenced by `nameof(...)` from outside the file (just the metadata), but you can't actually use the type — useful trivia for source-generator authors.

**Sources:**
- [Access Modifiers – C# language reference (Microsoft Learn)](https://learn.microsoft.com/en-us/dotnet/csharp/language-reference/keywords/access-modifiers)
- [The `file` modifier – C# language reference (Microsoft Learn)](https://learn.microsoft.com/en-us/dotnet/csharp/language-reference/keywords/file)

## Q4. Difference between interface and abstract class

The textbook line — "interface = contract, abstract class = partial implementation" — is still true, but C# 8 (default interface members) and C# 11 (static abstract members) blurred it. The remaining hard differences are about **state, identity, and value-type compatibility**.

### What still separates them

| Concern                        | `interface`                                                          | `abstract class`                                  |
|-------------------------------|----------------------------------------------------------------------|--------------------------------------------------|
| Instance fields                | **No** (only properties — backed by the implementer's storage)        | Yes                                              |
| Constructors / finalizers      | No                                                                   | Yes (constructors only run via derived `: base()`) |
| Inheritance arity              | Many interfaces per type                                             | Exactly one base class                           |
| Inheritable by structs?        | Yes                                                                  | No (structs derive only from `System.ValueType`) |
| Default access of members      | `public` (since C# 8 you can opt into others when there's a body)    | `private` for fields, varies for members         |
| Versioning ergonomics          | Adding a member = breaking unless it has a default implementation   | Adding a non-abstract member = non-breaking      |
| Operator/static contract       | Yes — via static abstract/virtual members (C# 11)                    | Limited — operators must be `public static`      |

### What changed in modern C#

- **C# 8 — Default Interface Members (DIM).** An interface member can have a body, so adding a method to a published interface no longer breaks every implementer. *But* the body has no access to instance state — it can only call other members of the interface. DIMs are dispatched via the interface vtable, so calling them through the concrete type requires a cast to the interface (`((IFoo)x).DefaultedMethod()`).
- **C# 11 — Static abstract / virtual interface members.** Interfaces can now declare static API surface — operators, static factories, parsing — that generic code can call through a type parameter. This is what `INumber<T>` and the BCL's "generic math" hierarchy are built on.

```csharp
public interface IAdditionMonoid<T> where T : IAdditionMonoid<T>
{
    static abstract T Zero { get; }
    static abstract T operator +(T a, T b);
}

public readonly struct Sum : IAdditionMonoid<Sum>
{
    public int Value { get; init; }
    public static Sum Zero => default;
    public static Sum operator +(Sum a, Sum b) => new() { Value = a.Value + b.Value };
}

public static T Total<T>(IEnumerable<T> items) where T : IAdditionMonoid<T>
{
    var acc = T.Zero;
    foreach (var x in items) acc += x;
    return acc;
}
```

This **is not possible with abstract classes** — operators and static methods aren't virtual on classes.

### Choosing between them

Use an **abstract class** when:
- The hierarchy needs **shared instance state or invariants** (a base ctor that validates inputs, common fields).
- You're modeling an "is-a" relationship with strong identity (e.g., `Stream`, `DbCommand`).
- You need a **finalizer**, partial trust on constructor sequencing, or to reuse a non-trivial protected helper.

Use an **interface** when:
- You want **multiple-inheritance** of capability ("can-do") — `IDisposable`, `IAsyncEnumerable<T>`, `IComparable<T>`.
- The contract must be implementable by **structs**.
- You need static-shape generic constraints (numeric/parsing abstractions).
- You're publishing an API and need the freedom to add members later via DIM without breaking implementers.

**Senior-level gotchas:**
- Abstract classes can declare `protected` constructors — useful for forcing instantiation through derived types only. Interfaces have no equivalent enforcement.
- A class can implement two interfaces that have the same default-member signature. The compiler will require the implementer to **explicitly** disambiguate or it's an error — DIM is not a free win.
- Implementing an interface on a `struct` and then calling its members through the interface boxes the struct unless you constrain a generic to the interface (`where T : IFoo, struct`) and call through `T`. This is exactly the pattern static-abstract members enable for zero-allocation polymorphism.
- "Abstract class with no abstract members" is a legitimate signal that the type should be inherited but never instantiated — different from a sealed concrete class with private ctors.

**Sources:**
- [Static abstract members in interfaces (Microsoft Learn)](https://learn.microsoft.com/en-us/dotnet/csharp/language-reference/proposals/csharp-11.0/static-abstracts-in-interfaces)
- [Explore static virtual members in C# interfaces (Microsoft Learn)](https://learn.microsoft.com/en-us/dotnet/csharp/advanced-topics/interface-implementation/static-virtual-interface-members)
- [Interfaces – define behavior for multiple types (Microsoft Learn)](https://learn.microsoft.com/en-us/dotnet/csharp/fundamentals/types/interfaces)

## Q5. When is a static constructor called?

A static constructor (also called a **type initializer** in CLR/IL terms — `.cctor`) is run **at most once** per type, by the runtime, before that type is "used" for the first time within the current process / load context. The runtime — not the application code — calls it; you can't invoke it directly.

### What "first use" means

The CLR triggers the type initializer immediately before the *first* of these happens:

1. **An instance of the type is created** (via `new`, `Activator.CreateInstance`, deserialization, etc.).
2. **A static member is referenced** — a static field is read or written, a static method is called, a static event is subscribed/raised, or a static property is accessed.

You can also explicitly force it via `RuntimeHelpers.RunClassConstructor(typeof(T).TypeHandle)` — useful when a type's static ctor has important side effects (e.g., registering with a static registry) and you want to run it deterministically.

### `beforefieldinit` — the timing relaxation

The CLR distinguishes two flavors of type initializer:

- **Strict (no `beforefieldinit`)** — type initializer runs *exactly* before first use, no earlier. This is what you get when you write an explicit `static C() { }` constructor.
- **Lax (`beforefieldinit` set)** — the runtime is *allowed* to run the initializer "any time before the first reference to a static field." It usually still happens at first use, but the JIT may also defer past a method entry that touches non-static members. The compiler emits `beforefieldinit` when there's no explicit static ctor, only static field initializers.

```csharp
public class Strict      { public static readonly int X = Compute(); static Strict() {} }   // no beforefieldinit
public class Lax         { public static readonly int X = Compute(); }                       // beforefieldinit
```

**Why this matters**: with `beforefieldinit` the JIT can elide the type-init check on every static-field access, which is faster. So *don't write an empty `static C() {}` "just to be safe"* — you're paying for stricter timing semantics you don't need. Only declare an explicit static ctor when the body actually does something.

### Threading guarantees

The CLR guarantees a static ctor runs **exactly once** per type, even under concurrent first access from multiple threads. The runtime takes a per-type lock; other threads block until initialization completes. This makes `static readonly` lazy-singleton patterns trivially thread-safe:

```csharp
public sealed class Cache
{
    public static readonly Cache Instance = new();
    private Cache() { /* expensive setup, runs once, thread-safe */ }
}
```

### The exception trap

If a static constructor throws, the runtime wraps it in a **`TypeInitializationException`** and **caches the failure**. Every subsequent attempt to use the type — for the rest of the process / `AssemblyLoadContext` lifetime — re-throws the same `TypeInitializationException` immediately *without rerunning the ctor*. The type is permanently poisoned.

```csharp
public class Bad
{
    static Bad() => throw new InvalidOperationException("config missing");
}
// First access: TypeInitializationException with InnerException = InvalidOperationException
// Every subsequent access: same TypeInitializationException, no retry
```

This is the most common interview gotcha around static ctors. Treat them like a tiny, paranoid IO boundary: catch and log internally, fail fast with a helpful message, but don't let arbitrary code throw.

### Other rules

- A static constructor **cannot** have parameters or an access modifier (it's implicitly private to the runtime).
- It is allowed alongside instance constructors — they coexist; the static ctor runs first on first use.
- For generic types, **each closed generic instantiation has its own static ctor invocation** — `Foo<int>` and `Foo<string>` initialize independently.
- For `static class`, the static ctor still applies; the class itself is `abstract sealed` in IL but `.cctor` runs on first member access exactly the same way.

### Related modern feature: module initializers (C# 9)

If you need code to run *once per assembly load* rather than per type, decorate a parameterless static method with `[ModuleInitializer]`. The CLR runs it before any type-level access from that assembly. This is the right tool when "static constructor on some bootstrap type" feels like a workaround.

```csharp
internal static class StartupHooks
{
    [ModuleInitializer]
    internal static void Init() => GlobalRegistry.RegisterAll();
}
```

**Senior-level gotchas:**
- Don't put I/O, network calls, or anything that can deadlock in a static constructor. The per-type lock is held while the ctor runs; if your ctor awaits something that needs the same lock (or another type's static ctor that needs yours), you get a classic dual-lock deadlock that's very hard to diagnose.
- Order of execution between static field initializers and the explicit `static C()` body is **textual order**: field initializers run first, in source order, then the explicit static ctor body. If a field initializer throws, the ctor body never runs.
- Don't rely on static ctor execution timing for correctness when `beforefieldinit` is set — write logic that's correct regardless of when initialization happened.
- In ASP.NET Core, prefer DI singletons over `static readonly` for anything that has dependencies or needs disposal — static ctors run once per AppDomain/load context, which doesn't match the lifetime semantics you usually want.
