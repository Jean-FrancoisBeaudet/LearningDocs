## Q46. What is the difference between Auto and Manual reset events?

Both are kernel-backed wait handles that derive from `EventWaitHandle`. The single distinguishing axis is **what happens after `Set()` releases a waiter**:

- **`AutoResetEvent`** — releases **exactly one** waiter, then the handle resets itself to non-signaled. Models a one-permit handoff or a turnstile.
- **`ManualResetEvent`** — releases **all** current waiters and stays signaled until `Reset()` is called. Subsequent `WaitOne()` calls return immediately. Models a latch / "the gate is open" / one-shot broadcast.

```csharp
// Single-item handoff — one signal releases one consumer.
var doorbell = new AutoResetEvent(initialState: false);

_ = Task.Run(() => { doorbell.WaitOne(); Console.WriteLine("consumer A"); });
_ = Task.Run(() => { doorbell.WaitOne(); Console.WriteLine("consumer B"); });

doorbell.Set();  // releases ONE of them; auto-resets.
doorbell.Set();  // releases the other.
```

```csharp
// Latch — flip once, all current and future waiters fly through.
var ready = new ManualResetEvent(initialState: false);

_ = Task.Run(() => { ready.WaitOne(); Console.WriteLine("worker 1"); });
_ = Task.Run(() => { ready.WaitOne(); Console.WriteLine("worker 2"); });

ready.Set();  // releases ALL; stays signaled.
```

### When to actually reach for these

These types hold OS handles. Use them only when you genuinely need:

- **Cross-process synchronization** via named events.
- **Interop** with `WaitHandle.WaitAny/All`, `RegisteredWaitHandle`, or overlapped I/O.
- A real `WaitHandle` for legacy P/Invoke surfaces.

For in-process work, prefer the lighter alternatives:

| Need | Modern primitive |
|------|------------------|
| In-process latch | `ManualResetEventSlim` (spin-then-block, no kernel handle until contended) |
| Counted async handoff | `SemaphoreSlim` with `WaitAsync(CancellationToken)` |
| Single-shot async signal | `TaskCompletionSource<T>` |
| Producer/consumer stream | `Channel<T>` |

`ManualResetEventSlim` doesn't allocate the underlying `WaitHandle` until you actually call `.WaitHandle` or block long enough to need a kernel transition — that lazy allocation is the main reason to prefer it over `ManualResetEvent` for in-process code.

### Senior-level gotchas

- **Always dispose.** Both types implement `IDisposable` and leak OS handles otherwise.
- **No async waits.** Only `WaitOne` is offered. To await one, wrap with `RegisteredWaitHandle` + `TaskCompletionSource<T>` — or just use `SemaphoreSlim`/`Channel<T>` instead.
- **`AutoResetEvent.Set()` retains exactly one stored signal** if no one is waiting. This is *not* the same as `Monitor.Pulse`, which loses signals when no thread is waiting — easy bug if you mentally model one as the other.
- **Missed-wakeup with `ManualResetEvent`.** A racing `Set()`/`Reset()` pair around a slow consumer can leave the consumer blocked forever. If you need durable signaling, use a counted primitive or a `Channel<T>`.

## Q47. What are Channels in C#?

`System.Threading.Channels` is the modern in-process producer/consumer queue that ships in the BCL. It's the async-first successor to `BlockingCollection<T>` and the right primitive for any pipeline where one side produces and another consumes — background work, request batching, fan-out/fan-in, decoupling.

Two factory shapes:

- `Channel.CreateUnbounded<T>(...)` — never blocks the writer. Use only when the producer rate is bounded by something else.
- `Channel.CreateBounded<T>(BoundedChannelOptions)` — capacity cap; built-in backpressure. The `FullMode` knob picks the strategy:

| `FullMode` | Behavior when full |
|------------|--------------------|
| `Wait` (default) | Writer awaits until space frees. |
| `DropOldest` | Discard the oldest unread item. |
| `DropNewest` | Discard the newest buffered item. |
| `DropWrite` | Discard the incoming item. |

The channel is split into a writer half and a reader half:

```csharp
var ch = Channel.CreateBounded<Order>(new BoundedChannelOptions(capacity: 100)
{
    FullMode      = BoundedChannelFullMode.Wait,
    SingleReader  = true,   // unlocks lock-free fast paths
    SingleWriter  = false,
});

// Producers
async Task ProduceAsync(IAsyncEnumerable<Order> source, CancellationToken ct)
{
    await foreach (var order in source.WithCancellation(ct))
        await ch.Writer.WriteAsync(order, ct);

    ch.Writer.Complete();   // signals "no more items"
}

// Single consumer
async Task ConsumeAsync(CancellationToken ct)
{
    await foreach (var order in ch.Reader.ReadAllAsync(ct))
        await ProcessAsync(order, ct);

    // ch.Reader.Completion completes when writer signaled and the queue drained.
}
```

Key surface to know:

- **Writer**: `WriteAsync`, `TryWrite`, `WaitToWriteAsync`, `Complete()` / `TryComplete(Exception?)`.
- **Reader**: `ReadAsync`, `TryRead`, `WaitToReadAsync`, `ReadAllAsync()` (returns `IAsyncEnumerable<T>`), `Completion` (a `Task` that finishes when the writer completes and the buffer drains, faulted if the writer completed with an exception).

### When to choose Channels

- Decouple a fast-arriving signal from a slow processor (HTTP request → background pipeline).
- Batch work for downstream calls (write to channel, batch-read every N items or every T ms).
- Cap memory: a bounded channel with `Wait` gives you backpressure all the way to the producer for free.

When **not** to use Channels: across processes — that's a real broker (Kafka, RabbitMQ, Service Bus). For pure synchronous code with blocking semantics, `BlockingCollection<T>` still works.

### Modern note (.NET 10 / C# 14)

`Channel.CreateUnboundedPrioritized<T>(UnboundedPrioritizedChannelOptions<T>?)` (added in .NET 9, present in .NET 10) gives you a min-heap-ordered channel — the next item read is the lowest-priority value per the configured `IComparer<T>`. Useful for priority work queues without rolling your own.

### Senior-level gotchas

- **Always call `Complete()`** when production ends, otherwise consumers awaiting `ReadAllAsync()` block forever.
- **`SingleReader`/`SingleWriter` are *promises*, not enforcement.** Lying about them silently corrupts state.
- **`AllowSynchronousContinuations = true` is a footgun.** It lets the writer's thread run the consumer's continuation inline — fast, but a long continuation now blocks the producer.
- **`Channel<T>` is in-process only** — no transport, no persistence, no consumer-group semantics. Don't try to make it one.

**Sources:**
- [Channels — Microsoft Learn](https://learn.microsoft.com/en-us/dotnet/core/extensions/channels)
- [`Channel.CreateUnboundedPrioritized` — Microsoft Learn](https://learn.microsoft.com/en-us/dotnet/api/system.threading.channels.channel.createunboundedprioritized)

## Q48. Difference between `volatile` and `Interlocked`

Both touch the .NET memory model, but they solve different problems. **`volatile` is about ordering of reads/writes on a single field. `Interlocked` is about atomicity of read-modify-write.** Knowing which one you need is mostly about whether you're publishing a value or mutating one.

### `volatile`

A field modifier. Per the .NET memory model:

- A read of a `volatile` field has **acquire semantics** — no later memory operation may be reordered before it.
- A write to a `volatile` field has **release semantics** — no earlier memory operation may be reordered after it.

It does **not** make compound operations atomic:

```csharp
private volatile int _count;
// Still racy — read, +1, write are three separate steps.
_count++;
```

It's also restricted: you can apply `volatile` to references, `byte`/`sbyte`/`short`/`ushort`/`int`/`uint`/`char`/`float`/`bool`, enums with those underlying types, `IntPtr`/`UIntPtr`, and reference-constrained generic parameters. **Not allowed on `long`, `double`, `decimal`, or structs.**

The modern guidance is to skip the modifier and use the explicit fence APIs at the call site:

```csharp
// Per-access fences — works on any type, including long/double, and is local-reasoning friendly.
private long _lastTimestampTicks;

void Producer(long ticks) => Volatile.Write(ref _lastTimestampTicks, ticks);
long  Consumer()           => Volatile.Read (ref _lastTimestampTicks);
```

### `Interlocked`

A static class that performs an **atomic** read-modify-write on a memory location, typically lowering to a `lock`-prefixed CPU instruction (`lock cmpxchg`, `lock xadd`, etc.). Each operation also acts as a full memory barrier.

The kit:

- `Increment`, `Decrement`, `Add` — atomic counters.
- `Exchange<T>(ref location, value)` — atomic swap, returns the old value.
- `CompareExchange<T>(ref location, value, comparand)` — atomic CAS, the building block of every lock-free algorithm.
- `Read(ref long)` — torn-read-safe 64-bit read on 32-bit hosts.
- `And`, `Or` — atomic bitwise (added in .NET 5).
- `MemoryBarrier`, `MemoryBarrierProcessWide` — explicit fences.

```csharp
// Lock-free lazy initialization with CompareExchange.
private static Service? s_instance;
public static Service Instance
{
    get
    {
        var existing = Volatile.Read(ref s_instance);
        if (existing is not null) return existing;

        var fresh = new Service();
        // If s_instance is still null, install fresh; else discard fresh and use the winner.
        return Interlocked.CompareExchange(ref s_instance, fresh, null) ?? fresh;
    }
}
```

### How they compose

Use them together:

- Write a single field once, observe it everywhere → `Volatile.Write` for the publish, `Volatile.Read` for the load.
- Mutate a counter or swap a reference → `Interlocked`.
- Need both fences and atomic mutation → `Interlocked` already gives you the fence.

Don't reach for either if `lock` (or in C# 13+, `System.Threading.Lock`) gives you a clearer critical section. Lock-free code is genuinely hard to get right; the perf win rarely justifies the cognitive cost outside hot paths.

### Modern note (.NET 10 / C# 14)

C# 13 introduced the `System.Threading.Lock` reference type as a cheaper alternative to `lock` on `object`. The `lock(myLock) { … }` statement detects it and lowers to `myLock.EnterScope()` instead of `Monitor.Enter` / `Monitor.Exit`, avoiding the object header word and giving cleaner throughput. Still in .NET 10 — prefer it over `lock(new object())` for new lock fields.

### Senior-level gotchas

- **`volatile` is not a synchronization primitive.** Two volatile fields read in sequence give you no atomicity across them.
- **`volatile long` doesn't compile.** Use `Interlocked.Read` or `Volatile.Read(ref long)`.
- **`Interlocked.CompareExchange<T>` boxes for non-reference types** — there's no generic primitive overload. For `int`/`long`/`float`/`double` use the dedicated overloads — they're allocation-free.
- **`Thread.MemoryBarrier()` is a deprecated alias** for `Interlocked.MemoryBarrier()`; prefer the latter.
- **Don't reach for `volatile` to fix a Heisenbug.** If a field needs `volatile` to "work," it almost certainly needs a `lock` or `Interlocked` instead.

**Sources:**
- [Interlocked Class — .NET 10 Microsoft Learn](https://learn.microsoft.com/en-us/dotnet/api/system.threading.interlocked?view=net-10.0)
- [Lock Class — .NET 10 Microsoft Learn](https://learn.microsoft.com/en-us/dotnet/api/system.threading.lock?view=net-10.0)

## Q49. Explain how Async Streams work

Async streams are the C# 8 / .NET Core 3.0 feature that gives you `IAsyncEnumerable<T>` + `await foreach` + async iterators (`async IAsyncEnumerable<T> M() { … yield return …; await …; }`). They are the right shape for "values that arrive over time and each one needs awaiting."

```csharp
// Producer — a streaming pageable client.
public async IAsyncEnumerable<Order> StreamOrdersAsync(
    DateOnly since,
    [EnumeratorCancellation] CancellationToken ct = default)
{
    string? cursor = null;
    do
    {
        var page = await _client.GetPageAsync(since, cursor, ct);
        foreach (var order in page.Items)
            yield return order;

        cursor = page.NextCursor;
    } while (cursor is not null);
}

// Consumer.
await foreach (var order in StreamOrdersAsync(DateOnly.FromDateTime(DateTime.UtcNow.AddDays(-7)))
    .WithCancellation(ct)
    .ConfigureAwait(false))
{
    await ProcessAsync(order, ct);
}
```

### What the compiler builds

The method body is rewritten into a single state-machine that implements **both** `IAsyncEnumerable<T>` and `IAsyncEnumerator<T>` (the same instance is returned on the first `GetAsyncEnumerator(ct)` call). Each `MoveNextAsync()` returns a `ValueTask<bool>` — when the next value is already available synchronously the task completes inline with zero allocation via `IValueTaskSource<T>`. `DisposeAsync()` runs the iterator's `finally` blocks, including any `await using` cleanup.

### Cancellation

The cancellation story is the single thing the question usually misses. Three pieces have to line up:

```csharp
public async IAsyncEnumerable<T> M([EnumeratorCancellation] CancellationToken ct = default) { ... }
//                                  ↑ tells the compiler "thread the WithCancellation token in here"

await foreach (var x in M().WithCancellation(externalCt)) { ... }
//                          ↑ binds externalCt to the parameter at enumeration time
```

The `[EnumeratorCancellation]` attribute is what makes `WithCancellation()` flow into the iterator body. Without it, the consumer's cancellation token never reaches the producer — a silent bug, since the iterator just keeps producing.

### Comparing alternatives

| Shape | Returns | Push or pull | Multi-consumer | Backpressure |
|-------|---------|--------------|----------------|--------------|
| `Task<IEnumerable<T>>` | a fully materialized list | n/a | yes (snapshot) | none — buffers everything |
| `IAsyncEnumerable<T>` | one item at a time | pull | one (single-pass) | natural — consumer pulls when ready |
| `Channel<T>` | one item at a time | push | yes | configurable via `BoundedChannelOptions` |

Use `IAsyncEnumerable<T>` when the consumer is the rate-limiter and there's exactly one of them. Switch to `Channel<T>` when the producer runs independently or you need multiple readers/writers.

### Modern note (.NET 10 / C# 14)

.NET 10 ships **`System.Linq.AsyncEnumerable`** — a full set of LINQ operators over `IAsyncEnumerable<T>` directly in the BCL, replacing the community `System.Linq.Async` (Ix.NET) package as the default. You can now write:

```csharp
var totals = await StreamOrdersAsync(since, ct)
    .Where(o => o.Total > 100m)
    .Select(o => o.Total)
    .SumAsync(ct);
```

Two compatibility traps to know:

- The new BCL operators **do not accept async lambdas**. The Ix.NET equivalents (`WhereAwait`, `WhereAwaitWithCancellation`, etc.) are not part of the BCL surface. If you need async predicates/selectors, the Ix.NET package is still required.
- If both `System.Linq.Async` and the BCL surface are referenced in the same project, every overlapping call site fails with **CS0121 ambiguous-call**. Pick one.

EF Core has long surfaced `IAsyncEnumerable<T>` directly via `AsAsyncEnumerable()`, and ASP.NET Core minimal APIs serialize an `IAsyncEnumerable<T>` return as a streaming JSON array. The new BCL LINQ surface makes those endpoints far more pleasant to compose.

### Senior-level gotchas

- **Single-pass.** Don't enumerate the same `IAsyncEnumerable<T>` twice unless its source explicitly supports it; many sources are one-shot.
- **`finally` runs on consumer break.** Iterators correctly run `finally` blocks (and `await using` disposers) when the consumer breaks early — but only if the consumer disposes the enumerator (which `await foreach` does).
- **Don't fake-stream.** Returning `IAsyncEnumerable<T>` from a method that buffers the whole result internally is a lie about semantics and breaks consumer assumptions about memory.
- **`ConfigureAwait(false)` goes on the source**, not on each `MoveNextAsync` — `await foreach (var x in src.ConfigureAwait(false))`.

**Sources:**
- [Breaking change: System.Linq.AsyncEnumerable in .NET 10 — Microsoft Learn](https://learn.microsoft.com/en-us/dotnet/core/compatibility/core-libraries/10.0/asyncenumerable)
- [Ix.NET v7.0: .NET 10 and LINQ for IAsyncEnumerable — endjin](https://endjin.com/blog/2025/11/ix-v7-dotnet-10-linq-iasyncenumerable)

## Q50. Difference between `Task.Run` and `TaskFactory.StartNew`

`Task.Run(action)` is, in effect, a curated overload of `TaskFactory.StartNew`:

```csharp
// Task.Run is essentially this:
TaskFactory.StartNew(action,
    CancellationToken.None,
    TaskCreationOptions.DenyChildAttach,
    TaskScheduler.Default);
```

It is the right answer for "schedule this on the thread pool" 95% of the time. `StartNew` is the older, lower-level API that lets you pick scheduler and creation options.

### Why `Task.Run` is the default

Two reasons that bite production code:

**1. `Task.Run` unwraps async lambdas. `StartNew` does not.**

```csharp
// BUG: completes immediately when the OUTER lambda returns the inner Task.
Task<Task> wrong = Task.Factory.StartNew(async () =>
{
    await Task.Delay(1000);
});
await wrong;            // returns in microseconds; the Delay still runs in the background.

// Fix #1: Unwrap manually.
Task right = Task.Factory.StartNew(async () => { await Task.Delay(1000); }).Unwrap();

// Fix #2: just use Task.Run — overloads for Func<Task> and Func<Task<T>>.
Task best = Task.Run(async () => { await Task.Delay(1000); });
```

This `Task<Task>` trap is the single most common bug around the two APIs. The continuation never gets observed, exceptions go unhandled, and your "await" returns instantly.

**2. `Task.Run` denies child attachment.** `DenyChildAttach` blocks any inner task created with `TaskCreationOptions.AttachedToParent` from actually attaching. That keeps the wrapped delegate from leaking exception or completion semantics into surprise parent/child relationships.

### What `StartNew` actually offers on top

Reach for `StartNew` only when you genuinely need one of:

- **Custom `TaskScheduler`** — a UI-thread scheduler, a single-threaded scheduler, a `ConcurrentExclusiveSchedulerPair`, etc.
- **`TaskCreationOptions.LongRunning`** — a hint that the work is long and blocking. The default scheduler typically allocates a dedicated thread instead of borrowing from the pool, avoiding pool starvation. Useless for short async work.
- **`AttachedToParent`** — couples the task to its enclosing parent task. Almost never wanted in modern code.
- **`PreferFairness`** — FIFO instead of LIFO scheduling within the pool.
- **`HideScheduler`** — child tasks see `TaskScheduler.Default` instead of inheriting the current scheduler.
- **A non-`None` `CancellationToken`** — `Task.Run` has overloads for this too, so this alone isn't a reason.

```csharp
// Legitimate StartNew usage: long-blocking pump that should not steal a pool thread.
Task.Factory.StartNew(
    () => RunPumpUntilShutdown(token),
    token,
    TaskCreationOptions.LongRunning | TaskCreationOptions.DenyChildAttach,
    TaskScheduler.Default);
```

### Better alternatives to either

For long-lived blocking work, both APIs are second choices:

- **`new Thread(...) { Name = ... }.Start()`** — explicit, nameable, easy to find in dumps.
- **`BackgroundService` / `IHostedService`** — for app-lifetime work; gets cancellation, DI, structured shutdown for free.
- **A `Channel<T>` consumer loop** in a hosted service — for long-running pipelines.

For "I have a sync method on a pool thread and I want to call it from async code" — `Task.Run`, full stop.

For "I have CPU-bound work I want to await" — `Task.Run`. Don't use `Task.Run` to wrap an *already async* method just to "make it run in the background"; that's pure overhead.

### Senior-level gotchas

- **Never `Task.Run` inside library code** to pretend a sync API is async. The caller can do that themselves and you're spending their thread pool. Make the API truly async or be honest that it's sync.
- **`LongRunning` doesn't mean "run for a long time" — it means "this will block."** If your task is mostly `await`ing I/O, do not use `LongRunning`.
- **Both APIs run on the thread pool by default.** If you need to capture the current `SynchronizationContext` (e.g., the UI), use `TaskScheduler.FromCurrentSynchronizationContext()` with `StartNew`, or just `await` and rely on `ConfigureAwait(true)` (the default).
- **Exceptions thrown inside `Task.Run` show up when the returned task is awaited or `Wait`ed.** They don't crash the process unless you also wire up `TaskScheduler.UnobservedTaskException` and observe nothing.
