## Q40. Explain method overloading and overriding

They look similar — same method name appears in multiple places — but they answer two completely different questions.

**Overloading** = "which method do I call?" Resolved by the **compiler** at compile time using the static types of the arguments. Same method name, different signatures within the same scope (or across an inheritance chain).

**Overriding** = "which implementation runs?" Resolved by the **runtime** at call time using the dynamic type of the receiver. Derived class supplies a new body for an inherited `virtual`, `abstract`, or already-overridden method, declared with the `override` keyword.

```csharp
public class Logger
{
    // Overloads — chosen at compile time by argument types.
    public void Log(string message)              => Console.WriteLine(message);
    public void Log(string message, Exception e) => Console.WriteLine($"{message}: {e}");
    public void Log<T>(T value) where T : notnull => Console.WriteLine(value);

    // Virtual base implementation, ready to be overridden.
    public virtual void Write(string message) => Console.WriteLine(message);
}

public class JsonLogger : Logger
{
    // Override — same signature as the virtual base; chosen at runtime.
    public override void Write(string message)
        => Console.WriteLine($$"""{ "msg": "{{message}}" }""");
}

Logger l = new JsonLogger();
l.Write("hi");          // JsonLogger.Write — runtime dispatch
l.Log("hi");            // Logger.Log(string) — compile-time pick
l.Log<int>(42);         // Logger.Log<T> — compile-time pick over generic
```

### Overloading mechanics

Signature for overloading = method name + parameter list (types, count, modifiers `ref`/`out`/`in`/`params`). **Return type and parameter names are not part of the signature** — `int M()` and `string M()` are not overloads, they collide.

Overload resolution algorithm (simplified):

1. Build the candidate set: every accessible method named `M` in the receiver's type and base types, plus extension methods in scope.
2. Filter to those *applicable* with the given arguments (counts and implicit conversions match).
3. Pick the **better candidate** using tie-break rules: more specific parameter types win; non-generic beats generic when both work; non-`params` beats `params`; non-optional beats optional. Numeric conversions have a fixed "betterness" lattice.
4. If no single best candidate exists, you get **CS0121** ambiguity.

The order of these rules is exactly why `M(0)` calling a method with both `M(int)` and `M(long)` overloads picks `int` — `int` is the more specific applicable type for the literal.

### Overriding mechanics

`virtual` adds a slot to the type's vtable. `override` replaces the slot in the derived type's vtable. Calls to the method through any reference to the base type land on the derived implementation through that vtable lookup. This is the JIT's `callvirt` instruction (note: `callvirt` is also used for non-virtual instance methods to get the null-check, but for virtuals it walks the vtable).

Adjacent keywords:

- **`abstract`** — virtual with no body; the derived class **must** override.
- **`sealed override`** — overrides, but blocks further overrides in deeper subclasses.
- **`new`** — *hides* the base member without participating in the vtable. A reference of the base type calls the base; a reference of the derived type calls the derived. Different from override and almost always not what you want.
- **`base.M()`** — call the next-up implementation from inside an override. Required when the override augments rather than replaces.

```csharp
public class A      { public virtual void M() => Console.Write("A"); }
public class B : A  { public override void M() => Console.Write("B"); }
public class C : B  { public new void M()      => Console.Write("C"); }  // hides!

A a = new C();
a.M();   // "B"  — C's `new` does not participate in dispatch through A.
C c = (C)a;
c.M();   // "C"  — direct call to the hidden member.
```

### Modern features that intersect

- **Covariant return types** (C# 9): an override may return a more derived type than the base. Useful for fluent APIs and clone hierarchies.
  ```csharp
  public class Animal { public virtual Animal Clone() => new Animal(); }
  public class Dog : Animal { public override Dog Clone() => new Dog(); } // returns Dog, not Animal
  ```
- **Default interface methods** (C# 8): an interface can supply a body. Implementers may "override" by providing their own. The dispatch story is its own animal — DIM is *not* virtual through the class vtable; calling `((I)obj).M()` vs `obj.M()` can land on different implementations if the class doesn't explicitly implement `I.M`.
- **Primary constructor parameters** (C# 12 for class/struct) are not properties or fields — they don't get inherited or overridden. If you need polymorphic state, expose it as a `protected` property.
- **`params Span<T>`** (C# 13) participates in overload resolution: a method with `params Span<int>` is preferred over one with `params int[]` when both are applicable, because the language wants stack-allocated argument lists when possible.

### Senior-level gotchas

- Overloading + optional parameters is a known footgun. `void M(int x, int y = 0)` and `void M(int x)` both fit `M(5)`, and the rule "non-optional wins" picks `M(int)` — but flip the optional from one method to the other and the answer changes silently. Avoid mixing the two.
- Overload resolution does **not** consider generic constraints when picking between candidates. A constraint failure produces a hard error after the candidate is chosen, not a different pick. This trips people who think "the generic version won't apply because `T : struct`."
- Calling `virtual` from a constructor dispatches to the **most-derived** override — but at the time of the call, that derived class's own constructor body has not yet run, so its fields are at their default values. This is the canonical "virtual call from constructor" bug.
- `override` requires a matching signature *including parameter names matter for nothing, but defaults must match-or-omit*. C# lets you change a default value in an override; the **declared** default at the call site is taken from the static type, not the runtime type. So `((Base)x).M()` and `((Derived)x).M()` can pass different default values to the same physical method.
- An override's accessibility must equal the base's (you cannot widen). You can, however, narrow `protected internal` to `protected` in some specific scenarios — read the spec before you try.
- Operator overloading (`operator +`, `operator ==`, etc.) is method overloading in disguise — same resolution rules — but you must override `Equals` and `GetHashCode` together with `==`/`!=` or you'll get analyzer warnings (`CS0660`/`CS0661`).
- Hiding (`new`) silently breaks Liskov. Any code that holds a base reference to a derived instance gets the base behavior; this almost always becomes a bug. If you find yourself reaching for `new`, you usually want a different method name.

**Modern note (.NET 10 / C# 14):** C# 14's **extension members** ([Microsoft Learn](https://learn.microsoft.com/en-us/dotnet/csharp/whats-new/tutorials/extension-members)) bring new entries into the overload-resolution conversation — extension *properties*, extension *operators*, and *static* extension members. They use the new `extension(Receiver r)` block:
```csharp
public static class StringExt
{
    extension(string s)
    {
        public bool IsBlank => string.IsNullOrWhiteSpace(s);  // extension property
    }
}
```
Resolution still prefers instance members over extensions, and instance methods over extension methods — the rules just generalize to properties and statics. Overriding is unchanged in C# 14.

## Q41. Difference between `IEnumerable` and `IQueryable`

Both describe a deferred sequence of `T`. The split is *who* runs the query and *how* it's represented.

| | `IEnumerable<T>` | `IQueryable<T>` |
|---|---|---|
| Defined in | `System.Collections.Generic` | `System.Linq` |
| Represents | An iterator over an in-memory sequence (`MoveNext`/`Current`) | A query against an external provider, expressed as an expression tree |
| LINQ extensions live in | `Enumerable` static class | `Queryable` static class |
| Lambdas compile to | `Func<T, ...>` delegates | `Expression<Func<T, ...>>` trees |
| Where the work runs | In the C# process | Wherever the provider executes (SQL Server, Cosmos, OData service…) |
| When does work happen | Iteration time (foreach / `ToList`) | Materialization time (`ToList`, `FirstAsync`, `await foreach`…) |
| Inheritance | base | `IQueryable<T> : IEnumerable<T>` |

```csharp
DbSet<Order> orders = db.Orders;          // IQueryable<Order>

IQueryable<Order> q1 = orders.Where(o => o.Total > 100);   // expression tree
IEnumerable<Order> q2 = orders.AsEnumerable()              // boundary: now LINQ-to-Objects
                              .Where(o => Compute(o) > 100); // runs in C#

var list = await q1.ToListAsync(); // SELECT ... WHERE Total > 100
```

The important visual cue: every operator on `IQueryable<T>` returns another `IQueryable<T>` and *adds a node to the expression tree*. The provider only sees SQL when you ask for a result.

### Boundaries

- **`AsEnumerable()`** — turns an `IQueryable<T>` into an `IEnumerable<T>` reference. Future operators bind to `Enumerable.*`, not `Queryable.*`. From this point on, every row is pulled into memory and filtered/projected in C#.
- **`AsQueryable()`** — wraps an in-memory sequence in a queryable interface. Useful for testing — you can substitute an in-memory `IQueryable<T>` for a real `DbSet<T>` — but it does not give you provider semantics.
- **`ToList`/`ToArray`/`ToDictionary`/`First`/etc.`** materialize. For EF Core, prefer the `*Async` variants; they release the thread during the I/O.
- **`AsAsyncEnumerable()`** — gets you an `IAsyncEnumerable<T>` you can stream with `await foreach`, which is the right tool when you don't want to buffer all rows.

### The client-evaluation trap

The single most expensive mistake people make with `IQueryable`:

```csharp
// 1. Project entities — IQueryable<Order>
var pending = db.Orders.Where(o => o.Status == OrderStatus.Pending);

// 2. Now hand it to a method typed as IEnumerable<Order>:
public IEnumerable<Order> FilterRecent(IEnumerable<Order> source)
    => source.Where(o => o.CreatedAt > DateTime.UtcNow.AddDays(-7));

var recent = FilterRecent(pending).ToList();
```

`FilterRecent` upcasts to `IEnumerable<Order>`. The second `Where` binds to `Enumerable.Where` (not `Queryable.Where`), forcing EF to fetch every pending order from SQL and filter client-side. The `Status` predicate translated; the `CreatedAt` predicate did not. There is no warning — this is the EF Core "silent table scan" pattern.

EF Core 3.0+ throws on most non-translatable expressions in the *query tree*, but it cannot save you from someone re-typing the variable. The fix is contractual: keep the type `IQueryable<T>` all the way down to the materialization call.

### Async streaming

`IAsyncEnumerable<T>` is the streaming companion. EF Core exposes it through `AsAsyncEnumerable()`, and `await foreach` consumes a row at a time without buffering. Use this for very large result sets where holding the whole list defeats the database's streaming.

### Senior-level gotchas

- `Count()` on `IQueryable<T>` issues `SELECT COUNT(*)`. `Count()` on `IEnumerable<T>` enumerates the whole sequence. Same method name, completely different cost.
- A custom method (e.g., `IsEligible(o)`) inside a `Where` lambda on `IQueryable<T>` cannot be translated by EF — provider throws. The fix is either inline the predicate or rebuild it as an `Expression<Func<Order, bool>>` and use [LinqKit](https://github.com/scottksmith95/LINQKit) / `Expression.AndAlso` to compose.
- `IQueryable<T>` providers are not interchangeable. EF Core's translator differs from Cosmos LINQ from OData from MongoDB.Driver.Linq — write integration tests against the real provider, not in-memory `AsQueryable()`.
- `IQueryable<T>` does not equal "deferred" — both are deferred. The difference is *where* execution happens.
- Iterating an `IQueryable<T>` twice issues the SQL twice. Materialize once with `ToList()` if you need to reuse the results.
- LINQ's `Concat`, `Union`, etc. on two queryables from *different contexts* will fail at translation time. Materialize the smaller side.
- Returning `IQueryable<T>` from a repository / service method leaks the provider's evaluation surface to callers. That can be desirable (composable, efficient) or dangerous (callers can DDOS your DB with `.Where(o => o.Notes.Contains("x"))`). Pick a posture and stick with it; teams usually pick "return `IReadOnlyList<T>` after materialization" for boundaries and `IQueryable<T>` only inside the data layer.

**Modern note (.NET 10 / C# 14):** No language-level changes. Worth knowing on the runtime side: EF Core 10 continues investing in `IAsyncEnumerable<T>` ergonomics and JSON-column query translation, both of which surface as `IQueryable<T>` operators. The boundary rules are unchanged.

## Q42. What are expression trees in LINQ?

An **expression tree** is a runtime object that represents code as data — the abstract syntax tree of an expression, in `System.Linq.Expressions` types (`Expression`, `BinaryExpression`, `MethodCallExpression`, `LambdaExpression`, …). The compiler emits an expression tree (instead of IL) when a lambda is assigned to a variable of type `Expression<TDelegate>`:

```csharp
Func<Order, bool>             del  = o => o.Total > 100; // IL — a delegate
Expression<Func<Order, bool>> tree = o => o.Total > 100; // tree — data

// tree is roughly:
// LambdaExpression(
//   parameters: [ParameterExpression("o", typeof(Order))],
//   body: BinaryExpression(
//     NodeType: GreaterThan,
//     left:  MemberExpression(o, Order.Total),
//     right: ConstantExpression(100)))
```

That tree is walkable, transformable, and *compilable* back into a delegate at runtime via `Expression.Compile()`.

### Why they exist

`IQueryable<T>` is the canonical consumer. EF Core, Cosmos LINQ, OData, and dynamic ORM systems read the expression tree, translate it to a target language (SQL, JSONpath, Cosmos query, etc.), and execute it remotely. Without expression trees, you'd be back to writing string-concatenated SQL.

Other consumers:

- Mocking libraries (Moq) use them to encode "the call you expect" as data: `mock.Setup(m => m.Get(It.IsAny<int>()))`.
- Validation / mapping libraries (FluentValidation, AutoMapper) take `Expression<Func<T, TProp>>` to refer to a member without strings.
- High-perf reflection alternatives — building and compiling an expression tree once is faster than `PropertyInfo.GetValue` in a hot loop.

### Building, walking, rewriting

```csharp
var p = Expression.Parameter(typeof(Order), "o");
var body = Expression.GreaterThan(
    Expression.Property(p, nameof(Order.Total)),
    Expression.Constant(100m));

Expression<Func<Order, bool>> tree = Expression.Lambda<Func<Order, bool>>(body, p);
Func<Order, bool> compiled = tree.Compile();
```

The traversal pattern is `ExpressionVisitor` — subclass it, override the `Visit*` methods you care about, return a rewritten node:

```csharp
sealed class ParameterReplacer(ParameterExpression from, Expression to) : ExpressionVisitor
{
    protected override Expression VisitParameter(ParameterExpression node)
        => node == from ? to : base.VisitParameter(node);
}
```

This is how you compose two `Expression<Func<T, bool>>` predicates — you *cannot* just `&&` them at the language level (those are delegates), so you visit one tree, swap its parameter for the other's, and rebuild a `Lambda`.

### Limits — and the senior must know these

- **Expression-bodied only.** `Expression<Func<T, bool>> e = x => { return x > 0; };` — the **statement body** form is forbidden for `Expression<>`. (It compiles for `Func<>` because that's a delegate.)
- **No `await`, no `yield`, no local functions, no `ref`/`out` arguments, no tuple deconstruction in the body.**
- **No assignments** in `Expression<Func<...>>` until C# 9 added some support — the `Expression.Assign` API exists, but the *language compiler* won't translate `x = 1` inside a lambda assigned to `Expression<>`. You have to build assignments by hand.
- **Optional parameter defaults** are baked into the call expression as constants.
- **Captured locals** show up as `MemberExpression` over a synthesized closure class. Providers like EF Core then "constantize" them — that's why a parameterized SQL query is generated for `var x = 5; q.Where(o => o.Total > x);`.

### `Compile()` cost and caching

`Expression.Compile()` runs an in-process IL emitter (or, with `Expression.CompileToMethod` / `DynamicMethod`, slightly different paths) and is **slow**. Hundreds of microseconds per compile. The compiled delegate runs at near-native speed, but compiling per call wipes the win.

The fix: cache the compiled delegate, keyed by the tree's identity if you must, or by some structural key you control. EF Core has its own compiled-query cache (`EF.CompileQuery`, `EF.CompileAsyncQuery`) that bypasses retranslation entirely for hot queries — measurable on tight loops.

### Senior-level gotchas

- The compiler emits a **fresh tree on every call site evaluation** for a lambda `Expression<>`. Two adjacent assignments of the same lambda produce two different `Expression<>` instances. Don't compare expression trees with `==`.
- `expr.ToString()` gives a readable but lossy representation; `expr.DebugView` (an internal property surfaced in the debugger) is more accurate. For programmatic equality, walk both trees with a custom visitor.
- Expression trees do not capture the **closure semantics** of `var x = 5` perfectly — they capture the *closure field reference*. If you mutate `x` after building the tree, the next compile/execute reads the new value. This is occasionally what you want, often a bug.
- `Expression.Convert` is generated implicitly when you select a value-typed member into an `object`-returning lambda — providers must handle that conversion node. Some custom providers don't and silently produce wrong queries.
- `ExpressionVisitor` is **not** virtual on every method — overriding the wrong one (`VisitMember` vs `VisitMemberAccess` historically) is an easy slip. Always check the base method exists for what you want.
- Building an expression by hand and forgetting to call `body.Reduce()` on a node that's a `BlockExpression` or an extension expression can leak into the provider as an unsupported node.

**Modern note (.NET 10 / C# 14):** Expression trees themselves have not gained new node kinds in C# 14. The complementary tooling worth pairing with them in your head: Roslyn **incremental source generators** are now the preferred path when the goal is "rewrite or generate code at compile time" — choose generators over runtime expression trees when the data is known at build time, choose expression trees when the data only exists at runtime (user-supplied predicates, dynamic UI filters). EF Core 10's compiled query API (`EF.CompileAsyncQuery`) remains the right escape hatch for hot-path query reuse.

## Q43. How does exception handling work in C#?

Exception handling in C# is the runtime's structured-exception-handling (SEH on Windows, equivalent on other OSes), wrapped by the language as `try`/`catch`/`finally` and surfaced as `System.Exception` and its subclasses.

```csharp
try
{
    var content = await client.GetStringAsync(url, ct);
    return Parse(content);
}
catch (TaskCanceledException) when (!ct.IsCancellationRequested)
{
    // Timeout — distinguish from caller cancellation via the filter.
    return Fallback();
}
catch (HttpRequestException ex)
{
    log.LogWarning(ex, "Fetch failed");
    throw new ServiceUnavailableException("Upstream offline", ex);
}
finally
{
    metrics.IncrementFetch();
}
```

### Mechanics

1. `throw new SomeException(...)` allocates the exception, captures the current stack into `StackTrace`, and walks the call stack looking for a matching `catch`.
2. Each `catch` clause is matched in order; a clause matches if (a) the exception is assignable to its declared type and (b) its `when` filter (if any) returns `true`.
3. **`when` filters run *before* the stack unwinds.** This is the property that makes them special: the debugger and stack snapshots see the original frames. `catch (Exception e) when (Log(e))` with `Log` returning `false` is the canonical "log without unwinding" trick.
4. The matching `catch` runs. If it `throw;`s, the original exception continues propagating with its stack preserved.
5. `finally` blocks always run — on normal flow, on caught exceptions, on rethrows.

### Around `async`

The state machine generated for `async` methods stores any thrown exception inside the returned `Task`, where it stays until someone `await`s it. At that `await`, the exception is rethrown via `ExceptionDispatchInfo.Throw()`, which preserves the original stack and stitches the awaiter's stack frames on top.

```csharp
async Task<int> Outer()
{
    try { return await Inner(); }
    catch (InvalidOperationException ex)
    {
        // ex.StackTrace shows the original throw site in Inner
        // *and* the await chain through Outer.
        throw;
    }
}
```

`Task.WhenAll` aggregates child task exceptions into an `AggregateException` *on the task*, but `await` unwraps to the **first** exception. To see all of them, await the task to faulted state and read `task.Exception` (`AggregateException`) directly.

### Cancellation

`OperationCanceledException` (and its derived `TaskCanceledException`) gets special treatment:

- Constructor that takes a `CancellationToken` is the right one to throw — preserves the token for the caller to inspect.
- `await` on a canceled task throws `OperationCanceledException`, which is sometimes a category that needs a *different* catch than your "real" errors. The senior idiom: `catch (OperationCanceledException) when (ct.IsCancellationRequested) { throw; }` to let cancellation bubble cleanly.
- Don't swallow `OperationCanceledException` in middleware — it's a flow-control exception, not an error. ASP.NET Core treats it as request-aborted and skips the error pipeline.

### `try`/`finally` ordering with `using`

`using` and `await using` desugar to `try`/`finally`. Order matters when you nest them:

```csharp
await using var conn = await db.OpenAsync(ct);   // disposed last
await using var tx   = await conn.BeginAsync(ct); // disposed first
// On exception: tx rolled back, conn closed, in that order.
```

The "outer disposable disposed last" guarantee is what makes nesting transactions inside connections safe.

### Senior-level gotchas

- Catching `Exception` is sometimes correct (top-level handlers, Polly), but in libraries it almost always is not. Catch the type you can actually handle and leave everything else alone.
- `catch (Exception)` does **not** catch a `StackOverflowException` (uncatchable since .NET 2.0) and may not give you a chance to handle `OutOfMemoryException` cleanly. `ThreadAbortException` no longer exists in .NET Core onward.
- Exceptions in `finally` blocks **replace** the in-flight exception. The original is lost — log defensively. (`ExceptionDispatchInfo` can mitigate but rarely worth it.)
- A `catch` block's `when` filter that itself throws is treated as "filter returns false" *and* the new exception is swallowed. Filters must be safe.
- Throwing across an interop boundary (`DllImport`) does not work — managed exceptions cannot unwind native frames. Use return codes or `Marshal.GetLastWin32Error`.
- `async void` propagates uncaught exceptions to `SynchronizationContext.UnobservedException` / `AppDomain.UnhandledException`, generally crashing the process. Reserve `async void` for top-level event handlers and wrap with `try`/`catch`.
- Exceptions are slow on the *throw* path — first-chance exception capture, stack walking, security-info attach. Don't use exceptions for control flow (use `TryParse` / `Try*` patterns).
- `throw` as an expression (C# 7) lets you write `var x = arg ?? throw new ArgumentNullException(nameof(arg));` — convenient, idiomatic.
- ASP.NET Core's exception handling has its own pipeline: middleware → exception-handler middleware → `ProblemDetails`. Throwing a domain exception that surfaces as JSON `application/problem+json` requires either a `ProblemDetailsService` mapping or an `IExceptionHandler` (added in .NET 8). Don't reinvent error JSON in controllers.
- Polly is the resilience library: retry, circuit breaker, timeout, bulkhead, hedging. The "should I retry?" question is exception-shape work — `IsTransient` policies live here.

**Modern note (.NET 10 / C# 14):** No language-level changes to exception handling in C# 14. The cumulative ASP.NET Core 10 surface is mature: `IExceptionHandler` + global exception middleware + `ProblemDetails` + the `[ProblemDetails]`-friendly minimal API filters now compose cleanly without a custom middleware. C# 14's null-conditional assignment (`obj?.Prop = value`) is unrelated but pairs well with defensive code that previously needed a `try`-style guard.

## Q44. Name all the ways to rethrow an exception

Five variants worth knowing — three correct, one anti-pattern, one adjacent.

### 1. `throw;` — preserve the original stack

The right answer 90% of the time. Inside a `catch` block, a bare `throw` rethrows the current exception **with its stack trace intact**.

```csharp
catch (IOException)
{
    metrics.Increment("io_failure");
    throw;
}
```

Restrictions: only valid inside a `catch` block (or a method called from one, for `ExceptionDispatchInfo` scenarios — see below). The Roslyn analyzer **CA2200** ([Microsoft Learn](https://learn.microsoft.com/en-us/dotnet/fundamentals/code-analysis/quality-rules/ca2200)) flags any `throw ex;` inside a catch as "Rethrow to preserve stack details." Treat it as an error.

### 2. `throw new XException(message, inner)` — wrap and preserve via `InnerException`

Use this when you want to add domain context. The original is preserved as `ex.InnerException`, walked by logging frameworks and visible in `Exception.ToString()`.

```csharp
catch (HttpRequestException ex)
{
    throw new InvoiceFetchException($"Could not fetch invoice {id}", ex);
}
```

The wrapper carries its own (new) stack trace; the inner one keeps its original. Combine wisely — `Exception.ToString()` prints the chain.

### 3. `ExceptionDispatchInfo.Capture(ex).Throw()` — preserve the stack across boundaries

Use when you need to **store** an exception now and rethrow it **later, possibly on a different thread**, with the stack trace intact.

```csharp
ExceptionDispatchInfo? captured = null;
try { DoWork(); }
catch (Exception ex) { captured = ExceptionDispatchInfo.Capture(ex); }

// Some time later, possibly on another thread / async continuation:
captured?.Throw();
```

This is what `await` uses internally to surface a faulted `Task`'s exception. CA2200's docs explicitly cite this as the right pattern when rethrowing **outside** of a catch block. `ExceptionDispatchInfo.Throw(Exception)` (static, .NET 5+) is the same idea without a captured info.

Adjacent: **`ExceptionDispatchInfo.SetCurrentStackTrace(ex)`** (.NET 5+) lets you stamp a manually-constructed (not-yet-thrown) exception with the current stack trace before throwing — handy when wrapping a captured exception inside a new one and you want the wrapper's stack to match the throw site, not the wrapper's allocation site.

### 4. `catch (...) when (LogAndContinue(...))` — the filter trick

Strictly speaking this is *not* rethrowing — but it's adjacent and frequently asked alongside. A `when` filter that returns `false` causes the catch to be skipped, leaving the exception to propagate as if no handler were present. Crucially, the stack does not unwind during the filter, so logs and dumps see the original frames.

```csharp
catch (Exception ex) when (Log(ex)) { /* never reached */ }
static bool Log(Exception ex) { logger.LogError(ex, "captured"); return false; }
```

Why include this in a "rethrow" answer: it accomplishes the practical goal of "log and let propagate" with a *better stack* than `catch { Log(); throw; }` (which has unwound to the catch frame by the time logging runs).

### 5. `throw ex;` — the wrong way (mention only to call it out)

```csharp
catch (Exception ex)
{
    throw ex;   // CA2200 — original stack lost
}
```

Resets the stack trace to the current method. The exception object is reused but its `_stackTraceString` is rewritten on the next throw. Microsoft Learn's CA2200 rule documents this exactly: the stack list "between the original method that threw the exception and the current method is lost." Never ship this; the analyzer will flag it.

### Senior-level gotchas

- After `await`, `throw;` *does* preserve the original stack — because the awaiter already used `ExceptionDispatchInfo` to surface it. So in `try { await X(); } catch { throw; }` you keep both `X`'s stack and the awaiter's frames.
- Wrapping always loses your ability to filter on the inner exception in upper handlers unless they walk `InnerException`. Polly's policies *do* walk; many hand-rolled catches don't.
- Rethrowing from `Task.WhenAll`'s aggregate: `await Task.WhenAll(...)` throws the first exception; if you want all, capture and `throw new AggregateException(allErrors)` deliberately.
- `throw;` inside a `catch` filter (the `when` clause) is a compile error — filters must return `bool`, not throw.
- In libraries you write, prefer `throw new MyDomainException(msg, ex)` at boundaries (so callers handle a stable type) and bare `throw;` deep in implementation (so domain errors keep flying).
- `ExceptionDispatchInfo.Throw()` is what every well-built async helper uses internally; if you find yourself writing `try/catch/store/Throw()` by hand, write it as `ExceptionDispatchInfo` immediately.

**Modern note (.NET 10 / C# 14):** No language-level changes to rethrow semantics in C# 14. The platform-level relevance: `ExceptionDispatchInfo.SetCurrentStackTrace` continues to be the right way to mark a manually-constructed exception with a meaningful stack, and is increasingly used by source generators that build exceptions on your behalf.

## Q45. Explain generics

Generics let you write a type or method parameterized by one or more **type parameters**, instantiated by callers with concrete type arguments. The CLR makes type parameters real: at runtime, `List<int>` and `List<string>` are distinct types, with their own JIT-compiled methods. This is **reified generics** — fundamentally different from Java's erasure.

```csharp
public sealed class Cache<TKey, TValue> where TKey : notnull
{
    private readonly Dictionary<TKey, TValue> _map = new();
    public bool TryGet(TKey key, out TValue? value) => _map.TryGetValue(key, out value);
    public void Put(TKey key, TValue value) => _map[key] = value;
}

public static class Algos
{
    public static T Max<T>(T a, T b) where T : IComparable<T>
        => a.CompareTo(b) >= 0 ? a : b;
}
```

### What you can parameterize

Types (`class`, `struct`, `record`, `interface`), methods, delegates. Not properties, not events, not fields directly (a field can have a generic type, but cannot itself introduce a new type parameter).

### Constraints (the catalog)

The `where` clause restricts what the type argument can be. Constraints expand the operations the body can use.

| Constraint | Meaning |
|---|---|
| `where T : class` | Reference type. Allows null comparisons. |
| `where T : class?` | Reference type, nullability oblivious to caller's annotation. |
| `where T : struct` | Non-nullable value type (`Nullable<T>` does **not** satisfy this). |
| `where T : notnull` | Reference or value, just not nullable. |
| `where T : unmanaged` | Value type with no reference fields anywhere in the layout. Enables `sizeof(T)`, `stackalloc T[n]`, `fixed`. |
| `where T : new()` | Has a public parameterless ctor. Enables `new T()`. |
| `where T : SomeBase` | Inherits from `SomeBase`. |
| `where T : ISomeInterface` | Implements `ISomeInterface`. |
| `where T : U` | Type parameter dependency — `T` must be `U` or derived. |
| `where T : default` | Disambiguation in overrides for unconstrained vs constrained scenarios. |
| `where T : allows ref struct` | **C# 13** — anti-constraint that *permits* `T` to be a `ref struct` like `Span<T>`. |

The first three (`class`/`struct`/`notnull`) are mutually exclusive. Combining is otherwise allowed, with `new()` last.

### `allows ref struct` — the C# 13 unlock ([Microsoft Learn](https://learn.microsoft.com/en-us/dotnet/csharp/language-reference/keywords/where-generic-type-constraint))

`ref struct` types (`Span<T>`, `ReadOnlySpan<T>`, `Utf8.JsonReader`, `ReadOnlySequence<T>` enumerator) historically could not be used as generic type arguments at all — because generic infrastructure boxes, captures into closures, and stores in heap fields, which `ref struct`s cannot do.

C# 13's `where T : allows ref struct` opts a single type parameter into accepting ref structs. Inside the body, the compiler enforces ref-safety rules — no boxing, no closure capture, no heap field of type `T`. The canonical use site is the BCL's `string.Create`:

```csharp
public static string Create<TState>(int length, TState state, SpanAction<char, TState> action)
    where TState : allows ref struct;
```

Now the state can be a `ReadOnlySpan<char>`, threaded through the callback without copies. Several .NET 9/10 APIs adopt the same pattern.

### Variance

Generic interfaces and delegates may declare type parameters as **covariant (`out T`)** or **contravariant (`in T`)**. Generic *classes* never have variance.

```csharp
IEnumerable<Dog>     dogs    = new List<Dog>();
IEnumerable<Animal>  animals = dogs;            // covariant — Dog is-a Animal

Action<Animal> log = a => Console.WriteLine(a);
Action<Dog>    dl  = log;                       // contravariant — handler of Animal handles Dog
```

The rule of thumb: `out` for "produces T" (return positions), `in` for "consumes T" (parameter positions), invariant if both. Variance only works on reference type arguments — `IEnumerable<int>` is not `IEnumerable<object>`, because boxing isn't a reference conversion.

### Static abstracts in interfaces (C# 11) — generic math

C# 11 added `static abstract` and `static virtual` interface members. Combined with generics, this enables interfaces over operators and constants:

```csharp
public static T Sum<T>(IEnumerable<T> source) where T : INumber<T>
{
    T total = T.Zero;
    foreach (var x in source) total += x;
    return total;
}
```

`T.Zero` and `+` come from interfaces (`INumberBase<T>.Zero`, `IAdditionOperators<T,T,T>`). This widens "generic over data" into "generic over algebra" — one method works for `int`, `decimal`, `double`, `BigInteger`, custom money types, etc.

### How the runtime implements it

- **Value type T** — the JIT generates a separate code body specialized for that T. Inlining works through the type parameter. This is why `List<int>` outperforms `ArrayList` decisively — no boxing, full specialization.
- **Reference type T** — the JIT generates a single shared body called the "canon" body, which uses runtime type information to find type-specific data when needed. All `List<string>`, `List<MyClass>`, `List<Animal>` share machine code.
- **Reflection sees T**. `typeof(List<int>) != typeof(List<string>)`, both have a `GetGenericArguments()`. Contrast with Java where erasure removes the parameter at runtime.

### Senior-level gotchas

- `where T : struct` excludes `T = Nullable<U>` — there is **no** way to write a single generic that accepts "any struct *or* Nullable<that struct>"; you write two methods.
- `default(T)` for an unconstrained `T` is `null` for reference T and zero-initialized for value T. The C# 9+ analyzer treats it as maybe-null at every read site — you'll see `T?` appearing as the result type of a method that returns `default`.
- Generic constraints **do not** participate in overload resolution. A non-generic `void M(int)` and `void M<T>(T) where T : struct` both apply to `M(5)`; the rule "non-generic beats generic" picks the non-generic. Constraint failures only matter *after* the candidate is chosen.
- `class` constraint vs nullability: `where T : class` means non-nullable reference type; `where T : class?` allows the caller to pass `string?`. Use the latter when your generic is a passthrough container.
- Variance is shallow. `IList<Dog>` is not `IList<Animal>` — `IList<T>` uses T in both `get` and `set` positions, so it must be invariant. People expect array-like covariance (which exists for arrays but is broken — `Animal[] a = new Dog[1]; a[0] = new Cat();` throws `ArrayTypeMismatchException` at runtime, the original sin of .NET arrays).
- A `record class` with a generic type parameter still gets value-equality based on `EqualityComparer<TParam>.Default` — which is reference equality for class T unless T overrides `Equals`. Easy footgun for "why aren't my records equal?"
- Static fields of a generic type are **per-instantiation**: `Counter<int>.Count` and `Counter<string>.Count` are different storage locations. Looks like a bug, is the spec.
- Code size bloat is real with many distinct value-type Ts. If you have `Map<int>`, `Map<long>`, `Map<DateTime>`, `Map<Guid>` etc. all with hot methods, the JIT produces N copies. NativeAOT exposes this directly as binary size; for ahead-of-time scenarios, sometimes you specialize manually.
- Reflection over an open generic (`typeof(List<>)`) is a *type definition*, not a usable type. You construct via `MakeGenericType(typeof(int))`. Closed generics participate in reflection like any other type.

**Modern note (.NET 10 / C# 14):** No new generic features at the language level in C# 14; the cumulative picture is C# 11's static abstracts + C# 13's `allows ref struct`. C# 14's **extension members** can be generic (e.g., extension properties on a generic receiver), and C# 14's **partial constructors and events** apply to generic types as much as non-generic ones. Generic math (`INumber<T>`) is mature in .NET 10 and is now the expected idiom for "works for all numeric types."
