## Q6. How to create an extension method?

An extension method is a `static` method declared in a `static` (non-generic, non-nested) class, whose **first parameter is prefixed with `this`**. The C# compiler then lets callers invoke it with instance-call syntax (`receiver.Method(args)`), but at the IL level it is just a static call — there is no runtime polymorphism.

```csharp
namespace MyExtensions;

public static class StringExtensions
{
    public static bool IsNullOrBlank(this string? s) =>
        string.IsNullOrWhiteSpace(s);
}

// Caller side
using MyExtensions;
"  ".IsNullOrBlank();   // true — compiles to StringExtensions.IsNullOrBlank("  ")
```

Resolution rules worth knowing:

- **Instance methods always win.** If the receiver type (or a base type) defines a matching instance method, the extension is shadowed silently — you get no warning. Adding a method to a base class can therefore break extension callers in a downstream library.
- The extension's namespace must be **in scope via `using`** (or via global usings / implicit usings in SDK-style projects) for the call to resolve. This is why LINQ requires `using System.Linq;`.
- Extensions cannot add **state** (no instance fields), cannot **override** virtual members, and cannot be called via reflection as if they were instance methods.
- Generic extensions and constraints work as expected: `public static T? FirstOr<T>(this IEnumerable<T> src, T? fallback) where T : class`.

Idiomatic uses across the BCL: LINQ (`Enumerable`, `Queryable`), fluent builders (`IServiceCollection.AddSingleton<T>()`), `IApplicationBuilder.UseXxx()`, `Span<T>` / `Memory<T>` helpers.

**Modern note (.NET 10 / C# 14):** C# 14 ships **extension members** — not just methods. You can now declare extension **properties**, **indexers**, **static methods**, and **static properties** by grouping them inside an `extension` block on the receiver type. The classic `this`-parameter syntax still works and remains backward-compatible.

```csharp
public static class EnumerableExtensions
{
    extension<T>(IEnumerable<T> source)
    {
        // Instance extension property
        public bool IsEmpty => !source.Any();

        // Instance extension method (no more 'this T source' parameter)
        public IEnumerable<T> WhereNotNull() => source.Where(x => x is not null);
    }

    extension<T>(IEnumerable<T>)            // no receiver name → static extension
    {
        public static IEnumerable<T> Empty() => [];
    }
}

// Usage
new int[0].IsEmpty;                 // extension property
IEnumerable<string>.Empty();        // static extension method on the type itself
```

The receiver's type, generic parameters, and constraints are declared once on the `extension` block instead of being repeated per member, which finally fixes the "fluent helper class" boilerplate problem.

**Senior-level gotchas:**

- `this string s` accepts `null` — the method is not invoked through the reference, so no NRE is thrown automatically. Always validate or use `[NotNullWhen]` annotations.
- Two namespaces defining the same extension produce **CS0121** ambiguity at the call site — diamond-import bugs become a versioning hazard for libraries.
- Extension methods are not part of the type's reflection surface. `typeof(string).GetMethods()` will not list `IsNullOrBlank`. Code that relies on reflection (mappers, serializers) will not see them.
- Extension members on interfaces are not the same thing as **default interface methods** (DIM, C# 8). DIM lives on the type; extension members are external sugar.

## Q7. Does C# support multiple class inheritance?

**No.** A C# class has exactly one base class (implicitly `System.Object`). The CLR itself is single-inheritance for classes — this is a runtime constraint, not just a language one — chosen to avoid the diamond ambiguity problem (C++'s virtual base classes) and to keep method-table layout deterministic.

What C# *does* allow:

- **Multiple interface implementation.** A class (or struct) can implement any number of interfaces.
- **Default interface methods (DIM)** — C# 8 / .NET Core 3.0 onwards. An interface may carry a default body, so multiple inheritance of *behavior* (not state) is possible.
- **Composition** — hold the would-be-base as a field and forward calls. This is almost always the better answer for code reuse.

```csharp
public interface ICache { void Set(string k, object v); }
public interface IMetrics { void Increment(string name); }

public sealed class InstrumentedCache : ICache, IMetrics
{
    private readonly ICache _inner;
    private readonly IMetrics _metrics;

    public InstrumentedCache(ICache inner, IMetrics metrics)
        => (_inner, _metrics) = (inner, metrics);

    public void Set(string k, object v) { _metrics.Increment("cache.set"); _inner.Set(k, v); }
    public void Increment(string name)  => _metrics.Increment(name);
}
```

DIM diamond resolution: if two base interfaces both supply a default for the same member, the implementing class **must** provide its own override or explicitly disambiguate (`IA.M()` / `IB.M()`); otherwise CS8705. The CLR resolves at runtime via the method-impl table.

**Senior-level gotchas:**

- DIM cannot add state — interfaces still cannot hold fields. If you find yourself wanting "multiple inheritance for state", you want composition.
- **Records** can only inherit from other records; a record cannot inherit a non-record class and vice versa. This is a record-equality / `EqualityContract` requirement, not arbitrary.
- Calling a default interface method requires invoking it through the interface (`((IFoo)x).M()`), not through the concrete reference, unless the type re-declares the member. This catches people off guard in API design.
- Don't reach for DIM as "Java 8 default methods". The intended use case is **safely evolving published interfaces** (adding a new member without breaking existing implementers), not as a primary code-reuse mechanism.

## Q8. Explain boxing and unboxing

**Boxing** is the implicit conversion of a value type to `object` (or to an interface the value type implements). The runtime allocates a box on the **managed heap**, copies the value into it, and prepends the standard object header (sync block + method-table pointer). **Unboxing** is the explicit reverse: the runtime checks that the box's actual type matches `T` (throws `InvalidCastException` if not), then copies the payload back to a value-type slot.

At IL level these are the `box` and `unbox.any` opcodes — they are real instructions, not free.

```csharp
int i = 42;
object o = i;       // box     — heap allocation, ~24 bytes on 64-bit
int j = (int)o;     // unbox.any — type check + copy
```

Why it matters: every box is a **gen-0 allocation**, which means GC pressure. In a hot loop boxing can dominate CPU time long before the allocations themselves become visible in memory profilers — they show up as gen-0 GC count.

**Common silent boxing sources** (the things you actually have to watch for):

- Non-generic collections / APIs: `ArrayList`, `Hashtable`, `IEnumerable`, `IComparable.CompareTo(object)`.
- `string.Format`, `$"{value}"`, `Console.WriteLine(value)` for value types **before** `ISpanFormattable` / `IUtf8SpanFormattable` were implemented on the BCL primitives. Modern .NET (8/9/10) avoids the box for most primitives via `DefaultInterpolatedStringHandler` and span-formatting paths.
- `==` between `object` and a value type, or `Equals(object)` on a struct that has not overridden it (the inherited `ValueType.Equals` reflects fields and boxes both sides).
- Calling an inherited `object` method on a struct via an interface reference rather than via a constrained generic.
- `Enum.HasFlag` historically boxed both the flag and the enum value — fixed in CoreCLR by a JIT intrinsic, but worth knowing if you target older runtimes.

**Avoidance patterns:**

```csharp
// Use generics + constraints — the JIT specializes per value type, no box
T Max<T>(T a, T b) where T : IComparable<T> => a.CompareTo(b) >= 0 ? a : b;

// Span<T> / ReadOnlySpan<T> — value-type sequences without IEnumerable boxing
int Sum(ReadOnlySpan<int> xs) { int s = 0; foreach (var x in xs) s += x; return s; }

// Span-based formatting paths — no box, no intermediate string
Span<char> buf = stackalloc char[32];
int.TryFormat(value, out var written, format: default, provider: null);
```

**Senior-level gotchas:**

- `Nullable<T>` boxing is special: `((object)nullable)` boxes the **underlying value** if `HasValue`, otherwise it boxes to a `null` reference — *not* to a boxed `Nullable<T>`. Many devs assume the wrapper survives.
- Every interface call on a struct via an interface reference boxes. Use `where T : IFoo, struct` constrained generics so the JIT devirtualizes and no box happens.
- `dynamic` boxes value-type operands aggressively; the DLR call site holds the box for caching too.
- Generic math (.NET 7+, `INumber<T>`, `IAdditionOperators<T,T,T>`) was specifically introduced to eliminate boxing in numeric abstractions that previously had to go via `IComparable` / `IConvertible`.

## Q9. What is heap and stack?

Two distinct memory regions used by .NET to hold data with very different lifetime / cost characteristics.

**Stack (per-thread call stack):**

- LIFO frame structure managed by the OS / CPU. Allocation is just "subtract from `rsp`", so it is essentially free — no GC involvement.
- Holds **method frames** (return address, saved registers), **value-type locals**, **method parameters**, and **references** (the pointer itself; the object lives elsewhere).
- Default size is small — ~1 MB on Windows, ~8 MB on Linux. Deep recursion → `StackOverflowException`, which is **unrecoverable** (the CLR tears down the process; you cannot catch it generically).
- `stackalloc` lets you carve a `Span<T>` out of the stack frame — bounded, free, no GC, but escapes the frame at your peril (`ref struct` and span lifetime rules enforce this).

**Managed heap (process-wide, GC-tracked):**

- All **reference-type instances** live here. So do **boxed value types**, captured locals in closures, async state machines (when they cannot stay inline), and arrays.
- Logically partitioned by the GC into **Small Object Heap** (SOH — gen 0, gen 1, gen 2), **Large Object Heap** (LOH — objects ≥ 85 000 bytes), and the **Pinned Object Heap** (POH — added in .NET 5, lets you pin without fragmenting SOH).
- Allocation in gen 0 is a bump-pointer increment — also nearly free in the fast path. The cost is **collection**, not allocation: gen 0 is collected often and quickly; gen 2 is collected rarely but stops more (or all) threads depending on GC mode.
- Two GC flavors: **Workstation** (single GC thread, lower latency, default for client) and **Server** (one GC heap per logical CPU, higher throughput, default for server). **Background GC** runs gen-2 work concurrently with the application threads in both modes.

**Where does a value type *actually* live?** The "value types go on the stack" mantra is false in general. A value type lives wherever its containing storage location lives:

| Where it lives | When                                                                           |
| -------------- | ------------------------------------------------------------------------------ |
| Stack          | Local variable / method parameter that does not get captured or boxed.         |
| Heap           | Field of a class; element of an array; boxed; captured by lambda; field of an `async`/`iterator` state machine. |
| Register       | The JIT may keep small values entirely in CPU registers, never touching memory. |

```csharp
struct Point { public int X, Y; }

class Owner { public Point P; }     // P lives on the heap, inside the Owner instance

void M()
{
    Point local = new(1, 2);        // stack (or register)
    object boxed = local;           // box on the heap; 'local' is copied
    Point[] arr = new Point[10];    // arr is a heap object; elements are value-type slots inside it
}
```

**`ref struct` (`Span<T>`, `ReadOnlySpan<T>`, `Utf8JsonReader`):** the compiler enforces stack-only lifetime. They cannot be a field of a class, cannot be captured by a lambda, cannot be a generic type argument (until C# 13's `allows ref struct`), cannot cross an `await`. This is how `Span<T>` can be a "view" over stack, heap, or unmanaged memory without any boxing or allocation.

**Modern note (.NET 10 / C# 14):** **DATAS (Dynamic Adaptation To Application Sizes)** is the **default GC mode in .NET 10** (it shipped on by default for SDK projects in .NET 9 and was broadened in .NET 10). DATAS sets the gen-0 budget based on the long-lived data size and observed throughput, so the heap shrinks aggressively under light load and expands under heavy load. The trade-off: highly throughput-sensitive workloads with bursty allocation can see worse p99 latency, and you can opt out via `DOTNET_GCDynamicAdaptationMode=0`, the `System.GC.DynamicAdaptationMode` runtimeconfig setting, or the `GarbageCollectionAdaptationMode` MSBuild property. Combined with the **regions** allocator that replaced segments and **NativeAOT** for ahead-of-time scenarios, the GC story in .NET 10 is meaningfully different from the .NET Framework / .NET Core 3 model interview material was written against.

**Senior-level gotchas:**

- `async` methods lift any local that crosses an `await` into a heap-allocated state machine — even `int` counters. This is why "value types are free" reasoning collapses in async code.
- A captured local in a lambda or local function is hoisted to a **closure class** on the heap — same effect.
- Large objects on LOH default to gen 2 and are collected only on full GC. Bursty large-array allocation is a classic source of "memory grows, gen 2 hits, p99 spikes".
- `stackalloc` inside a loop is fine — the stack pointer is reset per iteration — but `stackalloc` of unbounded user-controlled size is a stack-overflow vulnerability. Always cap and fall back to `ArrayPool<T>` past the threshold.
- The CoreCLR JIT does **not** do classical escape analysis to promote ref-types to the stack. Don't write code expecting "the JIT will figure out my object doesn't escape" — it won't.

## Q10. Difference between `string` and `StringBuilder`

`string` is **immutable**. Every operation that appears to modify a string (`+`, `Replace`, `Substring`, `ToUpper`) actually allocates a new `string` instance and copies characters. That makes strings safe to share across threads and cheap to compare for equality (string interning lets the runtime sometimes pointer-compare), but expensive to mutate iteratively.

`StringBuilder` is a **mutable** `char` buffer that exposes append / insert / replace operations against an internally growing array (chunked linked-list of arrays, actually — appending past `Capacity` allocates a new chunk rather than copying the whole buffer, since .NET 4).

The textbook trap:

```csharp
string result = "";
for (int i = 0; i < 10_000; i++)
    result += i;          // O(n²) total: each += allocates a fresh string and copies the prior one
```

vs

```csharp
var sb = new StringBuilder(capacity: 64_000);
for (int i = 0; i < 10_000; i++)
    sb.Append(i);         // O(n) total
string result = sb.ToString();
```

**When you do *not* need `StringBuilder`:**

- Constant folding: `"a" + "b" + "c"` is a single literal at compile time — zero cost.
- A **fixed**, small number of concatenations (≤ ~4 fragments): `string.Concat` / `+` is faster than spinning up a builder.
- `string.Join(separator, items)` is already optimal for joining a known sequence — uses a single buffer pass.
- **Interpolated strings** in C# 10+ go through `DefaultInterpolatedStringHandler`, which uses an `ArrayPool<char>`-backed buffer and span formatting under the hood. `$"User {id} at {time:O}"` is no longer the boxing/`ToString` disaster it was in older runtimes.

**When `StringBuilder` is the right answer:**

- Loop with **unknown or large** number of appends.
- Many incremental `Append`/`Insert`/`Replace` operations on the same logical buffer.
- Building output that exceeds a stackalloc-friendly size and where you want to avoid pooling complexity.

**Modern alternatives a senior should reach for first:**

```csharp
// 1. string.Create — preallocate the exact final length and write into a Span<char>
string Csv(int[] xs) => string.Create(xs.Length * 2, xs, (span, source) =>
{
    for (int i = 0; i < source.Length; i++) { span[i*2] = (char)('0'+source[i]); span[i*2+1] = ','; }
});

// 2. CompositeFormat (.NET 8+) — parse the format string once, reuse forever.
//    Eliminates repeated parsing of the same template in hot paths.
private static readonly CompositeFormat Template =
    CompositeFormat.Parse("User {0} at {1:O}");

string Render(int id, DateTime t) => string.Format(null, Template, id, t);

// 3. ArrayPool<char> + Span<char> for full-control, allocation-free formatting
//    when you can write directly to a destination buffer (logging, serialization).
```

**Senior-level gotchas:**

- `StringBuilder` is **not thread-safe**. One builder per thread, or external locking.
- Always pass a sensible `capacity` if you can estimate the final size — growth means new chunk allocations and (on `ToString`) a final copy.
- `sb.ToString()` allocates a fresh `string` and copies. If the consumer accepts `ReadOnlySpan<char>`, prefer writing to a span directly and skip the materialization.
- For very small outputs, the `StringBuilder` instance itself (~50 bytes) and its initial char array can outweigh a couple of `+` allocations. Profile, don't assume.
- C# 10+ interpolated string handlers are **lazy** — `LogInformation($"x={Expensive()}")` no longer pays for `Expensive()` if the log level is filtered out, *provided* the API uses an `[InterpolatedStringHandlerArgument]` parameter (Microsoft.Extensions.Logging does, since .NET 8).
- `string.Intern` exists and is occasionally useful for deduping high-cardinality repeated strings, but it pins the strings for the process lifetime — a memory-leak vector if used naively.
