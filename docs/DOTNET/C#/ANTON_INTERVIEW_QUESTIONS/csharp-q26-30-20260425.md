## Q26. Name the two forms of records

The two forms are:

1. **`record class`** — a reference type. The keyword `record` on its own is shorthand for `record class` (reference type is the default).
2. **`record struct`** — a value type. Added in C# 10 (.NET 6). Has a sub-flavor, **`readonly record struct`**, which makes the whole struct immutable.

That's it. There is no third form in .NET 10 / C# 14 — the language has not added a new record kind since C# 10.

```csharp
public record        Person(string Name, int Age);          // record class
public record class  Customer(Guid Id, string Email);       // explicit class form
public record struct Point(double X, double Y);             // mutable value record
public readonly record struct Money(decimal Amount, string Currency); // immutable value record
```

### What both forms give you for free

- **Value-based equality**: synthesized `Equals`, `GetHashCode`, `==`, `!=` over the public/instance state. Two records are equal iff every member is equal.
- **`ToString()` synthesis**: prints `TypeName { Prop1 = ..., Prop2 = ... }` — handy for logs.
- **`Deconstruct`** for positional records, so you can pattern-match: `if (p is (var x, var y))`.
- **`with`-expression support** (see Q27) for non-destructive copies.
- **Positional syntax** = primary constructor + properties (init-only on `record class`/`readonly record struct`, plain settable on `record struct`).

### Where the two forms diverge — beyond reference vs value

| | `record class` | `record struct` |
|---|---|---|
| Inheritance | Yes — supports a base record class, has a synthesized virtual `EqualityContract` | No (structs can't inherit) |
| Default mutability | Properties are **init-only** | Properties have **public set** by default; mark `readonly record struct` to lock them |
| Copy cost | Reference copy (cheap) | Full bitwise copy on every parameter pass / `with` |
| Identity in collections | Stable hash if members are stable | Mutable `record struct` is a hash-table foot-gun — its hash changes if you mutate it |
| `with` produces | A new heap allocation via the synthesized `Clone` virtual | A new struct value, no allocation |

### How a senior actually picks between them

- **DTOs, messages, events, API models** → `record class`. Heap allocation is a non-issue, value equality is gold for tests and dedup, and you usually want reference semantics anyway.
- **Small, immutable value-like types** (money, coordinates, identifiers, range, color) → `readonly record struct`. Stack-allocated, no GC pressure in hot loops, value equality matches the domain meaning.
- **Mutable `record struct`**: avoid unless you have a measured reason. Defensive copies + value semantics + mutation is the trifecta of "where did my change go?" bugs.
- **Never use a non-`readonly` record struct as a `Dictionary<,>` or `HashSet<>` key.** Mutating a member changes the hash and you lose the entry.
- **Big record structs are an anti-pattern.** Past ~16 bytes the BCL itself starts using `in` and ref returns to avoid copies; you're paying for value semantics you didn't want. If your `record struct` has more than three or four fields, it probably wants to be a `record class`.

### Senior-level gotchas

- `record` and `record class` are the *same thing*. Some teams require the explicit `record class` for symmetry with `record struct` — that is style, not semantics.
- The synthesized `Equals` on a `record class` includes a runtime type check (`other.EqualityContract == this.EqualityContract`). A `Person` and a `Customer : Person` with identical state are **not** equal. People expect this for entities, but it surprises people who expected pure structural equality.
- `record class` `Equals` walks **all instance fields**, including inherited ones. If you add a mutable list property, two records with logically-equal-but-different-instance lists won't be equal — `Equals` calls `EqualityComparer<T>.Default` which is reference equality on `List<T>`.
- `readonly record struct` is `readonly` end-to-end — every member is treated as `readonly`, which means calling a non-`readonly` method on a stored field forces a defensive copy. If a `readonly record struct` wraps another struct, watch perf.
- In `record struct`, the synthesized `GetHashCode` and `Equals` use *all* fields — including private ones if any are added via partial declarations. There's no opt-out except writing them yourself.

**Modern note (.NET 10 / C# 14):** Records themselves are unchanged in C# 14. The two language additions worth knowing in *adjacent* territory: the new `field` contextual keyword for auto-properties (not specific to records but it does interact — you can now express a property with custom logic that still uses an auto-generated backing field, which used to require giving up records' positional sugar), and partial constructors / events being allowed across partial type declarations (relevant for record + source generator pairings).

## Q27. What is `with` keyword used for?

The `with` expression performs **non-destructive mutation**: it produces a new instance copied from an existing one, with the listed members overwritten. The original is untouched.

```csharp
public record Person(string Name, int Age, string City);

var alice = new Person("Alice", 30, "Montreal");
var older = alice with { Age = 31 };
// alice.Age == 30
// older.Age == 31
// older.Name == "Alice", older.City == "Montreal"
```

It's the canonical way to "modify" an immutable type — and the only way to "change" an `init`-only property after construction.

### Where `with` works

| Type | `with` supported? |
|---|---|
| `record class` (incl. `record`) | Yes — calls a synthesized `protected virtual TypeName <Clone>$()` |
| `record struct` / `readonly record struct` | Yes — emitted as a struct copy + member overwrite |
| Plain `struct` | **Yes since C# 10** — no `record` keyword needed |
| Anonymous types | Yes — `new { x.Name, Age = 99 }`-style scenarios use `with` since C# 10 |
| Plain `class` (non-record) | **No.** This is the giveaway test in interviews. |

```csharp
// Plain struct since C# 10:
public struct Coord(int X, int Y);
var origin = new Coord { X = 0, Y = 0 };
var right  = origin with { X = 5 };

// Record struct copy is a new value, not shared state:
public record struct Counter(int N);
var a = new Counter(1);
var b = a with { N = 2 };  // a still 1
```

### How a senior thinks about it

- **`with` is a shallow copy.** A `record class Order(int Id, List<Item> Items)` cloned with `with { Id = 99 }` shares the *same `List<Item>` reference* as the original. Mutating `Items` on the clone mutates the source too. If you need deep semantics, the source members must themselves be immutable (use immutable collections, other records).
- **Polymorphism is preserved on `record class`.** The synthesized `Clone` is virtual and overridden in derived records, so `(Person)customer with { Name = "X" }` returns a `Customer`, not a sliced `Person`. This is *the* reason `EqualityContract` exists — it's how the compiler tracks the runtime type for cloning and equality.
- **`init`-only setters honor `with`.** That's literally why `init` exists — you can't `obj.Name = "X"` on a record, but `obj with { Name = "X" }` is the language-blessed equivalent.
- **`record with { }` (empty change set) is equal to the original** under value equality but not reference equal. Useful invariant for tests.
- **Pattern + `with`** is the modeling sweet spot for state machines: `state = state switch { Initial => new Loading(...), Loading l => l with { Progress = p }, ... };` — pure, allocation-aware, and exhaustive.

### Senior-level gotchas

- **Cost on big records**: every `with` allocates a new instance and copies all fields. In hot loops, profile. For a 6-field `record class` it's nothing; for a 50-field one it's a measurable cost.
- **`with` does not call your normal constructor**. It calls the synthesized clone path, then assigns the listed properties. So validation logic placed in your primary constructor body **does not re-run** on the cloned instance for the *unchanged* members, but it *does* run on the assignment of the changed property if you put validation in the property setter (and you can, on `record class` — set `init { ... }` body). If you put validation only in the constructor, `with` is a back door around it.
- **`with` on a struct returns a new struct** — assigning it to the same variable is fine, but if you're working through a property or array element you need to be aware you're rewriting the whole value: `arr[0] = arr[0] with { N = 1 };`.
- The `Clone` method is named `<Clone>$` in IL — you can't call it directly in C# source, and reflection/serialization code that walks methods may stumble on the angle-bracket name.
- `with` does **not** work on a plain `class` even if it has init-only properties. If an interviewer's "trick" version of this question asks about `class`, the answer is "no — convert it to a `record class` or copy by hand."

## Q28. What is the purpose of Primary Constructors?

Primary constructors put a parameter list directly on the type declaration. Those parameters are **in scope for the entire body** of the type — field initializers, property initializers, method bodies, base-call argument lists.

There are **two flavors** with very different semantics:

```csharp
// 1) Record primary constructor — C# 9.
//    Each parameter becomes a public init-only property.
public record Person(string Name, int Age);
// Auto-generated: public string Name { get; init; }
//                 public int    Age  { get; init; }
//                 Deconstruct, Equals, ToString, with-support

// 2) Class / struct primary constructor — C# 12.
//    Parameters are NOT properties. They're in-scope values.
//    The compiler synthesizes private capture fields only when needed
//    (i.e., when a parameter is referenced from a method body).
public class OrderService(IOrderRepo repo, ILogger<OrderService> log)
{
    public async Task<Order> GetAsync(int id, CancellationToken ct)
    {
        log.LogInformation("Loading order {Id}", id);
        return await repo.GetAsync(id, ct);
    }
}
```

The class form is the big ergonomic win for DI-heavy ASP.NET Core code: it kills the constructor + readonly field + assignment boilerplate that dominated services.

### What the two flavors share

- Parameters are visible everywhere in the type body.
- Base call uses the same parameter list: `class Derived(int x, string y) : Base(x, y)`.
- Cannot have multiple primary constructors. You can declare *additional* secondary constructors, but they must chain to the primary via `: this(...)`.

### Where they diverge — and this is where the bugs live

| | Record primary ctor | Non-record class/struct primary ctor |
|---|---|---|
| What parameters become | Public `init`-only properties | Captured into private synthesized fields (only when used in a method body), readonly *only if the compiler can prove it* — they are **not** explicitly readonly |
| Re-used in the body | As properties (`this.Name`) | As parameters (just `name`) |
| Equality / `with` impact | Full record machinery generated from them | None |
| Default value of the parameter | Becomes the property's default | Just a default for the parameter |
| Base call | Goes on the `: BaseRecord(...)` clause | Same: `: Base(...)` |

### How a senior thinks about it

- **The class-form primary constructor parameter is *not* a field.** Reading `repo` in a method body looks like a parameter access, but the compiler synthesizes a private field behind the scenes. Each method that references the parameter generates that capture. This is convenient but easy to misread.
- **No implicit `readonly`.** A primary constructor parameter is a *parameter* — methods inside the type can mutate the captured value: `private void Reset() { repo = new InMemoryRepo(); }` actually compiles. This is the single biggest "footgun" of class primary ctors. If you want immutability, assign to an explicit `private readonly` field:

```csharp
public class OrderService(IOrderRepo repo)
{
    private readonly IOrderRepo _repo = repo; // forces readonly capture
}
```

  Yes, this re-introduces a field declaration — but it's a deliberate "I want this to never change" annotation. Many style guides require it.

- **Validation has no clean home.** A `record Person(string Name, int Age)` with `Name` required-non-empty has no obvious place for the check. Two practical patterns:
  ```csharp
  // (a) Throw inside a property's init body via an explicit body:
  public record Person(string Name, int Age)
  {
      public string Name { get; init; } = Name ?? throw new ArgumentNullException(nameof(Name));
  }
  // (b) Use the C# 11 `required` keyword + a parameterless ctor + factory.
  ```
  In a class primary ctor, validate in a field initializer:
  ```csharp
  public class OrderService(IOrderRepo repo, ILogger<OrderService> log)
  {
      private readonly IOrderRepo _repo = repo ?? throw new ArgumentNullException(nameof(repo));
      private readonly ILogger    _log  = log  ?? throw new ArgumentNullException(nameof(log));
  }
  ```
  Or simpler with .NET 6+: `ArgumentNullException.ThrowIfNull(repo);` as the first statement of any method that uses it — but a senior usually prefers fail-fast at construction.

- **Attributes**: parameter-targeted attributes work via `[param: ...]`. Some attribute scenarios (e.g., `[FromServices]` on a Minimal API handler vs. on a primary ctor parameter) still need an explicit constructor; check generator support per framework.

- **Lambdas and capture**: a lambda inside a method that closes over the primary ctor parameter captures the *parameter*, which becomes a field capture — same as any other field reference. Don't expect parameter-style "fresh value per call"; the field is the type's, not the call's.

- **`record struct` parameters are mutable by default.** `record struct Point(int X, int Y)` lets you do `p.X = 5;`. Mark `readonly record struct` to forbid that — and you almost always want `readonly`.

### Analyzer signals worth turning on

- `IDE0290` ("Use primary constructor") — Roslyn's nudge to migrate boilerplate ctors.
- Primary-ctor parameter unused / captured-but-never-read warnings — useful for spotting dead DI.

### Senior-level gotchas

- A primary constructor parameter and a field *cannot* have the same name. The compiler will refuse to compile, which is the correct outcome but trips people coming from "underscore field" conventions. Either don't introduce a duplicate field, or pick a different name (`_repo` vs `repo`).
- Parameter shadowing in methods: a method-local variable named the same as a primary ctor parameter shadows it; the synthesized field is still there but inaccessible by name. Use `this.fieldName` is **not** an option — primary ctor captures don't expose a `this`-prefixed accessor in source.
- **No partial primary constructors split across `partial` declarations** until C# 14. C# 14 added `partial` support for *constructors and events* (helpful for source generators that wanted to extend a primary-ctor type), but you still have only one primary ctor per type.
- Records and primary constructors look syntactically identical but the parameters mean fundamentally different things — properties vs values. Don't conflate them in design discussions.

**Modern note (.NET 10 / C# 14):** C# 14 adds **partial constructors and partial events**, which lets source generators define one half and the user define the other — relevant when a generator wants to inject behavior into a primary-ctor-bearing type. The new **`field` contextual keyword** is also worth pairing in your head with primary ctors: in cases where the auto-property ergonomics of records were what you wanted but you also needed custom set-logic, you can now write a normal class with a primary ctor and use `field` inside a property to get a compiler-generated backing field — no more hand-rolling backing fields just to add validation.

## Q29. Explain how Nullable Reference Types work

Nullable Reference Types (NRT) are an **opt-in static analysis** the C# compiler runs on top of the type system. They are **not a runtime feature** — at runtime, `string` and `string?` are both `System.String`. The CLR knows nothing about nullability.

What NRT actually does: the compiler reads your `?` annotations, walks the code's control flow, tracks each reference's "flow state" (`NotNull`, `MaybeNull`, `Null`), and emits warnings when a possibly-null reference is dereferenced or passed where non-null is required.

```csharp
#nullable enable
public string GetCity(Person? p)
{
    // CS8602: Dereference of a possibly null reference.
    return p.City;

    // After narrowing — no warning:
    if (p is null) throw new ArgumentNullException(nameof(p));
    return p.City;
}
```

### How you turn it on

- Project-wide: `<Nullable>enable</Nullable>` in the csproj. Default in `dotnet new` templates since .NET 6.
- Per-file: `#nullable enable` (or `disable`/`restore`/`enable warnings`/`enable annotations`).
- Four modes: `enable` (annotations + warnings), `disable` (neither), `warnings` (warnings on, but `?` is meaningless), `annotations` (you can use `?`, but the compiler is silent — useful for public-API marking during gradual rollout).

### Annotations

- `T?` on a reference type means "may be null." On a value type it remains the old `Nullable<T>` (these are language-level different, surface-level similar).
- Method/parameter/return/field/property declarations all carry annotations.
- Generic `T?`:
  - For `T : class` constrained generics, `T?` means `T` with maybe-null state.
  - For `T : struct` constrained generics, `T?` means `Nullable<T>`.
  - For *unconstrained* `T`, `T?` is allowed (since C# 9) and means "the default of T, with maybe-null flow state for reference types." `default(T)` for unconstrained `T` produces a value the compiler treats as maybe-null at the call site.

### Flow-state attributes — the senior's toolkit

These live in `System.Diagnostics.CodeAnalysis` and are how you *teach* the analyzer about your custom helpers.

```csharp
public static void ThrowIfNull([NotNull] object? arg, string name)
{
    if (arg is null) throw new ArgumentNullException(name);
}
// Caller: ThrowIfNull(x, nameof(x));  // afterward, x is treated as not-null

public static bool TryGet([NotNullWhen(true)] out string? value) { ... }
// Caller: if (TryGet(out var v)) { v.Length; }  // v narrowed to not-null

[MemberNotNull(nameof(_log))]
private void Initialize() { _log = new Logger(); }

[MaybeNull] public T Find() => default;       // returns "T but may be null"
[NotNull]   public string Name { get; set; }  // setter accepts null but warns;
                                              //   the analyzer treats reads as not-null
```

The full set worth knowing: `[NotNull]`, `[MaybeNull]`, `[NotNullWhen]`, `[MaybeNullWhen]`, `[NotNullIfNotNull]`, `[MemberNotNull]`, `[MemberNotNullWhen]`, `[DoesNotReturn]`, `[DoesNotReturnIf]`. Most fluent helpers (`ArgumentNullException.ThrowIfNull`, `Debug.Assert`) are annotated so the analyzer cooperates.

### How a senior actually uses NRT

- **Treat warnings as errors** in CI for new code: `<TreatWarningsAsErrors>true</TreatWarningsAsErrors>` plus `<WarningsAsErrors>nullable</WarningsAsErrors>` if you want only NRT-related warnings to block.
- **Don't sprinkle `!` to silence the compiler.** Each `!` is a "trust me" — every use should be justifiable. Common legitimate uses: post-`MemberNotNullWhen` checks, init patterns the compiler can't follow (e.g., `[Required]` properties hydrated by a serializer), `DataRow` / `Dictionary` indexer accesses you've just guarded.
- **Public API contract first.** Annotate public types and method signatures even if internal code is messy. A `string?` return type is a documented contract — telling consumers "this can be null" is more important than internal cleanliness.
- **The serializer / EF / reflection problem.** `System.Text.Json`, EF Core, AutoMapper — these set fields via reflection and don't honor NRT contracts. A non-nullable property on a deserialized DTO can absolutely be `null` at runtime. Mitigations:
  - C# 11 `required` keyword to force initializer / object-initializer assignment.
  - Constructor-based binding (records or `[JsonConstructor]`) so the compiler sees the contract.
  - `[Required]` data annotations + validation pipeline as a runtime backstop.
  - On EF Core entities, use `required` or null-forgiving init (`= null!;`) on non-nullable navigation properties hydrated by the change tracker.
- **The unconstrained-generic gotcha.** `class Cache<T> { private T? _value; public T Get() => _value!; }` — the `!` is required because the compiler can't know `T` isn't a reference type. Better: use two methods or constrain.
- **NRT does not do runtime null checks.** It is purely compile-time. If you need argument validation, write it: `ArgumentNullException.ThrowIfNull(x);` at the top of the method. (The proposed `!!` parameter null-check syntax was removed before C# 11 shipped — don't use it; it isn't there.)

### Senior-level gotchas

- **Events default to nullable** (`event EventHandler? Click;`) because before any subscriber attaches, the delegate field is `null`. The compile-time analyzer will warn on `Click(this, e)`; the idiomatic invocation is `Click?.Invoke(this, e)`.
- **`new string[10]` is `string[]`, not `string?[]`** — but every slot is `null`. The compiler types the array as non-nullable; reading and using a slot that hasn't been assigned is a runtime `NullReferenceException` waiting to happen. For arrays of reference types you control, declare `string?[]` and tighten when you fill them.
- **Constructor flow analysis** warns on uninitialized non-nullable fields (`CS8618`). Three idiomatic fixes: assign at declaration, mark the field nullable, or add `required` on the property (C# 11).
- **`Dictionary<TKey,TValue>`'s indexer is annotated as returning non-null** even though it throws on miss. That is a deliberate pragmatism — use `TryGetValue` for the safe path.
- **Library boundaries: "oblivious" types**. Reference assemblies compiled without NRT are oblivious — you get neither warning nor protection. Auditing third-party deps for this is part of any serious migration.
- **The `?` after a generic type is structural**, not a runtime tag. `List<string?>` and `List<string>` are the same type at runtime — interop and reflection cannot distinguish them.
- **Nullable analysis vs JIT optimizations**: don't expect any perf change from enabling NRT. It is a design-time tool.

**Modern note (.NET 10 / C# 14):** Nothing fundamental about NRT changed in C# 14. The relevant ecosystem move: the new C# 14 **null-conditional assignment** (`obj?.Prop = value`, `dict?[k] = v`) is now valid — the assignment is skipped (and the right-hand side not evaluated) if the receiver is null. This is a small but pleasant nullability ergonomics win that pairs with NRT-narrowed code paths.

## Q30. Do switch expressions have any return type limitations?

Yes — and the question is sharper than it sounds, because a switch *expression* doesn't have a "return type" in the function-signature sense. Its result type is **inferred** from its arms. The rules and limits are:

```csharp
// Each arm is "pattern => expression". The whole thing is a single expression.
string label = code switch
{
    1   => "one",
    2   => "two",
    _   => "other",
};
```

### The limitations a senior should be able to list

1. **All arms must converge to a single result type.** The compiler computes the *best common type* across every arm's expression (plus the discard arm). Implicit conversions are considered. If no single type satisfies all arms, you get **CS8506** ("No best type was found for the switch expression") and have to either (a) cast one arm or (b) put the expression in a target-typed context.

   ```csharp
   // Fails — best common type is ambiguous between Circle and Square:
   var shape = name switch
   {
       "circle" => new Circle(1),
       "square" => new Square(1),
       _        => throw new ArgumentException(),
   };

   // Fix 1: target-type the expression. Compiler unifies via IShape.
   IShape shape = name switch
   {
       "circle" => new Circle(1),
       "square" => new Square(1),
       _        => throw new ArgumentException(),
   };

   // Fix 2: cast one arm to nudge the inference.
   var shape = name switch
   {
       "circle" => (IShape)new Circle(1),
       "square" => new Square(1),
       _        => throw new ArgumentException(),
   };
   ```

2. **A switch expression cannot be `void`.** Arms must produce a value — including the "no useful result" arms, which use `throw`. If you want side effects without a value, use a switch *statement*, not a switch expression. (A switch *statement* can be void; a switch *expression* cannot.)

3. **`throw` arms don't constrain the type.** `throw` is "no value of any type" so the compiler ignores throw-arms when inferring the result type. This is why most exhaustive-by-throw patterns work — `_ => throw new UnreachableException()` doesn't force a type.

4. **`null` literal arms are typed by the target.** `null => null` is fine when the result type is a reference type or `Nullable<T>`. Mixing `null` and a non-nullable `int` is rejected unless the target type is `int?`.

5. **Exhaustiveness is a *warning*, not an error**, but it changes runtime behavior. If the compiler proves no arm matches some input, it warns **CS8509** ("not exhaustive") and inserts a throw of `SwitchExpressionException` for the un-matched case. With pattern matching over open class hierarchies this is the most common shipping bug — your "exhaustive" switch is missing a `null` arm or a derived-class case.

6. **`ref` returns are not allowed from switch expressions.** A switch expression is not a ref-returning expression. If you need `ref` semantics, you need a regular `if/else` or a method.

7. **Pattern restrictions inside `ref struct` / `Span<T>` worlds.** Patterns over `ref struct` types (e.g., `Span<T>`) work, but the result of a switch expression cannot be of a `ref struct` type assigned somewhere illegal — you stay inside the same scoping rules as any `ref struct` value.

8. **`var` arm "type"** infers from the matched expression, not the result type. Often newcomers think `_` and `var x` are interchangeable; they're not — `var x` always matches and binds, useful when you want to use the value in the arm.

### Target-typing — the rule that makes most "limitation" questions go away

Since C# 9, the switch expression participates in target typing. That means `IShape s = expr switch { ... };` can unify arms that have no common type by themselves, because the target `IShape` drives the conversion. **Use this** rather than peppering casts in each arm:

```csharp
public Shape Build(string kind) => kind switch
{
    "circle" => new Circle(1),
    "square" => new Square(1),
    _        => throw new ArgumentOutOfRangeException(nameof(kind)),
};
// Method's return type IShape is the target. No CS8506.
```

### How a senior thinks about switch expressions vs statements

- **Switch *expression* = mapping** (input → value). Pure, total, easy to test.
- **Switch *statement* = dispatch with side effects** (do something per case). When you find yourself writing `case X: DoX(); break;` use the statement form.
- **Sealed hierarchies + switch expressions** is the closest C# gets to discriminated unions today. Combine `sealed` (or `abstract` with sealed leaves) with pattern arms and you get compile-time exhaustiveness — no `_` needed if the compiler sees you covered every leaf. Open hierarchies always need a `_` and you accept runtime risk.
- **Don't forget `null`.** For reference types, the `_` discard does *not* match `null` in modern C# pattern semantics — you need an explicit `null =>` or `null or _ =>` arm. A senior gets this wrong once and never again.

```csharp
public string Describe(Animal? a) => a switch
{
    null         => "nothing",
    Dog d        => $"dog {d.Name}",
    Cat c        => $"cat {c.Name}",
    _            => "unknown animal",
};
```

### Senior-level gotchas

- `case` order matters. The compiler picks the first matching arm — so `Animal a => "animal"` placed before `Dog d => "dog"` will swallow every dog. Most-specific first.
- `not null` and `null` are subtle: a pattern like `{ } x` is "non-null instance with any properties" and is the closest C# pattern to "not null." Use it deliberately.
- Pattern arms that look exhaustive often aren't: `enum Color { R, G, B }` switch over `Color` *can* be hit with `(Color)99` — the CLR allows arbitrary integer values for enums. You always need either `_` or a runtime trust contract.
- The compiler's exhaustiveness analysis is best-effort. It improved noticeably in C# 11/12 with list patterns, but it can still miss "obviously" exhaustive cases when you mix recursive patterns; don't take its silence as proof.
- `SwitchExpressionException` (in `System.Runtime.CompilerServices`) is what an unmatched switch expression throws — *not* `InvalidOperationException` and *not* `ArgumentOutOfRangeException`. Catch it precisely if you have specific recovery; otherwise let it propagate.

**Modern note (.NET 10 / C# 14):** Switch expressions themselves are unchanged in C# 14. The adjacent C# 14 features that show up in the same code: **extension members** (you can define extension properties, which often appear inside switch arm projections) and **null-conditional assignment** (`thing?.Prop = expr`) which interacts cleanly with `null`-arm switches. Nothing in C# 14 changed exhaustiveness analysis or pattern types.
