## Q11. How to create a date with a specific timezone?

The first thing a senior should say: **`DateTime` does not carry a timezone.** It only has a `Kind` (`Utc`, `Local`, `Unspecified`). "Local" means *whatever the host machine is set to* — that is the single biggest production bug magnet in this area. If you want a date that is anchored to a real-world zone, you need either `DateTimeOffset` (carries a UTC offset, but **not** DST rules) or `DateTime` + `TimeZoneInfo` (carries DST rules).

### Three legitimate ways

```csharp
// 1) DateTimeOffset — the right primitive for "an instant, observed at this offset"
var inUtc      = new DateTimeOffset(2026, 4, 25, 14, 30, 0, TimeSpan.Zero);
var inMontreal = new DateTimeOffset(2026, 4, 25, 10, 30, 0, TimeSpan.FromHours(-4));
// Note: an offset is NOT a timezone. -04:00 in April is Montreal in DST,
// in November it would be -05:00. Don't store an offset and call it a zone.

// 2) DateTime + TimeZoneInfo — when you actually need the zone (DST rules)
//    .NET 6+ accepts both Windows ("Eastern Standard Time")
//    and IANA ("America/New_York") IDs cross-platform.
TimeZoneInfo nyc = TimeZoneInfo.FindSystemTimeZoneById("America/New_York");

DateTime localWall = new(2026, 4, 25, 10, 0, 0, DateTimeKind.Unspecified);
DateTime utc       = TimeZoneInfo.ConvertTimeToUtc(localWall, nyc);
DateTime backLocal = TimeZoneInfo.ConvertTimeFromUtc(utc, nyc);

// 3) Build a DateTimeOffset bound to a zone (handles DST correctly)
var offsetForThatInstant = nyc.GetUtcOffset(localWall);
var anchored = new DateTimeOffset(localWall, offsetForThatInstant);
```

### How a senior actually designs this

- **Store and transport UTC.** Persist `DateTime` as `Utc` (or better, `DateTimeOffset`) and only convert to a user's local zone at the **presentation boundary**.
- **Never trust `DateTime.Now` in server code.** It is bound to the machine's clock and zone — fine for a console toy, dangerous in containers, where the host TZ is usually UTC but not always.
- **Don't conflate offset with zone.** `DateTimeOffset` does not know about DST. If you need "the same wall-clock time next year in Montreal," you need `TimeZoneInfo`, not just an offset.
- **Use `TimeProvider` for testability.** Since .NET 8, inject `TimeProvider` instead of calling `DateTime.UtcNow`/`DateTimeOffset.UtcNow` directly. In tests use `FakeTimeProvider` (`Microsoft.Extensions.TimeProvider.Testing`). In production inject `TimeProvider.System`. `TimeProvider` exposes `LocalTimeZone`, `GetUtcNow()`, `GetLocalNow()`, and timer creation.
- **`DateOnly` / `TimeOnly`** (.NET 6+) for date-only or time-of-day values — kills a whole class of "midnight in which zone?" bugs.

### Senior-level gotchas

- Ambiguous and invalid times around DST transitions: `TimeZoneInfo.IsAmbiguousTime` / `IsInvalidTime`. `ConvertTimeToUtc` will *throw* on an invalid local time (the spring-forward gap).
- EF Core maps `DateTime` differently per provider; SQL Server's `datetime2` is timezone-naive — round-tripping a `Local` `DateTime` will silently lose the kind.
- JSON: `System.Text.Json` serializes `DateTime` with the original `Kind`. Mixing `Local` and `Utc` across services is a classic interop bug — pick one (UTC) and enforce it at the boundary.
- Background services scheduled by clock time (e.g., "every day at 2 AM Eastern") must use `TimeZoneInfo` + recompute next-fire each iteration; don't precompute a `DateTime` once and add 24 hours forever.

**Modern note (.NET 10 / C# 14):** `TimeProvider` is now the idiomatic time abstraction across BCL libraries (e.g., `PeriodicTimer`, `Task.Delay`, `CancellationTokenSource` all accept a `TimeProvider`). Cross-platform IANA timezone resolution that started in .NET 6 is fully baked — you can use IANA IDs on Windows and Windows IDs on Linux without manual mapping.

## Q12. How to change current culture?

Two cultures live on every thread:

- **`CultureInfo.CurrentCulture`** — formatting (numbers, dates, currency, sort order).
- **`CultureInfo.CurrentUICulture`** — resource lookup (which `.resx` is loaded).

They are independent. A French user might run a French UI but expect ISO formatting — set them separately.

### Four ways to set them

```csharp
// 1) Per-thread, current value (writable since .NET Core).
CultureInfo.CurrentCulture   = CultureInfo.GetCultureInfo("fr-CA");
CultureInfo.CurrentUICulture = CultureInfo.GetCultureInfo("fr-CA");

// 2) Process-wide default for any thread that has not set its own.
CultureInfo.DefaultThreadCurrentCulture   = CultureInfo.GetCultureInfo("fr-CA");
CultureInfo.DefaultThreadCurrentUICulture = CultureInfo.GetCultureInfo("fr-CA");

// 3) Scoped swap with try/finally (or a using-disposable wrapper).
var previous = CultureInfo.CurrentCulture;
try
{
    CultureInfo.CurrentCulture = CultureInfo.InvariantCulture;
    // ... formatting-sensitive code (parsing config, serializing decimals) ...
}
finally
{
    CultureInfo.CurrentCulture = previous;
}

// 4) Per-call: prefer this when you only need it for one operation.
var s = 1234.56m.ToString("N2", CultureInfo.InvariantCulture);
DateTime.TryParseExact(input, "yyyy-MM-dd",
    CultureInfo.InvariantCulture, DateTimeStyles.None, out var d);
```

### How a senior thinks about this

- **Prefer `IFormatProvider` per call** to mutating thread state. Every BCL formatting/parsing API has an overload that takes a `CultureInfo`. It's allocation-free for cached cultures (`GetCultureInfo` returns interned instances) and avoids leaking culture into unrelated code.
- **Use `CultureInfo.InvariantCulture` for storage and protocol boundaries.** Configuration files, SQL parameters, JSON, log lines — never the user's culture. The `tr-TR` "Turkish I" bug (`"i".ToUpper() != "I"`) has shipped in real codebases; analyzers `CA1304` (specify culture), `CA1305` (specify `IFormatProvider`), `CA1310` (string comparison) catch this — leave them on.
- **`StringComparison.Ordinal`** for any code-path comparison (config keys, file paths, identifiers). Culture-sensitive comparison only at the user-facing boundary.
- **Async flow**: in modern .NET, `CurrentCulture` is backed by `AsyncLocal<T>`, so it flows across `await`. This was *not* true on .NET Framework 4.5 — there you had to set `DefaultThreadCurrentCulture`. If you read old StackOverflow answers, that is why.
- **ASP.NET Core**: don't set culture manually per request. Use `app.UseRequestLocalization(...)` with `IRequestCultureProvider`s (query string, cookie, `Accept-Language`). The middleware sets both cultures for the request scope and reverts on the way out.
- **Blazor WebAssembly**: each user's browser tab has its own culture; set it in `Program.cs` via `CultureInfo.DefaultThreadCurrentCulture` based on `NavigatorOptions` or persisted user preference.

### Senior-level gotchas

- `new CultureInfo("fr-CA")` allocates; `CultureInfo.GetCultureInfo("fr-CA")` returns a cached read-only instance. Use the latter in hot paths.
- The thread-pool may rent threads that already have a culture from a prior workload. Setting `CurrentCulture` mutates the borrowed thread; on .NET Framework this leaked between unrelated requests. On modern .NET, the `ExecutionContext`/`AsyncLocal` model insulates this — but only if you don't bypass it (e.g., raw `Thread.Start`, custom `TaskScheduler`).
- Logs and telemetry should always be `InvariantCulture`. Localized log timestamps are unsearchable.
- `decimal.Parse("1,5")` succeeds in `fr-CA` and fails in `en-US`. Always pass the culture explicitly when parsing data you didn't generate.

## Q13. What is the difference between `HashSet<T>` and `Dictionary<TKey, TValue>`?

Both are hash-based, both are O(1) average lookup, both are **not** thread-safe. The difference is what they store.

| | `HashSet<T>` | `Dictionary<TKey, TValue>` |
|---|---|---|
| Stores | unique elements | unique keys, each mapped to a value |
| "Is X present?" | `Contains(x)` — O(1) | `ContainsKey(x)` — O(1) |
| "Give me X" | n/a (the element *is* what you have) | `dict[key]` / `TryGetValue` — O(1) |
| Equality based on | `IEqualityComparer<T>` for the element | `IEqualityComparer<TKey>` for the key only |
| Set operations | `UnionWith`, `IntersectWith`, `ExceptWith`, `SymmetricExceptWith`, `IsSubsetOf`, etc. | none |
| Memory per slot | hash + element + next-bucket pointer | hash + key + value + next-bucket pointer |

```csharp
var seen = new HashSet<string>(StringComparer.OrdinalIgnoreCase);
seen.Add("Alice");           // true
seen.Add("ALICE");           // false — duplicate under the comparer
bool has = seen.Contains("alice");  // true

var byId = new Dictionary<int, User>(capacity: 1024);
byId[42] = new User(42, "Alice");
if (byId.TryGetValue(42, out var u)) { /* ... */ }
```

### How a senior actually thinks about this

- **Pick by intent, not by feel.** `HashSet<T>` is for "is this element in the set?" and set algebra. `Dictionary<K,V>` is for "look up a value associated with a key." If you find yourself doing `dict.Keys.Contains(x)` you wanted a `HashSet`.
- **The comparer is the contract.** Both honor `IEqualityComparer<T>`. Use `StringComparer.Ordinal` (or `OrdinalIgnoreCase`) for string keys — `EqualityComparer<string>.Default` is culture-sensitive and slower in some scenarios.
- **Pre-size when you know the count.** Both expose a `capacity` constructor. Resizing a hash table is allocation- and rehash-heavy.
- **`GetHashCode` must be stable.** If you override `Equals` on a custom type used as a key/element, override `GetHashCode` consistently — and never derive it from a mutable field. Mutating a key after insertion is *the* classic "where did my entry go?" bug.
- **Thread safety**: neither is safe for concurrent writers, even one writer + many readers. For high concurrency use `ConcurrentDictionary<TKey, TValue>`. There is no `ConcurrentHashSet<T>` in the BCL — the idiomatic substitute is `ConcurrentDictionary<T, byte>` and ignore the value, or use `ImmutableHashSet<T>` for snapshot semantics.
- **Enumeration order is undefined.** Don't depend on insertion order (that's `OrderedDictionary` / `List`).
- **`FrozenDictionary`/`FrozenSet`** (.NET 8+): build-once, read-many, faster lookup, no mutation. Use for static config tables.

### Senior-level gotchas

- `Dictionary<,>.Add` throws on duplicate; `dict[key] = value` overwrites. `HashSet<>.Add` returns `bool` (no throw). Pick the right verb for the right intent — silently overwriting is a category of bug.
- Indexer `dict[key]` throws `KeyNotFoundException` on miss. Prefer `TryGetValue` or `GetValueOrDefault` in hot/uncertain paths.
- Iterating *and* mutating a Dictionary throws `InvalidOperationException`. Snapshot via `.ToArray()` or use `ConcurrentDictionary`.
- Reference-type keys without an overridden `Equals`/`GetHashCode` use reference equality — usually not what you wanted.

## Q14. What is the purpose of the method `ToLookup`?

`ToLookup` produces an **`ILookup<TKey, TElement>`** — effectively a read-only **multimap**: one key → many values, all materialized eagerly and indexed for fast repeated lookup.

```csharp
Product[] products = await db.Products.ToArrayAsync(ct);

ILookup<string, Product> byCategory = products.ToLookup(p => p.Category);

IEnumerable<Product> electronics = byCategory["Electronics"];   // many
IEnumerable<Product> missing     = byCategory["DoesNotExist"];  // empty, no throw

foreach (var group in byCategory)            // IGrouping<string, Product>
    Console.WriteLine($"{group.Key}: {group.Count()}");
```

### How a senior frames it vs `GroupBy`

- `GroupBy` returns `IEnumerable<IGrouping<TKey, TElement>>` and is **deferred**. You iterate it once; re-iterating re-runs the source. There is no key-based lookup.
- `ToLookup` is **eager**: it walks the source once, builds the index, and now you can hit `lookup[key]` repeatedly in O(1).
- Use `ToLookup` when you have a one-to-many relationship to query repeatedly. Use `GroupBy` when you only want to project/aggregate each group once and stream the results.

A common pattern is replacing N+1 queries in memory: pull a parent list and a child list separately, build a `ToLookup` on the foreign key, then attach children in O(N).

```csharp
var ordersByUser = orders.ToLookup(o => o.UserId);
foreach (var u in users)
    u.Orders = ordersByUser[u.Id].ToList();
```

### Senior-level gotchas

- The indexer never throws — missing keys return an empty sequence. That's a feature (no need for `ContainsKey`), but it can mask a bug if you assumed the key exists.
- `ILookup` is **read-only** — there is no public mutator. If you need to append, you're using the wrong type; reach for `Dictionary<TKey, List<TElement>>` or `ILookup` rebuild.
- The lookup holds references to the source elements. If the source is huge and you only need a key→count, use `GroupBy(...).ToDictionary(g => g.Key, g => g.Count())` instead — same eager cost, less memory.
- Like all hash-keyed structures, the `IEqualityComparer<TKey>` matters; `ToLookup` has overloads to pass one. Default culture-sensitive string comparison is rarely what you want for IDs.

## Q15. Does LINQ `Cast<T>` method create a new object?

**Short answer: No — `Cast<T>` does not copy your collection or your reference-type elements.** It is a deferred iterator that, when enumerated, yields each element cast to `T`. For value types it does perform an unboxing copy of each element (which is unavoidable — that's what unboxing *is*).

```csharp
// Conceptually (close to the BCL implementation):
public static IEnumerable<T> Cast<T>(this IEnumerable source)
{
    if (source is IEnumerable<T> already) return already;   // fast path
    return CastIterator<T>(source);
}

static IEnumerable<T> CastIterator<T>(IEnumerable source)
{
    foreach (object obj in source)
        yield return (T)obj;     // throws InvalidCastException on mismatch
}
```

So what you get back is a new `IEnumerable<T>` *wrapper* (the iterator state machine — that single allocation), but no new collection and no new reference-type elements. Each enumeration walks the original source.

### What "new object" actually means here

- **Reference types**: no new object per element. The cast is identity-preserving — `obj.ReferenceEquals(original, casted)` is true.
- **Value types**: `Cast<int>` over an `IEnumerable` of *boxed* ints unboxes each one — that *is* a copy of the value. The boxed object on the heap continues to live until GC; you don't get a new boxed copy, you get the unboxed value.
- **The wrapper itself**: yes, one iterator object is allocated to track enumeration state. Cheap, but not literally zero.
- **Empty/no-op case**: if `source` already implements `IEnumerable<T>`, BCL short-circuits and returns the source directly — zero new allocation.

### `Cast<T>` vs `OfType<T>`

```csharp
object[] mixed = ["a", 1, "b", 2];

mixed.Cast<string>().ToList();    // throws InvalidCastException at the int
mixed.OfType<string>().ToList();  // ["a", "b"] — filters silently
```

`OfType<T>` skips elements that aren't `T`; `Cast<T>` throws. Pick by intent: "I assert all are T" → `Cast`. "I want the ones that are T" → `OfType`.

### Where you'd use `Cast<T>` in real code

The original purpose was bridging non-generic `IEnumerable` (think `XmlNodeList`, `DataRowCollection`, `ControlCollection`) into the generic LINQ surface. In modern code that is rare; most APIs return `IEnumerable<T>` already. If you find yourself reaching for `Cast<T>` over a generic source, you usually meant `Select(x => (T)x)` or a pattern match.

### Senior-level gotchas

- **Errors are deferred.** `xs.Cast<T>()` does not validate anything — the `InvalidCastException` lands at the call site that *enumerates* (`.ToList()`, `foreach`), often far from the source of the bug.
- **No null-tolerance for value types.** Casting a boxed `null` to `int` throws `NullReferenceException`. To `int?` it does not.
- **Custom conversion operators are ignored.** `Cast<T>` does an *unboxing* / reference-conversion cast (IL `castclass`/`unbox.any`), not a user-defined conversion. `Cast<MyType>` over a sequence of `string` will not call your `implicit operator MyType(string)`. Use `Select(s => (MyType)s)` for that — the C# compiler emits the user-defined conversion call there.
- **EF Core / `IQueryable<T>`**: `Cast<T>` on a queryable becomes part of the expression tree. The provider must translate it; many providers don't support arbitrary casts to SQL. Materialize first (`AsEnumerable()`) or use a typed projection.
