## Q16. Explain deferred execution in LINQ

**Deferred execution** means a LINQ query expression is *defined* when you write it but *not executed* until something actually enumerates it. The query is a recipe, not a result. Most LINQ-to-Objects operators (`Where`, `Select`, `Take`, `OrderBy`, `GroupBy`, …) return an `IEnumerable<T>` whose `GetEnumerator()` walks the source on demand via the iterator pattern (`MoveNext` → `Current`). Nothing happens until you iterate — `foreach`, `ToList()`, `ToArray()`, `Count()`, `First()`, `Single()`, `Aggregate()`, etc. are the **terminal** operators that pull on the chain.

```csharp
var query = numbers.Where(n => { Console.WriteLine($"check {n}"); return n > 2; });
// nothing prints yet
foreach (var n in query) { /* prints "check 1", "check 2", "check 3", ... */ }
foreach (var n in query) { /* prints them ALL OVER AGAIN — re-enumerated */ }
```

Two flavors of deferred operators:

- **Streaming** (lazy *and* incremental): `Where`, `Select`, `Take`, `Skip`, `SelectMany`. They yield as they consume.
- **Non-streaming / buffered** (lazy but must drain the source first): `OrderBy`, `GroupBy`, `Reverse`, `Distinct` (partial), `Join`. Still deferred — they don't execute until enumerated — but on first `MoveNext()` they read the **entire** source to build their internal structure.

**`IQueryable<T>` extends the same idea to expression trees.** With EF Core, every composed operator builds an `Expression` rather than a delegate; the provider walks the tree, translates it to SQL, and only executes when you enumerate / call a terminal. That's why `db.Users.Where(u => u.Name == name).Select(u => u.Email).ToListAsync()` issues exactly one parameterized SQL statement instead of pulling all users into memory.

**Senior-level gotchas:**

- **Captured-variable trap.** Lambdas close over variables by *reference*. Mutating the captured variable after defining the query but before enumerating changes the query's behavior. Old C# `foreach`-loop variable capture used to bite hard — fixed in C# 5+ for `foreach` but the trap still exists for `for` loops and reused locals.
- **Double enumeration.** Treating an `IEnumerable<T>` as a result is dangerous — every consumer re-runs the pipeline. If the source is a database query, file read, or one-shot iterator, results may differ each time, fail, or hammer the DB. Materialize with `.ToList()` once when crossing a boundary.
- **`Count()` vs `Any()`.** `Count() > 0` enumerates everything; `Any()` short-circuits on the first item. On `IQueryable` both translate to optimal SQL, but on in-memory pipelines the difference is real.
- **Side effects in selectors.** Because re-enumeration is silent, putting side-effects inside `Select`/`Where` produces non-determinism that's painful to debug. Keep selectors pure.
- **`AsEnumerable()` / `AsQueryable()` as cut points.** Use `AsEnumerable()` to deliberately switch from server-side (LINQ-to-SQL) to client-side evaluation when the next operator can't be translated — but be aware you've just promoted the rest of the pipeline to in-process.

## Q17. How does the `ImmutableList` work?

`ImmutableList<T>` (in `System.Collections.Immutable`) is a **persistent data structure** backed by a self-balancing binary tree (an AVL tree). Every "mutating" operation — `Add`, `Insert`, `RemoveAt`, `SetItem` — returns a *new* `ImmutableList<T>` while the original is untouched and still safe to use from any thread. The new instance shares most of its internal tree nodes with the old one (**structural sharing**), so the per-operation allocation is O(log n) nodes rather than copying the whole sequence.

```csharp
var a = ImmutableList.Create(1, 2, 3);
var b = a.Add(4);          // a is still {1,2,3}; b is {1,2,3,4}; nodes shared
var c = b.SetItem(0, 99);  // c is {99,2,3,4}; still shares with b
```

Complexity contract — and the trap — is that `ImmutableList<T>` does **not** match `List<T>`'s big-O:

| Op            | `List<T>` | `ImmutableList<T>` | `ImmutableArray<T>` |
| ------------- | --------- | ------------------ | ------------------- |
| Indexer `[i]` | O(1)      | **O(log n)**       | O(1)                |
| `Add`         | amort O(1)| O(log n)           | **O(n)** (copy)     |
| `Insert(i,x)` | O(n)      | O(log n)           | O(n)                |
| Memory        | low       | high (tree nodes)  | low (flat array)    |

Use `ImmutableList<T>.Builder` (via `ToBuilder()`) when you need to perform many mutations in a row — it allows in-place edits and then `ToImmutable()` produces the final immutable instance once, avoiding intermediate allocations.

```csharp
var builder = ImmutableList.CreateBuilder<int>();
foreach (var x in source) builder.Add(x);
ImmutableList<int> result = builder.ToImmutable();
```

**Senior-level gotchas:**

- **Default to `ImmutableArray<T>`** for read-mostly small-to-medium collections. It's a struct wrapping a regular array — O(1) indexer, no GC overhead per item, ideal for enumeration. Reach for `ImmutableList<T>` only when you need cheap insert/remove in the middle.
- **Thread safety = immutability, not concurrency.** Concurrent readers are fine because the instance never changes. To "update" a shared field, use the static `ImmutableInterlocked.Update` helpers — they handle the CAS-and-retry loop for you.
- **For read-heavy lookups that never change after build, prefer `FrozenDictionary` / `FrozenSet`** (see Q18). For a key/value store that does change but stays thread-safe, prefer `ImmutableDictionary` or `ConcurrentDictionary` depending on whether you need snapshot semantics.
- **`ImmutableList<T>.WithComparer` doesn't exist** — that's a sequence, not a set. The `WithComparer`/`SetItem` style applies to `ImmutableHashSet<T>` and `ImmutableDictionary<TKey,TValue>`.
- The `[CollectionBuilder]` attribute on the immutable types means **collection expressions** (`ImmutableList<int> x = [1,2,3];`) compile to the optimal builder path.

## Q18. What are the benefits of using Frozen collections?

`FrozenDictionary<TKey,TValue>` and `FrozenSet<T>` (namespace `System.Collections.Frozen`, shipped in .NET 8 and refined in .NET 9 / 10) are **read-optimized, build-once collections**. They're created from existing data and then never modified — there are no `Add`/`Remove` methods at all. In exchange for that contract, the build process inspects the keys and chooses the fastest possible lookup strategy (perfect hashing for small sets, specialized layouts for `string` keys with case-sensitive vs case-insensitive comparers, length-bucketed string lookups, etc.).

```csharp
using System.Collections.Frozen;

private static readonly FrozenDictionary<string, int> StatusCodes =
    new Dictionary<string, int>
    {
        ["OK"] = 200, ["Created"] = 201, ["NotFound"] = 404
    }.ToFrozenDictionary(StringComparer.OrdinalIgnoreCase);

int code = StatusCodes["created"]; // fastest possible string-keyed lookup in BCL
```

Benefits over the alternatives:

- **Faster reads than `Dictionary<TKey,TValue>`.** Microbenchmarks show ~30–60% faster lookups for `FrozenDictionary` and ~15–20% faster contains for `FrozenSet`, depending on key type and size. `string` keys see the biggest wins because of dedicated specializations.
- **Faster reads than `ImmutableDictionary`.** `ImmutableDictionary` is tree-backed (O(log n)); `FrozenDictionary` is hash-table-backed and tuned at build time, so it crushes immutable on lookup throughput.
- **Read-only by contract**, so passing one around guarantees no caller can mutate it — a stronger API guarantee than handing out `IReadOnlyDictionary` over a mutable backing store.

Mental model: **`Frozen` is the right choice when the collection is built once at startup (or rarely) and read on the hot path forever after**. Classic uses: route tables, status-code maps, config lookups, allow-lists, tokenizer keyword sets.

**Senior-level gotchas:**

- **Build cost is real and non-trivial** — sometimes 5–10× the cost of building a regular `Dictionary` of the same size, because the constructor analyzes keys and picks a strategy. If you rebuild the collection frequently, you'll be net slower than a `Dictionary`. Frozen pays off when (build-cost) « N × (read-cost-savings).
- **Don't confuse with `ImmutableDictionary`.** Immutable supports "mutations" via `Add`/`SetItem` that return a new instance with structural sharing (persistent). Frozen has *no* mutation surface — to "change" it you rebuild from scratch. Different mental models for different workloads.
- **Comparer choice locks in optimizations.** Always pass `StringComparer.Ordinal` or `OrdinalIgnoreCase` for `string` keys unless you genuinely need culture-sensitive comparison; the frozen builder has hand-tuned paths for those two.
- `FrozenDictionary<TKey,TValue>` and `FrozenSet<T>` are abstract — you always go through `.ToFrozenDictionary()` / `.ToFrozenSet()` factories, never `new`.
- For very small collections (≤ ~10 items), the perf delta vs `Dictionary` shrinks; the readability/intent signal is then the main win.

## Q19. Name thread-safe collections

The BCL groups thread-safe collections into three families. Pick by *workload*, not by name.

**1. `System.Collections.Concurrent` — lock-striped / lock-free for high-concurrency mutation.**

| Type                                  | Best for                                              |
| ------------------------------------- | ----------------------------------------------------- |
| `ConcurrentDictionary<TKey,TValue>`   | Shared key/value cache with frequent reads + writes   |
| `ConcurrentQueue<T>`                  | FIFO producer/consumer, lock-free                     |
| `ConcurrentStack<T>`                  | LIFO, lock-free                                       |
| `ConcurrentBag<T>`                    | Unordered, optimized when same thread adds *and* takes (per-thread local storage) |
| `BlockingCollection<T>`               | Bounded blocking producer/consumer wrapper around any `IProducerConsumerCollection<T>` |

**2. Immutable collections — thread-safe by virtue of immutability.** `ImmutableList<T>`, `ImmutableArray<T>`, `ImmutableDictionary<TKey,TValue>`, `ImmutableHashSet<T>`, `ImmutableSortedDictionary<TKey,TValue>`, `ImmutableQueue<T>`, `ImmutableStack<T>`. The instance never changes, so concurrent readers need zero synchronization. To publish updates atomically use `ImmutableInterlocked.Update`/`AddOrUpdate` against a shared field.

**3. Frozen collections — read-only after build (Q18).** `FrozenDictionary<TKey,TValue>`, `FrozenSet<T>`. Trivially thread-safe because no mutation API exists.

**Honourable mention: `Channel<T>`** (`System.Threading.Channels`). Not a collection per se, but the modern, async-friendly producer/consumer primitive. If you reach for `BlockingCollection<T>` in new code, consider `Channel<T>` instead — it integrates cleanly with `await` and supports bounded back-pressure.

**Senior-level gotchas:**

- **`ConcurrentDictionary.GetOrAdd(key, factory)` is *not* atomic** w.r.t. the factory. Under contention the factory can run multiple times for the same key — only one result wins. If construction is expensive or has side effects, wrap the value in `Lazy<T>`: `dict.GetOrAdd(key, k => new Lazy<T>(() => Build(k))).Value`.
- **`ConcurrentBag<T>` is a trap if produce and consume happen on different threads.** Its perf model assumes the thread that adds is also the thread that takes (work-stealing pattern). For pure producer/consumer, use `ConcurrentQueue<T>` or `Channel<T>`.
- **`Count` on concurrent collections takes a snapshot lock and is O(n) on some implementations.** Avoid in hot paths; use `IsEmpty` instead where possible.
- **Compound operations are still races.** `if (!dict.ContainsKey(k)) dict.Add(k, v);` is two operations — use `TryAdd` or `AddOrUpdate`.
- **Enumerating a concurrent collection gives a *moment-in-time snapshot*, not a live view.** Items added during enumeration may or may not appear; deletes won't throw `InvalidOperationException` like the non-concurrent counterparts would.

## Q20. How to perform a lock for asynchronous code?

**Short answer: `SemaphoreSlim(1, 1)` with `WaitAsync` and `Release` in `try`/`finally`.**

```csharp
private readonly SemaphoreSlim _gate = new(1, 1);

public async Task<T> GetOrLoadAsync(CancellationToken ct)
{
    await _gate.WaitAsync(ct).ConfigureAwait(false);
    try
    {
        // critical section — may await freely
        return await LoadAsync(ct).ConfigureAwait(false);
    }
    finally
    {
        _gate.Release();
    }
}
```

**Why not `lock` / `Monitor`?** Two reasons:

1. **The compiler forbids `await` inside a `lock` block** — `Monitor.Enter`/`Exit` is thread-affine; releasing on a different thread than acquired is undefined behavior. Continuations after `await` may resume on a different thread, so the lock would be "released" by the wrong thread.
2. **Even if you could, blocking on `Monitor` defeats the purpose of `async`.** If contention is real, you'd be sleeping a thread-pool thread instead of returning it, which leads to thread-pool starvation under load.

**Why `SemaphoreSlim(1, 1)`?** It's a counting semaphore initialized to allow exactly one holder at a time — a mutex with an `async`-aware acquire. `WaitAsync` returns a `Task` that completes when the slot is free, supports `CancellationToken`, and timeout overloads. It is **not** thread-affine, so any continuation on any thread can call `Release`.

**Important: `System.Threading.Lock` (.NET 9 / C# 13) does NOT solve this.** That new type is a faster, dedicated-object replacement for using `object` with `lock`/`Monitor` — and it's still purely synchronous. You cannot `await` inside `lock.EnterScope()` either. It improves perf and intent for sync critical sections; it does **not** help async code.

```csharp
// .NET 9+ — fine for SYNC critical sections only
private readonly System.Threading.Lock _sync = new();

public void DoSync()
{
    using (_sync.EnterScope())
    {
        // synchronous work only — NO await here
    }
}
```

**Other useful patterns:**

- **`AsyncLock` from `Nito.AsyncEx`** — wraps `SemaphoreSlim` in an `IDisposable`-style pattern (`using (await _lock.LockAsync()) { ... }`). Pure ergonomics.
- **Per-key locking** — instead of one global `SemaphoreSlim`, use a `ConcurrentDictionary<TKey, SemaphoreSlim>` so only callers contending for the same key serialize. Watch the dictionary growth: use a TTL or weak-reference cleanup.
- **Reader/writer scenarios**: there is no `AsyncReaderWriterLock` in the BCL. Either use the one in `Nito.AsyncEx`, or rebuild it with two `SemaphoreSlim`s — but most "reader/writer" workloads are better served by **immutable snapshots** (publish a new immutable instance on writes; readers take the current reference and never block).

**Senior-level gotchas:**

- **Always release in `finally`.** If the critical section throws and you skip `Release`, the semaphore is permanently held — a quiet deadlock that often surfaces as "the app hangs after a while".
- **`SemaphoreSlim` is not reentrant.** If a method that holds the gate calls another method that also tries to acquire it, you deadlock. `Monitor` is reentrant; `SemaphoreSlim` is not.
- **Plumb a `CancellationToken` into `WaitAsync`.** Otherwise a hung holder takes the whole pipeline down with no way to recover.
- **`ConfigureAwait(false)`** in libraries — important on ASP.NET Core (where the sync-context is null anyway, so it's mostly a no-op) and critical on UI/legacy contexts to avoid deadlocks when callers do `.Result`.
- **For very short critical sections protecting a single field**, prefer `Interlocked` operations or an immutable-snapshot swap (`Interlocked.CompareExchange` against an immutable reference) over any lock.
