## Q36. How is Encapsulation implemented in C#?

Encapsulation in C# is the deliberate hiding of internal representation behind a typed contract. It is enforced by the **compiler** at design time — there is no runtime "private" — and it is composed of several orthogonal mechanisms that together let you choose *who can see what* and *how state can change*.

### The mechanisms, ranked by how often a senior reaches for them

1. **Access modifiers** — the visibility lever.

   | Modifier | Visible from |
   |---|---|
   | `public` | Anywhere |
   | `internal` | Same assembly (default for top-level types) |
   | `protected` | Declaring class + derived classes |
   | `private` | Declaring type only (default for members) |
   | `protected internal` | Same assembly **or** derived classes (union) |
   | `private protected` | Derived classes **in the same assembly** (intersection) |
   | `file` | Declaring source file only (C# 11) — generator-friendly |

   Defaults are not symmetric: top-level types default to `internal`; class/struct members default to `private`; interface members default to `public`. A senior knows these so they don't accidentally narrow or widen scope by omission.

2. **Properties over fields.** A property is the encapsulation primitive. Even a one-line `public int Count { get; set; }` is more encapsulated than a public field, because you can change the implementation later without breaking ABI:

   ```csharp
   public sealed class Cart
   {
       private readonly List<Item> _items = new();
       public IReadOnlyList<Item> Items => _items;        // expose read-only view
       public int Count => _items.Count;                  // computed
       public decimal Total => _items.Sum(i => i.Price);  // invariant lives here
       public void Add(Item i) => _items.Add(i);          // controlled mutation
   }
   ```

   Returning `IReadOnlyList<T>` (or `IReadOnlyCollection<T>`, `IReadOnlyDictionary<K,V>`) is the canonical way to leak the iteration/length surface without leaking the mutation surface.

3. **`init`-only setters and `required` members** — encapsulation around *construction*. `init` blocks every assignment after the constructor / object initializer; `required` (C# 11) forces the caller to set the member, so you don't need a parameterless ctor that produces an invalid object:

   ```csharp
   public sealed class User
   {
       public required string Email { get; init; }
       public required string DisplayName { get; init; }
   }
   var u = new User { Email = "a@b", DisplayName = "A" }; // both required
   ```

4. **`readonly`** at every level: `readonly` field (assignable only at declaration or in a ctor), `readonly struct` (whole struct immutable), `readonly` member on a struct (the member promises not to mutate `this`). Ignoring this on structs is the source of the "defensive copy" perf trap — calling a non-`readonly` method on a `readonly` struct field forces the JIT to copy.

5. **Records and `with`.** Records bundle `init` properties, value equality, and a controlled mutation entry point (`with`) into one declaration. The encapsulation story here is "every change is a new instance," which sidesteps shared-mutable-state bugs entirely.

6. **Module-level encapsulation.** `internal` + `[InternalsVisibleTo("MyAssembly.Tests")]` is how seniors keep test-only APIs out of the public surface without resorting to `public` + "please don't call this." `file` (C# 11) goes one step further: scope a type to a single source file — primarily a source-generator pattern so two generated files can use the same helper name without colliding.

7. **Encapsulating *invariants*.** Encapsulation isn't just about visibility; it's about making *illegal states unrepresentable*. A senior pairs `private` setters with constructor validation so that *if you can hold the object, it is valid*:

   ```csharp
   public sealed class Email
   {
       public string Value { get; }
       public Email(string value)
       {
           if (string.IsNullOrWhiteSpace(value) || !value.Contains('@'))
               throw new ArgumentException("Invalid email", nameof(value));
           Value = value;
       }
   }
   ```

### How a senior thinks about it

- **Encapsulation is a design tool, not a security tool.** Reflection (`BindingFlags.NonPublic`), `UnsafeAccessor` (.NET 8+), `Mono.Cecil`, debugger evaluation — all of these can read or write your `private` state. If a value is sensitive (a secret, a credential), encapsulation is irrelevant; you need encryption or process isolation.
- **`private` is per-type, not per-instance.** Two `Account` instances can read each other's `private` fields. This is by design (think `Equals`, copy ctor, comparison) but it surprises people coming from languages where `private` means "this object only."
- **`internal` is permeable.** `[InternalsVisibleTo]` exists, and once granted, you have a contract obligation to that consumer assembly — the analyzer won't flag breaking changes to internals.
- **Properties beat fields even at "no logic" stage.** Going from `public int X;` to `public int X { get; set; }` is a binary-breaking change. Start with a property.
- **`readonly` collections are not immutable collections.** Returning a `List<T>` cast to `IReadOnlyList<T>` is consumer-side discipline, not enforcement — a malicious cast back to `List<T>` defeats it. For real immutability use `System.Collections.Immutable` or `ReadOnlyCollection<T>`.
- **Auto-property backing fields are still fields**: every `public string Name { get; set; }` has a compiler-generated `<Name>k__BackingField`. Serializers, EF, and reflection will see and write it.

### Senior-level gotchas

- **`protected` members on a `sealed` class are pointless** — analyzer `CA1822` won't catch it but a code reviewer should. Choose `private`.
- **A `private` constructor + `static` factory** is the canonical encapsulation pattern when construction can fail. Don't expose a public ctor that throws — return a `Result<T>` or use a factory and document the failure mode.
- **`virtual` undermines encapsulation.** Every `virtual` member is a contract you've handed to subclasses. If you don't intend extension, mark members `sealed override` or, better, mark the class `sealed` outright. The "default `sealed`" mindset is a senior tell.
- **`get; private set;` is *not* immutable.** Other instance methods of the same class can still set it. For true single-assignment, use `init` or a `readonly` backing field.
- **Encapsulation breaks across `partial`** — a partial class can have one half declare `private` state and the other half access it freely. Source generators rely on this.

**Modern note (.NET 10 / C# 14):** The new **`field` contextual keyword** is a meaningful encapsulation upgrade. Pre-C# 14, the moment you wanted a property with custom logic — validation, lazy init, raise-on-change — you had to write a private backing field by hand and your encapsulation story leaked by one identifier. With `field`, the compiler still synthesizes the backing storage and you reference it as `field` from the property body:

```csharp
public string Name
{
    get => field;
    set => field = value?.Trim() ?? throw new ArgumentNullException(nameof(value));
}
```

The property keeps a single authoritative storage location, the type doesn't grow a new private member name, and the validation isn't in a separate method or a hand-rolled field. C# 14 also added **partial constructors and partial events**, which lets generators contribute encapsulated behavior without forcing the user to expose new members.

## Q37. What is the difference between `ref` and `out` parameters?

Both are pass-by-reference: the parameter becomes an alias for the caller's storage location, not a copy. The difference is **definite-assignment direction**:

| | `ref` | `out` |
|---|---|---|
| Caller must assign before the call | **Yes** | No |
| Method must assign before returning | No | **Yes** |
| Method may read the parameter | Yes | **No** (until it has assigned it) |
| Intent | Two-way (read-modify-write) | One-way out (return-via-parameter) |
| Call-site syntax | `Method(ref x)` | `Method(out x)` or `out var x` |

```csharp
// out — classic Try-pattern
public static bool TryParseAge(string s, out int age)
{
    if (int.TryParse(s, out age) && age is >= 0 and <= 150) return true;
    age = 0;          // must assign on every path before return
    return false;
}

if (TryParseAge(input, out var a)) { /* a is in scope here */ }

// ref — read-modify-write
public static void Increment(ref int counter) => counter++;

int n = 5;
Increment(ref n);  // n == 6
```

### Restrictions both share

- **No properties or indexers** at the call site. `Method(ref obj.Prop)` won't compile because a property is a method, not a storage location. Stage to a local first.
- **Can't be used in `async` methods**. The state machine boxes parameters into a struct that lives on the heap; `ref`/`out`/`in` aliases would dangle. The compiler refuses.
- **Can't be used in iterators** (`yield return`) — same reason.
- **Can't be used on lambda/expression-tree parameters** in many contexts; certain delegate types don't support them at all.
- **Variance / overload disambiguation**: `void Foo(int x)` and `void Foo(ref int x)` are *different* signatures and can coexist; the call site picks via the `ref`/`out` keyword.

### The whole `ref`-family worth knowing alongside

| Modifier | Since | Meaning |
|---|---|---|
| `ref` | 1.0 | Aliased, read-write, must be assigned by caller |
| `out` | 1.0 | Aliased, write-only-then-read; method must assign |
| `in` | C# 7.2 | Aliased, **read-only** for callee; pass-large-structs-without-copy idiom |
| `ref readonly` *parameter* | C# 12 | Like `in` but the call site **must** use `ref`/`in` keyword (forces caller acknowledgement that they're aliasing). For API authors who want callers to know they're not copying. |
| `scoped` | C# 11 | Constrains a `ref`/`ref struct` parameter so it can't escape the method (ref-safety) |
| `ref` *return* / `ref` *local* | C# 7.0 / 7.2 | Return an alias; bind a local to one |

### How a senior actually uses these

- **`out` is for `Try*` patterns.** Anywhere you'd naturally return `(bool ok, T value)`, `out` is more allocation-friendly and has the bonus of `[NotNullWhen(true)]` — when you `if (TryGet(out var v))`, the compiler narrows `v` to non-null inside the branch. A tuple return loses that flow analysis.
- **`ref` for in-place mutation** is rare in modern C#. Most "modify" scenarios are better expressed by returning a new value or operating on `Span<T>` / `Memory<T>`. The legitimate `ref` cases left are interop with native APIs and tight `Span`-based numeric kernels.
- **`in` for large readonly structs.** When you have a 64-byte transform matrix, `in Matrix4x4 m` avoids the copy on every call. But `in` has a perf gotcha (next bullet).
- **The `in` defensive-copy trap.** If you call a non-`readonly` method on an `in` parameter (or any readonly ref), the JIT inserts a defensive copy because it can't prove the method won't mutate. The fix is to make the struct (or just the methods you call) `readonly`. Use `readonly struct` end-to-end or you've paid the parameter's complexity for nothing.
- **`ref readonly` parameter vs `in`.** They're the same to the callee. The difference is *call-site requirement*: `in` allows the implicit form `Method(value)`; `ref readonly` requires `Method(in value)` or the equivalent. Use `ref readonly` when you want the caller to *see* that they're aliasing — usually for APIs over `ref struct` types where copying would be wrong.
- **Discard `out`**: `Dictionary.TryGetValue(key, out _)` is the idiomatic "do you have it?" check.

### Senior-level gotchas

- **`out` parameters on `interface` methods don't get covariant return-style narrowing.** If `IRepo<T>.TryGet(int, out T)` is called via a generic, the compiler treats `T` as the interface's `T`, not whatever the caller knows; null-flow analysis follows the declared signature.
- **`ref` returns can outlive the local** if you're not careful. `ref int Get() { int x = 5; return ref x; }` is a compile error today, but composition through generic helpers can sneak past simpler ref-safety rules — that's why `scoped` exists.
- **`ref struct` parameters** (e.g., `Span<T>`) interact with `ref`/`out` modifiers via the `scoped` annotation. If you don't write `scoped`, the parameter is "unscoped," which the compiler may forbid in some contexts because it would let the ref escape.
- **`out` does not zero by default.** Pre-method, the storage may hold whatever was there (for value types it's the field's prior value, for reference types it's the prior reference). The compiler enforces "method must assign before return" so consumers don't observe garbage, but the *method's* first read after `out` parameter declaration is a use-of-unassigned-variable error — the parameter is treated as unassigned at method entry.
- **Don't use `ref`/`out` to "return multiple values."** Tuples (`(bool, T)`) or a small record are clearer in 90% of cases. Reserve `out` for the `Try*` pattern, where the bool drives flow.
- **Async wrappers around `Try*` methods are an anti-pattern.** You can't have `async Task<bool> TryFooAsync(out T value)`. Express it as `Task<(bool, T)>` or `Task<T?>` instead.

## Q38. How does the `using` statement work?

`using` is syntactic sugar for **deterministic disposal**: the compiler lowers it into a `try { ... } finally { x?.Dispose(); }` so a resource is released whether the block exits normally or via an exception.

```csharp
// using statement — explicit block scope
using (var conn = new SqlConnection(cs))
{
    conn.Open();
    // ...
}  // Dispose() called here, even on exception

// Compiler lowering (essentially):
{
    SqlConnection conn = new SqlConnection(cs);
    try { conn.Open(); /* ... */ }
    finally { if (conn != null) ((IDisposable)conn).Dispose(); }
}
```

### The two surfaces

```csharp
// (1) using statement — scope is the block
using (var f = File.OpenRead(path)) { /* ... */ }

// (2) using declaration (C# 8) — scope is the enclosing block
public async Task ProcessAsync(string path)
{
    using var f = File.OpenRead(path);
    // ...                       <- f disposed at end of method
}

// (3) await using — DisposeAsync()
await using var conn = new NpgsqlConnection(cs);
await conn.OpenAsync(ct);
```

### What types are dispose-able

- **`IDisposable`** — classic.
- **`IAsyncDisposable`** (since .NET Core 3.0) — paired with `await using`. If a type implements both, `await using` calls `DisposeAsync()`, plain `using` calls `Dispose()`.
- **Pattern-based dispose for `ref struct`** (C# 8). Since `ref struct` types can't implement interfaces, the compiler accepts a public `void Dispose()` method by signature. This is what enables `Span`-friendly resources without boxing.

### Multiple resources

```csharp
using var a = Acquire("A");
using var b = Acquire("B");
using var c = Acquire("C");
// Disposal order: c, then b, then a (reverse declaration order)
```

The reverse order is critical when later resources depend on earlier ones — a `SqlTransaction` must be disposed before the `SqlConnection` that owns it.

### How a senior thinks about it

- **`using` does not swallow exceptions.** It guarantees `Dispose` runs, then the exception continues to propagate. If you need to log/transform, wrap in a `try { using (...) { ... } } catch { ... }`.
- **`Dispose` itself can throw, and that exception will *replace* the original.** This is the C# analog of "exception during stack unwinding." If your disposable might throw on cleanup and you care about the original exception, you need a manual `try`/`finally` and `ExceptionDispatchInfo` or a `try`/`catch` inside the dispose path. Most app code doesn't, but library authors of disposables should make `Dispose` non-throwing.
- **`using (Disposable d = null)` is safe.** The lowering generates a null check, so it won't NRE.
- **Sync `using` over an `IAsyncDisposable`-only type.** If the type *only* implements `IAsyncDisposable` and you write `using` instead of `await using`, the compiler errors. If it implements both, `using` silently uses sync disposal — which can deadlock on a SynchronizationContext-bound thread (e.g., legacy ASP.NET) if the implementation expects async release. Always prefer `await using` for async types.
- **`using` declaration scope is wider than people expect.** It runs to the end of the *enclosing block* — usually the method. If you want a narrow scope, the explicit `using (...) { }` form is safer; or introduce an inner `{ ... }` block to bound a `using var`. A common bug: holding a database connection until the end of an action method when the work was done halfway through.
- **`Dispose` should be idempotent.** Code that releases a resource may be called from a finalizer and from `using` — the canonical `Dispose(bool disposing)` pattern handles this. Don't write a `Dispose` that breaks on second call.

### Senior-level gotchas

- **`yield return` inside a `using` block** keeps the resource alive across the iterator boundary. The resource is only disposed when the consumer either iterates to the end **or** disposes the enumerator (which a `foreach` does in its `finally`). If a consumer abandons the enumerator without disposing, the resource leaks. `IEnumerator<T>` implements `IDisposable` for exactly this reason.
- **`using var` + `try/catch` interplay**: a `using var` declared *inside* a `try` is disposed *before* the `catch` runs, which means you can't reference `_ = disposable.SomeProperty` inside the `catch`. Move the declaration outside `try` if needed.
- **Static analyzer friends**: `CA2000` ("Dispose objects before losing scope") flags missed `using` opportunities. `IDISP*` rules from the *IDisposableAnalyzers* package are stricter and worth turning on for resource-heavy code.
- **`HttpClient` is the famous "don't dispose"** — it's intended to be reused; disposing it kills connection pooling. Use `IHttpClientFactory` instead.
- **`using static`** (e.g., `using static System.Math;`) is a *different* feature — it's a directive to import a type's static members into the file's namespace, unrelated to disposal. The reuse of the keyword causes confusion.
- **`ConfigureAwait(false)` does not propagate through `await using`.** Each `await using` does its own `await DisposeAsync()` whose continuation respects the current sync-context unless you write `await using (var x = ... .ConfigureAwait(false))` — and even that is awkward. For library code, prefer `IAsyncDisposable` types whose `DisposeAsync` doesn't capture context.

## Q39. What is a delegate and how is it used?

A delegate is a **type-safe reference to one or more methods** with a matching signature. At the runtime level, every delegate type derives from `System.MulticastDelegate` → `System.Delegate` and carries an invocation list. At the language level, it's the building block for events, callbacks, LINQ, and any "pass behavior as a value" scenario.

```csharp
// (1) Custom delegate type
public delegate decimal PriceCalculator(Order o);

// (2) Built-in generic forms — usually preferred
Func<int, int, int> add = (a, b) => a + b;
Action<string>      log = Console.WriteLine;
Predicate<int>      isPositive = x => x > 0;

// (3) Method-group conversion
Func<string, int> parse = int.Parse;     // points to int.Parse(string)

// (4) Multicast
Action<string> sink = log;
sink += s => File.AppendAllText("log.txt", s + Environment.NewLine);
sink("hello");   // both subscribers run, in subscription order
```

### Where delegates show up in real code

- **Events.** An `event EventHandler<T> Foo;` is a delegate field with restricted access — outside the type, only `+=` and `-=` are allowed.
- **LINQ predicates and projections.** `Where`, `Select`, `OrderBy` take `Func<T, ...>` — that's a delegate (until you target an `Expression<Func<T, ...>>`, in which case the compiler builds an expression tree instead, which is the IEnumerable-vs-IQueryable distinction).
- **Callback / strategy injection.** `services.AddSingleton<IFoo>(sp => new Foo(...))`, `Polly` policies, `IHttpClientFactory.ConfigureHttpClient(Action<HttpClient>)`.
- **Functional composition.** `Func<A,B>` + `Func<B,C>` chain into a `Func<A,C>` — useful in pipelines.

### Multicast — useful but weird

```csharp
Func<int, int> chain = x => x + 1;
chain += x => x * 2;
int r = chain(3);    // r == 6 — only the LAST handler's return value is observed
```

- All handlers run, in subscription order.
- For non-void delegates, only the **final** handler's return value is the result of the invocation. Earlier returns are discarded.
- An exception in any handler stops the chain and the rest don't run. To get "run them all and aggregate exceptions," walk `GetInvocationList()` manually.

### How a senior thinks about it

- **Delegates vs lambdas vs expression trees.** A lambda is a *syntax*; what it compiles to depends on the **target type**. `Func<int,int> f = x => x+1;` becomes a delegate (compiled IL); `Expression<Func<int,int>> e = x => x+1;` becomes a parsed `Expression` tree (no IL run yet). The same source becomes radically different artifacts.
- **Closures cost allocations.** A lambda that captures a local variable allocates a *display class* on the heap (one per surrounding scope), and the delegate is allocated against it. In hot paths, lift to a static lambda or a method group. The C# 9+ `static` lambda modifier (`static x => ...`) prevents capture and turns the analyzer on for any accidental capture.
- **Method-group caching.** Since C# 11, the compiler caches the delegate produced by a method-group conversion at the call site, so `RegisterCallback(MyHandler);` doesn't allocate a new delegate every call. Before C# 11, this was a noticeable source of allocations. (Cached delegates from non-capturing lambdas have existed much longer.)
- **`Action` vs `Func<Task>` for async callbacks.** `Action` is `void`-returning — you cannot `await` it, and an `async () => ...` lambda assigned to `Action` becomes `async void`, which is the "kill the process on unhandled exception" foot-gun. For async callbacks, declare `Func<Task>` / `Func<CancellationToken, Task>` / `Func<ValueTask>` and `await` them.
- **`MulticastDelegate` invocation list is immutable.** `+=` and `-=` produce a *new* delegate; they do not mutate the existing one. That makes `event` raise patterns thread-safe at the assignment level — `var snapshot = MyEvent; snapshot?.Invoke(...);` won't lose handlers added concurrently, though it might miss them.
- **Delegate equality is structural-by-target-and-method.** Two delegates pointing at the same method on the same target are equal. This is how `-=` knows what to remove. A *capturing* lambda creates a new closure each time, so `e -= () => Foo()` rarely matches the original; store the delegate in a variable if you need to unsubscribe.

### Senior-level gotchas

- **`async void` event handlers**. Acceptable for top-level UI event handlers (because the framework expects a `void` return), but every other use is a bug — exceptions go unobserved and crash the process. Use `Func<..., Task>` instead.
- **Subscribing to a long-lived event from a short-lived object** is the classic .NET memory leak. The publisher holds a reference to your delegate's target, which holds your object. Use `WeakEventManager` (WPF) or unsubscribe explicitly in `Dispose`.
- **Delegate variance**: `Func<T>` is covariant in `T`, `Action<T>` is contravariant in `T`. This lets `Func<string>` flow into a `Func<object>` slot. Custom delegates need explicit `out`/`in` on type parameters to participate.
- **`Delegate.Combine`/`Remove` for void delegates** are how `+=`/`-=` work under the hood. For non-void delegates, "combine" still works but you usually shouldn't use multicast — use a list of `Func` and choose how to aggregate.
- **`Invoke` vs direct call**: `f.Invoke(x)` and `f(x)` are equivalent. Use `?.Invoke(...)` to tolerate null delegates (no subscribers).
- **Struct-target methods box.** Creating a delegate over an instance method on a `struct` boxes the struct. Static or method-group on a class instance is alloc-free (post C# 11 caching).

**Modern note (.NET 10 / C# 14):** Delegates are unchanged in C# 14. The adjacent C# 14 feature with the most spillover to delegate-style code is **extension members** (extension properties and indexers, not just methods) — useful when an API takes a delegate that operates on a third-party type and you want a fluent surface. Method-group caching (C# 11) and `static` lambdas (C# 9) remain the two performance levers a senior should reach for.

## Q40. Explain method overloading and overriding

These are two unrelated mechanisms that share the syllable "over." Confusing them in an interview is a tell — the senior answer makes the distinction crisp:

- **Overloading**: same name, **different signatures**, in the **same** type (or via extensions). Resolved at **compile time** by overload resolution. Provides ad-hoc polymorphism.
- **Overriding**: same signature, **derived** type provides a new implementation of a `virtual`/`abstract` member. Resolved at **runtime** via the method-table. Provides subtype polymorphism.

```csharp
// Overloading — compile-time dispatch by signature
public class Logger
{
    public void Log(string msg) { /* ... */ }
    public void Log(string msg, Exception ex) { /* ... */ }
    public void Log(LogLevel lvl, string msg) { /* ... */ }
}

// Overriding — runtime dispatch by actual type
public abstract class Shape { public abstract double Area(); }
public sealed class Circle(double r) : Shape
{
    public override double Area() => Math.PI * r * r;
}
Shape s = new Circle(2);
s.Area();   // calls Circle.Area at runtime
```

### Overloading — what counts as "different signature"

The signature is method name + parameter list + arity + parameter modifiers. **Return type is not part of the signature.** What does count:

- Type of each parameter.
- Number of parameters.
- `ref` / `out` / `in` modifiers (each makes a distinct overload).
- `params` is part of the signature in the sense that overload resolution treats `Foo(int[])` and `Foo(params int[])` differently *at the call site* but the method-table sees them the same.
- Generic arity: `Foo<T>(T x)` and `Foo<T,U>(T x, U y)` are different.

```csharp
void Foo(int x) {}
void Foo(int x, int y = 0) {}      // legal; second overload has default
// void Foo(int x) {return 1;}     // ILLEGAL — return type alone doesn't disambiguate
void Foo(ref int x) {}             // legal; differs by ref modifier
```

**Overload resolution** picks the *best* candidate by a betterness algorithm that considers exact type match, implicit conversions, generic vs non-generic (non-generic preferred when both apply), `params` expansion, optional parameter defaults. When tied, it errors with CS0121.

### Overriding — what's required

- Base member must be marked `virtual`, `abstract`, or already an `override`.
- Derived member must be `override` with the **same signature, same return type** (or a covariantly compatible one — see below).
- `sealed override` blocks further overriding in subclasses.
- Visibility must match — you can't widen `protected` to `public` via override; for that, hide with `new` (rarely advisable).
- `base.X(...)` calls the original implementation from inside the override.

```csharp
public class Animal { public virtual string Speak() => "..."; }
public class Dog : Animal { public override string Speak() => $"woof; {base.Speak()}"; }
public class Puppy : Dog { public sealed override string Speak() => "yip"; }
```

### Hiding (`new`) is *not* overriding

```csharp
public class Base    { public virtual void Greet() => Console.WriteLine("base"); }
public class Hider   : Base { public new void Greet() => Console.WriteLine("hider"); }

Base b = new Hider();
b.Greet();    // prints "base" — `new` doesn't participate in the v-table
```

Hiding bypasses polymorphism and is almost always a code smell — it produces dispatch behavior that depends on the *static* type. The compiler warns (CS0108) unless you write `new` explicitly.

### Covariant return types (C# 9)

```csharp
public class Animal {}
public class Dog : Animal {}

public class AnimalFactory { public virtual Animal Create() => new Animal(); }
public class DogFactory    : AnimalFactory { public override Dog Create() => new Dog(); }
// Caller of (DogFactory).Create() gets a Dog without a cast.
```

Useful for fluent APIs and clone patterns. Records use this internally for `with`.

### Interface members — overriding's cousin

- **Implicit implementation**: a public method on the class with matching signature satisfies the interface contract; can also be called via the class type directly.
- **Explicit interface implementation** (`void IFoo.Bar() { ... }`): callable only through the interface — useful for two interfaces with conflicting method names, or to "hide" a method from the class's normal surface.
- **Default interface methods** (C# 8): `interface IFoo { void Bar() { /* default body */ } }`. Lets you add methods to interfaces without breaking implementers. They are *not* overridable from a class via `override`; the class re-implements by declaring the method directly. Mostly used for evolving widely-implemented interfaces.
- **Static abstract members in interfaces** (C# 11): `interface INumber<T> { static abstract T Zero { get; } }`. Foundation of the .NET 7+ generic-math library. Implementers provide the static; callers reach it through a `where T : INumber<T>` constraint.

### How a senior thinks about it

- **Default to `sealed` classes.** Inheritance is a contract you've handed to consumers — every `virtual` member is a future maintenance constraint. If you don't *intend* to be overridden, mark the class `sealed`. The team will thank you when you can refactor without breaking implementers.
- **Prefer composition + interfaces over deep inheritance trees.** `override` is fine in shallow hierarchies (decorators, strategies) and noxious at depth (the classic 4-level "Animal → Pet → Dog → Puppy" trap).
- **Overload resolution + `null`** is the most common ambiguity. `Foo(string?)` and `Foo(object?)` both accept `null`; the call `Foo(null)` is ambiguous. Disambiguate at the call site with `Foo((string?)null)` or rename the overload.
- **Default-parameter overloads can collide silently.** `Foo(int x, int y = 0)` and `Foo(int x)` both match `Foo(5)`; the rules say the one without optional params wins. Adding a default to an existing overload is a binary-compatible change with subtle source consequences.
- **`base.Method()` from an `override`** is the only way to call the immediate base implementation; reflection-based "skip a level" is undefined behavior in normal C#. Don't try to reach `grandparent.Method()` via casts — it doesn't work the way you'd think; virtual dispatch still goes through the v-table.
- **`async` overrides must keep the async signature.** You can't override a `Task<T> Foo()` with `T Foo()` and vice versa — the return type must match (modulo covariance).

### Senior-level gotchas

- **Hiding via `new` + the wrong receiver type** is the biggest "huh?" bug. A test passing the derived type to a base-typed parameter sees the base behavior. Always `override`.
- **Optional parameters are baked at the call site.** `void Log(string msg, int level = 0)` — if a consumer compiles against v1 and you ship v2 with `level = 1`, the consumer still passes `0` until they recompile. Default values are not virtualized; they are constants in the calling assembly.
- **Generic overload resolution** picks the most specific generic instantiation but can be tricked by constraints; a generic `Foo<T>(T)` plus a non-generic `Foo(int)` will pick the non-generic for `Foo(5)`, but `Foo<int>(5)` forces the generic — which is rarely what people expect.
- **Operators are static and overloaded, not overridden.** `==` on a base class is static — invoking it through a derived reference still uses the base's static; you need `Equals` (which is `virtual`) for polymorphic equality. This is the single biggest reason custom equality logic should live in `Equals`/`GetHashCode`, not operators.
- **Covariant returns + interfaces**: as of C# 9, an interface implementation can use a more derived return type, but only via *implicit* implementation; explicit interface implementations have to match the declared signature exactly.
- **Default interface methods + diamond**: if two interfaces give conflicting defaults and a class implements both, the class **must** disambiguate by re-implementing — the compiler errors otherwise.
- **`new` accidentally**: forgetting `override` on a method that *should* override generates CS0108 ("hides inherited member") and silently turns it into hiding. Treat that warning as an error.

**Modern note (.NET 10 / C# 14):** No changes to overloading or overriding semantics in C# 14. The adjacent additions (**`field` keyword**, **partial constructors and events**, **extension members**) don't alter the resolution rules but expand what's expressible: extension *properties* mean "overloading by extension" now extends past methods, and partial constructors let source generators contribute alternate constructor overloads without colliding with user code.
