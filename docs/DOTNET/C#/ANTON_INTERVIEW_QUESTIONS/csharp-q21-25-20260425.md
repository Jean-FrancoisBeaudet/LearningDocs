## Q21. Name all the ways for creating a new thread

First, a senior framing: **"creating a new thread"** is not the same as **"scheduling work that may run concurrently."** The CLR distinguishes:

- **Dedicated OS thread** â€” a brand-new managed `Thread` mapped 1:1 to an OS thread. Expensive (~1 MB user-mode stack by default, kernel transitions to start/stop), but you fully control its lifetime and priority.
- **Thread-pool work item** â€” work scheduled onto the shared `ThreadPool`. No new thread is created in the common case; the pool reuses pre-warmed worker threads.

Most APIs below schedule onto the thread pool. Only the first creates a true new thread.

### Ways to spin up work

1. **`new Thread(...)`** â€” the only way to create a *dedicated* managed thread.
   ```csharp
   var t = new Thread(() => DoWork()) { IsBackground = true, Name = "worker" };
   t.Start();
   ```
   Use when you need long-running work that should not steal pool threads, custom stack size (`new Thread(start, maxStackSize)`), or you must set apartment state / priority.

2. **`ThreadPool.QueueUserWorkItem`** â€” fire-and-forget pool work; legacy entry point.
   ```csharp
   ThreadPool.QueueUserWorkItem(static state => DoWork(), null, preferLocal: false);
   ```

3. **`Task.Run`** â€” the modern default for offloading CPU-bound work to the pool. Returns a `Task`/`Task<T>` you can await.
   ```csharp
   var result = await Task.Run(() => Compute());
   ```

4. **`Task.Factory.StartNew`** â€” older, more explicit. Lets you pass `TaskCreationOptions.LongRunning` (which *does* spawn a dedicated thread, not a pool thread) and a custom `TaskScheduler`. Avoid unless you actually need those knobs â€” it does **not** unwrap async delegates by default, leading to nested `Task<Task>`.

5. **`Parallel.Invoke` / `Parallel.For` / `Parallel.ForEach` / `Parallel.ForEachAsync`** â€” data-parallel constructs that partition work across pool threads. `Parallel.ForEachAsync` (.NET 6+) is the right tool for bounded-concurrency async iteration.

6. **`System.Threading.Timer` / `PeriodicTimer`** â€” fire callbacks on the pool. `PeriodicTimer` (.NET 6+) is async-friendly: `await timer.WaitForNextTickAsync(ct)`.

7. **`async`/`await`** â€” strictly speaking does *not* create a thread. The compiler builds a state machine; continuations resume on the captured `SynchronizationContext` or the pool. Confusing this with "threading" is a classic mid-level mistake.

8. **`BackgroundWorker`** â€” legacy WinForms helper. Don't use in new code.

9. **`IHostedService` / `BackgroundService`** â€” the ASP.NET Core / Generic Host pattern for long-running background work, lifetime-managed by the host. Internally uses `Task.Run`/your own awaits, not a dedicated thread.

10. **PLINQ (`AsParallel()`)** â€” partitions a query across pool threads.

11. **TPL Dataflow / `Channel<T>`** â€” producer/consumer pipelines that schedule onto the pool.

### Senior-level gotchas
- `Task.Factory.StartNew(asyncDelegate)` returns `Task<Task>`. You almost always want `Task.Run` or to call `.Unwrap()`.
- `TaskCreationOptions.LongRunning` is the only `Task` API that quietly creates a dedicated thread â€” useful for blocking work you cannot make async.
- Don't `new Thread()` for short work â€” pool reuse is dramatically cheaper.
- Thread-pool starvation is the #1 cause of mysterious latency spikes; `dotnet-counters monitor System.Runtime` shows pool queue length.
- `Thread.IsBackground = true` lets the process exit without joining; foreground threads keep the process alive.

## Q22. How to execute multiple async tasks at once?

The core primitive is **`Task.WhenAll`**. Around it sit a small family of patterns; pick by what "at once" actually means.

### `Task.WhenAll` â€” wait for everything

```csharp
var t1 = httpClient.GetAsync(url1, ct);
var t2 = httpClient.GetAsync(url2, ct);
var t3 = httpClient.GetAsync(url3, ct);

HttpResponseMessage[] responses = await Task.WhenAll(t1, t2, t3);
```
- Tasks run concurrently because they are *started* before the first `await`. Awaiting `Task.WhenAll` only joins them.
- If multiple tasks fault, `WhenAll` rethrows the **first** exception, but the returned `Task.Exception` contains an `AggregateException` with all of them. A `try { await Task.WhenAll(...); } catch (Exception ex) { ... }` only sees the first â€” inspect the outer task to recover the rest.

### `Task.WhenAny` â€” race / first-to-finish

```csharp
var winner = await Task.WhenAny(primary, fallback);
```
Useful for timeouts, hedged requests, racing replicas. Always remember to observe the *losing* tasks (await or attach a continuation) to avoid unobserved exceptions.

### `Task.WhenEach` â€” stream completions as they arrive (.NET 9+)

```csharp
await foreach (var t in Task.WhenEach(tasks))
{
    var page = await t; // surfaces exceptions in arrival order
    Process(page);
}
```
Replaces the awkward "loop on `WhenAny` + remove" pattern. Excellent for fan-out where you want to start consuming the fastest results immediately.
Source: <https://learn.microsoft.com/en-us/dotnet/api/system.threading.tasks.task.wheneach>

### `Parallel.ForEachAsync` â€” bounded async fan-out

```csharp
await Parallel.ForEachAsync(
    items,
    new ParallelOptions { MaxDegreeOfParallelism = 8, CancellationToken = ct },
    async (item, token) => await ProcessAsync(item, token));
```
Use when the input is *unbounded* and you need to throttle concurrency â€” e.g., 100k URLs but only 8 in flight. Roll your own with `SemaphoreSlim` if you need more shaping (per-host limits, retries).

### Channels / Dataflow

For producer/consumer pipelines or backpressure, `System.Threading.Channels` is the modern primitive; TPL Dataflow when you need typed blocks (`TransformBlock`, `ActionBlock`).

### Senior-level gotchas
- **Exception aggregation** â€” `await Task.WhenAll(...)` only throws the *first* exception caught in a `catch`. To see all, inspect the task: `var t = Task.WhenAll(tasks); try { await t; } catch { throw t.Exception!; }`.
- **Cold vs hot tasks** â€” `Task.Run(...)` and `httpClient.GetAsync(...)` are hot (already running). Custom `Task` factories that wrap synchronous work into `new Task(...)` are cold and need `.Start()` â€” almost always a bug.
- **Don't `Task.WaitAll`/`Task.Wait`** â€” they block a thread synchronously and risk deadlock.
- **Cancellation** â€” `WhenAll` does not cancel other tasks if one fails. Wire a shared `CancellationTokenSource` and call `Cancel()` in the catch to short-circuit siblings.
- **Allocation** â€” `WhenAll(tasks)` allocates an array; for hot paths over fixed-size sets, the `WhenAll<T>(Task<T> t1, Task<T> t2)` overloads (.NET 9+) avoid that.

**Sources:**

- <https://learn.microsoft.com/en-us/dotnet/api/system.threading.tasks.parallel.foreachasync>

## Q23. Explain Inheritance vs Composition

Two ways one type reuses behavior from another:

| | Inheritance | Composition |
|--|--|--|
| Relationship | "is-a" | "has-a" / "uses-a" |
| Coupling | Tight (subclass depends on base internals) | Loose (depends on an interface) |
| Reuse mechanism | Subclassing | Holding a reference / DI |
| Polymorphism | `virtual` / `override` | Interface dispatch |
| Flexibility | Fixed at compile time, single base in C# | Swappable at runtime |
| Testability | Hard â€” must mock base or use real | Easy â€” inject a fake |

### Inheritance â€” when it earns its keep
Use it when there is a true behavioral hierarchy with a shared invariant the base enforces, e.g. an ASP.NET Core `Controller` base, an EF Core `DbContext`, or framework-defined templates (`BackgroundService`).

```csharp
public abstract class BackgroundService : IHostedService, IDisposable
{
    protected abstract Task ExecuteAsync(CancellationToken ct);
    // host machinery the subclass MUST NOT reimplement
}
```
The base owns the lifecycle; derivations only fill in the abstract slot. That's inheritance pulling its weight.

### Composition â€” the default

```csharp
public sealed class OrderService(IPaymentGateway payments, IInventory inventory, ILogger<OrderService> log)
{
    public async Task PlaceAsync(Order order, CancellationToken ct)
    {
        await inventory.ReserveAsync(order, ct);
        await payments.ChargeAsync(order, ct);
    }
}
```
`OrderService` is *composed* of an `IPaymentGateway` and an `IInventory`. Either dependency can be swapped (real, test double, decorator). Adding caching is a wrapping decorator, not a subclass.

### Why "prefer composition over inheritance" is repeated *ad nauseam*
- **Fragile base class problem** â€” changing a base method breaks subclasses in non-obvious ways.
- **Single inheritance** in C# means inheritance is a one-shot resource â€” spend it carefully.
- **Cross-cutting concerns** (logging, retries, caching) compose as decorators; they fight inheritance.
- **Tests** â€” inheriting a base that talks to a database makes every subclass an integration test.

### Senior-level gotchas
- Default to `sealed` on classes you don't intend to be inherited; this is enforced by analyzers like CA1052/CA1063 and is now the framework norm (e.g., `Span<T>`-based types, most `record` types).
- Interfaces + DI **is** composition. If you find yourself creating an abstract base just to share three lines, extract a service instead.
- Inheriting only to override one method is usually a strategy/pluggable behavior in disguise â€” pass a delegate or interface.
- "Inheritance for code reuse" is the antipattern; "inheritance for substitutability under an LSP-honoring contract" is fine.
- Records can `record Cat(string Name) : Animal(Name)`. Inheritance hierarchies of records have subtle equality rules â€” `Equals` checks the runtime `EqualityContract`, so a `Dog` is never equal to a `Cat` even with identical data. Useful, but surprising.

## Q24. Difference between `class` vs `record` vs `struct`

All three declare types; they differ on **memory model**, **identity**, and **mutability defaults**.

### `class` â€” reference type, identity-based equality

```csharp
public class Customer
{
    public required string Name { get; init; }
    public string? Email { get; set; }
}
```
- Allocated on the heap; variables hold references. Copying a variable copies the reference, not the data.
- `Equals`/`GetHashCode` are reference-based by default (`object.ReferenceEquals` semantics).
- Single inheritance, multiple interface implementation.
- The default for "an entity with identity that mutates over time" â€” domain entities, services, DI registrations.

### `record` (i.e. `record class`) â€” reference type, value-based equality

```csharp
public record Money(decimal Amount, string Currency);
```
The compiler generates:
- `Equals(object?)`, `Equals(Money?)`, `IEquatable<Money>`, `GetHashCode` â€” value-based across all positional and declared properties.
- `ToString()` â€” `"Money { Amount = 10.00, Currency = USD }"`.
- A protected `EqualityContract` property (the runtime type), which makes inherited records value-unequal across subtypes.
- A `Deconstruct` method for positional records.
- A copy constructor + `with` expression for non-destructive mutation:
  ```csharp
  var bumped = price with { Amount = price.Amount + 1 };
  ```
- Default property setter for positional parameters is `init` â€” shallow immutability.

Use when modeling **data**: DTOs, value objects, query results, messages.

### `struct` â€” value type, copy semantics

```csharp
public struct Point(double X, double Y);
```
- Lives where it's declared: stack for locals, inline inside the parent for fields, on the heap when boxed or inside a heap-allocated container.
- Assignment copies the bytes â€” no shared state, but every copy pays the cost.
- Default `Equals`/`GetHashCode` from `ValueType` use reflection over fields â†’ slow; **always** override or use `record struct`.
- No inheritance (only interface implementation). Implicit parameterless constructor exists; you can now define your own (C# 10+).
- Use `readonly struct` for true value-object immutability â€” also tells the compiler to elide defensive copies on member access.

### `record struct` â€” value type with record semantics

```csharp
public readonly record struct Point(double X, double Y);
```
- Same generated equality, `ToString`, `Deconstruct`, `with` as a record class.
- No copy constructor (because every copy *is* a copy of the bytes).
- `readonly record struct` is the idiomatic, allocation-free value object â€” the modern replacement for hand-rolled `struct` value objects.

### Quick decision matrix

| Scenario | Pick |
|--|--|
| Domain entity with identity + mutation | `class` (often `sealed`) |
| Immutable data carrier (DTO, message, query model) | `record` |
| Tiny (â‰¤16 byte), high-frequency value (coords, IDs, money) | `readonly record struct` |
| Performance-critical, no equality/`ToString` needed | `readonly struct` with manual `IEquatable<T>` |
| Inheritance hierarchy needed | `class` or `record` (records support inheritance) |
| Lives on the stack only (e.g. wraps `Span<T>`) | `ref struct` (see Q25) |

### Senior-level gotchas
- **Records are not immutable** â€” positional parameters are `init`, but you can declare mutable properties on a record. Mutating an object used as a dictionary key after insertion is the classic foot-gun (hash code changes).
- **Record equality is structural over declared members** â€” adding a property changes `GetHashCode`. This breaks distributed caches across deployments more often than you'd expect.
- **Struct size matters** â€” past ~16-24 bytes, copy cost dominates. Profile.
- **Boxing kills struct perf** â€” assigning to `object`, using as a non-`IEquatable` interface, or using in non-generic collections all box. Generic methods with `where T : struct, IEquatable<T>` are the safe pattern.
- **Default struct values are "valid but absurd"** â€” `default(DateTime)` is `0001-01-01`. Constructors don't always run (e.g., `new T[10]` for a `T : struct` initializes via `default`). Design structs to make `default` either obviously meaningful or obviously invalid.
- **`required` + records** (C# 11) lets you enforce property initialization without a positional ctor â€” useful for many-property DTOs.

## Q25. What are `ref struct`s used for?

A `ref struct` is a struct the compiler guarantees **lives only on the stack** (or as a temporary in a register / inline frame). The CLR enforces a set of restrictions so the type can never escape to the heap. They exist to make **safe, zero-allocation access to memory** possible â€” particularly to stack-allocated buffers, slices of arrays, and unmanaged memory.

### The canonical citizens
- **`Span<T>` / `ReadOnlySpan<T>`** â€” type-safe view over contiguous memory (managed array, stack-allocated buffer via `stackalloc`, or unmanaged pointer). Used pervasively in `System.Text`, JSON, networking.
- **`Utf8JsonReader`**, **`Utf8JsonWriter`** â€” forward-only parsers/writers that hold a `ref` to the underlying buffer.
- **`SpanSplitEnumerator<T>`**, **`MemoryExtensions.SpanSplitEnumerator`** â€” enumerators that hold a span.
- **`System.Buffers.SequenceReader<T>`** â€” multi-segment reader.
- **`Lock.Scope`** â€” the new `System.Threading.Lock` (.NET 9) returns a `ref struct` from `EnterScope()` so the disposal pattern stays stack-only.

### Restrictions (and why they exist)
A `ref struct` cannot:
- Be a field of a non-`ref struct` (would put it on the heap).
- Be boxed (`object`, dynamic, or non-`allows ref struct` interfaces).
- Be a generic type argument unless the constraint is `where T : allows ref struct` (C# 13+).
- Be captured by a lambda or local function (would hoist into a closure class on the heap).
- Implement `IDisposable` via boxing â€” but `ref struct` can have a public `Dispose()` and be used in `using` (the compiler binds it structurally).

These rules collectively guarantee the type can't outlive its stack frame â€” which is the whole point.

### Why bother â€” the performance story

```csharp
// Allocates: substring + char[] -> 2 heap allocations per call
public static int CountVowelsAlloc(string s) =>
    s.ToLowerInvariant().Count(c => "aeiou".Contains(c));

// Zero allocations: span over the original chars
public static int CountVowels(ReadOnlySpan<char> s)
{
    int n = 0;
    foreach (var c in s)
        if (c is 'a' or 'e' or 'i' or 'o' or 'u'
              or 'A' or 'E' or 'I' or 'O' or 'U') n++;
    return n;
}
```
`Span<T>` slices, parsers, and binary protocols routinely beat their array/string counterparts by 5-50Ă— in throughput because they remove allocations from hot paths. That's why ASP.NET Core's pipeline, `System.Text.Json`, gRPC, and Kestrel are full of them.

### `ref` fields and `scoped` (C# 11)
Since C# 11, a `ref struct` can have **ref fields** â€” references to other stack-resident state. This is what made it possible to implement `Span<T>` in pure C# instead of CLR magic. The new `scoped` keyword lets you constrain a `ref` parameter so it cannot escape:
```csharp
static ReadOnlySpan<int> First(scoped ReadOnlySpan<int> s) => s; // compile error: scoped can't escape
```
The compiler's ref-safety analysis prevents dangling references at compile time.

### `allows ref struct` constraint (C# 13)
Pre-C# 13, you couldn't write generic code that worked over both `Span<T>` and ordinary types. Now:
```csharp
static void Process<T>(T value) where T : allows ref struct { ... }
```
This unblocks generic algorithms (e.g. `string.Join`, `Random.GetItems`) over spans without overload duplication. It's why `System.Threading.Lock` could expose a generic-friendly disposal pattern.
Source: <https://learn.microsoft.com/en-us/dotnet/csharp/whats-new/csharp-13#allows-ref-struct>

### When to write your own
Reach for `ref struct` when you are:
- Wrapping a stack-allocated buffer (`stackalloc`) and exposing typed access.
- Building a forward-only reader/writer over `ReadOnlySpan<byte>` (binary protocols, custom serializers).
- Implementing an enumerator that should not allocate (`Span<T>` enumerators).

Otherwise stay with `struct` or `class` â€” `ref struct` propagates its restrictions to every API that takes one.

### Senior-level gotchas
- You **cannot `await`** in a method where a `ref struct` is in scope across the await â€” async state machines lift locals to the heap. Workaround: confine the `ref struct` to a non-async helper.
- You **cannot use a `ref struct` inside an iterator** (`yield return`) for the same reason.
- `using var reader = new Utf8JsonReader(...)` works because the compiler binds structurally to a `Dispose()` method â€” but it does **not** require implementing `IDisposable`.
- `Span<T>` over a `string` is `ReadOnlySpan<char>` â€” no copy, but you cannot mutate the string. `MemoryMarshal.AsMemory` and friends exist but bypass safety; rarely the right answer.
- `Memory<T>` is the heap-friendly cousin: usable in async, fields of classes, lambdas. The pattern is "store `Memory<T>`, slice into `Span<T>` for the hot loop."
