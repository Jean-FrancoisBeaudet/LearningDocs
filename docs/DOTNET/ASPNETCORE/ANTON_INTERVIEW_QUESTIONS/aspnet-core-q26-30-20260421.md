## Q26. What is HSTS?

**HSTS (HTTP Strict Transport Security, RFC 6797)** is an HTTP response header that tells a compliant browser: "for the next *N* seconds, never talk to this origin over plain HTTP — automatically upgrade every request to HTTPS, and refuse to connect if the TLS certificate is invalid." It is a defense against **SSL-stripping / man-in-the-middle downgrade attacks**, which `UseHttpsRedirection()` alone cannot fully prevent (the very first HTTP request is still exposed).

The header looks like this:

```
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
```

- `max-age` — seconds the browser must remember to force HTTPS. `0` clears the policy.
- `includeSubDomains` — applies to every subdomain too.
- `preload` — opt-in to the browser-shipped HSTS preload list (Chromium, Firefox, Safari, Edge). Once preloaded, even a first-ever visit is forced to HTTPS. Getting on the list is fast; getting off is slow.

### In ASP.NET Core

Two middlewares work together in the `Program.cs` pipeline:

```csharp
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddHsts(options =>
{
    options.Preload = true;
    options.IncludeSubDomains = true;
    options.MaxAge = TimeSpan.FromDays(365);
    // options.ExcludedHosts.Add("example.com");
});

builder.Services.AddHttpsRedirection(options =>
{
    options.HttpsPort = 443;
});

var app = builder.Build();

if (!app.Environment.IsDevelopment())
{
    app.UseHsts();                // adds the header
}

app.UseHttpsRedirection();         // 307/308 redirect to https://
```

Defaults worth knowing (ASP.NET Core 10):
- **Default `MaxAge` is 30 days** — intentionally conservative. Bump to 1 year only after you are fully confident HTTPS is sustainable, because clients will refuse HTTP for that entire window.
- `UseHsts()` **excludes the loopback** (`localhost`, `127.0.0.1`, `::1`) by default, which is why it's usually skipped in `Development` — dev certs don't chain to a trusted root and would cause `ERR_CERT_AUTHORITY_INVALID` failures with no override.
- The header is only honored by browsers when delivered **over HTTPS**. Sending it over plain HTTP does nothing.

### Senior-level gotchas

- **Preload is effectively irreversible.** The `preload` directive signals consent to browser vendors. Removal requires an unpreload request and propagates over months. Don't enable `preload` until you're sure every subdomain (now and future) will serve HTTPS.
- **TLS termination at a load balancer / reverse proxy.** If the LB terminates TLS and talks to Kestrel over HTTP, `UseHttpsRedirection` will see an HTTP request and loop/incorrectly redirect unless you wire up `ForwardedHeadersMiddleware` (`X-Forwarded-Proto`) *before* `UseHsts`/`UseHttpsRedirection`.
- **`includeSubDomains` covers every subdomain**, even ones you don't own the TLS story for (e.g. a legacy `reports.example.com`). Audit before enabling.
- **HSTS doesn't prevent the first-visit MITM** — preload does. Non-preloaded sites are vulnerable on the very first connection.
- **Don't set `max-age=0` accidentally** in config transforms — it wipes the policy for every user the moment they visit.

## Q27. How to return a file from an API Endpoint?

Two surfaces — controllers and Minimal APIs — with the same underlying `Microsoft.AspNetCore.Http.Results` machinery. The core decision is always **stream, don't buffer**: reading the full file into a `byte[]` wastes memory and, for anything over 85 KB, pins it on the Large Object Heap.

### Controllers

`ControllerBase` gives you helper methods:

```csharp
[HttpGet("report/{id:guid}")]
public IActionResult Download(Guid id, CancellationToken ct)
{
    // Streamed from disk — never buffered
    var stream = System.IO.File.OpenRead($"/var/reports/{id}.pdf");
    return File(
        stream,
        contentType: "application/pdf",
        fileDownloadName: "report.pdf",
        enableRangeProcessing: true);   // supports Range: bytes=
}
```

Variants:
- `File(byte[], ...)` — only for small, in-memory payloads.
- `File(Stream, ...)` — the default; framework disposes the stream.
- `PhysicalFile(path, ...)` — uses `SendFileAsync` (sendfile(2) on Linux, TransmitFile on Windows) — zero-copy, the most efficient option for on-disk files.
- `VirtualFile(virtualPath, ...)` — same, via the `IWebHostEnvironment.WebRootFileProvider`.

### Minimal APIs

```csharp
app.MapGet("/report/{id:guid}", (Guid id, CancellationToken ct) =>
{
    var stream = File.OpenRead($"/var/reports/{id}.pdf");
    return TypedResults.File(
        stream,
        contentType: "application/pdf",
        fileDownloadName: "report.pdf",
        enableRangeProcessing: true);
});
```

Prefer **`TypedResults`** over `Results` in .NET 10 — it returns a concrete `FileStreamHttpResult`, which gives you stronger typing, automatic OpenAPI metadata, and easier unit testing. Use `Results.*` only when you need to union multiple return types with `Results<Ok, NotFound, FileStreamHttpResult>`.

Other Minimal API helpers:
- `TypedResults.Stream(stream, ...)` / `TypedResults.Stream(pipeWriter => ...)` — write directly to the response body, no buffering.
- `TypedResults.Bytes(byte[], ...)` — tiny payloads only.
- `TypedResults.PhysicalFile(path, ...)` — `sendfile`-backed, best for large static files.

### Conditional GET and range requests

`TypedResults.File` accepts `lastModified` and `entityTag` parameters. When supplied, the framework handles `If-None-Match` / `If-Modified-Since` and returns **304 Not Modified**, saving the whole transfer. `enableRangeProcessing: true` makes partial `206 Partial Content` responses work — essential for video/audio seeking and resumable downloads.

```csharp
return TypedResults.File(
    stream,
    contentType: "application/pdf",
    fileDownloadName: "report.pdf",
    lastModified: fileInfo.LastWriteTimeUtc,
    entityTag: new EntityTagHeaderValue($"\"{fileInfo.Length:x}-{fileInfo.LastWriteTimeUtc.Ticks:x}\""),
    enableRangeProcessing: true);
```

### Senior-level gotchas

- **Do not `using` the stream** before returning it — the framework disposes it after the body is flushed. Wrapping in a `using` closes the stream before the response writer touches it and you get a `ObjectDisposedException` mid-response with a truncated body.
- **Avoid `ReadAllBytesAsync` / `MemoryStream` of the full file.** Every request allocates the whole file on the LOH; a 10 MB PDF × 50 concurrent clients = 500 MB LOH pressure and Gen-2 GC pauses. Always pipe from source (`BlobClient.OpenReadAsync`, `File.OpenRead`, S3 `GetObjectStreamAsync`) to response.
- **Set `Content-Length` when known.** Some helpers (`TypedResults.File(stream,...)`) will try to set it from `stream.Length`; non-seekable streams (network, gRPC) can't, so clients fall back to chunked encoding and can't show progress bars. If you know the size, pass it explicitly.
- **MIME sniffing.** Browsers will sniff content and possibly render a "downloaded" HTML as a page. Send `X-Content-Type-Options: nosniff` for user-uploaded files.
- **Authorization is per-endpoint, not per-file.** A user who can hit `/report/{id}` gets *that* file — always authorize by `id` ownership inside the handler, not just by `[Authorize]` attribute.
- **Path traversal.** Never concatenate user input into `File.OpenRead(path)` without `Path.GetFullPath` + base-path containment check. `PhysicalFile` does not sandbox you.

**Sources:**
- [Create responses in Minimal API applications — Microsoft Learn (aspnetcore-10.0)](https://learn.microsoft.com/en-us/aspnet/core/fundamentals/minimal-apis/responses?view=aspnetcore-10.0)

## Q28. How to accept a file in an API Endpoint?

Client uploads files as `multipart/form-data`. ASP.NET Core surfaces each part as **`IFormFile`** (single) or **`IFormFileCollection`** (multiple).

### Controllers

```csharp
[HttpPost("upload")]
[RequestSizeLimit(100_000_000)]                        // body cap
[RequestFormLimits(MultipartBodyLengthLimit = 100_000_000)]
public async Task<IResult> Upload(
    [FromForm] IFormFile file,
    [FromForm] string? description,
    CancellationToken ct)
{
    if (file.Length == 0) return TypedResults.BadRequest("Empty file.");

    await using var source = file.OpenReadStream();
    await using var target = File.Create(Path.Combine("/uploads", Path.GetRandomFileName()));
    await source.CopyToAsync(target, ct);

    return TypedResults.Ok(new { file.FileName, file.Length });
}
```

### Minimal APIs (.NET 8+, including .NET 10)

```csharp
app.MapPost("/upload", async (IFormFile file, CancellationToken ct) =>
{
    await using var stream = file.OpenReadStream();
    // ... persist ...
    return TypedResults.Ok(new { file.FileName, file.Length });
})
.DisableAntiforgery();    // only if the endpoint is not browser-origin
```

**Since .NET 8 (still the rule in .NET 10): Minimal API endpoints that bind `IFormFile`/`IFormFileCollection` require the antiforgery middleware** (`builder.Services.AddAntiforgery()` + `app.UseAntiforgery()`). If you forget, the app throws on startup. Non-browser clients (mobile, server-to-server) that can't carry an antiforgery token should call `.DisableAntiforgery()` on the endpoint — but only when the caller is authenticated by some other mechanism (JWT, API key), because this opts out of CSRF protection.

### Size limits — a stack of knobs

Uploads fail silently if you only tune one layer. You typically need all of these aligned:

| Layer | Setting | Purpose |
|---|---|---|
| Kestrel | `KestrelServerOptions.Limits.MaxRequestBodySize` | Connection-level body cap |
| Forms | `FormOptions.MultipartBodyLengthLimit` | Multipart parser cap |
| Forms | `FormOptions.ValueLengthLimit` | Per-field non-file cap |
| Endpoint | `[RequestSizeLimit(...)]` / `[DisableRequestSizeLimit]` | Per-endpoint override |
| Endpoint | `[RequestFormLimits(...)]` | Per-endpoint form override |
| Reverse proxy | nginx `client_max_body_size`, IIS `maxAllowedContentLength` | Often the silent killer in prod |

```csharp
builder.Services.Configure<FormOptions>(o =>
{
    o.MultipartBodyLengthLimit = 200L * 1024 * 1024; // 200 MB
    o.MemoryBufferThreshold    = 65 * 1024;          // before spilling to disk
});

builder.WebHost.ConfigureKestrel(o =>
{
    o.Limits.MaxRequestBodySize = 200L * 1024 * 1024;
});
```

### Streaming large uploads

`IFormFile` buffers each part: first to memory up to `MemoryBufferThreshold` (~65 KB), then to a temp file on disk. That's fine for small files but wastes IO for gigabyte uploads. For truly large files, read the multipart **manually** with `MultipartReader` and forward straight to the destination (blob storage, disk, S3) without touching temp files:

```csharp
app.MapPost("/upload/large", async (HttpRequest request, CancellationToken ct) =>
{
    if (!request.HasFormContentType ||
        !MediaTypeHeaderValue.TryParse(request.ContentType, out var mediaType) ||
        string.IsNullOrWhiteSpace(mediaType.Boundary.Value))
    {
        return Results.BadRequest("Expected multipart/form-data");
    }

    var boundary = HeaderUtilities.RemoveQuotes(mediaType.Boundary.Value).Value!;
    var reader   = new MultipartReader(boundary, request.Body);

    MultipartSection? section;
    while ((section = await reader.ReadNextSectionAsync(ct)) is not null)
    {
        if (!ContentDispositionHeaderValue.TryParse(section.ContentDisposition, out var disposition)
            || !disposition!.DispositionType.Equals("form-data")
            || string.IsNullOrEmpty(disposition.FileName.Value))
        {
            continue;
        }

        // Stream straight into blob storage — no temp file, no LOH.
        await blobClient.UploadAsync(section.Body, overwrite: true, ct);
    }

    return Results.Ok();
})
.DisableAntiforgery();
```

### Senior-level gotchas

- **Never trust `IFormFile.FileName` or `ContentType`.** Both are attacker-controlled. Validate by magic bytes (PDF: `%PDF-`, PNG: `\x89PNG`), generate your own storage key (GUID + known extension), and set `Content-Disposition: attachment` when serving it back.
- **`IFormFile.OpenReadStream()` returns a seekable stream backed by the temp file.** That makes validation (e.g., image dimensions) cheap, but means the disk is already touched — a DoS vector if size limits are wrong.
- **Antivirus.** Route uploads through an AV gateway (ClamAV sidecar, Defender for Storage) before your domain logic trusts the bytes.
- **`async void` / fire-and-forget in upload pipelines** silently eat exceptions. Always `await` the disposal and copy.
- **Don't hold the full file in a `MemoryStream` "to validate then save"** — you're back to LOH problems. Compute hash / magic-byte checks as the stream flows through (`CryptoStream` pattern).
- **Distinct quotas matter.** A chat app with 5 MB image limits and a bulk-import endpoint with 1 GB files should not share the same per-app Kestrel cap — use per-endpoint `[RequestSizeLimit]`.
- **CSRF + browser upload.** `.DisableAntiforgery()` is *not* safe for cookie-authenticated endpoints called from a browser. Issue a token via `IAntiforgery.GetAndStoreTokens(HttpContext)` and require it on form posts.

## Q29. How to access query string parameters in an API Endpoint?

Four ways, ordered from most idiomatic to most manual.

### 1. Minimal APIs — implicit binding

For primitives, enums, and types with a `TryParse`/`BindAsync` static method, parameter names are matched to query keys automatically. No attribute needed.

```csharp
app.MapGet("/products", (string? search, int page = 1, int pageSize = 20) =>
    TypedResults.Ok(new { search, page, pageSize }));

// GET /products?search=shoe&page=2
```

Use `[FromQuery]` only to **force** the source or **rename** the key:

```csharp
app.MapGet("/products", ([FromQuery(Name = "q")] string? search) => ...);
```

### 2. Minimal APIs — `[AsParameters]` for DTO-style binding

When you have many query params, group them into a record and flatten with `[AsParameters]`:

```csharp
public readonly record struct ProductQuery(
    string? Search,
    int Page = 1,
    int PageSize = 20,
    string? Sort = "name");

app.MapGet("/products", ([AsParameters] ProductQuery q) =>
    TypedResults.Ok(q));

// GET /products?search=shoe&page=2&pageSize=50&sort=price
```

Important caveat (true in .NET 10): `[AsParameters]` **flattens** the record — every property becomes a top-level parameter; it does **not** do nested/recursive binding. Minimal APIs don't support true complex-object query binding the way MVC does — you either flatten with `[AsParameters]` or write a `static TryParse`/`BindAsync` on the type for custom parsing.

### 3. Controllers — `[FromQuery]`

```csharp
public record ProductQuery([FromQuery] string? Search,
                           [FromQuery] int Page = 1,
                           [FromQuery] int PageSize = 20);

[HttpGet]
public IActionResult Get([FromQuery] ProductQuery q) => Ok(q);
```

MVC controllers support deep complex-object binding natively — no `[AsParameters]` needed.

### 4. Raw access — `HttpContext.Request.Query`

When keys are dynamic (e.g., arbitrary filter names) or you need to introspect everything:

```csharp
app.MapGet("/search", (HttpContext ctx) =>
{
    IQueryCollection q = ctx.Request.Query;
    StringValues ids = q["id"];          // ?id=1&id=2  → ["1","2"]
    string? raw      = q["q"].ToString(); // empty string if missing, never null
    return TypedResults.Ok(new { count = q.Count });
});
```

`IQueryCollection` keys are case-insensitive; values are `StringValues` which is an array-of-strings aware struct (handles `?id=1&id=2` naturally).

### Arrays / repeated keys

Both Minimal APIs and Controllers will bind `?tag=red&tag=blue` into `string[] tag` or `List<string> tag`. Comma-separated (`?tag=red,blue`) is **not** parsed by default — write a `TryParse` or a custom binder.

### Senior-level gotchas

- **Missing vs empty.** `?q=` gives you `""`, not `null`. Use a nullable reference type and check `string.IsNullOrEmpty`.
- **`StringValues` pitfalls.** Implicit conversion to `string` joins with commas. For `?id=1&id=2`, `ctx.Request.Query["id"].ToString()` returns `"1,2"`, not `"1"`. Use `.ToArray()`.
- **URL encoding.** `?q=a%26b` decodes to `a&b`. Do not double-decode.
- **Required vs optional.** Non-nullable primitives are **required** — a missing value yields a 400 before your handler runs. Give defaults or use `int?`.
- **`[AsParameters]` with `[FromQuery]` properties makes all properties required** — a long-standing quirk. Use defaults or nullable types to work around it.
- **Don't bind `[FromQuery]` onto a `Dictionary<string,string>`** in Minimal APIs expecting arbitrary filter bags — that isn't supported. Use `HttpContext.Request.Query`.
- **Filter / sort injection.** Never interpolate query values into SQL or LINQ `DynamicLinq` without an allowlist — `?sort=password` should not let the client reorder by arbitrary columns.

- [Parameter binding in Minimal API applications — Microsoft Learn (aspnetcore-10.0)](https://learn.microsoft.com/en-us/aspnet/core/fundamentals/minimal-apis/parameter-binding?view=aspnetcore-10.0)

## Q30. How to get current logged-in user information?

The authenticated caller is exposed as a **`ClaimsPrincipal`** after authentication middleware (`UseAuthentication()`) has populated `HttpContext.User`. Every piece of identity info — user id, name, email, roles, tenant, custom scopes — is a claim on that principal.

### Controllers

```csharp
[Authorize]
public class OrdersController : ControllerBase
{
    [HttpGet("me")]
    public IActionResult Me()
    {
        if (!User.Identity?.IsAuthenticated ?? true)
            return Unauthorized();

        var userId = User.FindFirstValue(ClaimTypes.NameIdentifier); // or "sub" for JWT
        var email  = User.FindFirstValue(ClaimTypes.Email);
        var roles  = User.FindAll(ClaimTypes.Role).Select(c => c.Value);

        return Ok(new { userId, email, roles });
    }
}
```

`ControllerBase.User` is a shortcut for `HttpContext.User`.

### Minimal APIs

Inject `ClaimsPrincipal` directly as a handler parameter — the framework resolves it from `HttpContext.User`:

```csharp
app.MapGet("/me", (ClaimsPrincipal user) =>
{
    var userId = user.FindFirstValue(ClaimTypes.NameIdentifier);
    return TypedResults.Ok(new { userId, user.Identity!.Name });
})
.RequireAuthorization();
```

### Accessing the user in services (not in controllers)

Controllers and Minimal API handlers get the `HttpContext` for free; services don't. Register **`IHttpContextAccessor`** and inject it:

```csharp
builder.Services.AddHttpContextAccessor();

public sealed class CurrentUser(IHttpContextAccessor accessor) : ICurrentUser
{
    private readonly ClaimsPrincipal? _user = accessor.HttpContext?.User;

    public Guid? Id =>
        Guid.TryParse(_user?.FindFirstValue(ClaimTypes.NameIdentifier), out var id)
            ? id : null;

    public bool IsAuthenticated => _user?.Identity?.IsAuthenticated == true;
    public bool IsInRole(string role) => _user?.IsInRole(role) == true;
}

builder.Services.AddScoped<ICurrentUser, CurrentUser>();
```

This pattern — a tiny `ICurrentUser` scoped wrapper — keeps domain/application services free of the `Microsoft.AspNetCore.Http` reference and is trivial to stub in unit tests.

### Senior-level gotchas

- **Don't capture `HttpContext` in a singleton.** `IHttpContextAccessor` is safe (it reads ambient state via `AsyncLocal<T>`), but if you grab `_accessor.HttpContext` in a constructor you bind yourself to whichever request happened to be in-flight at DI construction time. Resolve it per call.
- **`IHttpContextAccessor.HttpContext` is `null` outside a request** — `BackgroundService`, `IHostedService`, channel consumers, timers. If your background worker needs user context, pass it in via the enqueued message, not by reaching into `HttpContextAccessor`.
- **JWT claim-type remapping.** The legacy `JwtBearerHandler` / `JwtSecurityTokenHandler` rewrites `sub` → `ClaimTypes.NameIdentifier`, `email` → `ClaimTypes.Email`, etc. The modern `JsonWebTokenHandler` (default for new templates) can skip this. If you want raw JWT claim names (`sub`, `aud`, etc.), set `JsonWebTokenHandler.DefaultMapInboundClaims = false` (or `JwtBearerOptions.MapInboundClaims = false`). Be consistent across services or you'll hunt "why is `User.FindFirst("sub")` null in one API and populated in another" for an afternoon.
- **`User.Identity.Name`** comes from whichever claim you configured as `NameClaimType` in `TokenValidationParameters`. Default is `ClaimTypes.Name` — often unpopulated in token-based auth. Set `NameClaimType = "preferred_username"` (or similar) for a predictable value.
- **Role checks (`User.IsInRole`, `[Authorize(Roles="admin")]`) depend on `RoleClaimType`.** Same story as above — wire it explicitly in `TokenValidationParameters`.
- **Mutating the principal mid-request** (`HttpContext.User = newPrincipal`) doesn't persist beyond the request and is almost always a mistake — prefer re-issuing tokens / cookies.
- **`User` is never null**, but `User.Identity` can be an anonymous `ClaimsIdentity` — always check `IsAuthenticated` before trusting any claim.
- **Keep secrets out of claims.** Claims flow into logs (especially structured request logs); stuff like API keys, PII, or tokens bundled as claims will leak.
