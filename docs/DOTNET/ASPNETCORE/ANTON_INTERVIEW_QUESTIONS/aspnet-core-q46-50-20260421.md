## Q46. What are the `IHostApplicationLifetime` events and how can you attach callbacks to each?

`IHostApplicationLifetime` is the generic-host abstraction that surfaces the three lifecycle moments of the process. It's a singleton service injected from DI; in Minimal APIs it's also exposed as `app.Lifetime`.

The three events are exposed as `CancellationToken` properties, not CLR events — you register callbacks with `.Register()`:

| Event | When it fires | Typical use |
|---|---|---|
| `ApplicationStarted` | After `IHost.StartAsync()` completes. All hosted services have started; Kestrel is accepting traffic. | Warm caches, emit a "ready" trace, mark a readiness probe. |
| `ApplicationStopping` | At the *start* of graceful shutdown, before `IHostedService.StopAsync` runs. | Stop accepting new work, flip readiness to false, signal background loops. |
| `ApplicationStopped` | After all hosted services have stopped. | Final telemetry flush, close external sinks. |

Plus a method — `StopApplication()` — to trigger shutdown from within the process (the in-process equivalent of SIGTERM).

### Attaching callbacks

```csharp
public sealed class LifetimeReporter : IHostedService
{
    private readonly IHostApplicationLifetime _lifetime;
    private readonly ILogger<LifetimeReporter> _log;

    public LifetimeReporter(IHostApplicationLifetime lifetime, ILogger<LifetimeReporter> log)
        => (_lifetime, _log) = (lifetime, log);

    public Task StartAsync(CancellationToken ct)
    {
        _lifetime.ApplicationStarted.Register(() => _log.LogInformation("ready"));
        _lifetime.ApplicationStopping.Register(() => _log.LogInformation("draining"));
        _lifetime.ApplicationStopped.Register(() => _log.LogInformation("bye"));
        return Task.CompletedTask;
    }

    public Task StopAsync(CancellationToken ct) => Task.CompletedTask;
}
```

Minimal API style:

```csharp
var app = builder.Build();

app.Lifetime.ApplicationStarted.Register(() => cache.PrimeAsync().GetAwaiter().GetResult());
app.Lifetime.ApplicationStopping.Register(() => readinessProbe.MarkNotReady());

app.MapPost("/shutdown", (IHostApplicationLifetime l) =>
{
    l.StopApplication();        // programmatic, graceful shutdown
    return Results.Accepted();
});
```

### `IHostApplicationLifetime` vs `IHostLifetime`

Different abstractions — don't confuse them:

- **`IHostApplicationLifetime`** — the *application-facing* signal: "we're starting / stopping / stopped." This is what your code consumes.
- **`IHostLifetime`** — the *host-facing* driver: controls *how* the host listens for shutdown signals. Implementations include `ConsoleLifetime` (Ctrl+C, SIGINT, SIGTERM), `WindowsServiceLifetime`, `SystemdLifetime`. You swap this only when embedding the host into a non-standard process.

### Senior-level gotchas

- **Callbacks are synchronous.** `.Register(Action)` — there's no `RegisterAsync`. For async cleanup, put the work in a hosted service's `StopAsync`, which *is* awaited by the host.
- **`ApplicationStopping` runs inside the shutdown budget.** `HostOptions.ShutdownTimeout` (default 30s) is the *total* window for stopping. Long operations in a stopping callback eat into every other service's drain time.
- **Don't use `Task.Run` inside a stopping callback.** The host may exit before the task completes — you leak the work and corrupt in-flight state.
- **`BackgroundService` already has a stopping signal.** The `stoppingToken` passed to `ExecuteAsync` is cancelled when stopping begins. Wiring into `ApplicationStopping` from a background service is redundant and usually wrong.
- **`StopApplication()` is non-blocking.** It signals; it does not wait. If you need to await shutdown, you're probably in the wrong place — usually the calling endpoint should just return 202 and let the host wind down.
- **Kestrel request draining** is separate from hosted-service draining. `HostOptions.ShutdownTimeout` ends the hosted-service window; Kestrel's `KestrelServerOptions.Limits.RequestHeadersTimeout` and `RequestDrainTimeout` govern connection-level behavior.

**Sources:**
- [.NET Generic Host — IHostApplicationLifetime](https://learn.microsoft.com/en-us/dotnet/core/extensions/generic-host#ihostapplicationlifetime)

## Q47. How can you stop a `BackgroundService` from execution?

The framework contract: `BackgroundService` is stopped by **cancelling the `stoppingToken` passed to `ExecuteAsync`**. Everything else is either a way to *cause* that cancellation or an escape hatch for misbehaving services.

### 1. Cooperative cancellation (the only correct design)

Your `ExecuteAsync` observes `stoppingToken`. When the host begins shutdown, the token is cancelled; your loop exits.

```csharp
public sealed class OutboxDispatcher : BackgroundService
{
    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        using var timer = new PeriodicTimer(TimeSpan.FromSeconds(5));
        while (await timer.WaitForNextTickAsync(stoppingToken))
        {
            await DispatchBatchAsync(stoppingToken);
        }
        // WaitForNextTickAsync returns false on cancellation → clean exit, no exception
    }
}
```

Plumb the token into every awaitable: `await Task.Delay(1000, stoppingToken)`, `await db.SaveChangesAsync(stoppingToken)`, `await channel.Reader.ReadAsync(stoppingToken)`.

### 2. Host-level shutdown

Call `IHostApplicationLifetime.StopApplication()` from anywhere. The host walks `IHostedService.StopAsync` in *reverse* registration order; for `BackgroundService`, the base `StopAsync` cancels `stoppingToken` and awaits `ExecuteAsync` with the caller's shutdown ceiling.

```csharp
app.MapPost("/admin/shutdown", (IHostApplicationLifetime l) =>
{
    l.StopApplication();
    return Results.Accepted();
}).RequireAuthorization("AdminOnly");
```

### 3. Per-service stop

If you hold a reference to the service (singleton resolved from DI), call `StopAsync` directly. Stops *only that service*.

```csharp
var dispatchers = app.Services.GetServices<IHostedService>().OfType<OutboxDispatcher>();
await dispatchers.Single().StopAsync(CancellationToken.None);
```

Uncommon — usually a sign the service should instead be managed with the signal pattern below.

### 4. Signal pattern (linked token)

When business logic — not host shutdown — needs to stop a service, share a cancellation source between the caller and the service. Link it with `stoppingToken` so host shutdown *also* stops the service.

```csharp
public sealed class StopSignal
{
    private readonly CancellationTokenSource _cts = new();
    public CancellationToken Token => _cts.Token;
    public void Fire() => _cts.Cancel();
}

public sealed class LicenseCheck(StopSignal signal) : BackgroundService
{
    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        using var linked = CancellationTokenSource.CreateLinkedTokenSource(stoppingToken, signal.Token);
        try { await DoWorkAsync(linked.Token); } catch (OperationCanceledException) { }
    }
}
```

### Graceful cleanup via `StopAsync`

Override `StopAsync` for work that must run *after* `stoppingToken` is cancelled but before the host disposes the container.

```csharp
public override async Task StopAsync(CancellationToken cancellationToken)
{
    await _pendingBatch.FlushAsync(cancellationToken);
    await base.StopAsync(cancellationToken);   // cancels stoppingToken, awaits ExecuteAsync
}
```

### Senior-level gotchas

- **Ignoring `stoppingToken` blocks shutdown.** The host waits up to `HostOptions.ShutdownTimeout` (default 30s), then moves on — in containers/K8s that means SIGKILL after the grace period with truncated work.
- **`await Task.Delay(...)` without a token** is the most common offender. In a long-lived loop, that call becomes uncancellable for the delay duration.
- **`BackgroundServiceExceptionBehavior` defaults to `StopHost` since .NET 6.** An unhandled exception in `ExecuteAsync` crashes the entire host. That's the right default (fail fast), but means you must catch-and-log inside the loop if you want resilience:

    ```csharp
    while (await timer.WaitForNextTickAsync(stoppingToken))
    {
        try { await DoWorkAsync(stoppingToken); }
        catch (OperationCanceledException) { throw; }  // respect shutdown
        catch (Exception ex) { _log.LogError(ex, "iteration failed"); }
    }
    ```

    Only set `Ignore` if you consciously want a silent-failure service (rare — log first).
- **Scoped deps in a singleton.** `BackgroundService` is singleton; `DbContext`, `IMediator`, per-request services must be resolved with `IServiceScopeFactory.CreateScope()` *per iteration*, not cached in a field.
- **`PeriodicTimer` over `Task.Delay` loops.** No drift, supports cancellation directly, disposes cleanly. Since .NET 8, `WaitForNextTickAsync` is the idiomatic tick primitive.
- **`StopAsync` has its own ceiling** — the `cancellationToken` parameter is your cleanup budget. Don't start brand-new long-running work in it.
- **Don't `await` the application lifetime from inside `StopAsync`.** You're *in* shutdown — there's nothing left to wait for.

**Sources:**

- [Worker Services in .NET](https://learn.microsoft.com/en-us/dotnet/core/extensions/workers)

## Q48. What are the default return types you can use in controller actions?

Five forms, from least to most expressive:

| Return form | Implicit 200 wrap? | Typed OpenAPI? | Mixed status codes? |
|---|---|---|---|
| `T` (POCO) | Yes | Yes | No |
| `ActionResult<T>` | Yes (implicit operator) | Yes | Yes |
| `IActionResult` / `ActionResult` | You construct the result | No, unless `[ProducesResponseType]` | Yes |
| `Results<T1, T2, …>` (TypedResults union) | You return `TypedResults.Xxx(...)` | Yes — one arm per status | Yes |
| Async wrappers (`Task<…>`, `ValueTask<…>`) | Mirrors inner type | Mirrors inner type | Mirrors inner type |

Examples:

```csharp
// 1. POCO — framework wraps in 200 OK
[HttpGet("{id:int}")]
public async Task<Order> Get(int id, CancellationToken ct)
    => await _orders.GetAsync(id, ct);

// 2. ActionResult<T> — typed + mixed statuses via implicit conversion
[HttpGet("{id:int}")]
public async Task<ActionResult<Order>> Get(int id, CancellationToken ct)
{
    var o = await _orders.FindAsync(id, ct);
    return o is null ? NotFound() : o;        // both arms convert implicitly
}

// 3. IActionResult — most flexible, but metadata is attribute-driven
[HttpPost]
[ProducesResponseType<Order>(StatusCodes.Status201Created)]
[ProducesResponseType(StatusCodes.Status409Conflict)]
public async Task<IActionResult> Create(CreateOrderRequest req, CancellationToken ct)
{
    var result = await _orders.CreateAsync(req, ct);
    return result.IsSuccess
        ? CreatedAtAction(nameof(Get), new { id = result.Value.Id }, result.Value)
        : Conflict();
}

// 4. Results<...> — compile-time-checked unions, no attributes required
[HttpGet("{id:int}")]
public async Task<Results<Ok<Order>, NotFound>> GetTyped(int id, CancellationToken ct)
{
    var o = await _orders.FindAsync(id, ct);
    return o is null ? TypedResults.NotFound() : TypedResults.Ok(o);
}
```

### When to use which

- **`ActionResult<T>`** — modern default in new controllers. Best ergonomics/metadata trade-off without attribute noise.
- **`Results<...>`** (from `Microsoft.AspNetCore.Http.HttpResults`) — when the result set is small and fixed, and you want strict compile-time OpenAPI arms. Works in both controllers and Minimal APIs.
- **`IActionResult`** — when you need `File(...)`, `Redirect(...)`, streaming, or a delegated pipeline where the result type isn't fixed.
- **Plain `T`** — only for endpoints with *exactly one* successful outcome and no error path surfaced as a status code (rare in real APIs; usually you still want to return 404 for missing resources).

### Senior-level gotchas

- **Returning a domain entity leaks persistence concerns** — EF change-tracking proxies, lazy-loading graphs, navigation cycles that crash `System.Text.Json`. Always project to a DTO or record.
- **`ActionResult<T>` forbids `T = IActionResult`** — the compiler catches it, but not the subtler case where `T` is an abstract base type you didn't mean to expose as the schema.
- **Implicit 204 for `null` is brittle.** A controller that returns `null` for a reference `T` produces a `204 No Content` under some formatter configurations and an empty `200` under others. Be explicit: `NotFound()` or a ProblemDetails payload.
- **`[ApiController]` auto-400s on invalid model state** via ProblemDetails. Don't write `if (!ModelState.IsValid) return BadRequest(ModelState)` — it's noise and you lose the unified ProblemDetails schema.
- **`CreatedAtAction` uses the method *name*, not the route name.** Typo = runtime failure when it tries to build the Location header.
- **`ValueTask<ActionResult<T>>`** is not a free win. The allocation savings vanish if the result is awaited more than once — and for a single-await endpoint the win is marginal versus the added cognitive cost.
- **Exception-driven control flow** (`throw new NotFoundException()` intercepted by middleware) is convenient but expensive — exceptions cost orders of magnitude more than a typed `NotFound()`. Reserve for truly exceptional paths; use typed results for expected branches.
- **Don't `async void`** in controllers. Ever. Breaks exception handling, breaks the return-type contract, breaks your life.

**Sources:**
- [Controller action return types in ASP.NET Core](https://learn.microsoft.com/en-us/aspnet/core/web-api/action-return-types)
- [TypedResults / Results<> unions](https://learn.microsoft.com/en-us/dotnet/api/microsoft.aspnetcore.http.httpresults)

## Q49. What is model binding and how does it work in ASP.NET Core controllers?

Model binding is the pipeline that materializes action-method *parameters* from raw HTTP data — route values, query string, headers, body, form fields, DI services. It runs after routing and before the action executes; filters and validation plug in around it.

### The pipeline, phase by phase

1. **Binding source selection.** For each parameter, the framework decides *where* the value comes from. Under `[ApiController]`, the inference table is:

    | Parameter shape | Inferred source |
    |---|---|
    | Token matches a route template token | `[FromRoute]` |
    | Primitive / `IParsable<T>` / enum not in the route | `[FromQuery]` |
    | Complex reference type | `[FromBody]` |
    | `IFormFile` / `IFormFileCollection` / `IFormCollection` | `[FromForm]` |
    | Type registered in DI (.NET 7+) | `[FromServices]` |
    | `CancellationToken`, `HttpContext`, `HttpRequest`, `HttpResponse`, `ClaimsPrincipal` | Special-cased |

    Explicit attributes override inference.

2. **Value providers** supply raw strings/bytes: `RouteValueProvider`, `QueryStringValueProvider`, `FormValueProvider`, `HeaderValueProvider`.

3. **Model binders** convert raw values to CLR types:
    - `SimpleTypeModelBinder` — primitives, enums, `IParsable<T>`.
    - `ComplexObjectModelBinder` — recurses into POCO properties, binding each from its own (possibly different) source.
    - `ArrayModelBinder` / `CollectionModelBinder` / `DictionaryModelBinder`.
    - `BodyModelBinder` — delegates to input formatters.
    - Custom `IModelBinder` for bespoke formats (see Q51).

4. **Input formatters** deserialize the body for `[FromBody]`. `SystemTextJsonInputFormatter` is default; XML is opt-in (`.AddXmlDataContractSerializerFormatters()`). Content negotiation picks one by `Content-Type`.

5. **Validation** runs after binding, populating `ModelState` via `DataAnnotations` (`[Required]`, `[Range]`, etc.). `[ApiController]` auto-returns 400 ProblemDetails if `ModelState.IsValid == false`.

### Concrete example

```csharp
// POST /orders/2026?priority=high
// Body: { "items": [...] }
[HttpPost("{year:int}")]
public async Task<ActionResult<Order>> Create(
    int year,                         // [FromRoute] inferred
    [FromQuery] string? priority,     // explicit for clarity
    CreateOrderRequest request,       // [FromBody] inferred (complex type)
    IOrderService orders,             // [FromServices] inferred (DI-registered)
    CancellationToken ct)             // special-cased
{
    var result = await orders.CreateAsync(year, priority, request, ct);
    return result;                    // 200 OK via implicit conversion
}
```

### Minimal APIs bind differently

- No `ModelState`, no `[ApiController]`-style inference for forms.
- Complex types default to body; primitives to route/query; `IFormFile` requires explicit `[FromForm]` and `.DisableAntiforgery()` (or a token) in .NET 10.
- `[AsParameters]` groups related parameters into a struct/record for organization.
- Custom primitives bind via `IParsable<T>` / `ISpanParsable<T>` — or the legacy static `TryParse` / `BindAsync` hooks.

### Senior-level gotchas

- **Only one `[FromBody]` parameter per action.** The body stream is consumed once. Need "two bodies"? You're looking at `multipart/form-data` and `[FromForm]`.
- **`[FromQuery]` on a complex type binds *properties individually*** (`?Filter.Name=foo&Filter.Age=30`). It does **not** parse a JSON blob in the URL.
- **Two `JsonOptions` types.** Minimal APIs use `Microsoft.AspNetCore.Http.Json.JsonOptions` configured via `ConfigureHttpJsonOptions`; controllers use `Microsoft.AspNetCore.Mvc.JsonOptions` configured via `services.AddControllers().AddJsonOptions(...)`. Forgetting this leads to "why is my naming policy different across endpoints?"
- **Empty body binds to `null`** even with NRTs on — the compiler doesn't know the framework might hand you `null`. Combine `[FromBody]` with `[Required]`, or trust model validation to reject it.
- **Implicit `[FromServices]` is .NET 7+.** Before that, you attributed it. If a type isn't DI-registered, ASP.NET Core tries to treat it as `[FromQuery]` and produces confusing binding errors — double-check your `AddXxx` call.
- **`[FromHeader]` is case-insensitive for the header name** but the property/parameter name needs to match the header or use `Name = "..."`. `X-Correlation-Id` → `CorrelationId` param works; custom names need the attribute.
- **`IParsable<T>` in C# 11+ (mature in C# 14) makes strongly typed IDs first-class.** Implement `IParsable<OrderId>` and the binder uses it automatically — no custom `IModelBinder` required.
- **Binders run before filters.** Authorization can't sanitize what's already bound to your model. Do input validation; don't rely on an auth filter to "clean" a payload.
- **Large forms**: `[FromForm]` buffers the *entire* body. For uploads above tens of MB, bypass model binding and read `HttpRequest.ReadFormAsync` (or a streaming `MultipartReader`) directly.

- [Parameter binding in Minimal API applications](https://learn.microsoft.com/en-us/aspnet/core/fundamentals/minimal-apis/parameter-binding)

## Q50. Explain the difference between `[FromBody]`, `[FromQuery]`, `[FromRoute]`, and `[FromForm]` attributes

All four are `BindingSourceAttribute`s. They tell the binder *which value provider* to use for that parameter. The difference is what part of the request they read and how that part is parsed.

| Attribute | Source | Content-Type | Typical use |
|---|---|---|---|
| `[FromRoute]` | URL path tokens | — | Resource identifiers in `/orders/{id}` |
| `[FromQuery]` | URL query string | — | Filters, paging, optional params |
| `[FromBody]` | Request body via input formatter | `application/json` (or XML if configured) | JSON/XML payloads |
| `[FromForm]` | Form fields + files | `application/x-www-form-urlencoded` or `multipart/form-data` | HTML forms, file uploads |

One endpoint using three of them:

```csharp
[HttpPost("tenants/{tenantId:guid}/orders")]
public async Task<ActionResult<Order>> Create(
    [FromRoute] Guid tenantId,
    [FromQuery] string? correlationId,
    [FromBody] CreateOrderRequest request,
    CancellationToken ct)
{
    // POST /tenants/0f0…/orders?correlationId=abc
    // Body: { "items": [...] }
}
```

File upload with `[FromForm]`:

```csharp
[HttpPost("upload")]
[Consumes("multipart/form-data")]
public async Task<IActionResult> Upload(
    [FromRoute] Guid tenantId,
    [FromForm] IFormFile file,
    [FromForm] string category,
    CancellationToken ct)
{
    await using var s = file.OpenReadStream();
    await _blobs.SaveAsync(tenantId, category, s, ct);
    return NoContent();
}
```

### Why the distinction matters (under the hood)

- **`[FromBody]`** runs **input formatters** with content negotiation. Mismatched `Content-Type` → `415 Unsupported Media Type`. Body stream consumed once — **only one `[FromBody]` per action**.
- **`[FromQuery]` / `[FromRoute]`** use the **simple binding pipeline**: string → target type via `TypeConverter` or `IParsable<T>`. No formatter, no content negotiation.
- **`[FromForm]`** reads `HttpRequest.Form`, which forces the body to parse as form data. **You cannot combine `[FromForm]` with `[FromBody]` on the same action** — the body can be interpreted as one format at a time.

### Senior-level gotchas

- **`[FromQuery]` on complex types binds *properties*, not JSON.** `?Address.City=X&Address.Zip=Y` works; `?address={…}` does not.
- **`[FromRoute]` parameter names must match route tokens.** `int id` on an endpoint with no `{id}` token falls back to `[FromQuery]` under `[ApiController]`. Use `[FromRoute(Name = "id")]` if the names diverge.
- **`IFormFile` requires `[FromForm]`.** Putting it inside a `[FromBody]` JSON model silently fails — JSON has no concept of binary parts.
- **Anti-forgery + `[FromForm]`.** On state-changing verbs in Razor/MVC contexts, `[FromForm]` triggers anti-forgery token validation. Pure Web APIs usually omit anti-forgery middleware, but **do not disable it blindly** on cookie-authenticated endpoints — you reopen CSRF. For Minimal APIs in .NET 7+, anti-forgery is the default when `[FromForm]` + cookies meet; use `.DisableAntiforgery()` only on endpoints that aren't browser-reachable with credentials.
- **Missing body with non-nullable `[FromBody]` parameter** binds to `null` despite NRTs. Annotate with `[Required]` or let ProblemDetails 400 handle it via validation.
- **Buffering cost of `[FromForm]`** is the entire body. For large uploads use `MultipartReader` against `HttpRequest.Body` directly — don't rely on model binding to stream a 2 GB file.
- **`[Consumes("...")]` narrows formatters.** Add it when you need a crisp 415 instead of a cryptic binding failure.
- **Typed parameters beat string parameters.** Binding `Guid tenantId` rejects malformed input at the binder with a 400; binding `string tenantId` pushes the validation into your method body.
- **Minimal APIs parity** — same attributes exist, different inference rules. No `[FromForm]` inference without explicit attribute, and forms require `.DisableAntiforgery()` or a valid token on the endpoint.
- **`[FromHeader]` is sometimes the right answer.** If the value lives in a header (idempotency key, correlation ID), don't shoehorn it into the query string.
