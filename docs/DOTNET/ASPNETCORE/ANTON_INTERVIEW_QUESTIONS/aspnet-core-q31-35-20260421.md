# ASP.NET Core — Questions 31–35

_Source: `docs/DOTNET/ASPNETCORE/ANTON_INTERVIEW_QUESTIONS/aspnetcore-interview-questions.md`. Answers target .NET 10 / C# 14._

## Q31. How to inject dependencies in Minimal APIs?

Minimal APIs resolve dependencies **through the handler's parameter list**. The `RequestDelegateFactory` (and its source-generated counterpart, `RequestDelegateGenerator`, enabled via `<EnableRequestDelegateGenerator>true</EnableRequestDelegateGenerator>`) inspects each parameter and binds it from the appropriate source: route values, query string, body, headers, or the **request DI scope**. Since .NET 7, `[FromServices]` is optional — any parameter whose type is registered in the container is resolved automatically.

```csharp
builder.Services.AddScoped<IOrderService, OrderService>();
builder.Services.AddKeyedSingleton<IClock>("utc", new UtcClock());

app.MapPost("/orders",
    async (CreateOrderDto dto,                      // [FromBody] inferred (complex type, non-service)
           IOrderService orders,                    // [FromServices] inferred (registered in DI)
           [FromKeyedServices("utc")] IClock clock, // keyed services need the explicit attribute
           ClaimsPrincipal user,                    // special: current user
           CancellationToken ct) =>                 // special: request-aborted token
{
    var id = await orders.CreateAsync(dto, user.GetUserId(), clock.UtcNow, ct);
    return TypedResults.Created($"/orders/{id}", new { id });
});
```

**Inference rules** (the part that trips people up):

1. Built-in special types (`HttpContext`, `HttpRequest`, `HttpResponse`, `ClaimsPrincipal`, `CancellationToken`, `Stream`, `PipeReader`, `IFormFileCollection`, `IFormFile`) are bound from the request.
2. Primitive / string-parsable types come from route first, then query.
3. Types registered in DI come from the service container — no attribute needed.
4. Complex types not in DI come from the body (`[FromBody]`).
5. Ambiguous cases (a DTO also registered in DI, for example) require an explicit `[FromBody]` or `[FromServices]` — otherwise you get a startup exception.

**`AsParameters` for clean signatures.** When a handler grows too many parameters, group them into a `record struct` tagged `[AsParameters]`:

```csharp
public readonly record struct GetOrdersRequest(
    [FromRoute] Guid TenantId,
    [FromQuery] int Page,
    [FromServices] IOrderService Orders,
    CancellationToken Ct);

app.MapGet("/tenants/{tenantId:guid}/orders",
    ([AsParameters] GetOrdersRequest r) => r.Orders.ListAsync(r.TenantId, r.Page, r.Ct));
```

**Scope is per-request.** Each request opens its own `IServiceScope`; scoped services resolved in the handler live for that request. The same holds inside endpoint filters (`AddEndpointFilter`) — they receive the request's `HttpContext` and can resolve scoped services via `context.HttpContext.RequestServices`.

**Modern note (.NET 10):** RDG source generation is the AOT-friendly path. If you compile with `PublishAot=true`, all handler parameter types must be statically analyzable — runtime-only type registrations (e.g. `services.AddSingleton(sp => BuildFromReflection())`) still work, but reflection-heavy binding patterns do not, and you'll see `IL2026`/`IL3050` trim warnings at build. Keep handler signatures concrete and interfaces closed.

**Senior-level gotchas:**
- Don't inject `IServiceProvider` into a handler "just in case" — it's service-locator smell and defeats DI verification. If you need runtime resolution, inject a typed factory.
- `async void` handlers won't compile (Minimal APIs require `Task`/`ValueTask`/`IResult`), but `async Task` returning a plain `string` with a caught-but-swallowed exception still 200s — always propagate or map explicitly.
- A scoped service captured inside a lambda passed to `AddEndpointFilter<T>` registered as a **singleton** filter becomes a captive dependency — ASP.NET Core's scope validator catches this in Development but not in Production unless you enable it (`ValidateScopes = true`).
- When two endpoints in the same `MapGroup` need different DI lifetimes of the same abstraction, prefer **keyed services** (`[FromKeyedServices]`) over conditional resolution inside a factory.

**Sources:**
- https://learn.microsoft.com/aspnet/core/fundamentals/minimal-apis/parameter-binding?view=aspnetcore-10.0
- https://learn.microsoft.com/aspnet/core/fundamentals/minimal-apis?view=aspnetcore-10.0

## Q32. How would you structure your Minimal API endpoints?

The default Minimal API template puts everything in `Program.cs` — fine for a demo, an anti-pattern past ~10 endpoints. Three realistic structures, in order of preference:

### 1. Extension methods on `IEndpointRouteBuilder` + `MapGroup` (recommended)

One file per feature, each exposing a `MapXxx` extension that registers the feature's endpoints on a group. Cross-cutting metadata (auth, rate limiting, output cache, OpenAPI tags) lives at the group level; endpoint declarations stay lean.

```csharp
// Features/Orders/OrdersEndpoints.cs
public static class OrdersEndpoints
{
    public static IEndpointRouteBuilder MapOrders(this IEndpointRouteBuilder app)
    {
        var g = app.MapGroup("/api/orders")
                   .WithTags("Orders")
                   .RequireAuthorization()
                   .AddEndpointFilter<ProblemDetailsFilter>();

        g.MapGet("/{id:guid}", GetById).WithName("Orders.GetById");
        g.MapPost("/", Create).AddEndpointFilter<ValidationFilter<CreateOrderDto>>();
        g.MapDelete("/{id:guid}", Delete).RequireAuthorization("CanDeleteOrders");

        return app;
    }

    private static async Task<Results<Ok<OrderDto>, NotFound>> GetById(
        Guid id, IOrderService orders, CancellationToken ct) =>
        await orders.FindAsync(id, ct) is { } o ? TypedResults.Ok(o) : TypedResults.NotFound();

    private static Task<Created<Guid>> Create(
        CreateOrderDto dto, IOrderService orders, CancellationToken ct) => /* ... */;

    private static Task<NoContent> Delete(
        Guid id, IOrderService orders, CancellationToken ct) => /* ... */;
}

// Program.cs stays short
app.MapOrders();
app.MapProducts();
app.MapShipments();
```

Handlers are `static` — easier to unit-test by passing fakes directly, and no accidental `this` capture in filter chains.

### 2. Vertical-slice / feature folders

Organize by feature, not by technical layer. Each slice owns its endpoint, DTOs, validator, handler, and (if using MediatR/Wolverine) the command/query type:

```
src/
  Features/
    Orders/
      Create/
        CreateOrderDto.cs
        CreateOrderValidator.cs
        CreateOrderHandler.cs
        CreateOrderEndpoint.cs   // contains MapCreateOrder extension
      GetById/
        ...
    Products/
      ...
```

This scales far better than "Controllers/, Services/, Repositories/" once you have >30 features — changes stay local to one folder.

### 3. Endpoint-module interface (Carter, FastEndpoints)

```csharp
public interface IEndpointModule
{
    void MapEndpoints(IEndpointRouteBuilder app);
}

// at startup
foreach (var m in app.Services.GetServices<IEndpointModule>())
    m.MapEndpoints(app);
```

Clean, but requires reflection-based discovery (or manual registration), which **fights NativeAOT** unless you wire registrations explicitly. Carter/FastEndpoints both add conveniences on top (validation, response negotiation, endpoint-per-class) but at the cost of pulling in a framework. Prefer plain extension methods unless you're onboarding a team that needs more rails.

### Cross-cutting concerns

- Apply `RequireAuthorization`, `CacheOutput`, `RequireRateLimiting`, `WithOpenApi`, `Produces<T>`, `ProducesValidationProblem` at the **group** level, not per endpoint.
- Use `AddEndpointFilter<T>` for validation, problem-details normalization, tenant resolution. It's the Minimal API analog to MVC filters.
- Version APIs via groups: `app.MapGroup("/api/v1")` vs `/api/v2`, or use `Asp.Versioning.Http`.

**Modern note (.NET 10):** `TypedResults.Extensions` returning `Results<T1, T2, ...>` unions gives you per-endpoint OpenAPI response documentation for free — prefer it over `IResult`. Pair with the built-in `AddOpenApi()` (no more Swashbuckle for simple cases).

**Senior-level gotchas:**
- `MapGroup` metadata is **inherited**; opting out is asymmetric. `.AllowAnonymous()` on a specific endpoint under a `.RequireAuthorization()` group works, but overriding a group-level output-cache policy is awkward — make the group policy the least restrictive and tighten per endpoint.
- Don't register endpoints inside a hosted service or singleton constructor; `app.Map*` must be called on the builder before `app.Run()` or the endpoint data source is frozen.
- Reflection-based endpoint discovery (`Assembly.GetExportedTypes()`) breaks trimming — annotate discovery code with `[DynamicallyAccessedMembers]` and the assembly with `[assembly: DynamicDependency]`, or drop the pattern for AOT apps.
- If you share DTOs between Minimal APIs and MVC controllers, keep model-validation attributes DRY — MVC auto-validates on bind, Minimal APIs do not. Add an `IEndpointFilter` that runs the same `ValidationContext` or use FluentValidation's `AddFluentValidationAutoValidation`.

## Q33. What is Output Caching?

**Output caching** is the server-side middleware introduced in .NET 7 that stores **rendered HTTP responses** in a cache (in-memory by default, pluggable to Redis) keyed by request characteristics, and serves subsequent matching requests directly from the cache — short-circuiting the endpoint entirely. It is *distinct* from the older **response caching** middleware, which only emits `Cache-Control` headers hinting downstream caches; response caching doesn't store anything server-side.

### Wire-up

```csharp
builder.Services.AddOutputCache(options =>
{
    options.AddBasePolicy(b => b.Expire(TimeSpan.FromSeconds(10)));
    options.AddPolicy("Catalog", b => b
        .Expire(TimeSpan.FromMinutes(5))
        .SetVaryByQuery("page", "size")
        .Tag("catalog"));
});

var app = builder.Build();
app.UseOutputCache();     // must sit after UseRouting and before endpoints

app.MapGet("/products", (CatalogService c) => c.ListAsync())
   .CacheOutput("Catalog");

app.MapGet("/health", () => "ok")
   .CacheOutput(b => b.Expire(TimeSpan.FromSeconds(2)).NoCache()); // lambda per-endpoint

// MVC:
[OutputCache(PolicyName = "Catalog")]
public IActionResult Products() => View();
```

### What you get out of the box

- **Vary-by**: headers, query, route values, host, or arbitrary values via `SetVaryByValue((ctx) => ...)`.
- **Tag-based invalidation**: the big reason to pick Output Caching over response headers. Attach tags to responses, then evict whole groups:
  ```csharp
  app.MapPost("/products", async (ProductDto p, CatalogService c, IOutputCacheStore store, CancellationToken ct) =>
  {
      await c.CreateAsync(p, ct);
      await store.EvictByTagAsync("catalog", ct);   // bust all cached product pages
      return Results.Created($"/products/{p.Id}", p);
  });
  ```
- **Stampede protection**: concurrent requests for the same cache key wait for a single execution rather than all piling through.
- **Pluggable store**: default is in-process `MemoryCacheOutputCacheStore`; swap for Redis with `Microsoft.AspNetCore.OutputCaching.StackExchangeRedis` to share across instances.

### Output Caching vs Response Caching — quick framing

| | Output Caching | Response Caching |
|---|---|---|
| Storage | Server-side (memory or Redis) | No storage — only headers |
| Invalidation | Tags, programmatic eviction | Wait for TTL |
| Auth-aware | Configurable (off by default) | Respects `Cache-Control: private` |
| Use when | API reads, server-rendered pages, aggregated data | CDN/edge caching, static-ish assets |

### Modern note (.NET 10)

Output Caching is stable from .NET 7 onward; the Redis-backed store `Microsoft.AspNetCore.OutputCaching.StackExchangeRedis` reached stable on .NET 9 and ships for .NET 10. The policy API gained first-class support for `TypedResults` and no longer requires custom `IOutputCachePolicy` classes for common patterns.

**Senior-level gotchas:**
- **Authenticated requests are not cached by default.** The middleware looks at `HttpContext.User.Identity.IsAuthenticated` and bypasses. Enable per-user caching explicitly with a custom `IOutputCachePolicy` that sets `context.AllowCacheLookup = true` and varies by user id/claim — and be very sure you've keyed it correctly, or you'll serve one user's data to another.
- Middleware **order matters**: `UseOutputCache()` must come after `UseRouting()` so endpoint metadata is resolved, and before the terminal endpoint. Put it before `UseAuthentication`/`UseAuthorization` only if you intend anonymous caching (the default).
- Responses that set `Set-Cookie` or write to `Response.Cookies` are dropped from the cache by default (cookies are per-user by nature). Easy to miss in a redirect-after-login flow.
- Tags are strings — build them with a convention (`$"product:{id}"`, `"products"`) and evict hierarchically. Don't overload one tag with millions of entries; eviction scans the tag index.
- For very short TTLs on very hot endpoints, the overhead of cache key computation + lock acquisition can erase the benefit. Profile before cargo-culting `CacheOutput` everywhere.

## Q34. Difference between `IMemoryCache` and `IDistributedCache`?

Both live under `Microsoft.Extensions.Caching.*`, both solve "don't recompute this," but they're semantically different tools.

### `IMemoryCache`

- **In-process.** Lives inside the current app domain's heap.
- **Typed value API**: `cache.Set<T>(key, value, options)` stores object references — no serialization on get, no allocation on hit beyond dictionary lookup.
- **Cheap**: reads are hash-lookup fast (hundreds of nanoseconds).
- **Expiration**: absolute, sliding, `CancellationChangeToken`, post-eviction callbacks.
- **Size tracking** is opt-in — you pass `SetSize()` and configure `SizeLimit`; without it, the cache can grow unbounded.

```csharp
builder.Services.AddMemoryCache(o => o.SizeLimit = 1024);

public async Task<Product?> GetAsync(int id, CancellationToken ct)
{
    if (_mem.TryGetValue<Product>(id, out var hit)) return hit;
    var p = await _repo.FindAsync(id, ct);
    _mem.Set(id, p, new MemoryCacheEntryOptions()
        .SetSlidingExpiration(TimeSpan.FromMinutes(5))
        .SetSize(1));
    return p;
}
```

### `IDistributedCache`

- **Out-of-process.** Typically backed by Redis, SQL Server, NCache, Garnet, or CosmosDB.
- **Byte-array API**: `GetAsync(key)` returns `byte[]?`, `SetAsync(key, byte[], options)`. You serialize on the way in, deserialize on the way out — JSON by default, but you can plug MessagePack/Protobuf.
- **Costly by comparison**: network hop → microseconds-to-milliseconds even colocated, more over WAN.
- **Shared** across app instances — that's the entire point. Scale-out apps need it to avoid each node rebuilding the cache after deploy/rotation.

```csharp
builder.Services.AddStackExchangeRedisCache(o => o.Configuration = redisConn);

public async Task<Product?> GetAsync(int id, CancellationToken ct)
{
    var bytes = await _dist.GetAsync($"product:{id}", ct);
    if (bytes is not null) return JsonSerializer.Deserialize<Product>(bytes);

    var p = await _repo.FindAsync(id, ct);
    await _dist.SetAsync($"product:{id}",
        JsonSerializer.SerializeToUtf8Bytes(p),
        new DistributedCacheEntryOptions { AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(30) },
        ct);
    return p;
}
```

### Picking between them

| Criterion | `IMemoryCache` | `IDistributedCache` |
|---|---|---|
| Single instance, hot-path compute | ✅ | overkill |
| Multi-instance, consistency required | stale risk | ✅ |
| Needs sharing across processes | ❌ | ✅ |
| Survives restart | ❌ | ✅ (Redis with persistence) |
| Zero serialization cost | ✅ | ❌ |
| Cross-node invalidation | ❌ | ✅ |
| Both L1+L2 combined | — | — → use **HybridCache** (see Q35) |

**Senior-level gotchas:**
- `IMemoryCache` stores **references**. Mutating a cached POCO mutates the cache. Cache immutable types (records) or defensively clone on read — otherwise you'll chase the "sometimes the price is wrong" bug for days.
- Without `SizeLimit` + `SetSize()`, a `MemoryCache` can happily OOM your process. Always size it explicitly in a server app.
- The default JSON serializer for `IDistributedCache` honors camelCase and keeps private setters, but payload schema is a **wire contract** between your app instances during a rolling deploy. Add `[JsonIgnoreCondition]` / version the keys (`"product:v2:{id}"`) when you change shape.
- Redis latency adds up: at 2ms RTT, a single page doing 20 cache lookups spends 40ms in cache *alone*. Either batch (`Redis MGET`, pipelining) or put an L1 in front — which is exactly HybridCache.
- `IDistributedCache` doesn't have stampede protection out of the box. Two hosts missing simultaneously will both call the backend. Use a lock (`SemaphoreSlim` per key) or migrate to HybridCache.

## Q35. Explain how HybridCache/FusionCache works

Both are **multi-level caches**: an in-process L1 (fast, per-node) sitting in front of a distributed L2 (shared, slow). They hide the complexity of keeping both consistent, add stampede protection, and ship a single ergonomic API.

### Microsoft HybridCache (`Microsoft.Extensions.Caching.Hybrid`)

GA in .NET 9 (March 2025); package version `10.x` tracks .NET 10. It's the Microsoft-supported successor to the "roll your own two-level cache with `IMemoryCache` + `IDistributedCache`" pattern that everyone wrote badly.

```csharp
builder.Services.AddHybridCache(options =>
{
    options.DefaultEntryOptions = new HybridCacheEntryOptions
    {
        Expiration = TimeSpan.FromMinutes(30),         // L2 TTL
        LocalCacheExpiration = TimeSpan.FromMinutes(1) // L1 TTL (shorter — local replicas drift)
    };
    options.MaximumPayloadBytes = 1024 * 1024;         // guard against huge entries in L1
});

// Optional: swap the serializer from System.Text.Json to MessagePack, etc.
builder.Services.AddSingleton<IHybridCacheSerializer<Order>, MyOrderSerializer>();

public sealed class OrderService(HybridCache cache, IOrderRepo repo)
{
    public ValueTask<Order?> GetAsync(Guid id, CancellationToken ct) =>
        cache.GetOrCreateAsync(
            $"order:{id}",
            factory: ct2 => new ValueTask<Order?>(repo.LoadAsync(id, ct2)),
            tags: ["orders", $"order:{id}"],
            cancellationToken: ct);

    public ValueTask EvictAsync(Guid id, CancellationToken ct) =>
        cache.RemoveByTagAsync($"order:{id}", ct);
}
```

**Mechanics:**
1. `GetOrCreateAsync` checks L1 (in-process `MemoryCache`); hit → return immediately.
2. Miss → check L2 (`IDistributedCache`); hit → deserialize, populate L1, return.
3. Miss → **stampede protection**: concurrent callers for the same key join a single in-flight factory invocation (per-key `SemaphoreSlim`), not N calls to the database.
4. Factory runs, result is written to L2 then L1, tagged for later eviction.
5. `RemoveByTagAsync("orders")` evicts by tag across L1 and L2.

**Requirements & pieces:**
- If no `IDistributedCache` is registered, HybridCache runs as L1-only (still useful for stampede protection and the nicer API).
- Register any `IDistributedCache` implementation (`AddStackExchangeRedisCache`, SQL Server cache, etc.) to get the L2 tier.
- Serialization defaults to `System.Text.Json`; override per-type via `IHybridCacheSerializer<T>` for binary formats (MessagePack, Protobuf) on hot paths.
- `HybridCacheEntryOptions.Flags` lets you skip tiers per call (`DisableLocalCacheRead`, `DisableDistributedCacheWrite`, etc.) — useful when you know a value is one-off.

### FusionCache (community, ZiggyCreatures)

Predates HybridCache and remains feature-richer. Same core idea (L1 + L2) with extras that matter in production:

- **Fail-safe**: if the factory throws or times out, return the last known value from an extended grace window instead of bubbling the exception — guards dependent services during backend outages.
- **Soft/hard timeouts**: factory timeout below the overall request timeout; falls back to fail-safe or background refresh rather than 504'ing.
- **Eager refresh**: when an entry is close to expiring, trigger a background refresh on the next read so no user pays the miss penalty.
- **Backplane**: publish eviction events (Redis pub/sub) so other nodes drop their L1 copies when L2 changes — HybridCache currently does not invalidate L1 on remote evictions out of the box, which is the biggest real-world gap.
- **Observability**: first-class events for hit/miss/fail-safe/background refresh and OpenTelemetry integration.

```csharp
builder.Services.AddFusionCache()
    .WithDefaultEntryOptions(new FusionCacheEntryOptions
    {
        Duration = TimeSpan.FromMinutes(30),
        FactorySoftTimeout = TimeSpan.FromMilliseconds(100),
        FactoryHardTimeout = TimeSpan.FromMilliseconds(500),
        IsFailSafeEnabled = true,
        FailSafeMaxDuration = TimeSpan.FromHours(2),
        FailSafeThrottleDuration = TimeSpan.FromSeconds(30),
        EagerRefreshThreshold = 0.9f,
    })
    .WithSerializer(new FusionCacheSystemTextJsonSerializer())
    .WithDistributedCache(sp => sp.GetRequiredService<IDistributedCache>())
    .WithBackplane(new RedisBackplane(new RedisBackplaneOptions { Configuration = redisConn }));
```

### When to pick which

- **HybridCache** if you want the Microsoft-supported API, minimal surface, and you're OK with slightly weaker cross-node L1 invalidation. It's the safe default for new apps on .NET 9/10.
- **FusionCache** if you need fail-safe semantics, soft timeouts, eager refresh, or a backplane — all non-trivial to bolt onto HybridCache today.

**Modern note (.NET 10):** HybridCache's API is stable but its ecosystem is still maturing — custom serializers, OpenTelemetry taps, and backplane-style L1 invalidation are either preview or DIY. FusionCache remains a legitimate production choice, especially for services that must degrade gracefully when Redis/the database hiccups.

**Senior-level gotchas:**
- **Stampede protection is per-process.** HybridCache de-dupes concurrent factory calls on one node; N nodes cold-starting at once still produce N factory invocations. If the factory is expensive (a minute-long query) put a global lock (Redis `SET NX` or a semaphore in the DB) inside the factory itself.
- **Tags aren't free.** HybridCache maintains a tag index; `RemoveByTagAsync` is O(entries with that tag). Don't tag every entry with `"everything"` — you'll serialize all evictions through one huge set.
- **L1 drift.** HybridCache's `LocalCacheExpiration` should be shorter than `Expiration` precisely because remote evictions don't immediately reach local caches. Pick the shortest L1 TTL you can tolerate; prefer 30–60 seconds for entities that change.
- **Payload size.** `MaximumPayloadBytes` protects L1 memory but silently bypasses caching for large items — log when it fires, or you'll wonder why the cache hit rate tanks.
- **FusionCache's fail-safe masks real incidents.** Wire OpenTelemetry or the `FailSafeActivated` event to alerts; serving stale data for six hours while the backend is down is a feature, not a fact you want to discover during a postmortem.
- Both caches serialize across deploys. When a `record` gains a new required property, older cached blobs fail to deserialize. Either version the key (`order:v2:{id}`) or make new properties nullable with defaults.
