## Q36. What Caching patterns do you know?

Caching is a *coordination problem* â€” keeping a fast copy consistent with a slow source of truth. The pattern defines **who reads**, **who writes**, and **what happens on miss**. The named patterns:

- **Cache-Aside (Lazy Loading)** â€” the app checks the cache first; on miss it loads from the source, populates the cache, and returns. Writes go to the source, then invalidate (or update) the cache. This is the default; it's what `IMemoryCache.GetOrCreateAsync` gives you.
- **Read-Through** â€” the cache sits *in front* of the source. The app asks the cache; if it's missing, the cache transparently loads from the source. Simpler app code, but the cache provider must support it (HybridCache effectively is this â€” `GetOrCreateAsync` with a factory).
- **Write-Through** â€” on write, the app updates the cache and the source synchronously in one call. Consistency is strong; every write pays both latencies.
- **Write-Behind (Write-Back)** â€” writes land in the cache and are asynchronously flushed to the source. Fast writes, but you can lose data if the cache dies before the flush. Almost always the wrong pattern for a persistence layer; rare in ASP.NET Core applications.
- **Refresh-Ahead** â€” proactively reload hot keys before expiry so the next read still hits the cache. Prevents the "cold-entry stampede" on popular keys.
- **Request coalescing / single-flight** â€” when many concurrent requests miss the same key, only one runs the factory; the rest wait on that one result. `HybridCache` does this natively; `IMemoryCache` does not, which is the classic source of thundering-herd bugs.

Orthogonal concerns you choose alongside the pattern:

- **Eviction policy** â€” absolute TTL, sliding TTL, size-based LRU, tag-based (OutputCache/HybridCache).
- **Topology** â€” in-process (`IMemoryCache`, fast, per-instance), distributed (`IDistributedCache` via Redis/SQL, shared, serialization tax), **layered** (L1 in-proc + L2 distributed â€” the app of the pattern).

```csharp
// Cache-aside with HybridCache (layered + stampede-safe)
public sealed class ProductReader(HybridCache cache, IProductRepository repo)
{
    public ValueTask<Product?> GetAsync(int id, CancellationToken ct) =>
        cache.GetOrCreateAsync(
            key: $"product:{id}",
            factory: async c => await repo.FindAsync(id, c),
            options: new HybridCacheEntryOptions { Expiration = TimeSpan.FromMinutes(5) },
            tags: ["products", $"product:{id}"],
            cancellationToken: ct);
}
```

**Modern note (.NET 10):** `HybridCache` (`Microsoft.Extensions.Caching.Hybrid`, 10.x) is the recommended primitive now â€” it's layered (L1 `IMemoryCache` + optional L2 `IDistributedCache`), does request coalescing, supports **tag-based invalidation** (`RemoveByTagAsync`), and is AOT-compatible. It replaces most hand-rolled cache-aside code and the `LazyCache`/`FusionCache`-style wrappers that existed pre-.NET 9.

**Senior-level gotchas:**
- **`IMemoryCache.GetOrCreateAsync` has no stampede protection** â€” N concurrent misses run N factories. Either add a `SemaphoreSlim` keyed per entry, switch to HybridCache, or accept it for truly cheap factories.
- **Write-through consistency is a lie in a multi-instance deployment** â€” your write updates the local L1, but peer instances still serve stale until their TTL expires. Tag-eviction with a distributed store + pub/sub is what you actually need.
- **Don't cache mutable reference types** you then hand back to callers â€” a caller mutating the cached object changes what the next caller sees. Cache immutable DTOs/records.
- **Distributed cache payload = serialization tax.** For hot objects, measure; sometimes "no distributed cache, just call the DB with a good index" wins.
- **Cache key design** â€” include tenant, version, culture as needed. Silent multi-tenant bleed via a shared key is a data leak, not a bug.

## Q37. What is Rate Limiting used for and what types do you know?

Rate limiting caps how many requests a caller (user, tenant, IP, API key) can make in a given window. The motivations, in roughly decreasing order of how often they actually matter:

1. **Protect downstream dependencies** â€” your API fans out to a DB, a 3rd-party API with its own quota, a SaaS billing endpoint. Rate limiting *inbound* prevents you from hammering *outbound*.
2. **Fair-use / noisy-neighbor** â€” in a multi-tenant system one aggressive client must not starve the rest.
3. **Cost control** â€” each request costs CPU/egress/LLM tokens; cap the bleed.
4. **Abuse mitigation** â€” brute-force against `/login`, scraping, enumeration. Rate limiting is the cheap first line; it is **not** a DDoS defense at L7 (put Cloudflare/Front Door in front for that).
5. **SLA enforcement** â€” different tiers get different quotas.

ASP.NET Core ships the `Microsoft.AspNetCore.RateLimiting` middleware (built-in since .NET 7, still the baseline in .NET 10). Four algorithms:

- **Fixed Window** â€” N requests per fixed window (e.g. 100/min). Simple, but a caller can send 100 at 59.9s and another 100 at 60.1s â€” a 2Ã— burst at the boundary.
- **Sliding Window** â€” the window is divided into segments; as time advances, the oldest segment's requests "fall off". Smooths the fixed-window boundary spike at the cost of memory.
- **Token Bucket** â€” tokens refill at a fixed rate up to a bucket size; each request consumes one. Allows bursts up to the bucket size, then throttles to the refill rate. Best when you want "usually smooth, bursts OK."
- **Concurrency** â€” caps *in-flight* requests, not rate. Use for a downstream with a connection-pool-sized capacity (e.g. "never more than 20 concurrent calls to the CRM API"). This is the right tool when latency dominates, not throughput.

```csharp
builder.Services.AddRateLimiter(o =>
{
    o.RejectionStatusCode = StatusCodes.Status429TooManyRequests;

    o.AddPolicy("per-user", ctx =>
        RateLimitPartition.GetTokenBucketLimiter(
            partitionKey: ctx.User.Identity?.Name ?? ctx.Connection.RemoteIpAddress?.ToString() ?? "anon",
            factory: _ => new TokenBucketRateLimiterOptions
            {
                TokenLimit = 100,
                ReplenishmentPeriod = TimeSpan.FromSeconds(10),
                TokensPerPeriod = 20,
                QueueLimit = 0,                                 // no queueing -> instant 429
                AutoReplenishment = true,
            }));

    o.OnRejected = async (ctx, ct) =>
    {
        if (ctx.Lease.TryGetMetadata(MetadataName.RetryAfter, out var retry))
            ctx.HttpContext.Response.Headers.RetryAfter = ((int)retry.TotalSeconds).ToString();
        await ctx.HttpContext.Response.WriteAsync("Rate limit exceeded.", ct);
    };
});

app.UseAuthentication();
app.UseAuthorization();
app.UseRateLimiter();                                           // AFTER auth

app.MapPost("/orders", CreateOrder).RequireRateLimiting("per-user");
```

**Senior-level gotchas:**
- **Place `UseRateLimiter` after `UseAuthentication`** so you can partition by the authenticated identity. Partitioning anonymous traffic by IP is leaky (shared NAT, corporate proxies) â€” per-user is vastly better when you have it.
- **`QueueLimit > 0` hides the problem.** Queueing requests until a token is available looks polite but increases p99 latency and thread-pool pressure. For APIs, prefer `QueueLimit = 0` and let clients retry with backoff.
- **Always emit `Retry-After`** on 429. The `MetadataName.RetryAfter` metadata on the lease gives you the value for free â€” use it.
- **The middleware is per-process.** Two instances = two independent limiters = 2Ã— the limit. For hard global limits you need a distributed store (Redis + a library like `RedisRateLimiting`) or push the limit to the API gateway.
- **Don't rate-limit health endpoints.** `/healthz` and liveness/readiness probes should be on an unlimited policy â€” limiting them breaks orchestrators.
- **L7 DDoS is not this middleware's job.** An attacker with 10k IPs blows through any per-IP partition. Terminate DDoS upstream.

**Sources:**
- https://learn.microsoft.com/aspnet/core/performance/rate-limit?view=aspnetcore-10.0

## Q38. How to invalidate data in OutputCache?

OutputCache (introduced in .NET 7, still the shipped middleware in .NET 10) supports two invalidation strategies: **expiration** and **tag-based eviction**. There is no first-class API for "evict this exact key" â€” tags are the unit of invalidation.

**Step 1 â€” tag cached responses.** Either in the policy builder or via attribute:

```csharp
// Minimal API
app.MapGet("/products", GetProducts)
   .CacheOutput(p => p.Tag("products").Expire(TimeSpan.FromMinutes(10)));

app.MapGet("/products/{id:int}", GetProduct)
   .CacheOutput(p => p.Tag("products")
                     .SetVaryByRouteValue("id")
                     .Expire(TimeSpan.FromMinutes(10)));

// Controllers
[OutputCache(Tags = new[] { "products" }, Duration = 600)]
public IActionResult Get() { ... }
```

**Step 2 â€” evict by tag on mutation.** Inject `IOutputCacheStore` and call `EvictByTagAsync` after the write succeeds:

```csharp
app.MapPost("/products", async (ProductDto dto, IProductService svc,
                                IOutputCacheStore cache, CancellationToken ct) =>
{
    var created = await svc.CreateAsync(dto, ct);
    await cache.EvictByTagAsync("products", ct);               // fire AFTER the write commits
    return Results.Created($"/products/{created.Id}", created);
});
```

For finer-grained eviction, tag per-entity: `.Tag("products").Tag($"product:{id}")`, then evict `product:42` on a single-item update.

```csharp
// Combine with expiration; the cache auto-clears on TTL too, so a missed EvictByTagAsync is not fatal
builder.Services.AddOutputCache(o =>
{
    o.AddBasePolicy(b => b.Expire(TimeSpan.FromMinutes(5)));
    o.AddPolicy("catalog", b => b.Tag("products").Expire(TimeSpan.FromMinutes(30)));
});
```

**Modern note (.NET 10):** The default `IOutputCacheStore` is backed by `MemoryCache` â€” per-instance. `IDistributedCache` is **not** recommended as a drop-in backing store because it lacks the atomic tag-set operations OutputCache needs. For a multi-instance deployment where you need shared invalidation, write a custom `IOutputCacheStore` on top of Redis (using a Redis set per tag for the tagâ†’key index, and Lua scripts for atomicity), or use a community package that provides one. This is the single biggest trap with OutputCache in production.

**Senior-level gotchas:**
- **Evict AFTER the write commits**, not before. Otherwise a concurrent read between eviction and commit repopulates the cache with stale data. In EF Core terms: `SaveChangesAsync` first, *then* `EvictByTagAsync`.
- **Tag-eviction is not distributed pub/sub** with the default store. Instance A evicting on PUT does not invalidate instance B. TTL is still your safety net â€” set a reasonable `Expire`.
- **`EvictByTagAsync("products")` evicts every entry carrying that tag**, across all policies. Use narrower tags (`product:42`) for surgical invalidation.
- **OutputCache caches the *serialized response bytes*** â€” content-negotiation headers, compression, localization all affect the cache key. Use `SetVaryByHeader` / `SetVaryByQuery` / `VaryByValue` to avoid cross-user/cross-locale bleed.
- **Don't cache authenticated responses by default.** OutputCache explicitly excludes responses where `Authorization` headers or `Set-Cookie` are present unless you opt in â€” good; leave it that way unless you understand the cache-key dimensions.
- **Not the same as HybridCache.** OutputCache caches *HTTP responses* at the middleware layer; HybridCache caches *objects* inside your app. They coexist; don't double-cache the same data in both.

**Sources:**
- https://learn.microsoft.com/aspnet/core/performance/caching/output?view=aspnetcore-10.0

## Q39. How to implement API versioning?

Use the **`Asp.Versioning.*`** packages (successor to the older `Microsoft.AspNetCore.Mvc.Versioning`, now maintained as `dotnet/aspnet-api-versioning`):

- `Asp.Versioning.Http` â€” Minimal APIs
- `Asp.Versioning.Mvc` â€” controllers
- `Asp.Versioning.Mvc.ApiExplorer` â€” integration with Swagger/OpenAPI

Four standard strategies (pick based on client ergonomics and caching):

| Strategy | Example | When to pick it |
|---|---|---|
| **URL segment** | `GET /v1/products` | Most discoverable, simplest for humans; breaks HTTP caches on version bump; forces clients to change URLs. |
| **Query string** | `GET /products?api-version=1.0` | URL stays stable; easy to add later; plays poorly with some CDNs unless you vary by query. |
| **Header** | `X-Api-Version: 1.0` | Clean URLs; invisible to casual testing; awkward in a browser/curl. |
| **Media type** | `Accept: application/json;v=1.0` | REST-purist; rarely used in practice â€” tooling friction outweighs the aesthetic. |

```csharp
// Program.cs â€” Minimal API
builder.Services.AddApiVersioning(o =>
{
    o.DefaultApiVersion = new ApiVersion(1, 0);
    o.AssumeDefaultVersionWhenUnspecified = true;
    o.ReportApiVersions = true;                                // emits api-supported-versions, api-deprecated-versions
    o.ApiVersionReader = new UrlSegmentApiVersionReader();
})
.AddApiExplorer(o =>
{
    o.GroupNameFormat = "'v'VVV";                              // e.g. "v1", "v2", "v2.1"
    o.SubstituteApiVersionInUrl = true;                        // turns {version:apiVersion} into the literal
});

var app = builder.Build();

var versionSet = app.NewApiVersionSet()
    .HasApiVersion(new ApiVersion(1, 0))
    .HasApiVersion(new ApiVersion(2, 0))
    .ReportApiVersions()
    .Build();

var v1 = app.MapGroup("/v{version:apiVersion}")
            .WithApiVersionSet(versionSet)
            .HasApiVersion(1, 0);

v1.MapGet("/products", () => new[] { "v1-product" });

var v2 = app.MapGroup("/v{version:apiVersion}")
            .WithApiVersionSet(versionSet)
            .HasApiVersion(2, 0);

v2.MapGet("/products", () => new[] { new { Id = 1, Name = "v2-product" } });
```

For controllers:

```csharp
[ApiController]
[ApiVersion("1.0")]
[ApiVersion("2.0")]
[Route("v{version:apiVersion}/products")]
public sealed class ProductsController : ControllerBase
{
    [HttpGet, MapToApiVersion("1.0")]
    public IActionResult GetV1() => Ok(/* v1 shape */);

    [HttpGet, MapToApiVersion("2.0")]
    public IActionResult GetV2() => Ok(/* v2 shape */);
}
```

For Swagger, the `ApiExplorer` integration lets you register one `SwaggerDoc` per discovered version via `IApiVersionDescriptionProvider` â€” one dropdown per version in the UI.

**Senior-level gotchas:**
- **Version on breaking change, not on every release.** Additive changes (new optional fields, new endpoints) go in the existing version. Bump the version only when you *remove* or *change semantics*.
- **Don't use `[ApiVersion("1.0-beta")]` for prod clients.** Use stable integers/decimals; keep preview work out of the public surface or gate it behind a feature flag.
- **`AssumeDefaultVersionWhenUnspecified = true` is a compat trap.** It silently promotes old clients to a new default when you change it. In a public API, force clients to opt in.
- **Minimal APIs require the version route token** (`{version:apiVersion}`) for URL-segment versioning â€” don't hand-write `/v1/...` as a literal.
- **Don't proliferate shapes.** If V1 and V2 only differ by one field, often you can solve it with a content-negotiation header or an `Accept` extension rather than forking the whole endpoint.

**Sources:**
- https://github.com/dotnet/aspnet-api-versioning/wiki
- https://www.nuget.org/packages/Asp.Versioning.Http

## Q40. How to add API versioning to the existing API if you're not allowed to change the API URLs?

Constraint rules out URL-segment versioning. Two compatible answers: **header-based** and **query-string-based** â€” both preserve the existing routes. In practice you usually enable both, let old clients keep calling the URL as-is under the default version, and let new clients opt into newer versions.

```csharp
builder.Services.AddApiVersioning(o =>
{
    o.DefaultApiVersion = new ApiVersion(1, 0);
    o.AssumeDefaultVersionWhenUnspecified = true;              // existing callers stay on v1 with zero change
    o.ReportApiVersions = true;

    o.ApiVersionReader = ApiVersionReader.Combine(
        new HeaderApiVersionReader("X-Api-Version"),
        new QueryStringApiVersionReader("api-version"));
})
.AddApiExplorer(o => o.GroupNameFormat = "'v'VVV");
```

The endpoints keep their URLs; `HasApiVersion` on each endpoint / attribute on each controller decides which version handles the request:

```csharp
// Same URL, multiple versions
var set = app.NewApiVersionSet()
             .HasApiVersion(1, 0)
             .HasApiVersion(2, 0)
             .ReportApiVersions()
             .Build();

app.MapGet("/products", () => new[] { "v1" })
   .WithApiVersionSet(set).HasApiVersion(1, 0);

app.MapGet("/products", () => new[] { new { Id = 1, Name = "v2" } })
   .WithApiVersionSet(set).HasApiVersion(2, 0);
```

With the config above, `/products` with no header â†’ v1. With `X-Api-Version: 2.0` or `?api-version=2.0` â†’ v2. Existing clients are unaffected.

**Senior-level gotchas:**
- **Cache-key hazard.** When the same URL serves multiple versions, downstream caches (CDN, OutputCache, browser) will mix versions unless you vary on the version dimension. Emit `Vary: X-Api-Version` on responses and include `api-version` in your OutputCache `VaryBy`. This is the #1 bug shipped with this pattern.
- **`ReportApiVersions = true`** emits `api-supported-versions` and `api-deprecated-versions` headers. Publish the deprecation window via `[ApiVersion("1.0", Deprecated = true)]` and clients can see it without reading your release notes.
- **Swagger generation needs ApiExplorer** to produce one document per version, otherwise you'll render a single doc with duplicate paths â€” unusable. Register one `SwaggerDoc` per `IApiVersionDescriptionProvider.ApiVersionDescriptions` entry.
- **Don't parse the version yourself** from `HttpContext.Request.Headers["X-Api-Version"]` in a middleware. Use the library â€” it handles case, multiple readers, invalid values, and deprecated reporting for you.
- **`AssumeDefaultVersionWhenUnspecified` pins silent upgrades.** When you later promote v2 to default, old-behavior-expecting clients suddenly get v2. Either never change the default, or move the default forward only when you've measured zero v-unspecified traffic.
- **Query-string + POST bodies interact awkwardly** if your clients cache/normalize URLs without query; prefer the header reader as the canonical path and keep the query-string reader as a fallback for convenience testing.

- https://github.com/dotnet/aspnet-api-versioning/wiki/API-Version-Reader
