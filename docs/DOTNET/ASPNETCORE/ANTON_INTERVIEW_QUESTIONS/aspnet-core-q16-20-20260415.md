# ASP.NET Core — Questions 16–20

_Source: `DOTNET/ASPNETCORE/aspnetcore-interview-questions.md`. Answers target .NET 10 / C# 14._

## Q16. Explain action filters and how you might use them with controllers

Action filters are one stage of the MVC **filter pipeline**, which runs *inside* the endpoint execution after routing and model binding. The ordering of filter types is: Authorization → Resource → Action → Exception → Result. `IActionFilter` / `IAsyncActionFilter` surrounds the action method invocation, so it can see the bound arguments, short-circuit the call, or replace the returned `IActionResult`.

```csharp
public sealed class AuditActionFilter : IAsyncActionFilter
{
    private readonly ILogger<AuditActionFilter> _log;
    public AuditActionFilter(ILogger<AuditActionFilter> log) => _log = log;

    public async Task OnActionExecutionAsync(
        ActionExecutingContext ctx, ActionExecutionDelegate next)
    {
        var sw = ValueStopwatch.StartNew();
        var executed = await next();            // runs the action
        if (executed.Exception is null)
            _log.LogInformation("Action {Action} took {Ms}ms",
                ctx.ActionDescriptor.DisplayName, sw.GetElapsedTime().TotalMilliseconds);
    }
}
```

Registration options (scope determines DI lifetime and ordering):

- **Globally** — `builder.Services.AddControllers(o => o.Filters.Add<AuditActionFilter>());`
- **Per controller/action** — `[ServiceFilter(typeof(AuditActionFilter))]` (DI-resolved) or `[TypeFilter(typeof(AuditActionFilter))]` (DI + ad-hoc args). Plain `[AuditActionFilter]` only works if the filter is parameterless and does not need DI.

Typical senior uses: enforcing `ModelState` validity before the action runs, auditing/telemetry, tenant resolution, setting response headers based on the action result, replacing results for feature-flag gating. For cross-cutting concerns that don't need MVC context (claims on the principal, raw request/response bodies), prefer **middleware** — it's cheaper and runs for Minimal APIs too.

**Modern note (.NET 10):** Minimal APIs don't use MVC filters; they use **endpoint filters** (`IEndpointFilter`, `AddEndpointFilter`) introduced in .NET 7 and now the recommended cross-cutting hook for Minimal APIs. The two pipelines are separate — don't expect `IActionFilter` to fire on a `MapGet`.

**Senior-level gotchas:**
- `[ServiceFilter]` needs the filter registered in DI; `[TypeFilter]` does not but creates a new instance each request.
- Filter `Order` is independent of *type* ordering — authorization filters always precede action filters regardless of `Order`.
- Short-circuiting by setting `context.Result` skips the action *and* subsequent action filters but still runs result filters.
- Don't do heavy I/O in an `OnActionExecuting` sync filter — you'll block the thread pool; use the async variant.

## Q17. Why are Minimal APIs faster than Controllers?

Minimal APIs skip the MVC runtime. Requests hit an endpoint delegate directly through the routing/endpoint-routing middleware, without MVC's action-invoker state machine, filter pipeline resolution, model-binder/value-provider stack, `IActionResult` executor indirection, content-negotiation defaults, and the per-request `ControllerContext`/`ActionDescriptor` bookkeeping. The RequestDelegateFactory emits a compiled delegate (optionally via a **source generator** `RequestDelegateGenerator`, enabled with `<EnableRequestDelegateGenerator>true`) that binds parameters and writes the response with minimal allocation.

Concretely, Minimal APIs:

1. Use **source-generated** parameter binding (trim- and AOT-friendly) instead of MVC's reflection-based model binder.
2. Write primitives/`IResult` directly via `Results.*` — no `IActionResultExecutor<T>` dispatch, no `OutputFormatter` negotiation unless you opt in.
3. Avoid the filter pipeline unless you add `AddEndpointFilter`.
4. Have a flatter call stack → better inlining and lower allocations per request.

For a trivial `GET /hello` the delta is real but modest; for high-RPS JSON APIs with a focused contract it's meaningful. For complex controllers doing validation, content negotiation, and filters, the gap shrinks — MVC is paying for features you're actually using.

**Modern note (.NET 10):** NativeAOT support for ASP.NET Core targets Minimal APIs specifically; MVC controllers are still not fully AOT-compatible. If you need AOT or the smallest cold-start (Azure Functions/Container Apps scale-to-zero), Minimal APIs are the only realistic option.

**Senior-level gotchas:**
- "Faster" ≠ "better." MVC's conventions, filters, and model validation are a lot of code you'd re-implement badly in Minimal APIs. Pick based on feature need, not micro-benchmarks.
- `AddEndpointFilter` allocations can erase the advantage if you chain many filters — profile.
- Avoid boxing in Minimal API return types: return `Results.Ok(dto)` / `TypedResults.Ok(dto)` rather than plain objects for AOT + OpenAPI inference.

## Q18. How to add authorization to the project?

Authorization in ASP.NET Core is two-step: **authentication** establishes *who* the caller is (`ClaimsPrincipal`), **authorization** decides *what* they may do. You wire both in `Program.cs`:

```csharp
var builder = WebApplication.CreateBuilder(args);

builder.Services
    .AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(o =>
    {
        o.Authority = builder.Configuration["Auth:Authority"];
        o.Audience  = builder.Configuration["Auth:Audience"];
        o.TokenValidationParameters = new()
        {
            ValidateIssuer = true,
            ValidateAudience = true,
            ValidateLifetime = true,
            ValidateIssuerSigningKey = true,
        };
    });

builder.Services.AddAuthorizationBuilder()
    .SetFallbackPolicy(new AuthorizationPolicyBuilder()
        .RequireAuthenticatedUser().Build())        // secure by default
    .AddPolicy("AdminsOnly", p => p.RequireRole("admin"))
    .AddPolicy("CanEditOrders", p => p.RequireClaim("scope", "orders:write"));

var app = builder.Build();

app.UseAuthentication();   // order matters
app.UseAuthorization();

app.MapControllers();
app.MapGet("/admin", () => "ok").RequireAuthorization("AdminsOnly");
```

Applying it:

- Attribute-based: `[Authorize]`, `[Authorize(Policy = "AdminsOnly")]`, `[Authorize(Roles = "admin,ops")]`, `[AllowAnonymous]`.
- Minimal APIs: `.RequireAuthorization("AdminsOnly")` on the endpoint or group.
- Global gate: `SetFallbackPolicy` forces everything to require auth unless the endpoint opts out via `[AllowAnonymous]`.

For claim/resource logic that can't be expressed as `RequireRole`/`RequireClaim`, use **`IAuthorizationRequirement` + `AuthorizationHandler<T>`** and register with `services.AddSingleton<IAuthorizationHandler, MyHandler>()`.

**Modern note (.NET 10):** `AddAuthorizationBuilder()` (net8+) is the preferred fluent API over the older `services.AddAuthorization(o => ...)`. `SetFallbackPolicy` is the idiomatic way to make the app secure-by-default rather than scattering `[Authorize]` everywhere.

**Senior-level gotchas:**
- `UseAuthentication` must come **before** `UseAuthorization`, both after `UseRouting`.
- `[Authorize]` without a scheme uses the default scheme — be explicit when you mix cookies + JWT (`[Authorize(AuthenticationSchemes = "Bearer")]`).
- Roles in JWT require the correct claim type mapping; JwtBearer maps `role`/`roles` → `ClaimTypes.Role` only if `MapInboundClaims` is on (default true). Turning mapping off breaks `RequireRole` unless you set `RoleClaimType` in `TokenValidationParameters`.
- For resource-based checks (does user X own order Y?), inject `IAuthorizationService` and call `AuthorizeAsync(user, resource, "policy")` — never encode ownership checks in the handler statically.

## Q19. How to add authorization to all Controller's methods except one?

Three idiomatic options, from most-preferred to least:

**1. Controller-level `[Authorize]` + action-level `[AllowAnonymous]`** — the original MVC pattern:

```csharp
[ApiController, Route("api/[controller]"), Authorize]
public sealed class OrdersController : ControllerBase
{
    [HttpGet("public-summary"), AllowAnonymous]
    public IActionResult PublicSummary() => Ok(/* ... */);

    [HttpPost] public IActionResult Create(...) => ...; // requires auth
}
```

**2. Fallback policy + `[AllowAnonymous]` on the exception.** Best when "everything needs auth" is an app-wide invariant:

```csharp
builder.Services.AddAuthorizationBuilder()
    .SetFallbackPolicy(new AuthorizationPolicyBuilder()
        .RequireAuthenticatedUser().Build());
```

Then only the anonymous endpoint carries `[AllowAnonymous]`. The controller stays attribute-free.

**3. Convention / endpoint group.** For Minimal APIs: `app.MapGroup("/api").RequireAuthorization();` then the one public endpoint is mapped outside the group or uses `.AllowAnonymous()`.

`[AllowAnonymous]` **always wins** over `[Authorize]` at any scope — even a fallback policy — because the authorization middleware checks for `IAllowAnonymous` metadata on the endpoint and skips policy evaluation. That's what makes this pattern safe.

**Senior-level gotchas:**
- `[AllowAnonymous]` bypasses authorization, **not authentication**. If an `Authorization: Bearer` header is present it's still validated; an invalid token can still 401. Use `[AllowAnonymous]` on login endpoints but don't assume `HttpContext.User` is unauthenticated there.
- Don't layer multiple `[Authorize]` attributes expecting "OR" — multiple policies combine with AND. For "admin OR owns-the-resource" build a single policy with an `IAuthorizationRequirement` and multiple handlers (handlers are OR'd; requirements within a policy are AND'd).
- If you use `SetFallbackPolicy`, remember endpoints without *any* endpoint metadata (e.g. static files before UseRouting) aren't affected — placement of `UseStaticFiles` matters for what gets protected.

## Q20. How would you implement Log-In functionality?

There are three realistic patterns; pick based on the client and the threat model.

### A. ASP.NET Core Identity with cookies (server-rendered or same-origin SPA)

Use when the frontend is Razor/Blazor Server or an SPA on the same origin. Identity handles password hashing (PBKDF2/Argon2id via `IPasswordHasher<TUser>`), lockout, two-factor, email confirmation.

```csharp
builder.Services
    .AddDbContext<AppDb>(o => o.UseNpgsql(cs))
    .AddIdentityCore<AppUser>(o =>
    {
        o.Password.RequiredLength = 12;
        o.Lockout.MaxFailedAccessAttempts = 5;
        o.SignIn.RequireConfirmedEmail = true;
    })
    .AddEntityFrameworkStores<AppDb>()
    .AddSignInManager();

builder.Services.AddAuthentication(IdentityConstants.ApplicationScheme)
    .AddCookie(IdentityConstants.ApplicationScheme, o =>
    {
        o.Cookie.HttpOnly = true;
        o.Cookie.SecurePolicy = CookieSecurePolicy.Always;
        o.Cookie.SameSite = SameSiteMode.Lax;
        o.ExpireTimeSpan = TimeSpan.FromHours(8);
        o.SlidingExpiration = true;
    });

app.MapPost("/login", async (LoginDto dto, SignInManager<AppUser> signIn) =>
{
    var result = await signIn.PasswordSignInAsync(
        dto.Email, dto.Password, isPersistent: dto.Remember, lockoutOnFailure: true);
    return result.Succeeded ? Results.NoContent()
         : result.IsLockedOut ? Results.StatusCode(423)
         : Results.Unauthorized();
}).AllowAnonymous();
```

Since **.NET 8**, `services.AddIdentityApiEndpoints<AppUser>()` + `app.MapIdentityApiEndpoints<AppUser>()` provide drop-in `/register`, `/login`, `/refresh`, `/confirmEmail` endpoints — use them if the defaults fit; they emit bearer tokens by default but support cookies via `?useCookies=true`.

### B. External IdP (OIDC) — production default

Delegate credential handling to Entra ID / Auth0 / Keycloak / Okta. You never store passwords. Flow: Authorization Code + PKCE for SPAs and mobile; Authorization Code for server-rendered apps.

```csharp
builder.Services.AddAuthentication(o =>
    {
        o.DefaultScheme = CookieAuthenticationDefaults.AuthenticationScheme;
        o.DefaultChallengeScheme = OpenIdConnectDefaults.AuthenticationScheme;
    })
    .AddCookie()
    .AddOpenIdConnect(o =>
    {
        o.Authority = "https://login.microsoftonline.com/<tenant>/v2.0";
        o.ClientId = "...";
        o.ClientSecret = builder.Configuration["Oidc:Secret"];
        o.ResponseType = "code";
        o.UsePkce = true;
        o.SaveTokens = true;
        o.Scope.Add("openid"); o.Scope.Add("profile"); o.Scope.Add("email");
    });
```

### C. Custom JWT issuance (first-party API clients)

Only when A and B don't fit — e.g. machine-to-machine or a dedicated auth service you own. Verify credentials against the user store, then mint a short-lived access token plus a refresh token stored server-side (hashed), rotated on use, revocable on logout.

```csharp
app.MapPost("/auth/login", async (LoginDto dto, IUserService users, ITokenService tokens) =>
{
    var user = await users.FindAsync(dto.Email);
    if (user is null || !users.VerifyPassword(user, dto.Password))
        return Results.Unauthorized();

    var (access, refresh) = await tokens.IssueAsync(user);
    return Results.Ok(new { access, refresh });
}).AllowAnonymous();
```

**Senior-level gotchas:**
- **Never** roll your own password hashing — use `IPasswordHasher<T>` (PBKDF2) or Identity's Argon2 configuration. No MD5, no SHA-256-unsalted.
- Always `lockoutOnFailure: true` or equivalent — otherwise online brute force is trivial. Pair with rate limiting on the login endpoint (`AddRateLimiter`).
- For SPAs, prefer **cookies with `SameSite=Lax` + `HttpOnly` + `Secure` + CSRF tokens** over storing JWTs in `localStorage` (XSS steals `localStorage` tokens in one line). If you *must* use bearer tokens, keep them short-lived (≤ 15 min) and use refresh-token rotation with reuse detection.
- Return **generic "invalid credentials"** for unknown-user and wrong-password cases — distinguishing them is a username enumeration vulnerability.
- Log login failures with structured fields (user id, IP, user-agent) but **never** the password or token. Hash emails in logs if GDPR scope applies.
- Put login behind `[AllowAnonymous]` explicitly even when no fallback policy exists — future-you will add one and wonder why login 401s.
