## Q56. What's the difference between `ControllerBase` and `Controller`? When would you use each?

`ControllerBase` is the API base class. `Controller` **inherits from** `ControllerBase` and layers view-rendering concerns on top:

| Feature | `ControllerBase` | `Controller` |
| --- | --- | --- |
| Model binding, validation, `ActionContext`, `Ok()/NotFound()/Problem()` etc. | ✅ | ✅ |
| `View()`, `PartialView()`, `RedirectToAction()`, `ViewData`, `ViewBag`, `TempData`, `ViewComponent()` | ❌ | ✅ |
| Pulls in view-engine services (Razor) | ❌ | ✅ |

```csharp
// Web API — JSON only. Correct choice:
[ApiController]
[Route("api/[controller]")]
public sealed class OrdersController : ControllerBase
{
    [HttpGet("{id:guid}")]
    public ActionResult<OrderDto> Get(Guid id) => Ok(...);
}

// Server-rendered MVC — needs views:
public sealed class HomeController : Controller
{
    public IActionResult Index() => View();
}
```

**Rule of thumb:** use `ControllerBase` unless you genuinely need to return a Razor view. Pair it with `[ApiController]`, which turns on automatic 400 responses for invalid models, `[FromBody]` inference for complex types, attribute-routing enforcement, and `ProblemDetails`-shaped error responses.

**Senior-level gotchas:**

- Dragging `Controller` into an API project isn't a compile error, but it signals confusion and pulls in view-engine services unnecessarily. Reviewers will flag it.
- `ControllerBase.User` / `HttpContext` / `Request` / `Response` / `TempData`-like state is there — you don't need the view-rendering base for that.
- Razor Pages use `PageModel` — a separate hierarchy, not `Controller`. Don't mix them up in interviews.
- `[NonController]` exists to *exclude* a type that would otherwise be discovered as a controller by convention — useful when a base class lives in a shared package.

## Q57. How can you handle exceptions globally for all controllers?

Three modern mechanisms, listed in the order a senior would reach for them on .NET 10.

### 1. `IExceptionHandler` + `UseExceptionHandler()` + `ProblemDetails` — the idiomatic pattern

Introduced in .NET 8, promoted to the default pattern since. Implement `IExceptionHandler`, register with `AddExceptionHandler<T>()`, activate middleware with `UseExceptionHandler()`, and add `AddProblemDetails()` so unhandled failures produce RFC 9457 JSON.

```csharp
public sealed class NotFoundExceptionHandler(IProblemDetailsService pds) : IExceptionHandler
{
    public async ValueTask<bool> TryHandleAsync(
        HttpContext ctx, Exception ex, CancellationToken ct)
    {
        if (ex is not NotFoundException nf) return false;

        ctx.Response.StatusCode = StatusCodes.Status404NotFound;
        return await pds.TryWriteAsync(new ProblemDetailsContext
        {
            HttpContext = ctx,
            ProblemDetails =
            {
                Title   = "Resource not found",
                Detail  = nf.Message,
                Type    = "https://httpstatuses.io/404",
                Status  = 404,
                Extensions = { ["traceId"] = ctx.TraceIdentifier }
            }
        });
    }
}

// Program.cs
builder.Services.AddProblemDetails();
builder.Services.AddExceptionHandler<NotFoundExceptionHandler>();
builder.Services.AddExceptionHandler<FallbackExceptionHandler>(); // order matters

var app = builder.Build();
app.UseExceptionHandler();   // no args — picks up registered handlers
```

Handlers are singletons, tried in registration order until one returns `true`. Unhandled → default 500 + ProblemDetails.

### 2. `UseExceptionHandler("/error")` lambda or error endpoint — simpler cases

```csharp
app.UseExceptionHandler(errorApp => errorApp.Run(async ctx =>
{
    var feature = ctx.Features.Get<IExceptionHandlerFeature>();
    ctx.Response.StatusCode = 500;
    await ctx.Response.WriteAsJsonAsync(new { error = feature?.Error.Message });
}));
```

Fine for small services; doesn't scale when you need polymorphic handling.

### 3. MVC exception filters (`IExceptionFilter` / `IAsyncExceptionFilter`) — controller-scoped

Runs inside the MVC filter pipeline, has access to `ExceptionContext` / `ActionContext`. Use for MVC-specific concerns (e.g., switching `Result` based on `ActionDescriptor` metadata). Do **not** make it your primary mechanism — middleware-thrown exceptions never reach it.

**Modern note (.NET 10 / C# 14):** Starting in .NET 10, when an `IExceptionHandler` returns `true`, the runtime **suppresses the default exception logs and metrics** — the previous behavior (always emit) meant your "handled" exceptions still showed up as unhandled in diagnostics. If you want to keep metrics, emit them yourself inside the handler. ProblemDetails output now conforms to RFC 9457 (which obsoleted RFC 7807). ([Handle errors in ASP.NET Core](https://learn.microsoft.com/en-us/aspnet/core/fundamentals/error-handling?view=aspnetcore-10.0))

**Senior-level gotchas:**

- Order of `AddExceptionHandler<T>()` calls **is the dispatch order**. Put specific handlers before the catch-all.
- Always log with the original exception + `ctx.TraceIdentifier` (and `Activity.Current?.TraceId` if OTel is on) so clients can quote the `traceId` and you can join it in logs.
- Do **not** swallow `OperationCanceledException` when the client disconnected — return `499`-ish / `StatusCodes.Status499ClientClosedRequest` or just let it propagate; logging a stack trace for every disconnect is noise.
- Exceptions thrown *before* the exception-handler middleware (e.g., from `UseRouting` decisions or earlier middleware) won't be caught — so put `UseExceptionHandler()` at the very top of the pipeline.
- Developer exception page (`UseDeveloperExceptionPage`) is environment-gated and never for production — it leaks stack traces and env vars.
- For minimal APIs the same middleware applies; you don't need a filter equivalent.

**Sources:**
- [Handle errors in ASP.NET Core](https://learn.microsoft.com/en-us/aspnet/core/fundamentals/error-handling?view=aspnetcore-10.0)
- [Handle errors in ASP.NET Core APIs](https://learn.microsoft.com/en-us/aspnet/core/fundamentals/error-handling-api?view=aspnetcore-10.0)

## Q58. What is structured logging and why should you use it?

**Structured logging** means emitting each log event as a **message template + named properties**, not a pre-formatted string. The template is stable; the values are captured as separate, typed, queryable fields.

```csharp
// Unstructured (bad) — a string blob
logger.LogInformation($"Order {orderId} placed by user {userId} for {amount:C}");

// Structured (good) — template + properties
logger.LogInformation(
    "Order {OrderId} placed by user {UserId} for {Amount}",
    orderId, userId, amount);
```

The second form lets a structured sink (Seq, Elastic, Loki, OpenTelemetry Collector, App Insights) index `OrderId`, `UserId`, `Amount` as real columns you can filter on — `OrderId = "abc"` — instead of regex-grepping strings.

### Why it matters

1. **Queryability** — grouping, aggregation, alerting.
2. **Stable grouping key** — the template itself becomes the event identifier. All `"Order {OrderId} placed..."` events are the same class even when values differ.
3. **Correlation** — pairs naturally with OpenTelemetry `Activity` (trace/span IDs are attached automatically by ASP.NET Core).
4. **Allocation-free with source generation** — use `[LoggerMessage]` so the compiler writes the boilerplate:

```csharp
public static partial class OrdersLog
{
    [LoggerMessage(
        EventId = 1001,
        Level = LogLevel.Information,
        Message = "Order {OrderId} placed by user {UserId} for {Amount}")]
    public static partial void OrderPlaced(
        ILogger logger, Guid orderId, Guid userId, decimal amount);
}

// call site
OrdersLog.OrderPlaced(logger, id, userId, amount);
```

The generator emits a method that avoids parameter boxing, skips formatting when the level is disabled, and produces a single allocation-free call. It also integrates with redaction (see Q60).

### Scopes — ambient structure

```csharp
using (logger.BeginScope(new Dictionary<string, object> { ["TenantId"] = tenantId }))
{
    // every log emitted inside here carries TenantId
}
```

**Senior-level gotchas:**

- **Never interpolate into the template** (`$"Order {orderId}..."`). The runtime sees a unique string every call, destroys the grouping key, defeats the duplication cache, and turns every value into part of the message text rather than a field.
- Use stable, PascalCase property names. Don't rename them casually — dashboards and alerts depend on them.
- Pick levels deliberately: at scale, `Information` is *not* free. Hot-path traces belong at `Debug`/`Trace`, gated.
- Message templates are how you enforce a *schema* without one. Reviewers should push back on free-form log text.
- Don't log whole DTOs — see Q60. Use `[LogProperties]` with classification instead.

## Q59. How can you log request and response data for incoming HTTP requests?

The built-in tool is **`Microsoft.AspNetCore.HttpLogging`**. Prefer it over hand-rolled middleware.

```csharp
builder.Services.AddHttpLogging(o =>
{
    o.LoggingFields = HttpLoggingFields.RequestPropertiesAndHeaders
                    | HttpLoggingFields.ResponsePropertiesAndHeaders
                    | HttpLoggingFields.Duration;
    o.RequestHeaders.Add("x-correlation-id");
    o.ResponseHeaders.Add("x-ratelimit-remaining");
    o.MediaTypeOptions.AddText("application/json");
    o.RequestBodyLogLimit  = 4 * 1024;
    o.ResponseBodyLogLimit = 4 * 1024;
    o.CombineLogs = true; // single event per request/response pair
});

var app = builder.Build();
app.UseHttpLogging();          // place early, after UseExceptionHandler
```

Two ergonomic features worth knowing:

- **`CombineLogs = true`** — emits one log entry covering both request and response rather than two separate entries, which is much easier to correlate under concurrency.
- **Per-endpoint control** — `WithHttpLogging(...)` on minimal APIs and the `[HttpLogging]` / `[DisableHttpLogging]` attributes on MVC to opt in/out or override fields (e.g., only log `RequestBody` on one endpoint).

```csharp
app.MapPost("/webhooks/stripe", Handle)
   .WithHttpLogging(HttpLoggingFields.RequestBody | HttpLoggingFields.ResponseStatusCode);
```

For classic access-log shape, register **`AddW3CLogging(...)` + `UseW3CLogging()`** — writes a standards-compliant W3C Extended Log File.

### When hand-rolled middleware is justified

You almost never need it. The case is: you want to log a field HttpLogging doesn't surface (e.g., a parsed JSON property). Then:

```csharp
app.Use(async (ctx, next) =>
{
    ctx.Request.EnableBuffering();          // allow re-reading the body
    // read once, reset, then call next
    using var reader = new StreamReader(ctx.Request.Body, leaveOpen: true);
    var body = await reader.ReadToEndAsync();
    ctx.Request.Body.Position = 0;
    // log… then
    await next();
});
```

**Senior-level gotchas:**

- Body logging at `Information` explodes log volume and cost. Gate via level, sampling, or per-endpoint opt-in.
- Default header allowlist is deliberately minimal (no `Authorization`, no `Cookie`). Don't "just add them" to debug — that's a secret leak (see Q60).
- Without `EnableBuffering()`, reading the request body in middleware consumes the stream; the MVC model binder downstream will see an empty body.
- If you already have OpenTelemetry instrumentation (`AddAspNetCoreInstrumentation`), prefer OTel semantic-convention attributes on the activity over duplicate logs — same data, half the cost.
- Streaming responses + full body logging = broken streaming. HttpLogging handles this correctly; hand-rolled middleware usually doesn't.

**Sources:**
- [HTTP logging in ASP.NET Core](https://learn.microsoft.com/en-us/aspnet/core/fundamentals/http-logging/?view=aspnetcore-10.0)

## Q60. How would you handle sensitive data in logs to ensure security and compliance (e.g., GDPR)?

Treat logs as a data store — they have retention, access controls, and a GDPR footprint. Your defenses stack:

### 1. Don't capture it in the first place (design)

- Log IDs, not payloads. `UserId = "abc-123"` is fine; `Email = "alice@x.com"` is personal data under GDPR Art. 4(1).
- Use DTO projections for logging — don't shovel incoming request models through `JsonSerializer.Serialize` into a log line.
- Record `ToString()` synthesizes a string that includes **every** property. Override it or use a logger-specific projection for types that carry PII.

### 2. Classify + redact with `Microsoft.Extensions.Compliance.*`

First-party, integrates with the `[LoggerMessage]` source generator, and lets you apply redaction declaratively.

```csharp
// Define (or reuse) a classification
public sealed class PiiAttribute()
    : DataClassificationAttribute(new DataClassification("Contoso", "PII"));

public sealed record Customer(
    Guid Id,
    [property: Pii] string Email,
    [property: Pii] string PhoneNumber);

// Source-generated logging method with property logging
public static partial class CustomerLog
{
    [LoggerMessage(LogLevel.Information, "Customer {Id} updated")]
    public static partial void Updated(
        ILogger logger, Guid id, [LogProperties] Customer customer);
}

// Program.cs wiring
builder.Services.AddLogging(b => b.EnableRedaction());

builder.Services.AddRedaction(redaction =>
{
    redaction.SetRedactor<ErasingRedactor>(
        new DataClassificationSet(new DataClassification("Contoso", "PII")));
    // or HmacRedactor for deterministic pseudonymization:
    // redaction.SetHmacRedactor(o => { ... }, piiSet);
});
```

Now any `[Pii]`-classified property emitted through a `[LoggerMessage]`-generated method flows through the registered redactor before hitting the sink. `ErasingRedactor` blanks the value; `HmacRedactor` produces a deterministic HMAC so you can join records without seeing raw values. Redaction is only wired for properties decorated with `[LogProperties]` / `[TagProvider]` in source-generated logs — hand-written `logger.LogInformation(...)` calls bypass it.

### 3. HTTP logging — allowlists + redactors

Per Q59, `AddHttpLogging` has explicit header allowlists. Combine with `HttpLogging.Redaction` to classify entire headers:

```csharp
builder.Services.AddHttpLogging(o =>
{
    o.RequestHeaders.Add("x-correlation-id");
    // do not add Authorization / Cookie / Set-Cookie
});
```

Never turn on body logging on endpoints that handle credentials, tokens, card data, or health information.

### 4. Storage & access controls

- **Retention** — cap log TTL per sensitivity tier. Personal data with no business need shouldn't sit in logs for a year.
- **RBAC on the sink** — Seq signals / App Insights workspaces / Loki multi-tenancy.
- **DSR deletion** — data-subject requests under GDPR Art. 17 must include log stores; an HMAC redactor makes this easier (no raw identifiers to delete, but you can still correlate).
- **Region** — log sink data residency matters; don't ship EU personal data to a US-only analytics stack without a transfer mechanism.

### 5. Exceptions are a classic leak vector

Stack traces and bound parameters end up in logs. `LogException(ex)` can include `ex.Data`, argument values captured by the debugger-visible frames, and the serialized `ToString()` of the surrounding context. Scrub exception messages that you construct yourself (e.g., `throw new InvalidOperationException($"email {email} is banned")`).

**Modern note (.NET 10 / C# 14):** `Microsoft.Extensions.Compliance.Redaction` 10.4.0 is the current shipping version and the recommended redaction stack. Combined with `[LoggerMessage]` / `[LogProperties]` / `[TagProvider]`, the source generator emits the redactor lookup and call — no reflection at runtime. ([Data redaction in .NET](https://learn.microsoft.com/en-us/dotnet/core/extensions/data-redaction), [Compile-time logging source generation](https://learn.microsoft.com/en-us/dotnet/core/extensions/logger-message-generator))

**Senior-level gotchas:**

- Redaction is opt-in per property. A new field added to a DTO without `[Pii]` silently flows through in the clear. Guard with analyzers or code review, not trust.
- `HmacRedactor` is pseudonymization, **not anonymization** under GDPR — it's still personal data if the key exists, so protect the key.
- Log aggregators often cache/index *before* your retention job runs. Redact at the source.
- Test redaction with a contract test: inject an instrumented logger, assert redacted output for each classified field. Manual log inspection is an anti-process.
- Don't forget `query string` — it's often logged in access logs and carries PII from GET-based forms; move those to POST.

- [Compliance libraries in .NET](https://learn.microsoft.com/en-us/dotnet/core/extensions/compliance)
- [Microsoft.Extensions.Compliance.Redaction on NuGet](https://www.nuget.org/packages/Microsoft.Extensions.Compliance.Redaction)
