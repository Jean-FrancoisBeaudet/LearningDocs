# ASP.NET Core — Questions 6–10

_Source: `DOTNET/ASPNETCORE/aspnetcore-interview-questions.md`. Answers target .NET 10 / C# 14._

## Q6. Explain how `appsettings.json` configuration layering works

ASP.NET Core configuration is a **last-writer-wins key/value store** built by stacking providers in order. When you call `builder.Configuration["Key"]`, the host walks the providers in **reverse order** and returns the first hit. Later providers override earlier ones.

The default `WebApplicationBuilder` (in .NET 10, same as 8/9) registers providers in this order:

1. `appsettings.json` (optional, reload-on-change)
2. `appsettings.{Environment}.json` (e.g. `appsettings.Development.json`)
3. User secrets (only in `Development`)
4. Environment variables (prefixed by `DOTNETCORE_` and unprefixed)
5. Command-line arguments
6. (Azure Key Vault, App Configuration, etc. when you add them explicitly)

So `ConnectionStrings:Db` in `appsettings.Production.json` overrides the value in `appsettings.json`, and an env var `ConnectionStrings__Db` (note the **double underscore** — that's the hierarchical separator on non-Windows) overrides both.

```csharp
var builder = WebApplication.CreateBuilder(args);

// Add an extra layer — e.g. a shared config file
builder.Configuration
    .AddJsonFile("shared.json", optional: true, reloadOnChange: true)
    .AddAzureAppConfiguration(o => o.Connect(connString));

var timeout = builder.Configuration.GetValue<int>("Http:TimeoutSeconds");
```

**Key mechanics worth knowing:**

- **Hierarchy uses `:`** in code (`"Logging:LogLevel:Default"`) and **`__`** in env vars. Arrays are index-based: `Servers:0:Host`.
- `reloadOnChange: true` uses a file watcher and pushes changes through `IOptionsMonitor<T>` / `IConfiguration` without restart. `IOptions<T>` does **not** see reloads — it's a singleton snapshot captured at first resolution.
- `GetSection("X").Exists()` vs `GetValue`: `GetSection` never returns null; check `.Exists()` or `.Value is not null`.
- Binding is case-insensitive, keys are stored as strings — type safety comes from the Options pattern, not from `IConfiguration` itself.

**Senior-level gotchas:**

- In containers, `DOTNET_ENVIRONMENT` / `ASPNETCORE_ENVIRONMENT` controls which `appsettings.{env}.json` loads. Mis-setting this is the #1 "why does prod have dev settings?" bug.
- Secrets in `appsettings.json` is a smell — prefer user-secrets locally and Key Vault / App Configuration in prod.
- The `WebApplicationBuilder` already includes `ChainedConfigurationProvider` so calling `AddConfiguration(builder.Configuration)` again double-stacks providers and confuses precedence.

## Q7. What is the difference between Singleton, Scoped, Transient services?

These are the three **service lifetimes** the built-in DI container supports (`Microsoft.Extensions.DependencyInjection`):

| Lifetime     | Instances per…                         | Registered via                               |
|--------------|----------------------------------------|----------------------------------------------|
| Transient    | Every resolution / injection           | `services.AddTransient<T>()`                 |
| Scoped       | One per DI scope (per HTTP request in ASP.NET Core) | `services.AddScoped<T>()`         |
| Singleton    | One for the entire app lifetime        | `services.AddSingleton<T>()`                 |

In ASP.NET Core, a **scope** is created per HTTP request by the request-services middleware (`IServiceScopeFactory`). Inside a request, every `AddScoped` service resolves to the same instance; across requests, they're different.

```csharp
services.AddSingleton<IClock, SystemClock>();        // pure, thread-safe, no state
services.AddScoped<IUnitOfWork, EfUnitOfWork>();     // DbContext, per-request
services.AddTransient<IEmailSender, SmtpSender>();   // cheap, short-lived
```

**Rules of thumb:**

- **Singleton** for stateless, thread-safe services (clocks, config snapshots, HTTP clients via `IHttpClientFactory`, logger factories). Must be thread-safe because concurrent requests share them.
- **Scoped** for anything tied to a request: `DbContext`, unit-of-work, current-user services. `DbContext` is scoped because it is **not thread-safe** and accumulates change-tracking state.
- **Transient** for lightweight stateless helpers. Don't use for anything holding resources (disposal still happens, but only at scope end — so transient inside a singleton leaks until app shutdown).

**Captive-dependency problem:** a longer-lived service holding a shorter-lived one *captures* it. A singleton holding a scoped `DbContext` pins one `DbContext` for the app's lifetime — catastrophic. The default container validates this in `Development` when you set `ValidateScopes = true` (on by default for `CreateBuilder`).

**Senior-level gotchas:**

- `AddHttpClient<T>()` registers `T` as **transient**, but the underlying `HttpMessageHandler` is pooled by `IHttpClientFactory` — this is why you don't leak sockets.
- Factory-style registration (`AddSingleton<IFoo>(sp => ...)`) lets you defer construction but the `sp` is the **root** provider — don't resolve scoped services from it.
- `IServiceProvider` itself resolves to the **current scope's** provider inside a request. Store the `IServiceScopeFactory`, not the provider, for deferred work (see Q8).

## Q8. How to use a Scoped service inside a Singleton service?

**You cannot inject a scoped service into a singleton's constructor** — that creates a captive dependency and the framework will throw on startup with `ValidateScopes = true`:

> "Cannot consume scoped service 'IFoo' from singleton 'IBar'."

The fix is to inject `IServiceScopeFactory` (itself a singleton) and create a **short-lived scope** each time you need the scoped service:

```csharp
public sealed class OrderPublisher(IServiceScopeFactory scopeFactory, ILogger<OrderPublisher> logger)
{
    public async Task PublishAsync(Order order, CancellationToken ct)
    {
        await using var scope = scopeFactory.CreateAsyncScope();
        var db = scope.ServiceProvider.GetRequiredService<AppDbContext>();

        db.Outbox.Add(new OutboxMessage(order));
        await db.SaveChangesAsync(ct);
    }
}
```

Key points:

- `CreateAsyncScope()` (since .NET 6) returns an `AsyncServiceScope` that disposes async — important when the scope contains services with `IAsyncDisposable` (like `DbContext` in EF Core 8+).
- Scope lifetime must be **short and well-defined**. A scope per message / per work unit is the pattern. Never store the scope as a field of the singleton.
- In a `BackgroundService`, the same pattern applies: the hosted service itself is a singleton, so wrap each iteration in its own scope.

**Alternative:** expose the scoped dependency via a **factory delegate** registered as singleton:

```csharp
services.AddSingleton<Func<AppDbContext>>(sp =>
    () => sp.CreateScope().ServiceProvider.GetRequiredService<AppDbContext>());
```

This is shown in some tutorials but is almost always worse than `IServiceScopeFactory` — it hides the scope lifecycle and invites leaks. Prefer the explicit scope.

**Senior-level gotchas:**

- One scope per *operation*, not per *service*. If a message handler calls three services that all need a `DbContext`, they should share **one** scope so change tracking works.
- `HttpContext` is scoped; accessing `IHttpContextAccessor.HttpContext` from a singleton outside a request returns `null`. Don't use it in background work — pass the data in explicitly.
- `ILogger<T>` is singleton-safe; `ILoggerFactory` is singleton. You never need a scope just for logging.

## Q9. How to execute code when the application is starting and stopping?

Three mechanisms, each with a distinct purpose:

### 1. Startup: run code *before* the host starts serving requests

Put it after `builder.Build()` and before `app.Run()`:

```csharp
var app = builder.Build();

using (var scope = app.Services.CreateScope())
{
    var db = scope.ServiceProvider.GetRequiredService<AppDbContext>();
    await db.Database.MigrateAsync();        // one-off init
}

app.Run();
```

This is synchronous-path startup: the HTTP listener isn't accepting yet. Good for migrations, warm-up, schema validation.

### 2. `IHostedService` / `BackgroundService`: long-running work tied to host lifetime

`StartAsync` runs during host startup (after DI is built, before the server accepts traffic if you await it). `StopAsync` runs on graceful shutdown with a token that trips after `HostOptions.ShutdownTimeout` (default 30s in .NET 8+, was 5s pre-.NET 6):

```csharp
public sealed class WarmupService(ICache cache, ILogger<WarmupService> log) : IHostedService
{
    public async Task StartAsync(CancellationToken ct) => await cache.PrimeAsync(ct);
    public Task StopAsync(CancellationToken ct) => Task.CompletedTask;
}
services.AddHostedService<WarmupService>();
```

### 3. `IHostApplicationLifetime`: lifecycle event hooks

Three `CancellationToken`s you can register callbacks against:

```csharp
var lifetime = app.Services.GetRequiredService<IHostApplicationLifetime>();

lifetime.ApplicationStarted.Register(()  => log.LogInformation("Up."));
lifetime.ApplicationStopping.Register(() => log.LogInformation("Draining..."));
lifetime.ApplicationStopped.Register(()  => log.LogInformation("Done."));
```

- `ApplicationStarted` — fires after the server starts listening.
- `ApplicationStopping` — first signal of shutdown; use it to stop accepting new work.
- `ApplicationStopped` — all hosted services have been stopped; last chance to flush.

You can also call `lifetime.StopApplication()` to trigger a graceful shutdown from code.

**Senior-level gotchas:**

- Don't do slow work in `StartAsync` without awaiting it explicitly — the host waits for it synchronously, blocking readiness probes. Offload to `ExecuteAsync` of a `BackgroundService` if it can run after the app is live.
- On containers, SIGTERM → graceful shutdown → `HostOptions.ShutdownTimeout` → SIGKILL. Tune the timeout to match your longest in-flight request (e.g. `services.Configure<HostOptions>(o => o.ShutdownTimeout = TimeSpan.FromSeconds(60));`).
- `IHostedService`s are started **in registration order** and stopped in **reverse** order. If service B depends on A being up, register A first.

## Q10. What is a Background Service?

A `BackgroundService` is an abstract base class that implements `IHostedService` and exposes a single `ExecuteAsync(CancellationToken stoppingToken)` method — the canonical way to run continuous or periodic work inside an ASP.NET Core / Generic Host app.

```csharp
public sealed class OutboxDispatcher(
    IServiceScopeFactory scopeFactory,
    ILogger<OutboxDispatcher> logger) : BackgroundService
{
    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        using var timer = new PeriodicTimer(TimeSpan.FromSeconds(5));
        while (await timer.WaitForNextTickAsync(stoppingToken))
        {
            try
            {
                await using var scope = scopeFactory.CreateAsyncScope();
                var dispatcher = scope.ServiceProvider.GetRequiredService<IOutbox>();
                await dispatcher.DispatchPendingAsync(stoppingToken);
            }
            catch (OperationCanceledException) { break; }
            catch (Exception ex)
            {
                logger.LogError(ex, "Outbox tick failed; continuing.");
            }
        }
    }
}

services.AddHostedService<OutboxDispatcher>();
```

How it fits into the host lifecycle:

- The host calls `StartAsync`, which on `BackgroundService` starts `ExecuteAsync` on the thread pool and returns immediately (it does **not** await the task). Your app starts accepting requests in parallel.
- `stoppingToken` is tripped when shutdown begins. Your loop must observe it and return.
- `StopAsync` waits for `ExecuteAsync` to finish, bounded by `HostOptions.ShutdownTimeout`.

**Use it for:** message pump workers, scheduled jobs, outbox/inbox dispatchers, cache warmers, telemetry flushers.

**Don't use it for:** one-shot startup init (do that inline after `builder.Build()`), or CPU-bound batch work (use a real job system like Hangfire/Quartz/DurableTask).

**Modern note (.NET 10):** .NET 8 introduced `BackgroundService.StartAsync` surfacing unhandled `ExecuteAsync` exceptions via `HostOptions.BackgroundServiceExceptionBehavior` (default: **crash the host**, vs. legacy "swallow and stop the service silently"). In .NET 10 this remains — always wrap your loop body in try/catch or a single unhandled exception tears down the process. See <https://learn.microsoft.com/en-us/dotnet/core/extensions/workers>.

**Senior-level gotchas:**

- `BackgroundService` is itself a **singleton**. Use `IServiceScopeFactory` for any scoped dependency (see Q8).
- Prefer `PeriodicTimer` over `Task.Delay` loops — it doesn't drift under load and cancels cleanly (see Q25).
- Never throw out of `ExecuteAsync` without catching first in production — `BackgroundServiceExceptionBehavior.StopHost` will kill the app. Log and continue unless the failure is unrecoverable.
- If you need multiple workers of the same type with different config, register them by distinct derived classes or use a keyed singleton factory — registering the same hosted-service type twice runs it twice.
