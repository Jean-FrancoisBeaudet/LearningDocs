## Q41. What is Swagger used for?

"Swagger" is shorthand for the tooling ecosystem built around the **OpenAPI Specification** — a language-agnostic JSON/YAML contract that describes what an HTTP API exposes: routes, verbs, parameters, request/response shapes, status codes, auth schemes. Once you have that contract, you get three concrete benefits:

1. **Interactive exploration UI** (Swagger UI, Scalar, ReDoc, NSwag Studio) — humans hit endpoints without Postman.
2. **Client SDK generation** (OpenAPI Generator, NSwag, Kiota) — typed clients in C#, TypeScript, Go, etc., from the spec.
3. **Contract testing & governance** — diff specs between versions to catch breaking changes; lint with Spectral; gate CI on it.

In ASP.NET Core, Swagger is how teams deliver a browsable API for frontend devs, partners, and internal tools, and how they generate clients so consumers aren't hand-rolling HTTP calls.

### The important distinction in .NET 9/10

Up through .NET 8, the default Web API template shipped **Swashbuckle.AspNetCore**, which produced *both* the OpenAPI document *and* the UI. Starting with .NET 9, Microsoft split the concerns:

- **`Microsoft.AspNetCore.OpenApi`** (first-party) — generates the OpenAPI document only. No UI.
- **UI is your choice** — the default template now wires up **Scalar** for the UI, but you can still use Swashbuckle or NSwag.

```csharp
// Program.cs (.NET 10 default)
var builder = WebApplication.CreateBuilder(args);
builder.Services.AddOpenApi();            // <- generates /openapi/v1.json

var app = builder.Build();

if (app.Environment.IsDevelopment())
{
    app.MapOpenApi();                     // spec endpoint
    app.MapScalarApiReference();          // UI (separate package: Scalar.AspNetCore)
}
```

**Modern note (.NET 10):** The generated document now defaults to **OpenAPI 3.1** (not 3.0) and can be served as **YAML** by suffixing the endpoint with `.yaml`/`.yml` (`app.MapOpenApi("/openapi/{documentName}.yaml")`). OpenAPI 3.1 aligns with JSON Schema draft 2020-12 — notably, `nullable: true` is gone; nullability is expressed with a type array (`"type": ["string", "null"]`). If you have custom transformers from .NET 9, the underlying library jump to OpenAPI.NET 2.0 is a breaking change: `OpenApiAny` is replaced by `JsonNode`.

**Senior-level gotchas:**

- OpenAPI ≠ Swagger UI. Don't expose the UI in production unless it's an intentionally public API — it's an information-disclosure vector.
- Per-version document: `builder.Services.AddOpenApi("v1")` / `"v2"`; wire one document per API version, not one giant doc.
- If you need vendor-specific extensions (x-ms-*, x-nullable), use `IOpenApiDocumentTransformer` / `IOpenApiOperationTransformer` / `IOpenApiSchemaTransformer` — don't hand-edit generated JSON.

## Q42. How to add documentation of endpoints, models and fields in Swagger?

There are three layers — use them together, not in competition.

### 1. XML doc comments (the free one)

In .NET 10, `Microsoft.AspNetCore.OpenApi` reads XML doc comments natively via a **compile-time source generator**. No runtime cost, no `IncludeXmlComments(...)` boilerplate (that was the Swashbuckle pattern).

Enable documentation generation in the csproj:

```xml
<PropertyGroup>
  <GenerateDocumentationFile>true</GenerateDocumentationFile>
  <NoWarn>$(NoWarn);CS1591</NoWarn> <!-- suppress "missing XML" noise -->
</PropertyGroup>
```

Then write comments the way you'd write them for IntelliSense:

```csharp
/// <summary>Creates a new order for the authenticated user.</summary>
/// <param name="request">Order payload (items, shipping address).</param>
/// <response code="201">Order created; Location header points to the new resource.</response>
/// <response code="400">Validation failed.</response>
/// <response code="409">Idempotency conflict — key already used.</response>
[HttpPost]
public async Task<Results<Created<Order>, ValidationProblem, Conflict>> Create(
    CreateOrderRequest request, CancellationToken ct) => /* ... */;
```

XML docs flow into: the operation `summary`/`description`, parameter descriptions, response descriptions, and **schema property descriptions** on DTOs. `<inheritdoc />` works. XML from referenced projects is picked up if those projects also set `GenerateDocumentationFile`.

### 2. Fluent metadata on Minimal APIs / endpoint conventions

Minimal APIs prefer metadata via fluent builders — these win over XML when both are present, and they're discoverable in code:

```csharp
app.MapPost("/orders", CreateOrder)
    .WithName("CreateOrder")
    .WithTags("Orders")
    .WithSummary("Create an order")
    .WithDescription("Creates an order for the authenticated user and returns the new resource URL.")
    .Accepts<CreateOrderRequest>("application/json")
    .Produces<Order>(StatusCodes.Status201Created)
    .ProducesValidationProblem()
    .ProducesProblem(StatusCodes.Status409Conflict)
    .RequireAuthorization();
```

For controllers, the attribute-based equivalents are `[ProducesResponseType]`, `[Consumes]`, `[Tags]`, plus XML for prose.

### 3. Transformers for structural changes

When XML + fluent metadata can't express it (custom security schemes, global headers, polymorphic schemas, vendor extensions), use the transformer pipeline:

```csharp
builder.Services.AddOpenApi(options =>
{
    options.AddDocumentTransformer<BearerSecuritySchemeTransformer>();  // adds auth to components
    options.AddOperationTransformer((op, ctx, _) =>                      // adds x-rate-limit header
    {
        op.Parameters ??= [];
        op.Parameters.Add(new OpenApiParameter
        {
            Name = "X-Correlation-Id",
            In = ParameterLocation.Header,
            Schema = new OpenApiSchema { Type = JsonSchemaType.String }
        });
        return Task.CompletedTask;
    });
});
```

### Describing the schema itself

For DTOs, XML summaries on properties become schema descriptions. Validation attributes carry type info: `[Required]`, `[Range]`, `[StringLength]`, `[RegularExpression]`, `[EmailAddress]` are reflected as OpenAPI schema facets (`required`, `minimum`/`maximum`, `maxLength`, `pattern`, `format`). Nullable reference types → `nullable` info in the schema. Records work identically — put XML on the primary constructor parameters.

```csharp
/// <summary>An order line item.</summary>
public record OrderLine(
    /// <summary>SKU identifier (e.g. "ABC-123").</summary>
    [Required, RegularExpression("^[A-Z]{3}-\\d{3}$")] string Sku,
    /// <summary>Quantity; must be 1–1000.</summary>
    [Range(1, 1000)] int Quantity);
```

**Senior-level gotchas:**

- Don't duplicate. If `[ProducesResponseType(201)]` is already on the endpoint, don't also `<response code="201">` — pick one source of truth per fact.
- XML-comment summaries on a controller *class* become the tag description; you rarely want that. Use `[Tags("Orders")]` + an `IOpenApiDocumentTransformer` to set tag descriptions explicitly.
- If you need examples in the spec, attach them via `WithOpenApi(op => { op.RequestBody.Example = ...; return op; })` — there's no clean attribute for it.
- Swashbuckle users: `[SwaggerOperation]`, `[SwaggerResponseExample]`, `ISchemaFilter` still work if you stay on Swashbuckle, but they do *not* interop with `Microsoft.AspNetCore.OpenApi`. Pick one pipeline.

**Sources:**
- [ASP.NET Core OpenAPI XML documentation comment support](https://learn.microsoft.com/en-us/aspnet/core/fundamentals/openapi/openapi-comments?view=aspnetcore-10.0)
- [Include OpenAPI metadata in an ASP.NET Core app](https://learn.microsoft.com/en-us/aspnet/core/fundamentals/openapi/include-metadata?view=aspnetcore-10.0)
- [Generate OpenAPI documents](https://learn.microsoft.com/en-us/aspnet/core/fundamentals/openapi/aspnetcore-openapi?view=aspnetcore-10.0)

## Q43. How to get a connection string from the configuration?

The idiomatic one-liner:

```csharp
var cs = builder.Configuration.GetConnectionString("Default");
// shorthand for builder.Configuration["ConnectionStrings:Default"]
```

`GetConnectionString(name)` is an extension in `Microsoft.Extensions.Configuration` that reads from the conventional `ConnectionStrings` section:

```json
// appsettings.json
{
  "ConnectionStrings": {
    "Default": "Server=.;Database=App;Trusted_Connection=True;TrustServerCertificate=True"
  }
}
```

But the interesting question is **where the value actually lives** in each environment, because hardcoding a production connection string in `appsettings.Production.json` is a security smell.

### Layering (lowest to highest precedence, default host)

1. `appsettings.json`
2. `appsettings.{Environment}.json`
3. **User Secrets** (`dotnet user-secrets`) — Development only
4. Environment variables (including container env)
5. Command-line args
6. Custom providers: **Azure Key Vault**, **Azure App Configuration**, AWS Parameter Store, etc.

Environment-variable mapping uses `__` as the separator (works across OSes):

```bash
# maps to ConnectionStrings:Default
ConnectionStrings__Default="Server=...;Database=...;..."
```

### Wiring Key Vault / App Configuration (production pattern)

```csharp
if (!builder.Environment.IsDevelopment())
{
    // DefaultAzureCredential -> managed identity in production, azd/VS in dev
    builder.Configuration.AddAzureKeyVault(
        new Uri(builder.Configuration["KeyVault:Uri"]!),
        new DefaultAzureCredential());
}
```

Now your code still does `GetConnectionString("Default")` — Key Vault stores a secret named `ConnectionStrings--Default` (double-dash maps to colon). No code change between environments.

### Prefer strongly-typed binding + validation for non-trivial configs

For anything richer than a single string, bind into an options class and fail fast:

```csharp
public sealed class DatabaseOptions
{
    [Required] public required string ConnectionString { get; init; }
    [Range(1, 600)] public int CommandTimeoutSeconds { get; init; } = 30;
}

builder.Services
    .AddOptions<DatabaseOptions>()
    .Bind(builder.Configuration.GetSection("Database"))
    .ValidateDataAnnotations()
    .ValidateOnStart();   // throws at host startup, not on first request
```

### .NET Aspire

If you're in an Aspire solution, the AppHost resolves connection strings declaratively:

```csharp
// AppHost
var db = builder.AddSqlServer("sql").AddDatabase("app");
builder.AddProject<Projects.Api>("api").WithReference(db);
```

The service project then does `builder.AddSqlServerDbContext<AppDbContext>("app")` — connection string discovery, health checks, and OpenTelemetry are wired automatically.

**Senior-level gotchas:**

- **Never** check connection strings into source control — pre-commit hooks / `git-secrets` / GitHub push protection are cheap insurance.
- `GetConnectionString` returns `string?`. Don't null-forget it — throw a specific startup error if missing: `?? throw new InvalidOperationException("ConnectionStrings:Default not configured")`.
- Don't re-read `IConfiguration` per request in hot paths. Bind once, inject `IOptions<DatabaseOptions>`.
- EF Core: prefer `builder.Services.AddDbContext<AppDbContext>(o => o.UseSqlServer(cs))` over constructing `DbContext` manually — pooling, scoping, and lifetime are handled for you.
- Rotating secrets: with `IOptionsMonitor<T>` and a reloading provider (App Configuration), you can rotate without a deploy, *but* DI-scoped services that captured the old value won't see the change until a new scope. Plan rotation accordingly.

## Q44. How can you deploy an ASP.NET Core application?

There are two independent choices: **what artifact to produce** and **where to run it**.

### Artifact choices (`dotnet publish`)

| Mode | Command | When to use |
|---|---|---|
| Framework-dependent | `dotnet publish -c Release` | Default. Smallest artifact (~few MB); requires the .NET runtime on the host. |
| Self-contained | `... -r linux-x64 --self-contained` | No runtime on host; artifact ~70–80 MB. Useful for appliances, air-gapped boxes. |
| Single-file | `... -p:PublishSingleFile=true` | One executable; bundles assemblies. Faster scp/upload. |
| Trimmed | `... -p:PublishTrimmed=true` | Ships only reachable IL. Requires trim-safe code; reflection-heavy libs break. |
| **NativeAOT** | `... -p:PublishAot=true` | Ahead-of-time compilation to native. Sub-second cold start, no JIT, lower memory. Limits: no runtime code gen, no dynamic assembly loading, EF Core has constraints. Mature in .NET 10 for Minimal APIs, gRPC, Worker Services. |

### Hosting targets

**1. Azure PaaS (the path of least resistance)**
- **Azure App Service** — `az webapp deploy` or zip deploy; integrates with Key Vault, App Insights, managed identity. Slots for blue/green.
- **Azure Container Apps** — serverless containers with KEDA autoscaling, revisions, Dapr; best fit for microservices that need scale-to-zero.
- **AKS** — full Kubernetes when you outgrow Container Apps.
- **Azure Functions** (isolated worker model) — for event-driven workloads.

**2. Containers (the portable default)**

In .NET 10, you don't need a Dockerfile. Container publishing is built into the SDK (since 8.0.200, mature in 10):

```bash
dotnet publish -t:PublishContainer \
    -p:ContainerRegistry=myregistry.azurecr.io \
    -p:ContainerRepository=myorg/api \
    -p:ContainerImageTag=$(git rev-parse --short HEAD)
```

MSBuild properties in the csproj can pin the base image, set labels, add ENV, expose ports, etc. For a hand-written Dockerfile, prefer **chiseled Ubuntu** base images — they're ~15 MB, no shell, no package manager:

```dockerfile
# syntax=docker/dockerfile:1.7
FROM mcr.microsoft.com/dotnet/sdk:10.0 AS build
WORKDIR /src
COPY *.sln ./
COPY src/ ./src/
RUN dotnet publish src/Api/Api.csproj -c Release -o /app

FROM mcr.microsoft.com/dotnet/aspnet:10.0-noble-chiseled AS runtime
WORKDIR /app
COPY --from=build /app .
USER $APP_UID            # non-root by default in chiseled images
ENTRYPOINT ["./Api"]
```

**3. IIS on Windows** — ASP.NET Core Module (ANCM) in **in-process** (default; app runs in w3wp) or **out-of-process** (IIS reverse-proxies to Kestrel). In-process is faster; out-of-process is required for some legacy modules.

**4. Linux with a reverse proxy** — Kestrel behind **nginx**, **YARP**, or **HAProxy**, managed by `systemd`:

```ini
# /etc/systemd/system/myapi.service
[Service]
WorkingDirectory=/var/www/myapi
ExecStart=/var/www/myapi/MyApi
Restart=always
Environment=ASPNETCORE_ENVIRONMENT=Production
User=www-data
```

**5. Windows Service** — `builder.Host.UseWindowsService()` (from `Microsoft.Extensions.Hosting.WindowsServices`) + `sc.exe create` or `New-Service`.

### Deploy-time concerns a senior wires up

- **Readiness vs liveness** endpoints via `app.MapHealthChecks("/healthz/ready", new() { Predicate = r => r.Tags.Contains("ready") })` — Kubernetes and Container Apps need the distinction to avoid cycling pods that are just warming up.
- **Zero-downtime deploys**: slots (App Service) or rolling updates with `preStop` hooks (Kubernetes) — give Kestrel time to drain (`HostOptions.ShutdownTimeout`).
- **Forwarded headers**: behind any L7 proxy, call `app.UseForwardedHeaders()` early — otherwise `Request.Scheme`/`RemoteIpAddress` are wrong and auth callbacks break.
- **Config separation**: never bake secrets into the image. Env vars, Key Vault references, or mounted secret volumes.
- **Telemetry**: OpenTelemetry for traces/metrics/logs; ship to App Insights, Grafana Tempo, Honeycomb, etc. Use `OTEL_*` env vars rather than hardcoding exporters.

**Senior-level gotchas:**

- NativeAOT + EF Core: EF Core 10 supports AOT with *compiled models* — you must generate them at build (`dotnet ef dbcontext optimize`). Runtime query compilation is limited.
- `dotnet publish /t:PublishContainer` tags as `latest` by default — set `ContainerImageTag` explicitly, never deploy `:latest` to production.
- App Service Windows: in-process ANCM pins your process to a single worker; if you need scale-out, use Linux + containers or tune `maxProcesses`.
- Health-check predicates matter: a readiness probe that hits the database means DB blips restart pods. Keep readiness on *your* dependencies, liveness on *your* process.

**Sources:**
- [Containerize an app with dotnet publish](https://learn.microsoft.com/en-us/dotnet/core/containers/sdk-publish)
- [.NET SDK container creation overview](https://learn.microsoft.com/en-us/dotnet/core/containers/overview)

## Q45. How to configure logging in ASP.NET Core?

`WebApplication.CreateBuilder(args)` wires up logging defaults for you: the `Logging` section of configuration is read, and the providers **Console**, **Debug**, **EventSource**, and (on Windows) **EventLog** are registered. So the zero-config path is already structured logging to stdout.

### Configuration-driven filters

```json
// appsettings.json
{
  "Logging": {
    "LogLevel": {
      "Default": "Information",
      "Microsoft.AspNetCore": "Warning",
      "Microsoft.EntityFrameworkCore.Database.Command": "Warning"
    },
    "Console": {
      "FormatterName": "json",
      "FormatterOptions": {
        "IncludeScopes": true,
        "TimestampFormat": "O",
        "UseUtcTimestamp": true
      }
    }
  }
}
```

Filters cascade: "most specific category wins". `Microsoft.EntityFrameworkCore.*` filters inherit from `Microsoft.EntityFrameworkCore` unless a more specific one exists.

### Adding / replacing providers

```csharp
builder.Logging.ClearProviders();      // drop defaults if you're standardizing elsewhere
builder.Logging.AddJsonConsole(o =>    // structured JSON to stdout (best for containers)
{
    o.IncludeScopes = true;
    o.TimestampFormat = "O";
    o.UseUtcTimestamp = true;
});

builder.Logging.AddOpenTelemetry(o =>  // forward to OTLP collector / App Insights / Datadog
{
    o.IncludeFormattedMessage = true;
    o.IncludeScopes = true;
    o.ParseStateValues = true;
});
```

OpenTelemetry logs are **the** modern integration point: one exporter, three signals (logs, traces, metrics), vendor-neutral. Configure it at the services level:

```csharp
builder.Services.AddOpenTelemetry()
    .WithLogging(l => l.AddOtlpExporter())
    .WithMetrics(m => m.AddAspNetCoreInstrumentation().AddOtlpExporter())
    .WithTracing(t => t.AddAspNetCoreInstrumentation().AddOtlpExporter());
```

### Structured logging — the non-negotiable part

Always use the message template with placeholders. **Never** interpolate:

```csharp
// ❌ loses structure, forces string allocation even when disabled
logger.LogInformation($"Order {orderId} created for {userId}");

// ✅ placeholders become indexed fields in the log record
logger.LogInformation("Order {OrderId} created for {UserId}", orderId, userId);
```

The template is the message identity for search and aggregation — every sink can filter by `OrderId` as a field, not by regexing the rendered string.

### High-performance: `LoggerMessage` source generator

For hot paths, the `[LoggerMessage]` source generator wins on three dimensions: no boxing of value-type args, no per-call template parse, compile-time diagnostics if you mess up placeholder names.

```csharp
public sealed partial class OrderService
{
    private readonly ILogger<OrderService> _logger;
    public OrderService(ILogger<OrderService> logger) => _logger = logger;

    [LoggerMessage(EventId = 1001, Level = LogLevel.Information,
        Message = "Order {OrderId} created for {UserId} with {ItemCount} items")]
    private partial void LogOrderCreated(long orderId, Guid userId, int itemCount);

    [LoggerMessage(EventId = 1002, Level = LogLevel.Warning,
        Message = "Idempotency conflict on key {IdempotencyKey}")]
    private partial void LogIdempotencyConflict(string idempotencyKey);

    public async Task CreateAsync(CreateOrderRequest req, CancellationToken ct)
    {
        // ...
        LogOrderCreated(order.Id, req.UserId, req.Items.Count);
    }
}
```

This is the **recommended** style in modern .NET — `LoggerMessage.Define` (the older pattern) is still valid but noisier.

### Scopes and correlation

`BeginScope` attaches structured fields to every log inside a `using` block. Combine with `Activity.Current` for automatic trace-id correlation:

```csharp
using var scope = _logger.BeginScope(new Dictionary<string, object?>
{
    ["CorrelationId"] = correlationId,
    ["TenantId"] = tenantId
});
// every log in here carries CorrelationId and TenantId
```

Enable `IncludeScopes = true` on the console/JSON formatter for scopes to actually render.

### Dynamic level changes

`IOptionsMonitor<LoggerFilterOptions>` reloads when `appsettings.json` changes on disk (or App Configuration pushes). You can flip a category to `Debug` in production without redeploying — invaluable for diagnosing prod issues.

**Senior-level gotchas:**

- `logger.LogInformation($"...")` defeats the entire point of `ILogger`. Enforce it: enable analyzer **CA2254** ("Template should be a static expression").
- `ILogger<T>` takes `T` as the *category*. Don't inject `ILoggerFactory` just to create a logger — use the typed one so categories are stable.
- Async method logging: `LogDebug("start {Op}")` / `LogDebug("end {Op} in {Elapsed}ms")` bookends cost real CPU at high RPS. Use `LoggerMessage` source-generated methods and check `_logger.IsEnabled(LogLevel.Debug)` only when constructing args is expensive.
- Exception logging: `logger.LogError(ex, "Failed to process {OrderId}", orderId)` — *first* arg is the exception, *not* concatenated into the message.
- Don't log request bodies blindly — see Q60 on GDPR/sensitive data. Prefer a middleware that redacts known sensitive fields before emission.
- Sink selection: console-to-stdout + OpenTelemetry exporter is the containerized default. Avoid file logging inside containers (write to stdout, let the orchestrator collect it).
