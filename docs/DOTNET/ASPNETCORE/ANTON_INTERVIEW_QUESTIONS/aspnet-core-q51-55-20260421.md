## Q51. What is a custom model-binder class and how do you implement one?

**Model binding** is the controller-pipeline step that turns raw request data (route values, query string, form, body, headers) into action-method parameters. The default set of binders handles primitives, complex types, files, `JsonElement`, collections, and `CancellationToken`. A **custom model binder** is what you write when none of that is enough — typically for a domain primitive whose wire format doesn't map cleanly to JSON deserialization or a built-in converter.

### The pipeline

Three interfaces collaborate:

```
Request → IModelBinderProvider.GetBinder(ctx) → IModelBinder.BindModelAsync(ctx) → ctx.Result
```

- `IModelBinderProvider` decides *which* binder handles a given `ModelBinderProviderContext` (usually based on `ctx.Metadata.ModelType` or an attribute).
- `IModelBinder.BindModelAsync(ModelBindingContext)` does the actual conversion and sets `bindingContext.Result = ModelBindingResult.Success(value)` or `.Failed()`.
- Providers are ordered; first match wins.

### Minimal, realistic example — binding a composite `TenantId` from an `X-Tenant` header plus a route value

```csharp
public readonly record struct TenantId(Guid Region, Guid Tenant);

public sealed class TenantIdBinder : IModelBinder
{
    public Task BindModelAsync(ModelBindingContext ctx)
    {
        var region = ctx.HttpContext.Request.Headers["X-Region"].ToString();
        var tenant = ctx.ValueProvider.GetValue("tenantId").FirstValue;

        if (!Guid.TryParse(region, out var r) || !Guid.TryParse(tenant, out var t))
        {
            ctx.ModelState.TryAddModelError(ctx.ModelName,
                "X-Region header and tenantId route value must be GUIDs");
            ctx.Result = ModelBindingResult.Failed();
            return Task.CompletedTask;
        }

        ctx.Result = ModelBindingResult.Success(new TenantId(r, t));
        return Task.CompletedTask;
    }
}

public sealed class TenantIdBinderProvider : IModelBinderProvider
{
    public IModelBinder? GetBinder(ModelBinderProviderContext ctx) =>
        ctx.Metadata.ModelType == typeof(TenantId) ? new TenantIdBinder() : null;
}
```

Registering it globally:

```csharp
builder.Services.AddControllers(o =>
{
    o.ModelBinderProviders.Insert(0, new TenantIdBinderProvider()); // front of queue
});
```

Or per-parameter, skipping the provider:

```csharp
public IActionResult Get([ModelBinder(typeof(TenantIdBinder))] TenantId id) => Ok(id);
```

### What a senior reaches for **first** — before writing a custom binder

Custom model binders are heavy. Nine times out of ten one of these is the right answer:

| Alternative | When it fits |
|---|---|
| **`TypeConverter`** | Converting a single string token (route/query value) into a value type. Low ceremony, works across MVC/Minimal APIs/configuration binding. |
| **`IParsable<T>` / `ISpanParsable<T>`** (C# 11+) | The modern replacement for `TypeConverter`. Minimal APIs auto-detect `TryParse` — zero registration. |
| **`static BindAsync(HttpContext) → ValueTask<T?>`** | Minimal API equivalent of a custom binder. Full access to the request, async friendly, AOT-safe. |
| **`JsonConverter<T>`** | Shape lives inside the JSON body. Don't write a model binder for JSON — write a converter. |
| **Action filter** (`IAsyncActionFilter`) | Enriching/validating *after* normal binding succeeds (e.g., hydrating a related entity from an ID). |
| **Custom `IValueProvider`** | You need a new *source* of values (e.g., an RFC-ish request header format across many binders), not a new conversion. |

### `[FromBody]` vs model binders — the common confusion

`[FromBody]` **doesn't go through a model binder at all** — it uses the configured `InputFormatter` (`SystemTextJsonInputFormatter` by default). If you need to customize JSON body parsing, register a `JsonConverter<T>` (or write an `IInputFormatter`); writing an `IModelBinder` for a `[FromBody]` parameter is a dead end.

### Minimal API story

MVC model binders do **not** work with Minimal APIs. The Minimal API surface is intentionally narrower and AOT-first:

```csharp
public readonly record struct OrderId(long Value)
{
    public static bool TryParse(string? s, IFormatProvider? _, out OrderId r)
    {
        if (long.TryParse(s, out var v) && v > 0) { r = new OrderId(v); return true; }
        r = default; return false;
    }

    // For whole-request bindings (body + headers + …), use BindAsync instead:
    public static ValueTask<OrderId?> BindAsync(HttpContext ctx, ParameterInfo _) { /* … */ }
}

app.MapGet("/orders/{id}", (OrderId id) => /* … */);
```

### Senior-level gotchas

- **Don't `await` synchronous work** inside `BindModelAsync` — return `Task.CompletedTask`. Async model binders allocate a state machine per request.
- **Always set `ctx.Result`**, even on failure. Leaving it as the default `None` silently falls through to the next provider and produces baffling "parameter is null" errors.
- **Write to `ModelState`, don't throw.** Throwing inside a binder surfaces as a 500, not a 400 — the whole point of binding is graceful rejection.
- **Providers are instantiated once**. Don't stash per-request state in the provider or binder — they're reused across requests.
- **Per-parameter `[ModelBinder(…)]` beats global registration** when only one action needs the binder. Avoids poisoning the global order.
- **AOT / trimming**: classic `IModelBinderProvider` uses reflection heavily; Minimal APIs with `TryParse`/`BindAsync` are AOT-friendly by design. If AOT matters, prefer the latter.

## Q52. How do you enable or disable a controller or action? Which attribute is used?

Two dedicated attributes handle the "remove from the pipeline entirely" case, and a few orthogonal tools handle the softer "hide it" / "gate it" cases. They solve different problems — don't confuse them.

### `[NonController]` — exclude a class

Applied to a class, it tells the controller feature provider "this is not a controller, even if the name says so":

```csharp
[NonController]
public class ReportController // won't be routed; won't appear in ApiExplorer
{
}
```

Useful when you have a base class that ends in `Controller` for naming convention but isn't meant to be routed itself, or a legacy class you want to keep compiled but dead.

### `[NonAction]` — exclude a method

Applied to a public method on a controller, it stops the action selector from treating it as an action:

```csharp
public class OrdersController : ControllerBase
{
    [HttpGet("{id:long}")]
    public IActionResult Get(long id) => Ok(/* … */);

    [NonAction]                              // public helper, still testable, not routed
    public Order HydrateFromCache(long id) => /* … */;
}
```

Without `[NonAction]`, any `public` method on a controller is a candidate action — which is why helpers on controllers should normally be `private` or `protected`.

### Conditional registration at startup

The two attributes above are compile-time decisions. For environment- or config-driven toggling, exclude at service registration:

```csharp
builder.Services.AddControllers(o =>
{
    if (!builder.Configuration.GetValue<bool>("Features:Admin"))
        o.Conventions.Add(new RemoveControllerConvention<AdminController>());
});

sealed class RemoveControllerConvention<T>(): IApplicationModelConvention
{
    public void Apply(ApplicationModel m) =>
        m.Controllers.RemoveAll(c => c.ControllerType == typeof(T).GetTypeInfo());
}
```

Or, equivalently, use `IActionDescriptorChangeProvider` to re-evaluate routes when a feature flag flips without restart.

### Feature flags (the production-correct pattern)

`Microsoft.FeatureManagement.AspNetCore` gives you `[FeatureGate]` — returns **404** when the named flag is off, so probes don't leak the endpoint's existence:

```csharp
[FeatureGate("NewCheckout")]
[HttpPost("/api/v2/checkout")]
public Task<IActionResult> CheckoutV2(CheckoutRequest r) => /* … */;
```

Combined with Azure App Configuration, you can flip features in production without a redeploy.

### Hide from docs but keep the endpoint live

`[ApiExplorerSettings(IgnoreApi = true)]` — removes the action from OpenAPI / Swagger / Scalar. The route still works; it's just invisible to the spec. Use for internal or deprecated endpoints you can't delete yet.

### Summary table

| Goal | Tool |
|---|---|
| Permanently prevent a class from being a controller | `[NonController]` |
| Permanently prevent a public method from being routed | `[NonAction]` |
| Toggle by environment / config at startup | `IApplicationModelConvention` |
| Toggle at runtime | `[FeatureGate]` + `IFeatureManager` |
| Hide from OpenAPI only | `[ApiExplorerSettings(IgnoreApi = true)]` |
| Gate behind auth policy | `[Authorize(Policy = "…")]` |

### Senior-level gotchas

- `[NonAction]` doesn't make a method `private` — test frameworks and reflection can still reach it. Don't rely on it for security.
- `[FeatureGate]` returns 404 by default; flipping to 403 leaks existence. Keep the default unless you have a specific reason.
- Convention-based removal runs **once** at startup. If your flag needs to react to runtime changes, use `[FeatureGate]` or gate inside the action — don't try to mutate the controller model on the fly.
- `[ApiExplorerSettings]` affects doc generation, not auth — hidden ≠ private. Reverse-engineering the URL still calls the endpoint.

## Q53. What is Method Injection in a controller and how does it differ from constructor injection?

Both serve the same goal — resolving a dependency from DI — but at different scopes.

### Constructor injection — the default

Resolved **once when the controller instance is created** (per request in MVC):

```csharp
public sealed class OrdersController(IOrderService orders, IClock clock) : ControllerBase
{
    [HttpGet("{id:long}")]
    public Task<IActionResult> Get(long id) => /* uses orders + clock */;

    [HttpPost]
    public Task<IActionResult> Create(CreateOrderRequest r) => /* uses orders + clock */;
}
```

Primary-constructor syntax (C# 12+) is idiomatic here — concise, the fields are auto-captured.

### Method injection — per action

`[FromServices]` on an action parameter resolves the service from the request scope **only for that action**:

```csharp
public sealed class OrdersController(IOrderService orders) : ControllerBase
{
    [HttpGet("/orders/{id:long}/export")]
    public Task<IActionResult> Export(
        long id,
        [FromServices] IOrderExporter exporter,
        [FromServices] IBlobStorage blobs,
        CancellationToken ct) => /* … */;
}
```

### When to prefer which

- **Constructor injection**: dependency is used by most (or all) actions. Fewer parameters at the call site, cleaner signatures, better testability.
- **Method injection**: dependency is used by **one** action and is either (a) expensive to construct, (b) architecturally weird to require of a whole controller (think: an S3 client used only by `Export`), or (c) keyed — `[FromKeyedServices("pdf")] IExporter`.

The second reason is the underrated one: a controller that needs `IBlobStorage` only for one export endpoint pays the resolution cost on every other action if you put it in the constructor. In practice that's usually negligible, but it matters when the service has a non-trivial constructor.

### `[FromServices]` is often implicit in .NET 10

Since .NET 7, if a parameter's type is registered in DI and the controller has `[ApiController]`, the attribute is inferred — you can drop it:

```csharp
[HttpGet("/orders/{id:long}/export")]
public Task<IActionResult> Export(long id, IOrderExporter exporter, CancellationToken ct)
    => /* implicit FromServices for IOrderExporter */;
```

If a complex type is registered both in DI **and** would otherwise be bound from body, the DI resolution wins — which is almost always what you want, but can surprise you. You can disable the inference:

```csharp
builder.Services.Configure<ApiBehaviorOptions>(o =>
    o.DisableImplicitFromServicesParameters = true);
```

### Keyed services (.NET 8+)

`[FromKeyedServices]` works in both slots:

```csharp
// Constructor
public OrdersController([FromKeyedServices("retry")] IOrderService orders) { }

// Action
public IActionResult Export([FromKeyedServices("pdf")] IOrderExporter e) => /* … */;
```

### Senior-level gotchas

- **Don't use `[FromServices]` to escape a growing constructor.** A constructor with 10 dependencies is a design smell, not a binding smell — split the controller.
- **Scoping still applies.** `[FromServices] ISomethingScoped` is resolved from the *current request scope*, not a fresh one. You can't use it to get "a new transient each time" if the registration is scoped.
- **Property injection is not supported by the built-in DI container.** Don't confuse it with method injection. If you see `[Inject]` on a property, that's Blazor, not MVC.
- **Integration testing**: method-injected services are still fully swappable via `IConfigureOptions<…>` and `WebApplicationFactory<T>.WithWebHostBuilder(b => b.ConfigureTestServices(…))`. No special handling needed.
- **Minimal APIs**: the same `[FromServices]` / implicit-DI rules apply to endpoint delegates, including keyed services.

## Q54. How do you implement validation in controller actions? How do you handle invalid models?

Validation in ASP.NET Core has two phases:

1. **During model binding** — the binder materializes the parameter, then runs validation on it, populating `ModelState` with errors keyed by property path.
2. **In the action method** — you inspect `ModelState.IsValid` (or rely on the automatic 400 response from `[ApiController]`).

### Validators: what runs, in order

- **`System.ComponentModel.DataAnnotations`** attributes on the model (`[Required]`, `[StringLength]`, `[Range]`, `[RegularExpression]`, `[EmailAddress]`, custom `ValidationAttribute`). These are the built-in default.
- **`IValidatableObject`** on the model — implement `Validate(ValidationContext)` for cross-field validation that DataAnnotations can't express.
- **Custom attributes** derived from `ValidationAttribute`.
- **FluentValidation** — third-party library; registered via `builder.Services.AddValidatorsFromAssemblyContaining<…>()`. It replaces DataAnnotations at the MVC layer if you wire `AddFluentValidationAutoValidation()`. Mature ecosystems almost universally adopt it for non-trivial rules.

### The `[ApiController]` behavior you get for free

With the default API template:

```csharp
[ApiController]
[Route("api/[controller]")]
public sealed class OrdersController : ControllerBase
{
    [HttpPost]
    public IActionResult Create(CreateOrderRequest request)
    {
        // We never get here if request is invalid — the framework returned 400 already.
        return Created(/* … */);
    }
}

public sealed record CreateOrderRequest(
    [Required, StringLength(64)] string Sku,
    [Range(1, 1000)] int Quantity,
    [EmailAddress] string? NotifyEmail);
```

`[ApiController]` installs `ModelStateInvalidFilter`, which short-circuits invalid requests with a **`ValidationProblemDetails`** response (RFC 7807 / 9457) — 400 status, `application/problem+json`, errors grouped by property:

```json
{
  "type": "https://tools.ietf.org/html/rfc9110#section-15.5.1",
  "title": "One or more validation errors occurred.",
  "status": 400,
  "errors": {
    "Sku":      ["The Sku field is required."],
    "Quantity": ["The field Quantity must be between 1 and 1000."]
  }
}
```

### Manual handling — MVC controllers without `[ApiController]` or when you need custom behavior

```csharp
[HttpPost]
public IActionResult Create(CreateOrderRequest request)
{
    if (!ModelState.IsValid)
        return ValidationProblem(ModelState); // same RFC 9457 shape

    // …
    return CreatedAtAction(nameof(Get), new { id = order.Id }, order);
}
```

### Customizing the invalid-model response

Replace the default factory to add correlation IDs, extensions, custom status, etc.:

```csharp
builder.Services.Configure<ApiBehaviorOptions>(o =>
{
    o.InvalidModelStateResponseFactory = ctx =>
    {
        var problem = new ValidationProblemDetails(ctx.ModelState)
        {
            Type = "https://api.example.com/errors/validation",
            Title = "Request validation failed",
            Status = StatusCodes.Status400BadRequest,
            Instance = ctx.HttpContext.Request.Path,
        };
        problem.Extensions["traceId"] = Activity.Current?.Id ?? ctx.HttpContext.TraceIdentifier;
        return new BadRequestObjectResult(problem)
        {
            ContentTypes = { "application/problem+json" }
        };
    };
});
```

### **Modern note (.NET 10 / C# 14)** — built-in Minimal API validation via source generator

New in .NET 10: `builder.Services.AddValidation()` enables DataAnnotations-based validation for **Minimal APIs**, backed by a **source generator** that emits concrete validator code at compile time. No reflection at runtime → **AOT-safe**, faster startup, smaller allocations. The generator discovers validatable types referenced by endpoint handlers (explicitly via `[ValidatableType]` or transitively through parameters/base types) and wires an endpoint filter that produces `ValidationProblemDetails` on failure.

```csharp
var builder = WebApplication.CreateBuilder(args);
builder.Services.AddValidation();              // new in .NET 10
builder.Services.AddProblemDetails();          // consistent error shape

var app = builder.Build();

app.MapPost("/orders", (CreateOrderRequest req) => Results.Created())
   .DisableValidation();                       // opt out for a specific endpoint if needed

public sealed record CreateOrderRequest(
    [Required, StringLength(64)] string Sku,
    [Range(1, 1000)] int Quantity);
```

Controllers continue to use the reflection-based DataAnnotations validator today — the source-generator path is currently a Minimal API feature. For full AOT of a controller-based app, FluentValidation with its own source generator, or hand-rolled validators, remain the options.

### Senior-level gotchas

- **Validation runs after binding.** If binding fails (e.g., `"abc"` sent where `int` was expected), the action still short-circuits with a 400 — but the error is a *binding* error, not a *validation* one. The `ValidationProblemDetails` shape is the same either way; don't write tests that distinguish them.
- **Required reference types**: with nullable reference types enabled, non-nullable reference properties are treated as `[Required]` implicitly since .NET 6. Disable with `options.SuppressImplicitRequiredAttributeForNonNullableReferenceTypes = true` if it's noisy.
- **`IValidatableObject.Validate` runs only if DataAnnotations validation passed**. Don't rely on it for critical rules that should also fire when simpler validators fail.
- **Don't validate DTOs as domain objects.** Input validation is "is this message well-formed?" — business rules ("can this user place this order?") belong in the domain layer, not in `[Range]`.
- **Error localization**: `[Range]` / `[Required]` messages are resource-backed; set `builder.Services.AddLocalization()` and configure `DataAnnotationsLocalizationOptions` if you ship multiple languages.
- **FluentValidation + `[ApiController]`**: `AddFluentValidationAutoValidation()` plugs into the same 400-short-circuit behavior — you don't lose the automatic response factory.

## Q55. Explain the lifecycle of a controller in ASP.NET Core. When is it instantiated and disposed?

A controller instance is **scoped to a single HTTP request**: the activator creates it after routing has selected the action, the filter pipeline runs around it, and the framework disposes it when the response has finished writing.

### Step-by-step

```
Request arrives
  ↓
Routing → endpoint selected (controller + action descriptor)
  ↓
IControllerActivator.Create(context)
  ├── DefaultControllerActivator (default)
  │      └── ITypeActivatorCache → ActivatorUtilities.CreateInstance
  │             └── resolves constructor deps from the request's IServiceScope
  │
  └── ServiceBasedControllerActivator (opt-in via AddControllersAsServices())
         └── IServiceProvider.GetRequiredService(controllerType)
  ↓
Authorization filters
  ↓
Resource filters              (OnResourceExecuting)
  ↓
Model binding                 (ControllerContext wired, action args materialized)
  ↓
Action filters                (OnActionExecuting)
  ↓
Action method executes        ← your code
  ↓
Action filters                (OnActionExecuted)
  ↓
Result filters / IActionResult.ExecuteResultAsync
  ↓
Resource filters              (OnResourceExecuted)
  ↓
IControllerActivator.Release(context, controller)   ← disposes if IDisposable / IAsyncDisposable
  ↓
Response flushed, request scope disposed
```

Every exception path funnels through **Exception filters** (action-level) or the global exception handler middleware.

### Activation — `DefaultControllerActivator` vs `ServiceBasedControllerActivator`

- **Default**: `DefaultControllerActivator` uses `ITypeActivatorCache` → `ActivatorUtilities`. The controller *itself* is not registered in DI; only its constructor dependencies are. This is the zero-config default.
- **Service-based**: `builder.Services.AddControllers().AddControllersAsServices()` registers every discovered controller in the DI container, and the `ServiceBasedControllerActivator` resolves them from there. This unlocks:
  - Decoration (Scrutor-style) of controllers.
  - Keyed controller registration.
  - Lifetime overrides (though scoped is really the only sane option).
  - Interception via DI container features (Autofac interceptors, etc.).

Most teams never need it — but it's the right answer when you're asked "can I decorate a controller?"

### Disposal

After the response is written, the activator calls `Release(context, controller)`. The default implementation does:

```csharp
if (controller is IAsyncDisposable asyncDisposable)
    await asyncDisposable.DisposeAsync();
else if (controller is IDisposable disposable)
    disposable.Dispose();
```

Additionally, when the request scope itself is disposed (immediately after), any **scoped services** resolved during the request (including those injected into the controller) are disposed by the DI container — you don't need to dispose them manually.

### What this implies for senior design choices

1. **Controllers are not thread-safe-by-default and don't need to be.** They handle exactly one request; you can hold per-request state in fields without locks.
2. **Never do long-running work in the constructor.** Constructor runs synchronously, on the request thread, before the action. If a dependency is expensive to construct, either make it a scoped/singleton lightweight wrapper over a heavy resource, or method-inject it so only the actions that need it pay.
3. **`HttpContext` is reachable via `ControllerBase.HttpContext`.** Don't inject `IHttpContextAccessor` into a controller — it's already there. Save `IHttpContextAccessor` for services below the controller that happen to need `HttpContext`.
4. **`CancellationToken ct` as an action parameter** maps to `HttpContext.RequestAborted`. Plumb it into every downstream async call; a controller that ignores it leaks thread-pool work when clients disconnect.
5. **`IAsyncDisposable` is honored.** If you need async cleanup (flushing a background channel, etc.) make the controller `IAsyncDisposable`, not `IDisposable` that does `.Wait()`.

### Senior-level gotchas

- **Controllers are not pooled.** A new instance per request means one GC allocation per controller object + its captured dependencies per request — this is fine, but don't try to "optimize" by making fields `static`. You'll share state across requests and introduce subtle races.
- **Captured singletons can outlive the controller.** If a singleton service retains a reference to the controller (e.g., via a subscribed event), you leak every request's state. Don't subscribe from within the controller constructor.
- **`ServiceBasedControllerActivator` changes the shape of errors.** A missing-registration bug surfaces at request time as "no service for X" instead of an activation exception — makes startup validation (`ValidateOnBuild`) even more valuable.
- **Filters are *not* per-request by default.** `[TypeFilter]` / `[ServiceFilter]` respect DI scope; `[MyCustomFilter]` (attribute instance) is shared. Don't store per-request state in a filter without intentional scoping.
- **`OnActionExecuting` can short-circuit the action** by setting `context.Result`; the controller is still constructed and disposed, but the action method doesn't run. Useful for early-exit filters (feature flags, rate-limiting); surprising if you expected the action to be skipped entirely.
- **Don't confuse with Razor Pages**: a PageModel has a very similar lifecycle, but Razor Pages swap `OnGet`/`OnPost` handler discovery for the action descriptor step — the filter/activator pattern is the same.
