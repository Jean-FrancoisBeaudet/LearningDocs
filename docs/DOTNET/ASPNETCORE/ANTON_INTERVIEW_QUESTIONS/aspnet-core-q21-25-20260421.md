## Q21. Explain how JWT tokens work

A JWT is a compact, URL-safe, **self-contained** security token. Wire format is three base64url-encoded parts joined by dots:

```
header.payload.signature
```

- **Header** — `{ "alg": "RS256", "typ": "JWT", "kid": "2026-key-1" }`. `alg` is the signing algorithm; `kid` points at the key so validators can pick the right one during rotation.
- **Payload** — JSON set of *claims*. Registered claims: `iss` (issuer), `sub` (subject/user id), `aud` (intended audience/API), `exp` (expiry), `nbf` (not-before), `iat` (issued-at), `jti` (unique id). Plus any custom claims (`roles`, `tenant`, `scope`).
- **Signature** — `Sign(base64url(header) + "." + base64url(payload), key)`. This is what makes the token tamper-evident. **The payload is not encrypted** — it's base64, anyone can read it. Never put PII or secrets in a JWT.

### Signing algorithms

| | HS256 (symmetric) | RS256 / ES256 (asymmetric) |
|---|---|---|
| Key | Shared secret | Private key signs, public key verifies |
| Distribution | Every validator needs the secret | Validators fetch public JWKS endpoint |
| Risk | One compromised service leaks the signing key | Only the IdP holds the private key |
| When | Single-service monoliths, internal tokens | Microservices, anything with >1 validator |

In modern ASP.NET Core projects, default to **RS256 or ES256** with a JWKS endpoint. HS256 is almost always the wrong choice past a single service.

### Validation pipeline in ASP.NET Core

```csharp
builder.Services
    .AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.Authority = "https://idp.example.com";     // auto-discovers JWKS
        options.Audience  = "api.orders";
        options.MapInboundClaims = false;                  // keep original claim names
        options.TokenValidationParameters = new()
        {
            ValidateIssuer           = true,
            ValidateAudience         = true,
            ValidateLifetime         = true,
            ValidateIssuerSigningKey = true,
            ClockSkew                = TimeSpan.FromSeconds(30),
            NameClaimType            = "sub",
            RoleClaimType            = "roles"
        };
    });
```

Under the hood, the `JwtBearerHandler`:

1. Reads the `Authorization: Bearer <token>` header.
2. Pulls signing keys from the OIDC discovery document (`Authority + /.well-known/openid-configuration` → `jwks_uri`), cached with automatic refresh.
3. Hands the token to **`Microsoft.IdentityModel.JsonWebTokens.JsonWebTokenHandler`** (the modern replacement for the legacy `JwtSecurityTokenHandler` — faster, allocates less, and is the default in .NET 8+).
4. Verifies `alg` is whitelisted, signature matches, `exp`/`nbf`/`iss`/`aud` pass, then materialises a `ClaimsPrincipal` and attaches it to `HttpContext.User`.

### Revocation

JWTs are validated **offline**. That's the selling point — and the trap. There's no built-in way to revoke a token before `exp`. Standard mitigations:

- Keep access-token lifetime short (5–15 min).
- Pair with a refresh token you *can* revoke server-side (see Q22).
- For emergency revocation, maintain a deny-list (`jti`) in Redis and check it in a `JwtBearerEvents.OnTokenValidated` handler.

**Modern note (.NET 10 / C# 14):** .NET 10 keeps the `AddJwtBearer` surface stable; the newer `AddBearerToken` (introduced in .NET 8 for `MapIdentityApi`) uses an opaque token format, not JWTs — don't confuse them. If you need JWTs, stay on `AddJwtBearer`.

**Senior-level gotchas:**

- `alg: none` attack — never disable `RequireSignedTokens`; never accept unsigned tokens.
- Default `ClockSkew` is 5 minutes, which silently masks expiry bugs in tests. Drop it to 30s.
- Key rotation: rely on `ConfigurationManager<OpenIdConnectConfiguration>` (built into the handler). Set `RefreshOnIssuerKeyNotFound = true` so an unknown `kid` triggers a JWKS refresh instead of failing every request until the cache TTL expires.
- `MapInboundClaims = false` is worth setting explicitly — otherwise `sub` gets silently renamed to `http://schemas.xmlsoap.org/.../nameidentifier` and you chase ghosts.

## Q22. Explain Refresh tokens and how they work

Access tokens are deliberately short-lived so that a leaked token loses value quickly. A **refresh token** is a longer-lived credential the client exchanges for a new access token without re-prompting the user. The refresh token is opaque (random 256-bit value, *not* a JWT) and — unlike a JWT — is stored server-side so it can be revoked.

### Canonical flow

```
1. POST /token (password grant / auth code)     → access_token (15 min) + refresh_token (14 days)
2. Client calls API with access_token until 401
3. POST /token (grant_type=refresh_token, refresh_token=...)
4. Server validates, marks old refresh as rotated,
   issues NEW access_token + NEW refresh_token
5. Repeat
```

### Server-side storage

Store a **hash** of the refresh token (SHA-256), never plaintext:

```csharp
public sealed class RefreshToken
{
    public required string Id             { get; init; }
    public required byte[] TokenHash      { get; init; }   // SHA-256 of plaintext
    public required Guid   UserId         { get; init; }
    public required Guid   FamilyId       { get; init; }   // chain id for rotation
    public required string DeviceId       { get; init; }
    public required DateTimeOffset ExpiresUtc { get; init; }
    public DateTimeOffset? RevokedUtc     { get; set; }
    public string? ReplacedByTokenId      { get; set; }
}
```

### Rotation + reuse detection (OAuth 2.1 recommendation)

Every refresh issues a new pair; the old refresh is marked rotated and linked to the new one via `ReplacedByTokenId`. Both tokens in a chain belong to the same `FamilyId`. If a client ever presents a refresh token that has already been rotated, that means **someone else still has the old one** — revoke the entire family:

```csharp
public async Task<TokenPair> RefreshAsync(string presented, CancellationToken ct)
{
    var hash = SHA256.HashData(Encoding.UTF8.GetBytes(presented));
    var record = await _db.RefreshTokens
        .SingleOrDefaultAsync(t => t.TokenHash == hash, ct)
        ?? throw new UnauthorizedAccessException();

    var now = _time.GetUtcNow();

    if (record.RevokedUtc is not null || record.ExpiresUtc < now)
    {
        // Reuse of an already-rotated token => treated as theft.
        await _db.RefreshTokens
            .Where(t => t.FamilyId == record.FamilyId && t.RevokedUtc == null)
            .ExecuteUpdateAsync(s => s.SetProperty(t => t.RevokedUtc, now), ct);
        throw new UnauthorizedAccessException();
    }

    record.RevokedUtc = now;
    var next = IssueRefreshToken(record.UserId, record.FamilyId, record.DeviceId);
    record.ReplacedByTokenId = next.Id;
    _db.RefreshTokens.Add(next);
    await _db.SaveChangesAsync(ct);

    return new TokenPair(
        AccessToken:  IssueAccessToken(record.UserId),
        RefreshToken: next.Plaintext);
}
```

### Delivery

- **Browsers** → `HttpOnly; Secure; SameSite=Strict` cookie. Never `localStorage` (XSS-exfiltrable).
- **Mobile** → response body, stored in Keychain / Keystore.
- **Native SPAs** (Electron) → OS credential manager, not the renderer.

**Modern note (.NET 10 / C# 14):** the `MapIdentityApi<TUser>()` endpoints include `/refresh` out of the box, but they **do not implement rotation with reuse detection** — they issue a new access+refresh pair without invalidating reuse. Fine for demos; roll your own for production, or delegate to OpenIddict / Duende IdentityServer.

**Senior-level gotchas:**

- Absolute vs sliding expiry: sliding alone lets a stolen token live forever. Enforce a hard ceiling (e.g. 30 days absolute) on top of any sliding window.
- Logout must *revoke server-side*, not just delete the cookie. Clearing client state is cosmetic — a copy of the token on the attacker's machine still works.
- Multi-instance APIs need the refresh-token store in a **shared** backing service (SQL / Redis), never in-process memory.
- Don't sign refresh tokens with the same key as JWTs. They aren't JWTs — make them opaque random bytes (`RandomNumberGenerator.GetBytes(32)`).
- Bind refresh tokens to a device/session when you can — one stolen cookie shouldn't grant indefinite access across devices.

**Sources:**
- [JWT Validation and Authorization in ASP.NET Core – .NET Blog](https://devblogs.microsoft.com/dotnet/jwt-validation-and-authorization-in-asp-net-core/)

## Q23. How would you allow access to resources only if a user has specific permissions?

Short answer: **policy-based authorization with permission requirements**, hydrated from your data store into the `ClaimsPrincipal`, wired via a custom `AuthorizationPolicyProvider` so endpoints can declare permissions by name.

### Roles vs claims vs permissions

| | Coarseness | Where stored | Change granularity |
|---|---|---|---|
| Role (`admin`) | Coarse | Token / DB | Per role |
| Claim (`department=finance`) | Identity attribute | Token | Per user |
| Permission (`invoices:write`) | Fine-grained operation | DB → mapped to claims | Per capability |

A mature system models **Users → Roles → Permissions**. Authorize on the permission, not the role — so swapping a role's capabilities doesn't touch code.

### 1. Permission requirement

```csharp
public sealed class PermissionRequirement(string permission) : IAuthorizationRequirement
{
    public string Permission { get; } = permission;
}
```

### 2. Handler

```csharp
public sealed class PermissionHandler : AuthorizationHandler<PermissionRequirement>
{
    protected override Task HandleRequirementAsync(
        AuthorizationHandlerContext ctx, PermissionRequirement req)
    {
        if (ctx.User.HasClaim("permission", req.Permission))
            ctx.Succeed(req);

        return Task.CompletedTask;
    }
}
```

### 3. Dynamic policy provider

Hard-coding `AddPolicy("invoices:write", ...)` for every permission doesn't scale. A custom provider creates policies on demand when an `[Authorize("invoices:write")]` attribute is hit:

```csharp
public sealed class PermissionPolicyProvider(IOptions<AuthorizationOptions> options)
    : DefaultAuthorizationPolicyProvider(options)
{
    public override async Task<AuthorizationPolicy?> GetPolicyAsync(string name)
    {
        var existing = await base.GetPolicyAsync(name);
        if (existing is not null) return existing;

        return new AuthorizationPolicyBuilder()
            .AddRequirements(new PermissionRequirement(name))
            .Build();
    }
}
```

### 4. Claims transformation (hydrate permissions per request)

Don't stuff every permission into the JWT — tokens bloat, and revocation is painful. Instead, hydrate permissions from the DB *after* authentication:

```csharp
public sealed class PermissionClaimsTransformation(IPermissionStore store, IMemoryCache cache)
    : IClaimsTransformation
{
    public async Task<ClaimsPrincipal> TransformAsync(ClaimsPrincipal principal)
    {
        if (principal.Identity is not ClaimsIdentity { IsAuthenticated: true } id)
            return principal;

        if (id.HasClaim(c => c.Type == "permission")) return principal;  // already done

        var userId = id.FindFirst("sub")!.Value;
        var perms = await cache.GetOrCreateAsync($"perms:{userId}", async e =>
        {
            e.AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(5);
            return await store.GetPermissionsAsync(userId);
        });

        foreach (var p in perms!)
            id.AddClaim(new Claim("permission", p));

        return principal;
    }
}
```

### 5. Wire-up and usage

```csharp
builder.Services.AddSingleton<IAuthorizationPolicyProvider, PermissionPolicyProvider>();
builder.Services.AddScoped<IAuthorizationHandler, PermissionHandler>();
builder.Services.AddScoped<IClaimsTransformation, PermissionClaimsTransformation>();

// Endpoint
app.MapDelete("/invoices/{id:guid}", DeleteInvoice)
   .RequireAuthorization("invoices:delete");

// Controller
[Authorize("invoices:write")]
public sealed class InvoicesController : ControllerBase { /* ... */ }
```

### Resource-based authorization

When the decision depends on the specific record (owner-only edits), inject `IAuthorizationService` and evaluate against the resource:

```csharp
public async Task<IResult> EditInvoice(Guid id, InvoiceEditDto dto,
    ClaimsPrincipal user, IAuthorizationService authz, AppDbContext db, CancellationToken ct)
{
    var invoice = await db.Invoices.FindAsync([id], ct);
    if (invoice is null) return Results.NotFound();

    var result = await authz.AuthorizeAsync(user, invoice, "invoices:edit");
    if (!result.Succeeded) return Results.Forbid();

    // ... apply edit
    return Results.NoContent();
}
```

Your handler signature becomes `AuthorizationHandler<PermissionRequirement, Invoice>` so it sees the resource.

**Senior-level gotchas:**

- `IClaimsTransformation.TransformAsync` runs on **every** request and can even run multiple times per request across authentication middlewares. Guard with an idempotency check (`HasClaim(...)`) and cache the DB lookup — otherwise it's a performance landmine.
- Permissions change mid-session: since access tokens are self-contained, changes don't propagate until the token refreshes. For critical revocations (user fired), either shorten token lifetime, maintain a `jti` deny-list, or invalidate the refresh token and force re-auth.
- Don't invent your own permission string format and then sprinkle typos — use a static class or source-generated constants (`Permissions.Invoices.Write`) so the compiler catches mistakes.
- `Forbid()` vs `Unauthorized()`: if the user is authenticated but lacks permission, it's **403 Forbid**. `401` means "identity failed or absent", which restarts the auth challenge. Using the wrong one breaks SPAs.
- Role-based `[Authorize(Roles = "Admin")]` is fine for trivial cases, but don't mix styles — stick with policies throughout and model "admin" as a role requirement in a policy if needed.

**Sources:**

## Q24. What is `HostedService` used for?

`IHostedService` is the contract the .NET Generic Host uses to run long-lived workloads tied to the application lifecycle. It defines exactly two methods:

```csharp
public interface IHostedService
{
    Task StartAsync(CancellationToken cancellationToken);
    Task StopAsync (CancellationToken cancellationToken);
}
```

The `IHost` (i.e. the ASP.NET Core `WebApplication`) calls `StartAsync` on every registered hosted service *before* the HTTP server starts accepting traffic, and `StopAsync` during graceful shutdown.

### `BackgroundService` — the base you'll actually use

Hand-implementing `IHostedService` for a long-running loop is awkward because you have to return from `StartAsync` quickly but still keep work going. `BackgroundService` folds the two into one override:

```csharp
public sealed class OutboxPublisher(
    IServiceProvider sp,
    TimeProvider time,
    ILogger<OutboxPublisher> log) : BackgroundService
{
    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        using var timer = new PeriodicTimer(TimeSpan.FromSeconds(5), time);

        while (await timer.WaitForNextTickAsync(stoppingToken))
        {
            try
            {
                await using var scope = sp.CreateAsyncScope();
                var publisher = scope.ServiceProvider.GetRequiredService<IOutboxPublisher>();
                await publisher.PublishPendingAsync(stoppingToken);
            }
            catch (OperationCanceledException) { break; }
            catch (Exception ex)
            {
                log.LogError(ex, "Outbox publish failed");
            }
        }
    }
}
```

Register with `builder.Services.AddHostedService<OutboxPublisher>();`. Hosted services are singletons; scoped dependencies must be resolved inside a manually created scope.

### Typical use cases

- Queue/topic consumers (Service Bus, RabbitMQ, Kafka).
- Scheduled jobs (with `PeriodicTimer` or Quartz.NET for cron).
- Cache warmup at startup.
- Outbox-pattern publishers, dead-letter re-drivers.
- SignalR backplane flushers.

### `IHostedLifecycleService` — fine-grained hooks

Inherits `IHostedService` and adds four extra hooks executed in parallel across all registered services:

```
StartingAsync  → StartAsync  → StartedAsync
StoppingAsync  → StopAsync   → StoppedAsync
```

Use these when you need pre-start or post-stop coordination — e.g. warming feature flags *before* the first `StartAsync`, or flushing telemetry *after* every hosted service has stopped. Available since .NET 8 and present in .NET 10.

**Senior-level gotchas:**

- `StartAsync` is **awaited synchronously in startup**. Long work there delays the HTTP server coming up and breaks readiness probes. Move it into `ExecuteAsync` or `StartedAsync`.
- Unhandled exceptions in `ExecuteAsync` — `HostOptions.BackgroundServiceExceptionBehavior` defaults to `StopHost` (.NET 6+). That's usually what you want (crash and restart under orchestrator), but surprising if you expected the loop to keep running. Set to `Ignore` and wrap per-iteration if you'd rather log and continue.
- The `stoppingToken` passed to `ExecuteAsync` fires when shutdown begins. If you don't observe it, shutdown hangs until `HostOptions.ShutdownTimeout` (default 30s) elapses and the process is killed — noisy on Kubernetes, shows up as `SIGKILL` in logs.
- Scoped-dependency trap: never resolve a scoped `DbContext` in the constructor; create a scope per iteration.
- One hosted service per concern. Don't combine an inbox consumer + scheduler + health publisher into one class — you lose independent testability and restart semantics.
- On IIS / Azure App Service, the worker process can recycle without calling `StopAsync` on time. Design work to be idempotent and checkpoint progress.

- [Introducing the new IHostedLifecycleService Interface in .NET 8 — Steve Gordon](https://www.stevejgordon.co.uk/introducing-the-new-ihostedlifecycleservice-interface-in-dotnet-8)

## Q25. Explain the difference between `PeriodicTimer` and `await Task.Delay()`

Both let you run a loop on an interval, but they have *very different* timing and resource semantics.

### `Task.Delay` loop

```csharp
while (!ct.IsCancellationRequested)
{
    await DoWorkAsync(ct);
    await Task.Delay(TimeSpan.FromSeconds(5), ct);
}
```

Effective period is `work_duration + 5s`. If `DoWorkAsync` takes 2 s, the next iteration starts at T+7 s, then T+14 s, T+21 s — **drifts** with load. `Task.Delay` allocates a fresh `Task` each iteration, and cancellation throws `OperationCanceledException`.

### `PeriodicTimer`

```csharp
using var timer = new PeriodicTimer(TimeSpan.FromSeconds(5));
while (await timer.WaitForNextTickAsync(ct))
{
    await DoWorkAsync(ct);
}
```

Ticks fire on a **fixed schedule** relative to the previous tick. If work takes 2 s, the next `WaitForNextTickAsync` waits 3 s. No drift. `WaitForNextTickAsync` returns a reusable `ValueTask<bool>` — zero per-iteration allocation in the happy path.

### Side-by-side

| | `Task.Delay` loop | `PeriodicTimer` |
|---|---|---|
| Period | Drifts with work duration | Fixed period (no drift) |
| Allocation per tick | New `Task` + `CancellationTokenRegistration` | Reused `ValueTask<bool>` |
| Backpressure | Always inserts a fixed gap after work | If work outruns the period, the next `WaitForNextTickAsync` returns immediately; missed ticks collapse to one |
| Cancellation | Throws `OperationCanceledException` | Throws `OCE` on the token; returns `false` on `Dispose()` |
| Consumers | N/A | **Single consumer** — concurrent `WaitForNextTickAsync` calls are undefined |
| Lifetime | No resource | `IDisposable` — dispose to cancel pending waits |
| First tick | After the delay | After the period (not immediate) |

### Testability with `TimeProvider` (.NET 8+)

`PeriodicTimer` accepts a `TimeProvider`, so timed code becomes deterministic in tests:

```csharp
// Production
using var timer = new PeriodicTimer(TimeSpan.FromSeconds(5), TimeProvider.System);

// Unit test
var fake = new FakeTimeProvider();               // Microsoft.Extensions.TimeProvider.Testing
var sut  = new Worker(fake);
var task = sut.StartAsync(CancellationToken.None);

fake.Advance(TimeSpan.FromSeconds(5));           // fires tick deterministically
// assert side effects
```

This is the single biggest reason to prefer `PeriodicTimer` today: you can test scheduled code without `Thread.Sleep`.

**Modern note (.NET 10 / C# 14):** both APIs are unchanged in .NET 10. The `TimeProvider` overload has been the recommended form since .NET 8; treat `new PeriodicTimer(period)` as a code smell in new projects — always pass `TimeProvider`.

**Senior-level gotchas:**

- `PeriodicTimer` **does not queue missed ticks**. If your work takes 3× the period, you don't get a burst of three catch-up ticks — you get one. Usually desired, but don't rely on evenly-spaced side effects when throughput is uncertain.
- First tick isn't immediate. If you want "run now, then every N", invoke the work once before entering the loop, or check `await timer.WaitForNextTickAsync()` *after* an initial `DoWorkAsync()`.
- Wrap each iteration in `try/catch` — an unhandled throw exits the loop and silently stops your worker. Combined with `BackgroundServiceExceptionBehavior = Ignore`, that's how you get a worker that looks healthy but does nothing.
- Short periods (<15 ms on Windows) hit OS timer resolution limits — neither API can deliver sub-ms precision.
- Don't reach for `System.Timers.Timer` or `System.Threading.Timer` in new code — they predate async, compose poorly with cancellation tokens, and their callback exceptions crash the process by default (`System.Threading.Timer`) or get swallowed (`System.Timers.Timer`).
