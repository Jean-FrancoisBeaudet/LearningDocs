# ASP.NET Core — Questions 11-15

_Source: `DOTNET/ASPNETCORE/aspnetcore-interview-questions.md`. Answers target .NET 10 / C# 14._

## Q11. Name a few ways to read data from `appsettings.json` configuration

`IConfiguration` is the unified abstraction over **all** providers (JSON files, env vars, user secrets, Key Vault, command line). `appsettings.json` is just one provider registered by `WebApplication.CreateBuilder`. Ways to read it:

1. **Direct `IConfiguration` indexing** — quick, stringly-typed, no binding.
   ```csharp
   string? conn = builder.Configuration["ConnectionStrings:Default"];
   int timeout = builder.Configuration.GetValue<int>("Api:TimeoutSeconds");
   ```
2. **`GetSection` + `Bind`** — bind a subtree into a POCO.
   ```csharp
   var cfg = new ApiOptions();
   builder.Configuration.GetSection("Api").Bind(cfg);
   ```
3. **`Get<T>()`** — shorthand for bind-and-return.
   ```csharp
   var cfg = builder.Configuration.GetSection("Api").Get<ApiOptions>();
   ```
4. **Options Pattern** (`IOptions<T>`, `IOptionsSnapshot<T>`, `IOptionsMonitor<T>`) — the idiomatic way for DI-scoped consumption. See Q12.
   ```csharp
   builder.Services.Configure<ApiOptions>(builder.Configuration.GetSection("Api"));
   ```
5. **`GetConnectionString("Name")`** — sugar over `ConnectionStrings:Name`.
6. **Strongly-typed keys via `ConfigurationKeyName`** on properties when JSON names and POCO names differ.
7. **Record-based binding with primary-constructor POCOs** (nice with C# 14) — binder matches constructor parameters by name.

**Modern note (.NET 10 / C# 14):** the `ConfigurationBinder` source generator (`[ConfigurationKeyName]`, `<EnableConfigurationBindingGenerator>true</EnableConfigurationBindingGenerator>`) removes reflection-based binding, which is required for trimming and NativeAOT. In .NET 10 this is the default path for Minimal APIs published to AOT — prefer it over runtime binding when deploying AOT or trimmed apps.

**Senior-level gotchas:**
- Keys are **case-insensitive**; the separator is always `:` in code even though env vars use `__`.
- Reading via `IConfiguration` directly in business code bypasses the Options validation pipeline — don't do it for anything non-trivial.
- `GetValue<T>` does type conversion; unparseable values return `default(T)` silently unless you pass a default or throw.

## Q12. What is the Options Pattern?

The Options Pattern is the idiomatic way to expose **strongly-typed, DI-injected configuration sections** to application code, decoupling consumers from `IConfiguration`. You register a POCO bound to a config section, then inject one of three interfaces depending on change-tracking needs.

```csharp
public sealed class ApiOptions
{
    public const string SectionName = "Api";
    public required string BaseUrl { get; init; }
    public int TimeoutSeconds { get; init; } = 30;
}

// Registration
builder.Services
    .AddOptions<ApiOptions>()
    .Bind(builder.Configuration.GetSection(ApiOptions.SectionName))
    .ValidateDataAnnotations()
    .ValidateOnStart();

// Consumption
public sealed class WeatherClient(IOptions<ApiOptions> options)
{
    private readonly ApiOptions _opts = options.Value;
}
```

Three consumer interfaces:

| Interface | Lifetime | Refresh behavior |
|---|---|---|
| `IOptions<T>` | Singleton | Value captured once at first resolution; **never** reloads |
| `IOptionsSnapshot<T>` | Scoped | Rebound **per request/scope** — picks up file changes between requests |
| `IOptionsMonitor<T>` | Singleton | Live value + `OnChange` callback; reloads on provider change |

**Why it matters:**
- **Separation of concerns** — services depend on `ApiOptions`, not config layout.
- **Validation** — `ValidateDataAnnotations`, `Validate(...)`, `ValidateOnStart()` surface bad config at startup, not at first request.
- **Named options** — multiple configurations of the same type via `services.Configure<T>("name", ...)` and `IOptionsSnapshot<T>.Get("name")`. Essential for multi-tenant or per-provider setups.
- **Post-configuration** — `services.PostConfigure<T>(...)` runs after all `Configure` calls, useful for defaults that depend on DI.

**Senior-level gotchas:**
- Don't inject `IOptionsSnapshot<T>` into a singleton — captive dependency; use `IOptionsMonitor<T>` instead.
- `.Value` throws if validation fails; `ValidateOnStart()` (added in .NET 8) converts that into a startup failure instead of runtime.
- `required` members and primary-constructor POCOs are supported by the source-generator binder in modern .NET — no parameterless ctor needed.

## Q13. Name the use cases for using `IOptionsSnapshot` and `IOptionsMonitor`

_(The question as written says `ISnapshotMonitor` — this is a typo; the real interfaces are `IOptionsSnapshot<T>` and `IOptionsMonitor<T>`.)_

### `IOptionsSnapshot<T>` — **scoped**, recomputed per scope

**Use when:**
- Config can change at runtime (file reload, env-var refresh) and you want each **HTTP request** to see a consistent snapshot.
- You're in **request-scoped** code: controllers, Minimal API handlers, scoped services, MVC filters.
- You need **named options** that may vary per request (e.g. tenant resolution picks a named option at request start).

```csharp
app.MapGet("/limit", (IOptionsSnapshot<ApiOptions> opts) => opts.Value.TimeoutSeconds);
```

Reads are stable for the scope, so two reads within one request see the same value even if the file changes mid-request.

### `IOptionsMonitor<T>` — **singleton**, live value + change notifications

**Use when:**
- You're in a **singleton** (background service, hosted service, cache, HTTP handler, `IHttpClientFactory` handler) and can't take a scoped dependency.
- You need to **react** to configuration changes (`monitor.OnChange(newVal => ...)`) — e.g. rebuild a cached client, adjust a rate-limiter threshold, flush a cache.
- You want always-current values without creating a new scope.

```csharp
public sealed class FeatureFlagCache : IDisposable
{
    private readonly IDisposable? _sub;
    private FeatureFlags _current;

    public FeatureFlagCache(IOptionsMonitor<FeatureFlags> monitor)
    {
        _current = monitor.CurrentValue;
        _sub = monitor.OnChange(v => Interlocked.Exchange(ref _current, v));
    }

    public bool IsEnabled(string key) => _current.Map.GetValueOrDefault(key);
    public void Dispose() => _sub?.Dispose();
}
```

**Senior-level gotchas:**
- `OnChange` can fire **multiple times** for a single file save (editors do rename+write). Debounce if the side effect is expensive.
- The callback runs on a thread-pool thread — make handlers thread-safe and non-blocking.
- Always dispose the `IDisposable` returned by `OnChange` to avoid leaks in long-running hosts.
- `IOptions<T>` (no suffix) is the third flavor — singleton, **never** reloads. Use it when config is immutable post-startup.

## Q14. How to validate configuration?

Configuration validation ensures the app **fails fast at startup** when bound values are missing, malformed, or violate business rules — rather than exploding on the first request that happens to use the bad value.

### Mechanisms

1. **DataAnnotations** on the POCO — declarative, works for simple rules.
   ```csharp
   public sealed class ApiOptions
   {
       [Required, Url] public required string BaseUrl { get; init; }
       [Range(1, 300)] public int TimeoutSeconds { get; init; } = 30;
   }

   builder.Services.AddOptions<ApiOptions>()
       .Bind(builder.Configuration.GetSection("Api"))
       .ValidateDataAnnotations()
       .ValidateOnStart();
   ```

2. **Delegate validation** — ad-hoc predicate.
   ```csharp
   .Validate(o => o.TimeoutSeconds < o.MaxTimeoutSeconds, "Timeout must be < Max")
   ```

3. **`IValidateOptions<T>`** — full object, DI-injectable, supports multiple errors and named options.
   ```csharp
   public sealed class ApiOptionsValidator : IValidateOptions<ApiOptions>
   {
       public ValidateOptionsResult Validate(string? name, ApiOptions o)
       {
           var errs = new List<string>();
           if (!Uri.TryCreate(o.BaseUrl, UriKind.Absolute, out _)) errs.Add("BaseUrl invalid");
           if (o.TimeoutSeconds <= 0) errs.Add("TimeoutSeconds must be positive");
           return errs.Count == 0 ? ValidateOptionsResult.Success : ValidateOptionsResult.Fail(errs);
       }
   }
   builder.Services.AddSingleton<IValidateOptions<ApiOptions>, ApiOptionsValidator>();
   ```

4. **FluentValidation** — wire a `FluentValidation.IValidator<T>` into an `IValidateOptions<T>` adapter. Better for complex cross-field/async/business rules than DataAnnotations.

5. **Source-generated validator** — attribute `[OptionsValidator]` on a partial class implementing `IValidateOptions<T>`. The generator emits allocation-free validation code at build time — required for NativeAOT since DataAnnotations validation uses reflection.
   ```csharp
   [OptionsValidator]
   public sealed partial class ApiOptionsValidator : IValidateOptions<ApiOptions> { }
   ```

### `ValidateOnStart()` is the critical piece

Without it, validation is **lazy** — it runs the first time `.Value` is accessed, which might be during a request. `ValidateOnStart()` (via the `IStartupValidator` hosted service) forces validation at host startup and throws `OptionsValidationException` before Kestrel binds, so a bad config = failed deployment, not failed request.

**Senior-level gotchas:**
- Stack multiple validators on the same `AddOptions<T>()` chain — they all run.
- Validation runs against the **bound POCO**, so `[Required]` on a non-nullable `int` is meaningless (default 0 satisfies `[Required]`); use `[Range]` or make it nullable.
- For NativeAOT / trimming: combine the configuration-binding source generator with `[OptionsValidator]` — both are reflection-free.
- `IValidateOptions<T>` is resolved as **singleton**, so don't inject scoped services into it.

## Q15. What is the difference between DataAnnotations and FluentValidation?

Both validate models, but they sit at opposite ends of the declarative-vs-imperative spectrum.

### DataAnnotations (built-in, `System.ComponentModel.DataAnnotations`)

Attributes on the model itself:

```csharp
public sealed class CreateUserRequest
{
    [Required, EmailAddress] public required string Email { get; init; }
    [Required, StringLength(100, MinimumLength = 8)] public required string Password { get; init; }
    [Range(18, 120)] public int Age { get; init; }
}
```

- **Pros:** zero dependencies, integrated with ASP.NET Core model binding (`ModelState.IsValid`), Options Pattern, EF Core, Blazor forms. Works out of the box.
- **Cons:** attribute constants must be compile-time; poor support for **cross-field** rules, **conditional** rules, **async** (DB/HTTP) rules, or DI-injected validators. Extending means writing custom `ValidationAttribute` subclasses — clunky.

### FluentValidation (`FluentValidation` NuGet)

Separate validator classes using a fluent DSL:

```csharp
public sealed class CreateUserRequestValidator : AbstractValidator<CreateUserRequest>
{
    public CreateUserRequestValidator(IUserRepository users)
    {
        RuleFor(x => x.Email).NotEmpty().EmailAddress()
            .MustAsync(async (e, ct) => !await users.ExistsAsync(e, ct))
            .WithMessage("Email already registered");

        RuleFor(x => x.Password)
            .NotEmpty().MinimumLength(8)
            .Matches(@"[A-Z]").WithMessage("Need an uppercase letter")
            .Equal(x => x.PasswordConfirm).WithName("Password confirmation");

        When(x => x.Age < 18, () =>
            RuleFor(x => x.GuardianEmail).NotEmpty().EmailAddress());
    }
}
```

- **Pros:** validator is DI-enabled (inject repositories, clocks, HTTP clients), supports async, cross-field, conditional (`When`/`Unless`), reusable rule sets, collection rules (`RuleForEach`), testable in isolation. Model stays clean — no infrastructure concerns bleeding into domain/DTO types.
- **Cons:** extra dependency; since **FluentValidation 11.x the maintainer removed the auto-validation integration with ASP.NET Core** — you now call `validator.ValidateAsync(model)` manually (or via a package like `SharpGrip.FluentValidation.AutoValidation`). This is a deliberate design choice, not a gap.

### When to use which

- **DataAnnotations** — simple DTOs, Options classes, EF constraints, Blazor quick forms.
- **FluentValidation** — complex domain rules, validation requiring DB/HTTP/clock, keeping DTOs attribute-free (common in clean-architecture / DDD code), shared rule sets across endpoints.

They aren't mutually exclusive — it's fine to use DataAnnotations on Options and FluentValidation on request DTOs in the same app.

**Modern note (.NET 10 / C# 14):** Microsoft has been investing in **source-generated DataAnnotations validation** (`[OptionsValidator]`, and broader model-validation generators for Minimal APIs). This narrows the ergonomic gap somewhat — generated validators are AOT-safe and faster — but FluentValidation is still the clear winner for anything involving DI or async rules.

**Senior-level gotchas:**
- Don't put validation attributes on EF entities **and** re-use them as API DTOs — you conflate persistence constraints with API contract. Separate DTOs, FluentValidation on them.
- Since FV 11 dropped auto-integration, make sure your pipeline actually invokes the validator — a silent "no validator ran" is worse than no validation at all.
- FluentValidation's `MustAsync` runs the DB call on every request — cache or debounce if the lookup is hot.
- For Minimal APIs, a filter (`IEndpointFilter`) that resolves `IValidator<T>` from DI and short-circuits with `Results.ValidationProblem` is the canonical pattern.
